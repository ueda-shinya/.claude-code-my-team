# /feature-flow スキル

新機能開発の標準レール。すべての新機能はこのフローに乗せ、「早く、品質の良いものを作る」ことを目的とします。

## 適用範囲

**以下のいずれか1つ以上を満たす場合に `/feature-flow` を適用します：**

- 新規ファイル作成を伴う
- 新しいユーザー操作（CLIコマンド / トリガーワード / 画面等）が追加される
- 外部インターフェース（API / MCP / Notion等）の新規統合

**対象外:** 上記に当てはまらない既存機能の改修・バグ修正（都度判断）。

## 起動トリガー

- `/feature-flow <機能名>` コマンド
- シンヤさんが「新機能を作りたい」「〇〇（機能名）を作って」と発言したとき → アスカが自動起動

## 最重要ポリシー（シンヤさんの一番の関心事）

**修正・対策もれを絶対に出さない。**

開発中に発生した不具合・課題は、発見した瞬間に Notion 案件ページの「5. 不具合・課題リスト」にチェックボックス付きで追記すること。フロー完了前に未対応チェックボックスが0件であることを必ず確認する。**1件でも残っていたら完了報告しない**（クオリティゲート）。

## 標準9ステップフロー（2026-04-18: ステップ2.5 追加）

### ステップ1: ヒアリング（アスカ主導）

シンヤさんに以下を質問し、要件を引き出してください。

**必須項目:**
1. 目的（なぜこれを作るのか）
2. ユーザーストーリー（誰が、何のために、どう使うか）
3. 完了条件（何ができたら「完成」か）
4. 非対象（やらないこと）
5. 成功指標（どう測るか）
6. 想定利用頻度

質問はまとめて投げ、1往復で完結させてください。

**ヒアリング完了時点で Notion 案件を自動登録**（次節参照）。

### ステップ2: 要件定義（アスカが起案）

ヒアリング結果を「要件1枚」にまとめ、Notion 案件ページの「1. 要件定義」セクションに `notion-tasks.py --add-block` で追記。シンヤさんに確認を取り、合意を得てから次へ。

> **自動リナ召集について（CLAUDE.mdルール）**
> 要件定義に「合意する直前」、CLAUDE.mdの「Rina Auto-Invocation Rule」によりリナが自動召集され、要件の論理整合性・抜け漏れを局所検証します。これは維持してください。ステップ5の「リナ統合検証」とは目的が異なります（局所検証 vs 統合検証）。

### ステップ2.5: 既存資産リサーチ（アスカ主導・2026-04-18 追加）

要件定義が固まった後、設計に進む前に必ず実施する。**新規実装を避け、既存資産で代替可能な範囲を最大化するため**。

**背景**: 2026-04-18、gsc-ga4-analyzer 案件で発覚した問題。既存の `mcp__search-analytics` MCP と今日導入した `seo-audit` スキルで大部分が代替可能だったのに、要件定義から直接設計に入ってしまい、リナ7ラウンドの局所検証と KYT を実施後に機能重複が発覚。工数1〜2週間を浪費する寸前だった。

**確認対象（6種類、漏れなく）**:
1. **既存カスタムスキル**: `~/.claude/skills/*` 配下のトリガー語と要件を照合
2. **既存外部スキル・プラグイン**: skills.sh / plugin marketplaces で代替可能か
   - `npx skills add <repo> -l` でマーケ内一覧取得
   - `claude plugin marketplace list` ＋ 各マーケの `plugins/` ディレクトリ確認
3. **既存MCPサーバー**: `mcp__*` ツール群で機能を満たせるか
   - ToolSearch で `mcp__` をキーワードに全ツール一覧を取得
4. **既存Pythonスクリプト**: `~/.claude/scripts/*` 配下
5. **既存エージェント**: `agents/*` の組み合わせで実現できないか
6. **外部API・SaaS・OSツール等の外部資産**: Notion API / Google Workspace / Chatwork / Zapier / n8n / OS標準コマンド等で代替可能か（内部資産だけでなく既存契約中の外部サービスも対象）

**判定ルール**:
- **完全代替可能** → 開発中止、既存を使う（要件定義を「既存活用で解決」に更新）
- **部分代替可能** → 要件縮小、独自開発は代替不可の部分のみ（削減工数を Notion 案件に記録）
- **代替不可** → 予定通り独自開発（リサーチ結果を記録して設計へ）

**再実施の上限（ループ防止）**:
- 「部分代替可能」→ 要件縮小 → 2.5再実施 のループは **最大2回まで**（初回含めて計2回）
- 3回目の再実施は禁止。代わりに現時点の縮小要件で独自開発を強制開始（ステップ3へ進む）
- 再実施回数は Notion「1.5 既存資産リサーチ結果」に明記

**担当**: アスカが実施。範囲が広い場合は `/skill-finder` 経由でミオまたは claude-code-guide に委任可

**記録**: Notion案件ページの「1.5 既存資産リサーチ結果」セクションに以下フォーマットで追記

```
### 確認結果
- 既存スキル: （該当スキル名 + 代替可能性）
- 既存プラグイン/外部スキル: （該当名 + 代替可能性）
- 既存MCPサーバー: （該当ツール名 + 代替可能性）
- 既存スクリプト: （該当スクリプト名 + 代替可能性）
- 既存エージェント組合せ: （該当エージェント + 代替可能性）

### 判定
- [ ] 完全代替可能 → 開発中止
- [ ] 部分代替可能 → 要件縮小（縮小後の独自開発範囲を明記）
- [ ] 代替不可 → 独自開発継続

### 決定内容
（シンヤさん承認事項）
```

**所要時間目安**: 15〜30分（大型案件でも1時間以内）。この投資で後工程の大幅削減が期待できる。

**絶対にスキップしない**。「新規性が明らかだから不要」という判断は禁止。2026-04-18の gsc-ga4-analyzer 事例がそれで失敗した教訓。

### ステップ3: 設計（シュウ主担当）

シュウ（backend-engineer）に委任。UI系であればユイ・カイも巻き込む。

設計内容:
- 影響範囲
- データフロー
- インターフェース
- 代替案と選定理由

完成したら「2. 設計」セクションに記録。

> **自動リナ召集について（CLAUDE.mdルール）**
> 設計を「決定する直前」、CLAUDE.mdの「Rina Auto-Invocation Rule」によりリナが自動召集され、設計の論理検証・前提チェックが行われます。これも維持してください。ステップ5の「リナ統合検証」とは別目的です。

### ステップ4: KYT（危険予知）

`/kyt` スキルを呼び出し、ラウンドテーブルで危険予知を実施。

- **Win/Mac の差分は必ず観点に入れる**（Cross-Platform Pre-Verification Rule）
- 結果を「2.5 KYT結果」セクションに記録
- 「実装前に対処」項目が残っている間は次のステップに進まない

### ステップ5: リナ統合検証（logic-verifier）

ステップ2/3の自動リナ召集が「要件単体」「設計単体」の局所検証だったのに対し、ステップ5は **「KYT結果 + 要件 + 設計 + KYT対処策」を統合した最終論理検証** として位置付けます。重複ではなく、別目的の検証です。

リナに以下を依頼：
- 要件・設計・KYT対処策を **横断した整合性チェック**
- KYT対処策が設計に矛盾なく組み込めているか
- 統合された前提・制約に論理的な抜け穴がないか

結果を「3. リナ統合検証結果」セクションに記録。致命的指摘がなくなるまで次へ進まない。

### ステップ6: 実装（シュウ）

シュウに委任。**アスカは絶対にコードを書かない**（Asuka Never Codes Directly）。

- 日次の進捗を「4. 実装ログ」に追記
- **実装中に不具合が発生したら、その場で「5. 不具合・課題リスト」にチェックボックス付きで追記**（絶対に省略しない）
- `.env` 等の git-ignored ファイルを編集する場合は Safe Editing Rule（事前バックアップ）を遵守

### ステップ7: レビュー（サクラ）

サクラ（code-reviewer）にコードレビューを依頼。

- セキュリティ観点の対象（ファイル読み書き／認証フロー／ユーザー入力処理／外部API通信）に該当する場合は、セキュリティレビューも自動で含める
- **実装中に発見した不具合がすべて「5. 不具合・課題リスト」に反映されているか** をサクラに必ず確認させる（追記漏れ検出）
- 結果を「6. レビュー結果」セクションに記録
- 指摘対応がすべて完了してから次へ

### ステップ8: 動作確認＋ふりかえり

1. 完了条件のチェックリストを「7. 動作確認チェック」に記録
2. **完了報告前ゲート**: Notion ページの「5. 不具合・課題リスト」を `notion-tasks.py --show` で確認し、未対応チェックボックス（`- [ ]`）が0件であることを必ず確認する。1件でも残っていたら完了報告しない
3. `/kaizen` スキルを呼び出してふりかえりを実施。**振り返り項目に「不具合リストへの追記漏れがなかったか」を必ず含める**
4. 結果を「8. ふりかえり」セクションに記録
5. シンヤさんに完了報告
6. Knowledge Documentation Confirmation Protocol に従い、ナレッジ化するかをシンヤさんに確認

> **スキル・エージェント定義ファイルを作成/修正する案件の場合（重要）**
> ステップ5の「リナ統合検証」とは **別に**、ステップ8完了前に CLAUDE.md の「Rina Review After Rule Changes」ルールに基づく **スキル定義レビュー** をリナに依頼してください。リナを2回呼ぶことになりますが、目的が異なります（統合検証 vs スキル定義の論理レビュー）。スキップ不可。

## Notion案件自動登録（ステップ1完了時）

ヒアリングが完了したら、即座に `notion-tasks.py --add` を実行します。

```bash
python ~/.claude/scripts/notion-tasks.py --add "<機能名>" \
  --type 実装 \
  --priority P2-今週中 \
  --env <Windows or Mac> \
  --memo "<要件サマリ。Task Registration Content Standardの5項目を満たすこと>"
```

- 環境は `~/.claude/.env` の `PC_PLATFORM` から自動判定（`win` → Windows / `mac` → Mac）
- 優先度は P2-今週中 がデフォルト。シンヤさんが別途指示した場合はそれに従う
- メモは CLAUDE.md「Task Registration Content Standard」の5項目（背景／作業内容／注意点／関連ファイル／完了条件）を必ず満たす

### ページ本文の初期テンプレ

登録直後、`notion-tasks.py --add-block` を使って以下のセクション構造で本文を初期化してください。

```markdown
## 1. 要件定義
（ヒアリング後に記入）

## 1.5 既存資産リサーチ結果
（ステップ2.5 実施後に記入。判定結果と決定内容を必ず明記）

## 2. 設計
（設計後に記入）

## 2.5 KYT結果
（/kyt実行後に記入）

## 3. リナ統合検証結果
（ステップ5のリナ統合検証後に記入）

## 4. 実装ログ
（日次追記）

## 5. 不具合・課題リスト
発生した不具合・課題は以下に追記。完了前に全て解消すること。
- [ ] （未対応の課題）

## 6. レビュー結果
（サクラレビュー後に記入）

## 7. 動作確認チェック
（完了条件の☑）

## 8. ふりかえり
（/kaizen結果）
```

各セクションへの追記は、フェーズが進むごとに `notion-tasks.py --add-block` で行います。

## フェーズ遷移のクオリティゲート

| フェーズ | 次へ進む条件 |
|---|---|
| 1→2 | ヒアリング完了＋Notion案件登録済み |
| 2→2.5 | シンヤさんが要件定義に合意（直前のリナ自動局所検証をクリア） |
| 2.5→3 | 既存資産リサーチ完了＋シンヤさんが「独自開発継続 / 要件縮小 / 開発中止」のいずれかを判断済み |
| 3→4 | 設計ドキュメントがNotionに記録済み（直前のリナ自動局所検証をクリア） |
| 4→5 | KYTの「実装前に対処」項目がすべて対処完了 or 設計に反映済み |
| 5→6 | リナ統合検証で致命的指摘なし |
| 6→7 | シュウから実装完了宣言 |
| 7→8 | サクラレビュー指摘がすべて対応完了＋不具合リスト追記漏れなし確認済み |
| 8→完了 | 不具合リストの未対応0件 ＋ 完了条件すべて☑ ＋ /kaizen完了 |

各フェーズの遷移時、アスカは「ゲート条件を満たしているか」を明示的にチェックしてから次へ進んでください。

## CLAUDE.md 既存ルールとの整合性（必ず遵守）

- **Asuka Never Codes Directly**: アスカはコードを書かず、必ずシュウに委任する
- **Rina Auto-Invocation Rule**: ステップ2（要件合意直前）・ステップ3（設計決定直前）の自動リナ召集は維持。ステップ5の統合検証とは別目的
- **Rina Review After Rule Changes**: スキル/エージェント定義ファイルを変更する案件では、ステップ5とは別にスキル定義レビューをリナに依頼
- **Deliverable Quality Gate**: コード成果物はサクラレビュー必須。セキュリティ観点該当時は自動でセキュリティレビューも依頼
- **Safe Editing Rule for Git-Ignored Files**: `.env` 等を編集する場合は事前バックアップ必須
- **Cross-Platform Pre-Verification Rule**: Win/Mac 差分は KYT の観点に必ず入れる
- **Knowledge Documentation Confirmation Protocol**: フロー完了時に「ナレッジ化するか」を必ず確認
- **Immediate Recording Rule**: フロー進行中に決まった方針・ルールはその場で記録

---

## 注意事項

- 適用範囲（新規ファイル作成 / 新規ユーザー操作 / 新規外部統合 のいずれか）に該当しない案件にはこのフローを適用しない
- ステップを飛ばさない。「小さい機能だから設計は省略」などは禁止
- **ステップ2.5（既存資産リサーチ）は絶対にスキップしない**。「新規性が明らかだから不要」という判断は禁止（2026-04-18 gsc-ga4-analyzer 案件でこの判断ミスにより工数浪費の危機が発生。既存 MCP + seo-audit で大部分代替可能だったのに設計まで進んでしまった）
- 不具合・課題リストへの追記は、発見した瞬間に必ず行う（後でまとめてやる、は禁止）
- フロー完了前に必ず `notion-tasks.py --show "<機能名>"` で全セクションの状態を確認する
