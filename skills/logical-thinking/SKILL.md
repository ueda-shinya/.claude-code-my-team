# logical-thinking スキル

ロジカルシンキング（Logical Thinking）= 論理的に物事を考え、結論を導く思考法。事業戦略スクール「知足」のフレームワーク準拠で、4フレームワーク（MECE／WHY型／ピラミッド／SO型）と5実践ステップを統合適用する。

## 適用条件

- このスキルを使う場面：
  - 課題を構造的に分解し、明確な解決策を提示したい
  - 複雑な事象を「結論→理由→具体例」で説得力ある形に整理したい
  - 売上低下・業績不振等の原因を MECE+WHY型で深掘りしたい
- このスキルを使わない場面：
  - ピラミッド構造単体で結論→理由→詳細を整理したい → `pyramid-structure`
  - 既存の枠組みを疑い水平発想で新案を出したい → `lateral-thinking`
  - 前提や根拠を疑い妥当性を確認したい → `critical-thinking`
  - 意思決定で複数案を比較スコアリングしたい → `decision-making-framework`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 対象課題 | 必須 | 構造化したい問題・テーマ・目的 | 「売上が前年比20%減少した原因を特定したい」 |
| 関連データ | 任意 | 既存の数値・事実・顧客情報等 | 客単価／購入頻度／購入者数の月次推移 |
| 利用フレームワーク | 任意 | 4フレームから指定（無指定なら全適用） | 「MECEとWHY型のみで」 |
| アウトプット形式 | 任意 | レポート／ピラミッド図／結論サマリー | 「結論サマリー1ページ」 |

## 4つの基本フレームワーク

| フレームワーク | 役割 | 使う場面 | 例 |
|---|---|---|---|
| **MECE**（Mutually Exclusive, Collectively Exhaustive・漏れなくダブりなく） | 情報を重複なく網羅的に分類する原則 | 事象を全パーツに分解する初動 | 売上 = 単価 × 販売数量 ／ 売上 = 客単価 × 購入頻度 × 購入者数 |
| **WHY型思考**（Why So） | 問題の原因を「なぜ」を繰り返して深掘り | 根本原因の特定（トヨタの「なぜ5回」） | 売上減少→なぜ？→顧客単価低下→なぜ？→割引率が高すぎる |
| **ピラミッド構造** | 主張を先に示し、根拠を階層的に展開（結論→理由→具体例） | 説明・提案資料の構造化 | 結論「キャンペーン設計が必要」←理由3つ←各具体例 |
| **SO型思考**（So What・だから思考） | 主張や結論に対し、根拠を示して説得力を高める | アウトプット側の論理強化 | 新商品開発を進めるべき → だから、競争力を維持できるから |

**4つの関係性**：
- 縦軸：WHY型（下→上、Why So）／ SO型（上→下、So What）で結論と根拠を双方向に検証
- 横軸：MECE で各層を漏れなくダブりなく分類
- 全体構造：ピラミッド構造で結論→理由→具体例を3階層に整理

## 実行ステップ

### ステップ1：課題を明確化する

- 入力：対象課題（ユーザーから受領）
- 処理：
  1. 課題の全体像と目的を1文に言語化する（例：「売上が減少している理由を特定する」）
  2. 解決すべき問いの粒度を確認する（広すぎる場合は分解する：「全社売上減」→「BtoC事業のリピート率低下」等）
  3. ゴール（何が分かれば解決か）を明示する
- 出力：問い1文＋ゴール定義

### ステップ2：情報を収集・整理する（MECE）

- 入力：ステップ1の問い
- 処理：
  1. 必要なデータ・事実を洗い出す（顧客データ／売上データ／市場調査／顧客アンケート等）
  2. **MECE原則で分類する**：重複なく網羅的に。代表的な分解軸を提示：
     - 売上分解軸：`売上 = 客単価 × 購入頻度 × 購入者数` ／ `売上 = 単価 × 販売数量` ／ `売上 = 新規 + 既存リピート`
     - 顧客分解軸：新規／既存、年代別、地域別、チャネル別
     - 時系列軸：日次／週次／月次／前年同月比
  3. 各分類のデータ値を埋める（不足項目は「未取得」と明記）
- 出力：MECE分類表（各セグメントの数値が揃った状態）

### ステップ3：仮説を立てる

- 入力：MECE分類表
- 処理：
  1. 数値の異常値・偏りから「可能性の高い原因／解決策」を推測する
  2. 複数仮説を立て、優先順位を付ける（根拠の強さ＋影響度）
  3. 各仮説に「検証可能な形」で落とし込む（測定可能な条件）
- 出力：仮説リスト（仮説文／根拠／検証方法を1セット×3〜5本）

### ステップ4：検証する（WHY型・SO型）

- 入力：仮説リスト
- 処理：
  1. **WHY型思考で深掘り**：各仮説に「なぜそうなる？」を3〜5回繰り返し、根本原因へ降りる
     - 例：「割引キャンペーンが効果的でない」→なぜ？→「常連客の値引き利用が中心」→なぜ？→「新規流入導線が不足」→なぜ？→「広告チャネルが既存顧客リーチに偏っている」
  2. **SO型思考で確からしさを問う**：各根本原因に「だから何？」を当てて、結論として成立するか検証
     - 例：「広告チャネルが既存リーチに偏る」→だから何？→「新規CACが上昇しMQL不足」→だから何？→「結論: 新規流入チャネル拡張が必要」
  3. データや事実で裏付ける（アンケート／実測値／A/Bテスト結果）
  4. 矛盾があれば仮説に戻って修正
- 出力：検証結果テーブル（仮説ごとに：採択／却下／部分的に成立、根拠データ、結論案）

### ステップ5：結論を提示する（ピラミッド構造）

- 入力：検証結果テーブル
- 処理：
  1. 最上位に**結論**（1文・主張）を置く
  2. 第2層に**理由**（3つ程度・MECEになっているか確認）を配置
  3. 第3層に**具体例・データ・事実**を理由ごとに紐付ける
  4. 最後に「だから何？／本当に？」のセルフチェックで論理を確認
- 出力：ピラミッド構造（結論→理由3つ→具体例）

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 課題が広すぎて1つの結論に収束しない | ステップ1に戻り問いを分解する | MECEで分類しても結論が出ない原因はゴール曖昧 |
| MECE分解で「その他」が30%超 | 分解軸を選び直す | その他比率が高い＝軸選定ミスのサイン |
| WHY型を5回繰り返しても根本原因に到達しない | 別の分解軸（仮説）に切り替える | 1本の深掘りで詰まったら横展開する |
| 結論の根拠が1つしかない | 理由を追加するか結論の確度を下げる | ピラミッド構造は理由3つが標準（2では弱い、5以上は冗長） |
| データ不足で検証できない | ステップ2に戻り収集計画を立てる | 仮説のままアウトプットを出さない |
| 感情・偏見が混入している疑い | `critical-thinking` を併用する | ロジカルだけでは前提検証が弱い |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| MECEで分解せずいきなり仮説を立てる | 必ずステップ2でMECE分解してから仮説に進む | 抜け漏れに気づかず的外れな仮説に時間を使う |
| WHY型を1〜2回で止める | 最低3回、推奨5回深掘りする | 表層原因で止まると施策が効かない |
| 結論先行でデータを後付けする | データ→仮説→結論の順序を守る | 確証バイアスで結論を歪める |
| ピラミッド構造の理由が1つだけ | 理由は3つを目安に複数提示 | 1つだけだと「他の可能性は？」と問われて崩れる |
| SO型の「だから何？」を省く | 結論の前に必ず「だから何？」でセルフチェック | 根拠と結論が論理的に繋がっていない事故を防ぐ |
| 「その他」カテゴリで誤魔化す | MECE軸を選び直す | 分解軸の選定ミスを後段に持ち込まない |

## 連携スキル

- `pyramid-structure`：ピラミッド構造の3層整理（結論→理由→詳細）が単体で必要なときに使う。本スキルのステップ5で生成するピラミッドはこのスキルの出力フォーマットに準拠する
- `critical-thinking`：前提や根拠を疑う妥当性検証が必要な場合に併用。ロジカルだけでは「前提が間違っている」ケースを拾えない
- `lateral-thinking`：論理的に解決策が見つからない・既存枠組みを超えた新案が必要な場合に切り替え
- `decision-making-framework`：複数結論候補を比較してスコアリングで意思決定する場合に接続
- `pdca-cycle`：結論を実行→検証する継続改善が必要な場合に接続

## 5状態出力契約

このスキルは外部API・サブプロセス・外部設定依存を持たないため、CLAUDE.md「5状態出力契約」の対象判定基準には該当しない。ただし以下のケースでは明示マーカーを使うこと：

- `[OK:0件]`：仮説検証の結果すべて却下になった場合（採択結論なし）
- `[PARTIAL:N成功/M失敗]`：複数仮説のうち一部のみデータで裏付けられた場合
- `[SKIP]`：データ不足で検証ステップを実行できなかった場合（理由を明示）
