# /kyt スキル（危険予知ラウンドテーブル）

設計完了後・実装前に、関係エージェントを召集して設計のリスクを洗い出し、対策を確定するためのスキルです。アスカ（chief-of-staff）がファシリテーターを務めます。

## 起動トリガー

- `/kyt` コマンド実行時
- `/feature-flow` からの内部呼び出し
- シンヤさんが「KYTやって」「危険予知して」「pre-mortem」と発言したとき

## 前提

- 対象は「設計が固まっている」案件のみ。設計が未確定なら先に設計フェーズに戻ってください
- 対象案件のNotionページが既に存在していること（なければアスカが先に `notion-tasks.py --add` で作成）

## 参加者ルール

### 固定コア5名（常時召集・必須）

| エージェント | 役割 |
|---|---|
| シュウ（backend-engineer） | 実装観点 |
| サクラ（code-reviewer） | レビュー・セキュリティ観点 |
| リナ（logic-verifier） | 論理・前提検証 |
| ミオ（researcher） | 調査・プラットフォーム差分 |
| ハル（writer） | ドキュメント・整合性 |

### 案件特性に応じた追加召集（アスカ判断）

| 案件特性 | 追加召集 |
|---|---|
| UI / LP系 | ユイ（web-designer） / カイ（lp-designer） |
| 運用 / インフラ系 | ソウ（trouble-shooter） |
| マーケ / コピー系 | レン（marketing-planner） / コト（copywriter） |
| 営業関連 | タク（sales-consultant） |
| ビジネスモデル関連 | ナギ（business-consultant） |
| 法務リスクが関わる | ケン（legal-advisor） |

**参加者は最大8〜10名を想定。** アスカは案件概要を見て、コア5名にどの追加メンバーを足すかをセットアップ時に決定します。

## タイムボックス（参加人数別）

- **コア5名標準 → 45分構成**
- **7名以上 → 60分構成**
- フェーズ2（ラウンドロビン発表）の時間は **参加人数 × 1分** で算出する

## 進行手順

ファシリテーターはアスカ。各フェーズで以下の順番に進めてください。

### フェーズ0: セットアップ（5分）

1. アスカが対象案件の設計概要を全員に共有
2. アスカが参加者リスト（コア5名＋追加メンバー）を宣言
3. アスカが冒頭に次の問いを投げる（pre-mortem導入）：
   > 「このまま実装して、X日後に壊滅的な障害が起きました。何が原因だと思いますか？」
4. 観点リスト（13カテゴリ）を全員に共有

### フェーズ1: 現状把握 / 沈黙ブレインストーミング（10分）

- 全員が**沈黙でリスクを書き出す**（集団思考防止のため、発言禁止）
- 観点リスト（13カテゴリ）を見ながら、思いついたリスクを箇条書きで提出
- 各エージェントは主担当観点を最優先で見るが、他観点に踏み込んでも良い

**アスカの問いかけ例：**
- 「設計書を読んで、一番不安な箇所はどこですか？」
- 「これを他チームに引き継いだとき、何を警告しますか？」

### フェーズ2: 本質追究 / ラウンドロビン発表（参加人数 × 1分）

- 時計回りで**1人1リスクずつ発表**（1分以内）
- 重複は「Aさんのと同じ」でマージ終了
- 「今は思いつかない」はパス可（2周目で拾う）
- アスカは出てきたリスクを **カテゴリ / 内容 / 発生確率 / 影響度** の表形式で記録

**リスクが枯渇しかけたときのアスカの追加問い：**
- 「APIが落ちたら？」
- 「データが壊れたら？」
- 「Windowsで動かなかったら？」
- 「一番ベテランが休んでいる日に障害が起きたら？」
- 「ロールバックできますか？」

### フェーズ3: 対策樹立 / 3分類への振り分け（10〜15分）

発表されたリスクを以下の判断基準で3分類に振り分けます。

**判断マトリクスは「一次分類（基準）」：**

|              | 影響：小 | 影響：大 |
|--------------|----------|----------|
| **発生：高** | 監視     | **対処（必須）** |
| **発生：低** | 許容     | 監視 or 対処 |

**3分類の定義：**

- **実装前に対処**: リリース停止 / データ破損 / セキュリティ侵害につながるもの。必ず実装前に手を打つ
- **実装後に監視**: サービス継続は可能だが、放置で悪化するもの。アラート設定で対応
- **許容**: 発生確率極低 or 影響軽微。**意図的選択として記録必須**（「見なかったことにする」とは明確に区別）

**投票ルール（マトリクスの補正用）：**

投票は **「対処枠への昇格申請」にのみ使用** します。マトリクスの機械判定を、現場感覚で補正するための仕組みです。

- マトリクスによる一次分類が **基準**
- 各エージェントに3票配布
- 「マトリクス上は監視/許容だが、これは対処にすべき」というリスクに票を投じる
- **昇格には参加者の過半数（50%超）の票が必要**
- **降格（対処→監視等）には投票を使わない**（一度対処と判定されたものを下げない）
- **同点・可否が微妙な場合は、ファシリテーター（アスカ）が保守側に倒す**（＝対処へ昇格させる）

### フェーズ4: 13カテゴリ全網羅チェック（5分）

アスカが13カテゴリすべてを順番に読み上げ、各カテゴリについて「該当リスクあり（分類済み）」「該当リスクなし（議論の結果0件）」のいずれかを明示的に宣言します。

- **「議論しなかった」「該当リスクなし」「許容」を完全に区別する**
- 0件カテゴリは「議論の結果0件」と明示的に宣言してNotion記録に残す
- 議論漏れカテゴリがあった場合はその場で議論する

### フェーズ5: 目標設定 / オーナーと期日の確定（10分）

- 「対処」分類の各項目に**オーナー（担当エージェント）と期日**を割り当て
- 「監視」分類の各項目に**アラート条件**を定義
- 「許容」分類の各項目に**許容理由**を明記
- アスカが最終確認し、Notion記録フォーマットに整形

## 観点リスト（13カテゴリ）

既存の8パターンは `~/.claude/skills/kaizen/references/software-patterns.md` を参照してください。

**software-patterns.md由来の8カテゴリ：**
1. **Design Gap**
2. **Process Gap**
3. **Architecture Gap**
4. **Security**
5. **Data**
6. **Scaling**
7. **Concurrency**
8. **Policy Gap**

**プロジェクト固有5カテゴリ（必須）：**

9. **Platform Gap**: Win/Mac差分（`python3` vs `python`、パス区切り、`taskkill`/`open -a`等のOS固有コマンド、`%-m`/`%-d`等のフォーマット差）
10. **API Cost Risk**: Claude/Gemini API使用時のコスト閾値・`--test-mode`フラグ・`api-cost-history.json`への履歴記録
11. **Rollback Gap**: 実装を取り消せるか。git-ignoredファイル変更時のバックアップ手順（`.bak.YYYYMMDD-HHMMSS`形式、最大5世代）
12. **File Integrity Risk**: git-ignoredファイル書き込み時の破損復旧手段（Edit失敗時の `sed` フォールバック禁止）
13. **Review Bypass Risk**: 「小さい変更だから」でレビューをスキップしたくなる設計になっていないか（コードレビュー必須ルールの抜け穴）

## エージェント役割分担（13カテゴリ準拠）

| エージェント | 主担当カテゴリ |
|---|---|
| シュウ（backend-engineer） | Concurrency / Scaling / Architecture Gap |
| サクラ（code-reviewer） | Security / Review Bypass Risk / Policy Gap |
| リナ（logic-verifier） | Design Gap / Rollback Gap / API Cost Risk |
| ミオ（researcher） | Platform Gap / Process Gap（調査観点） / Data（外部データ観点） |
| ハル（writer） | File Integrity Risk / Process Gap（文書側） |
| アスカ | ファシリテーター専任（リスク発言はしない） |

主担当以外のカテゴリで気になったリスクも自由に発言可。役割は「最低限カバーすべき範囲」を示すもの。追加召集メンバーは案件特性に応じた観点を持ち込む。

## 効率化ルール

- **ラウンドロビン厳守**: 1人1リスク・1分以内・時計回り
- **沈黙ブレインストーミング先行**: フェーズ1は完全沈黙（集団思考防止）
- **パス可**: 「今は思いつかない」は2周目で拾う
- **重複マージ**: 「Aさんのと同じ」で発言終了
- **ファシリテーターは中立**: アスカはリスク発言をせず、進行と記録に専念

## Notion記録フォーマット

KYT終了後、アスカは以下の形式でMarkdownを生成し、`notion-tasks.py --add-block` で対象案件ページに記録します。

```bash
python ~/.claude/scripts/notion-tasks.py --add-block "<案件タイトル部分一致>" --text "$(cat <<'EOF'
## 2.5 KYT結果（YYYY-MM-DD実施）

### 参加者
コア: シュウ / サクラ / リナ / ミオ / ハル
追加: <案件特性に応じた追加メンバー>
ファシリテーター: アスカ

### 実装前に対処（N件）
- [ ] [カテゴリ] リスク内容 / 対策 / オーナー / 期日
- [ ] [カテゴリ] リスク内容 / 対策 / オーナー / 期日

### 実装後に監視（N件）
- [ ] [カテゴリ] リスク内容 / アラート条件
- [ ] [カテゴリ] リスク内容 / アラート条件

### 許容（N件・意図的選択）
- [カテゴリ] リスク内容 / 許容理由

### 13カテゴリ全網羅チェック結果
1. Design Gap: <該当N件 / 議論の結果0件>
2. Process Gap: <該当N件 / 議論の結果0件>
3. Architecture Gap: <該当N件 / 議論の結果0件>
4. Security: <該当N件 / 議論の結果0件>
5. Data: <該当N件 / 議論の結果0件>
6. Scaling: <該当N件 / 議論の結果0件>
7. Concurrency: <該当N件 / 議論の結果0件>
8. Policy Gap: <該当N件 / 議論の結果0件>
9. Platform Gap: <該当N件 / 議論の結果0件>
10. API Cost Risk: <該当N件 / 議論の結果0件>
11. Rollback Gap: <該当N件 / 議論の結果0件>
12. File Integrity Risk: <該当N件 / 議論の結果0件>
13. Review Bypass Risk: <該当N件 / 議論の結果0件>
EOF
)" --env <Windows or Mac> --assignee Asuka
```

**コマンド実行時の注意：**
- Windowsでは `python`、Macでは `python3` を使用（`PC_PLATFORM` で判定）
- 案件タイトルは部分一致でOK（完全一致不要）
- `--env` は実施環境を指定

## 完了後の動き

1. Notion記録完了をシンヤさんに報告
2. 「実装前に対処」項目があれば、各オーナーに個別タスクとして起票するか確認
3. `/feature-flow` から呼び出された場合は、対処項目の完了を待ってから次フェーズへ進む

## 制約・禁止事項

- **設計が未確定の状態でKYTを実施しない**（仮の設計に対する危険予知は時間の無駄）
- **アスカがリスク発言をしない**（ファシリテーター専任、中立性確保）
- **「許容」の意図的選択を省略しない**（暗黙の見逃しと意図的許容を明確に区別）
- **3分類に振り分けないリスクを残さない**（フェーズ3で必ず全件分類すること）
- **13カテゴリ全網羅チェックを省略しない**（フェーズ4必須）
- **「議論しなかった」と「該当リスクなし」と「許容」を混同しない**
- **降格投票を行わない**（対処判定を下げない）
- **Notion記録を省略しない**（口頭やチャットだけで終わらせない）
