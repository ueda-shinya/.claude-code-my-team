# client-expectation-management スキル

期待値調整 = クライアントの期待や要求を正確に把握し、現実的かつ実行可能な範囲に調整するスキル。過剰な期待や誤解を防ぎ、双方にとって満足度の高い結果を生み出すために必要。事業戦略スクール「知足」のフレームワーク準拠。

## 期待値調整の4つの目的

| 目的 | 内容 |
|---|---|
| 目的① 信頼関係の構築 | 期待値を合わせることで信頼が積み上がる |
| 目的② トラブルの回避 | 認識ズレによる炎上を予防 |
| 目的③ プロジェクト成功率の向上 | スコープ確定で納期・品質を達成しやすくなる |
| 目的④ 長期的な関係維持 | リピート受注・紹介につながる |

## 適用条件

- このスキルを使う場面：
  - クライアントが過剰な期待を持っている場面の調整
  - PJ開始時のスコープ・スケジュール・予算の合意形成
  - 納期遅延や仕様変更時の再合意
  - 既存クライアントとの中長期関係維持
- このスキルを使わない場面：
  - 提案書全体の生成 → `proposal-builder`
  - 商談ヒアリング項目の設計 → `hearing-questioning-skills`
  - 受注/失注分析 → `loss-analysis-kbf-ksf`
  - PJ収支管理・契約条件設計 → `project-finance-contract`
  - PM体制の構築 → `project-team-structure`
  - PJスケジュール設計 → `schedule-management`
  - ガントチャート設計 → `gantt-chart-design`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| プロジェクト概要 | 必須 | 案件の概要・規模・期間 | 「Webサイトリニューアル・3ヶ月・予算300万円」 |
| クライアント情報 | 必須 | 業界・役職・既知の要望 | 「製造業の経営企画部長・SEO/CV重視」 |
| 期待値の現状 | 必須 | 現在クライアントが持っている期待 | 「全機能をフェーズ1で実装したい」 |
| 自社の実行可能範囲 | 必須 | リソース・期間で実現可能な範囲 | 「フェーズ1は必須機能のみ・追加はフェーズ2」 |
| 想定される摩擦 | 任意 | 期待のズレ・予算超過・納期遅延等 | 「クライアントは納期短縮を要求」 |

## 期待値調整の5プロセス

| ステップ | 内容 | 実例 |
|---|---|---|
| 1. クライアントニーズの理解 | 要求をヒアリングし、具体的な目標や成功の定義を明確にする | 「このプロジェクトの目的は何ですか？」と質問でニーズを深掘り |
| 2. 現実的な提案の作成 | クライアントの期待に対し、実現可能な範囲を具体的に提示 | 「このスケジュールで達成可能なのはAとBです。Cは次フェーズで実施しましょう」 |
| 3. 明確な合意形成 | 提案内容・スケジュール・予算を文書化しクライアントと共有 | 契約書や議事録を活用 |
| 4. 定期的なコミュニケーション | 進捗や変更点を適宜報告し、期待値を調整 | 「現在の状況と課題は以下の通りです」 |
| 5. 納品後の確認とフォローアップ | 納品後にフィードバックを受け取り、次回プロジェクトに活かす | 「今回の成果物に対するご意見を伺います」 |

## 必要なスキル（5つ）

| # | スキル | 内容 |
|---|---|---|
| 01 | コミュニケーション能力 | クライアントの意図を正確に理解し、明確に伝える力 |
| 02 | 説得力 | 実現可能な提案を納得してもらう能力 |
| 03 | 問題解決能力 | 変更や追加要求に柔軟に対応し、解決策を提示する力 |
| 04 | リスク管理能力 | 潜在的なリスクを予測し、対応策を計画する能力 |
| 05 | 共感力 | クライアントの立場や感情を理解し、信頼を築く力 |

## 注意点（4つ）

| 注意点 | 内容 |
|---|---|
| 曖昧なコミュニケーションを避ける | 数値・期日・スコープを明確にする |
| 無理な約束をしない | できないことを「できる」と言わない |
| 変更点に迅速に対応する | 仕様変更・スコープ変更は即座に再合意 |
| クライアントを巻き込むプロセスを構築する | 一方通行ではなく協働関係を作る |

## 実践手法（3つ）

| 手法 | 内容 |
|---|---|
| 01. ヒアリングの強化 | クライアントのニーズを深く理解する質問を用意する |
| 02. 進捗報告の定期化 | 状況を見える化し、安心感を提供する |
| 03. データを活用した提案 | 根拠に基づく具体的な提案で説得力を高める |

## 実行ステップ

### ステップ1：クライアントニーズの理解（プロセス1）
- 入力：プロジェクト概要、クライアント情報、期待値の現状
- 処理：
  1. ヒアリング項目を設計（→ `hearing-questioning-skills` 連携）
  2. 「目的は何か」「成功の定義は何か」を明文化
  3. 暗黙の期待や前提を3つ以上書き出す
  4. 期待値ギャップを洗い出す（クライアント期待 vs 自社実行可能範囲）
- 出力：ニーズ整理シート＋期待値ギャップ表

### ステップ2：現実的な提案の作成（プロセス2）
- 入力：ステップ1の出力、自社リソース
- 処理：
  1. 「できる」「できない」「条件付きでできる」の3分類でスコープを整理
  2. フェーズ分割案を提示（フェーズ1: 必須／フェーズ2: 追加／フェーズ3: 拡張）
  3. 各フェーズの期間・予算・成果物を明示
  4. データを活用した根拠（過去事例・ROI試算・リスク予測）を添える
- 出力：フェーズ分割提案書（スコープ・期間・予算・成果物が明示された状態）

### ステップ3：明確な合意形成（プロセス3）
- 入力：ステップ2の提案
- 処理：
  1. 議事録テンプレに沿って合意内容を文書化
  2. 以下4点を明示：①スコープ ②納期スケジュール ③コスト・支払条件 ④責任分担
  3. クライアントから書面合意（契約書・発注書・メール承認）を取得
  4. 変更があった場合の手続き（変更管理プロセス）も合意に含める
- 出力：合意ドキュメント（契約書／議事録／発注書）

### ステップ4：定期的なコミュニケーション（プロセス4）
- 入力：合意ドキュメント、進捗実績
- 処理：
  1. 報告サイクルを決定（週次／隔週／月次）
  2. 報告フォーマットを統一（進捗％・課題・次アクション・リスク）
  3. 変更要求が出たら即座にステップ3に戻り再合意
  4. クライアント側の関係者を巻き込む仕組み（定例MTG・チャット参加）
- 出力：定期報告フォーマット＋報告サイクル

### ステップ5：納品後の確認とフォローアップ（プロセス5）
- 入力：納品物、クライアントフィードバック
- 処理：
  1. 納品後1週間以内にフィードバックMTGを設定
  2. 4軸でヒアリング：①期待通りだった点 ②期待と違った点 ③改善要望 ④次回への期待
  3. 振り返りログを社内ナレッジ化（→ 次回PJに活かす）
  4. アップセル／継続提案のタイミング設計
- 出力：振り返りログ＋次回提案案

### ステップ6：4注意点と5スキルの自己評価
- 入力：これまでのPJ運営記録
- 処理：以下チェックで自己点検
  - 4注意点：曖昧コミュ／無理な約束／変更対応遅延／巻き込み不足の有無
  - 5スキル：コミュ／説得／問題解決／リスク管理／共感の各5段階評価
  - 3実践手法：ヒアリング強化／進捗定期化／データ活用提案 の実施度
- 出力：自己評価＋改善計画

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| クライアントの期待が予算/期間を大幅に超える | フェーズ分割で「できる範囲」を提示し、Cは次フェーズへ | 全要求を呑むと炎上、断ると失注 |
| クライアントが「全機能を1フェーズで」要求 | 必須機能のみフェーズ1とし、追加は次期で具体的に分割提案 | 必須機能を優先し、追加機能は次期フェーズに分割（事例：ITPJ） |
| 仕様変更が多い | 変更管理プロセスを合意ドキュメントに含めておく | 後から再合意するコストが高い |
| 進捗が遅延しそう | 早期にクライアントへ報告＋再合意 | 黙っていると信頼が一気に崩れる |
| 文書化なしで口頭合意のみ | 議事録／メール承認を必ず残す | 認識ズレが後で炎上の火種になる |
| クライアントが感情的になっている | 共感力で受け止め、事実ベースで再合意 | 説得より先に共感が必要 |
| 納品後フィードバックを取らない | 必ず4軸（期待通り／違った／改善／次回）でヒアリング | 次PJのアップセル・改善ナレッジ機会を逃す |
| 期待値ギャップが解消できない | 受注を見送るか、条件変更を再交渉 | 無理な受注は中長期の信頼を破壊 |
| クライアントまたは自社提案で「競合がいない／少ない／ブルーオーシャン／空白市場」系の楽観主張が出た | CLAUDE.md「Competitive Absence Audit Rule」に従い `competitive-absence-audit` スキル発動を検討 | 楽観前提のまま期待値を組むと中長期で破綻するため強制チェック |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 全要求を呑んで受注 | フェーズ分割で「できる範囲」を提示 | 後で炎上・赤字 |
| 口頭合意のみ | 議事録／メール承認で必ず文書化 | 認識ズレが火種になる |
| 「できないこと」を「できる」と言う | できないことは正直に伝え代替案を提示 | 信頼の基盤が崩れる |
| 仕様変更を曖昧に呑む | 変更管理プロセスで再合意 | スコープクリープでPJ崩壊 |
| 進捗遅延を黙っている | 早期報告＋再合意 | 隠すほど信頼が崩れる |
| クライアントを巻き込まない | 定例MTG・チャット参加で協働関係 | 一方通行で温度差が出る |
| 納品後ヒアリングをしない | 4軸（期待通り／違った／改善／次回）で必ず実施 | 次PJの機会を逃す |
| 感情論で説得しようとする | データを活用した根拠で説得 | 経営層には数字が効く |

## 連携スキル

- `hearing-questioning-skills`：ステップ1のニーズ理解で質問項目を設計
- `proposal-builder`：ステップ2の現実的な提案を提案書に落とし込む
- `project-finance-contract`：ステップ3の合意ドキュメント（契約書・支払条件・責任分担）
- `project-team-structure`：体制管理・役割分担表
- `schedule-management`：ステップ4の進捗管理・スケジュール設計
- `gantt-chart-design`：定期報告でガントチャートを使う場合
- `loss-analysis-kbf-ksf`：ステップ5の振り返りログをKBF/KSF分析に活用

## 5状態出力契約

外部API・サブプロセス・外部設定依存を持たないため、CLAUDE.md「5状態出力契約」の対象判定基準には該当しない。ただし以下のマーカーを使うこと：

- `[OK:0件]`：期待値ギャップが0件で調整不要の場合（合意済み）
- `[PARTIAL:N成功/M失敗]`：5プロセスのうち一部のみ実施できた場合
- `[SKIP]`：クライアント情報が不明確で調整プランが組めない場合（前提不足）

※外部API依存なしのため `[FAIL]` は通常発生しない。前提不足は `[SKIP]` で扱う。
