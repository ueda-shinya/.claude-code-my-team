# llmo-audit

AI検索エンジン（ChatGPT / Copilot / Gemini / Perplexity）での引用・言及獲得を目的とした **LLMO監査**スキル。対象サイト／LP／記事を14項目（＋参考1項目）のチェックリストで点検し、issue一覧＋優先順位付き改善案を出力する。

## 実行主体

**ミオ（researcher）**。アスカは発動判定のみ行い、実行はミオに委任する。CLAUDE.md「開発ワークフロー（専門領域はエージェントに委任）」に準拠。

**ミオ委任の根拠**: 本スキルは WebFetch による外部ページ取得＋14項目の多軸判定＋クライアント業種情報との突き合わせを伴うため、リサーチ主体のミオが適任。seo-audit は外部スキル（coreyhaises31）で英語 description のためアスカ直接可だが、本スキルは日本語クライアント案件向けで運用コンテキスト依存が強い。

## 他スキルとの役割分担

| 状況 | 使うスキル |
|---|---|
| **AI検索エンジン（ChatGPT/Gemini/Perplexity/Copilot）向け最適化監査** | **`llmo-audit`（本スキル）** |
| Google検索エンジン向けの技術SEO・on-page監査 | `seo-audit` |
| 流入減少・CV減少の1次診断（GA4→GSC） | `ga-gsc-diagnosis` |
| 記事本文執筆・リライト | `/blog-post` |
| LP新規作成 | `/lp-create` |
| UI / アクセシビリティ監査 | `web-design-guidelines` |

**seo-audit との明示的な住み分け**: seo-audit の description にも「For AI search optimization, see ai-seo」と記載があり、両者は補完関係。技術SEO項目（Schema.org、XMLサイトマップ、見出し階層等）は一部重複するが、llmo-audit は**AI引用獲得の観点**で評価する（例: Perplexity は引用元を重視するため「ファクトベース引用元明記」の有無を最重要判定）。

## 適用条件

### このスキルを使う場面
- 既存サイト・LP・記事が AI 検索（ChatGPT 検索 / Copilot / Gemini AI Overview / Perplexity）で引用されているか点検したい
- 新規LP・記事公開前に LLMO 観点で事前チェックしたい
- クライアント案件で「AI検索での露出を高めたい」依頼を受けた
- Google向けSEOは一通り済んでいるが、AI検索での引用が取れていない

### このスキルを使わない場面
- Google検索の順位低下・流入減少（→ `ga-gsc-diagnosis` / `seo-audit`）
- 記事本文のライティングそのもの（→ `/blog-post`）
- 新規LP設計（→ `/lp-create`）
- UIデザイン・アクセシビリティ監査（→ `web-design-guidelines`）

## 実行ステップ

### ステップ1：対象ページの内容を取得する
- 入力：対象URL、監査範囲
- 処理：
  1. WebFetch で対象URLのHTMLを取得する
  2. `<head>` 内のメタ情報（title / description / canonical / JSON-LD）を抽出する
  3. **プラグイン・CMS痕跡の明示抽出（強制ガードの入力として必須）**：以下を明示的に探索し、検出有無を後続ステップに渡す：
     - `<meta name="generator" content="...">` の内容（WordPress / Wix / Squarespace / Shopify / Hugo / Jekyll / Astro 等）
     - HTMLコメント内のプラグイン署名（`<!-- All in One SEO ... -->` / `<!-- Yoast ... -->` / `<!-- RankMath ... -->` / `<!-- SEOPress ... -->`）
     - `<link>` や `<script>` の src/href に含まれる CMS/プラグイン特有のパス（`/wp-content/` / `/wp-includes/` / `aioseo` / `yoast` 等）
     - WebFetch の要約で上記が落ちている可能性がある場合は「抽出試行したが検出不能」とマークする（＝「痕跡なし」ではなく「取得不能」扱い）
  4. 本文の見出し構造（H1〜H6）、FAQ／Q&A セクションの有無、画像 alt の有無、更新日時表示の有無（**本文中の可視表記と JSON-LD の dateModified の両方**）を抽出する
  5. クライアント指定があれば `~/.claude/clients/<name>/README.md` を Read して業種・ターゲット情報を取得する
     - ファイル未存在（クライアント指定なし or README 未整備）→ `[SKIP]`（クライアント情報なしで一般前提監査を継続）
     - ファイル存在するが Read 権限エラー／ファイル破損等→ `[FAIL]` を記録のうえ一般前提で監査継続
  6. WebFetch 自体が取得失敗した場合は `[FAIL]` マーカー＋エラー要約＋ステップ2以降を `[SKIP]`（依存失敗）で記録して報告に進む
- 出力：ページ要素データ（JSON形式の構造化抽出結果）＋取得ステータス＋**プラグイン/CMS痕跡の抽出結果（検出／未検出／取得不能のいずれか）**

### ステップ2：14項目チェックリストで採点する（＋参考1項目）

- 入力：ページ要素データ、重視するAI（任意）、ユーザー提供情報（PageSpeed / 外部言及実績 / 権威リスト掲載実績）
- 処理：以下14項目（優先度「高」4項目・「中」10項目）を `OK` / `NG` / `要改善` / `判定不能` で採点する。参考項目（項目P1）は入力値があれば採点、なければ「ユーザー調査推奨」として案内のみ。

| # | 優先度 | 項目 | 判定基準 |
|---|---|---|---|
| 1 | **高** | 見出しタグ（H1-H6）階層の明確化 | H1がページ主題1つだけ・H2以降が論理階層を構成している→`OK`。H1が複数 or 階層が飛んでいる→`NG`。 |
| 2 | **高** | 主題タイプのSchema.org構造化データ実装 | ページ主題に対応する JSON-LD（Article / Product / HowTo / Recipe / LocalBusiness 等）が存在→`OK`。存在しない→`NG`。JS注入でWebFetch検出不可の場合→`判定不能`（Google Rich Results Test 利用を推奨として明記）。**※Person/FAQPage は項目5・9で判定するため本項目では評価対象外** |
| 3 | **高** | XMLサイトマップ送信 | `/sitemap.xml` が存在し200返す→`OK`。存在しない→`NG`。Bing Webmaster Tools への送信有無はユーザー確認。 |
| 4 | **高** | ブランド言及の増加戦略 | 入力「外部ブランド言及実績」が `あり`→`OK`、`なし`→`要改善`、未入力→`判定不能`（ユーザー調査推奨）。 |
| 5 | 中 | 著者・監修者情報の構造化データ連携（Person スキーマ専属） | Person スキーマで著者／監修者が明示→`OK`。名前だけ記載で構造化なし→`要改善`。著者情報自体なし→`NG`。 |
| 6 | 中 | ファクトベース引用元明記 | 統計・データ使用箇所に引用元URL＋調査時期が明記→`OK`。一部のみ→`要改善`。引用元記載なし→`NG`。**Perplexity 対策で最重要**。 |
| 7 | 中 | AIが解釈しやすい明確・簡潔なライティング | 一文25-35ワード（日本語50-80文字）目安・PREP構成→`OK`。長文・結論後置が目立つ→`要改善`。 |

| 10 | 中 | 画像コンテンツ最適化（alt / ファイル数 / ファイル名 / キャプション） | 主要画像すべてに意味のある alt **かつ** `<img>` タグ **2個以上**（目安：滞在181秒+グループ平均1.98）→`OK`。img 1個以下 or alt空／機械的→`要改善`。img 0個→`NG`。 |
| 11 | 中 | コンテンツの定期更新と鮮度維持 | 更新日時がページに明示され、直近12ヶ月以内に更新→`OK`。更新日記載なし or 12ヶ月超→`要改善`。**Copilot 対策で最重要**。 |

| 14 | 中 | E-E-A-T 徹底（著者経歴・実績・一次情報） | 著者プロフィール（経歴・資格・実績）が詳細＋一次情報（独自調査・事例・体験談）含む→`OK`。著者情報薄い→`要改善`。 |

|---|---|---|
| P1 | ページ表示速度（Core Web Vitals） | 入力「PageSpeed Insights スコア」が提供された場合のみ採点（LCP<2.5s / INP<200ms / CLS<0.1 で`OK`）。未提供なら「seo-audit または PageSpeed Insights での別途測定を推奨」と案内のみ。本項目はseo-auditと重複するため本スキルでは参考扱い。 |

#### 【強制ガード】JSON-LD関連項目の判定前チェックリスト（必須通過・スキップ禁止）

対象項目：
- **項目2（Schema.org）・項目5（Person スキーマ）・項目9（FAQPage スキーマ）**
- ~~項目11 は対象外~~：項目11は「本文中の可視表記による更新日」が優先判定材料。本文に更新日記載があれば JSON-LD の有無に関わらず `OK` 判定可。JSON-LD 側は副次経路（本文に更新日がない場合のみ参照）であり、本ガードの対象外とする。

##### 基本原則
前提：**ミオは WebFetch のみ使用可能**。JS実行後のDOMを直接観察する手段を持たない。したがって「JS注入された JSON-LD が存在しうる」可能性は**原理的に常に残る**。この原理的制約を前提として、以下のフロー判定を行う：

##### 判定フロー

**ステップA**：ステップ1で抽出した「プラグイン/CMS痕跡」を参照する。

```
痕跡=「検出」 → 判定不能（JS注入で見えない可能性大） → 改善提案欄に Rich Results Test 推奨を明記
痕跡=「取得不能」（WebFetch要約で落ちた可能性） → 判定不能 → 同上
痕跡=「未検出」（明示的に取得できたが何もない） → ステップBへ
```

**ステップB**：「NG判定を許す安全条件」のいずれかに該当するか確認する。

以下のいずれか1つでも該当すれば `NG` と断定してよい：

1. ユーザーが入力時点で「静的HTML・CMS不使用」と明示的に申告している
2. ユーザーが HTML ソースコードを直接提供した（ローカルファイル／テキスト貼付）
3. `<meta name="generator">` が明示的に以下のいずれか、**かつ** `<head>` 内 raw 抽出が成功している：
   - `Hugo` / `Jekyll` / `Astro`（SSG モード）/ `Eleventy` / `Gatsby`（純静的ビルド）
   - 「特定の静的サイトジェネレータ」と明示された他の SSG
4. `<head>` 内の全メタタグ・スクリプト・スタイルが raw 抽出できており、かつプラグイン署名・CMS痕跡・外部JS依存が一切ない

上記いずれにも該当しない → **`判定不能`** とする。Rich Results Test 推奨を改善提案欄に明記する。

##### なぜこの設計か
WebFetch で見えない = 不存在確定 ではない。5状態契約の「試みたか否か」原則に照らせば、「試みたが検証できなかった」は `[SKIP]`（判定不能）であり `NG` ではない。ただし過剰ガードで「NG が原理的に出せない」構造にすると真の NG 検出機能が死ぬため、**NG 判定を許す安全条件を明示列挙**することで両立を図る（2026-04-22 officeueda.com 試運転で NG 誤判定発生 → リナ検証で過剰ガード問題指摘 → 本設計に改訂）。

##### ガード通過の可観測性（必須）
**JSON-LD 関連項目を NG と判定する場合、項目別採点結果テーブルの「根拠」欄に必ず以下の形式でガード通過記録を明示する**：

```
ガード通過: 痕跡=未検出 / 安全条件=該当（条件3: generator=Hugo, head全抽出成功）
```

未記載のまま NG 判定された場合、アスカは差し戻してミオに再評価を依頼する。

### ステップ3：AI別の重点項目で重み付けする
- 入力：14項目の採点結果、重視するAI
- 処理：**優先度「高」4項目（項目1〜4）は常にハイライト対象とする**。これに加えて重視AI指定がある場合、以下のAI固有項目を追加ハイライトする：
  - `Gemini` 指定 → 項目 12（マルチモーダル）・14（E-E-A-T）を追加で「最重要」ハイライト
  - `Perplexity` 指定 → 項目 6（引用元明記）・13（権威リスト掲載）を追加で「最重要」ハイライト
  - `Copilot` 指定 → 項目 11（更新鮮度）を追加で「最重要」ハイライト（項目2は既にハイライト済み）
  - `ChatGPT` 指定 → 項目 9（FAQ）を追加で「最重要」ハイライト（項目2は既にハイライト済み）
  - `all`（デフォルト）→ 優先度「高」4項目のみがハイライト
- **重視AIで指定された最重要項目が `[SKIP]`（判定不能）の場合、レポート冒頭に「重視AI向け最重要項目の判定が不完全」と警告を明記**し、必要なユーザー追加入力（外部言及実績・権威リスト掲載実績等）を案内する
- 出力：AI別重点ハイライト付きの採点結果＋最重要項目のSKIP警告（該当時）

### ステップ4：優先順位付き改善案を生成する
- 入力：採点結果、AI別重点ハイライト
- 処理：
  1. `NG` 判定の項目を「即対応」カテゴリに分類
  2. `要改善` 判定の項目を「中期対応」カテゴリに分類
  3. `判定不能` の項目を「追加調査」カテゴリに分類（ユーザー側で Rich Results Test / PageSpeed Insights 等で確認依頼）
  4. 改善案の具体策提示時、`~/.claude/knowledge/ai-seo-reference/content-patterns.md`（AEO/GEOコンテンツテンプレート集）および `platform-ranking-factors.md`（プラットフォーム別ランキング要因）を参照して、プラットフォーム別の最重要項目やテンプレートを改善提案に反映する（該当する場合のみ・特に Gemini／Perplexity 指定時は優先参照）
  5. 各改善案に「推奨委任先」を付与。表記は「エージェント名（役割）＋ 該当スキル名」の形式で統一。専属スキルが存在するエージェントはスキル名を併記し、存在しないエージェントは「※専属スキルなし（エージェント直接依頼）」と明記する：
     - コード・マークアップ実装系 → `シュウ（backend-engineer）※専属スキルなし（エージェント直接依頼）`
     - 記事執筆・リライト系 → `ハル（writer）＋ /blog-post スキル`
     - LP新規作成系 → `ハル（writer）＋ /lp-create スキル`
     - マーケ戦略（ブランド言及・権威リスト掲載）系 → `レン（marketing-planner）※専属スキルなし（エージェント直接依頼）`
     - コピー文面系 → `コト（copywriter）＋ /humanizer スキル（AI臭い文面の自然化が必要な場合）`
- 出力：優先順位付き改善案リスト

### ステップ5：監査レポートを出力する
- 入力：全ステップの成果物
- 処理：出力フォーマットに従い監査レポートを組み立てる。以下のマーカー表示位置を遵守する：
  - ステップ1でWebFetch失敗していた場合 → レポート冒頭のサマリー直後に `[FAIL]` マーカー＋原因＋代替手段を明示
  - クライアントREADME が `[SKIP]` / `[FAIL]` の場合 → 監査レポート冒頭の「補足情報」サブセクションに明記（サマリーには影響させず、参考情報として扱う）
  - ステップ2で一部項目のみ判定不能 → サマリー末尾に `[PARTIAL:N成功/M失敗]` マーカー＋内訳を明示
  - 重視AI最重要項目がSKIP → サマリー内の警告行（⚠️）で明示
- 出力：監査レポート（Markdown）

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| WebFetch でHTML取得失敗 | レポート冒頭に `[FAIL]` マーカー＋エラー要約＋「対象URLへのアクセス可否をユーザーに確認」と記載。ステップ2以降は `[SKIP]`（依存失敗）で記録 | 失敗を黙って飛ばすとサイレント失敗になり誤判定の原因になる |
| JSON-LD 検出が不確実（JS注入の可能性） | `判定不能` とし、Google Rich Results Test 利用を推奨として明記 | seo-audit で確認済みの制約。WebFetch は JS実行後のHTMLを取得できない |
| クライアント情報なし or README 未存在 | 一般サイト前提で監査を実行。ただしレポート末尾に「業種・ターゲット情報があれば精度向上」と注記 | 業種依存の判断（YMYL等）は情報がある方が正確 |
| クライアント README Read 権限エラー／破損 | `[FAIL]` を記録のうえ一般前提で監査継続 | 試みた結果失敗なので `[SKIP]` ではなく `[FAIL]`。5状態契約の切り分け原則に準拠 |
| ステップ2で一部項目のみ判定不能 | レポートは `[PARTIAL:N成功/M失敗]` 相当として成功分と失敗分を両方明示。ステップ3・4は判定可能項目に対してのみ実行 | 部分成功を一律の成功扱いにすると誤判定を生む |
| 重視AI指定の最重要項目がSKIP | ステップ3でレポート冒頭に「重視AI向け最重要項目の判定が不完全」と警告を明記し、ユーザーに追加入力を案内 | Perplexity指定で項目6・13がSKIPだと結論が欠落するため明示的な警告が必要 |
| 全項目 `OK` | レポートで「現状維持＋月次モニタリング推奨」とする | LLMOはAI側のアルゴリズム変更が頻繁で、継続監視が必要 |
| 4項目以上が `NG` | 「基盤未整備」と判定し、優先度「高」4項目の即対応を最優先提案 | 全体底上げの方が個別最適化より効果的。採点対象14項目のうち3割近くがNGの水準 |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| JSON-LD 未検出で即「NG」と判定 | JS注入の可能性を考慮し、WebFetch で見えない場合は「判定不能」＋Rich Results Test 推奨 | WebFetch は JS実行後のHTMLを取得できない。seo-audit でも確認済みの制約 |
| WebFetch 失敗時にそのまま監査継続して「問題なし」扱い | 取得失敗は `[FAIL]` マーカーで明示し、ステップ2以降は `[SKIP]` に | CLAUDE.md「5状態出力契約」違反。サイレント失敗は誤判定を生む |
| 14項目すべて等しく評価してフラット（優先度高のハイライトなし）なレポート | 優先度「高」4項目＋重視AIの重点項目をハイライト | 14項目を横並びで提示すると「何から着手すべきか」が見えず実行に繋がらない |
| 「現状維持」「最適化推奨」など曖昧な改善案 | 具体的なタグ追加例・想定工数・委任先まで明記 | ユーザー／委任先が実装に移せる粒度が必要 |
| seo-audit と同じ項目を重複監査 | 住み分け表を参照し、技術SEO基本項目（robots.txt / Core Web Vitals の**詳細診断**）は seo-audit に委ねる。本スキルでは参考項目P1として入力提供時のみ参考採点し、総合評価・ハイライトには組み込まない | 重複監査は時間浪費、AI観点の評価に集中すべき |

## 5状態出力契約（本スキルの対象判定）

本スキルは WebFetch（外部ネットワーク呼び出し）を含むため、CLAUDE.md「スキル出力の5状態契約」の対象。

- **成功・データあり**: 通常の監査レポート出力
- **成功・該当なし**: `[OK:0件]` — 監査対象URLに該当項目（例: FAQ）が存在しない場合、該当項目のみ `[OK:0件]` を付与
- **部分成功**: `[PARTIAL:N成功/M失敗]` — 14項目中一部が WebFetch 取得失敗などで判定不能の場合、レポート冒頭に明記
- **失敗**: `[FAIL]` — WebFetch が対象URLへアクセスできない／全面的な取得失敗。エラー要約＋原因推定＋代替手段（ユーザーにHTML直接提供依頼等）を明記
- **未実行/スキップ**: `[SKIP]` — 依存する前段が `[FAIL]` の場合、後段ステップは `[SKIP]`（依存失敗）として扱う

**禁止事項**: 取得失敗項目を「OK」で埋めて出力すること、セクションごと消して無言で次に進むこと。
