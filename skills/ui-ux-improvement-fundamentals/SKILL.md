# ui-ux-improvement-fundamentals

**複数ページ構成のサイト全体のUI/UX改善**を、5原則＋CRAP4原則＋3ステップ改善フローで体系的に進めるスキル。**LP単体・LP制作・LP改善は本スキルの対象外**であり、`lpo-improvement-design`（LP特化）または `lp-create` に明確に分担する。コーポレートサイト／サービスサイト／ECサイト／会員サイト等、複数ページ構造を持つサイトの「ナビゲーション設計」「サイト全体のデザイン一貫性」「複数ページ間のUX体験」が主領域。

## 適用条件

- 使う場面（**複数ページ構造を持つサイトのみ**）：
  - コーポレートサイトの直帰率や離脱率が高く改善したい
  - サービスサイトのナビゲーションを再設計したい
  - ECサイトのカテゴリ階層・商品一覧→詳細フローのUX改善
  - 会員サイト・管理画面のUI一貫性を高めたい
  - 複数ページ間の体験が分断されている問題を解決したい
- 使わない場面：
  - **LP単体の改善（ファーストビュー・CTA・フォーム特化） → `lpo-improvement-design`**
  - **LPの設計レビュー → `lpo-improvement-design` または `lp-design-system`**
  - HP/LPどちらを作るかの判断 → `hp-lp-distinction-design`
  - LP制作フロー全般 → `lp-create`
  - LPセクションごとのデザイン仕様 → `lp-design-system`
  - UI監査（コードレビュー） → `web-design-guidelines`（Vercel Labs版・監査専用）
  - 流入減・CV減の1次診断 → `ga-gsc-diagnosis`

**判定の基準**：「対象が単一ページ完結（LP）か、複数ページ構造を持つサイトか」。単一ページなら `lpo-improvement-design`、複数ページなら本スキル。

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 対象サイトタイプ | 必須 | コーポレート／LP／サービス／キャンペーン／EC | 「BtoBコーポレートサイト」 |
| 改善目的 | 必須 | CV率向上／離脱率低下／ブランド価値向上／SEO効果 | 「直帰率を50%以下に」 |
| 想定デバイス | 推奨 | モバイル中心／PC中心／両方均等 | 「toC事業のためモバイル中心」 |
| 既存データ | 推奨 | GA/Clarity/ヒートマップの保有状況 | 「GA4設置済・Clarity未導入」 |

## 実行ステップ

### ステップ1：UI/UXの定義を共有しゴールを設定する

- 入力：対象サイトタイプ、改善目的
- 処理：
  - **UIとUXの定義**：
    - **UI（User Interface）**：ユーザーが操作するボタン、メニュー、フォームなどの設計のこと
    - **UX（User Experience）**：サービスを利用する際に得られる全体的な体験のこと
  - **UI/UXの目標**：
    - **UIの目標**：見やすく、使いやすく、わかりやすいインターフェースを提供
    - **UXの目標**：利便性と快適さを追求し、利用者満足度を向上させる
  - **サイトタイプ別ゴール例**：
    - コーポレートサイト：自社事業の周知・信頼感／問い合わせ増加／営業マンの商談ツール活用
    - LP：特定商品や無形商材への購買／問い合わせ促進
    - サービスサイト：自社サービスの魅力を伝える／申込増・リピート促進
    - キャンペーンサイト：自社キャンペーンや認知／SNSシェア・口コミ・バズる
    - EC・Webサービス：自社商品購買／リピート促進／サイトへのファン化
- 出力：定義の共有＋サイトタイプ別ゴール（数値KPI付きが望ましい）

### ステップ2：UI/UX改善の4メリットで投資判断を整理する

- 入力：改善目的
- 処理：以下4メリットのうち、優先度を整理
  1. **コンバージョン率の向上**：シンプルなデザインとわかりやすい導線で購入や問い合わせを促進
  2. **離脱率の低下**：スムーズな操作性でユーザーのストレスを軽減し、滞在時間を増やす
  3. **ブランド価値の向上**：心地よい体験を提供することで、企業や製品への信頼を高める
  4. **SEO効果の向上**：高品質なUI・UXは検索エンジンからの評価も向上させる
- ポイント：4メリットを最大限引き出すには「経営視点とマーケティング専門性を融合させた戦略的アプローチ」が不可欠
- 出力：投資判断の根拠（どのメリットを最優先するか）

### ステップ3：ユーザビリティ5原則で現状を点検する

- 入力：対象サイトの現状
- 処理：Jakob Nielsenの10原則から厳選した5原則で点検
- **抽出ロジック**：Jakob Nielsenの全10原則のうち、UI構造・操作フィードバック・直感性に直結する5原則を本スキルでカバーする。残り5原則（柔軟性と効率性／最小限主義デザイン／エラー認識・診断・回復／ヘルプドキュメンテーション／効率性のための加速性）は `web-design-guidelines`（監査スキル）でカバーするため、ここでは扱わない
  1. **視認性の高いシステム状態の維持**：ユーザーに現在の状況を明確に伝える（ローディング表示・進捗バー・送信完了通知など）
  2. **システムと現実世界のマッチング**：ユーザーの言葉や考え方に沿ったデザイン（専門用語の回避・自然なアイコン）
  3. **ユーザーのコントロールと自由度**：間違った操作からの回復手段を用意（戻る・キャンセル・元に戻す）
  4. **一貫性と標準化**：既存のデザインパターンや慣習を尊重（業界標準に従う・サイト内で統一）
  5. **エラーの防止**：間違いが起こりにくい設計にする（必須項目の明示・確認ダイアログ）
- 出力：5原則別の点検結果（OK/NG/要改善）

### ステップ4：モバイル/PCの違いを意識した設計方針を決める

- 入力：想定デバイス、対象サイトタイプ
- 処理：3観点でモバイル/PCの違いを整理
  - **画面サイズの違い**：モバイルでは縦スクロール、PCでは横の情報量が多い
  - **操作方法の違い**：モバイルはタップ・スワイプ、PCはクリック・ホバー
  - **コンテンツ設計の違い**：モバイルでは情報をシンプルに、PCでは詳細情報を提供
- 注意：toC事業ではモバイルアクセスがメイン。Googleもレスポンシブ対応を推奨しており、モバイルファーストでないサイトは評価されづらい
- 出力：モバイル/PC優先度＋レスポンシブ対応方針

### ステップ5：3ステップ改善フローを実行する

#### 5-1：現状分析

- 入力：既存データ、対象URL
- 処理：
  - **データの収集**：Google Analyticsで直帰率や訪問者の動向を分析
  - **ヒートマップの活用**：クリック箇所やスクロールの深度を可視化（Microsoft Clarityなどで無料計測可）
- 出力：現状の数値＋ヒートマップ可視化結果

#### 5-2：デザインの最適化

- 入力：5-1の分析結果
- 処理：
  - **レスポンシブデザイン**：どのデバイスでも見やすく操作しやすい設計
  - **CTAの改善**：目立つ色や位置で、ユーザーを行動に誘導するボタン設計
- 出力：デザイン改善案

#### 5-3：ユーザー視点の導入

- 入力：5-2のデザイン改善案
- 処理：
  - **ペルソナ作成**：ターゲットユーザーの具体的な像を定義し、それに基づいて設計
  - **ユーザビリティテスト**：実際の利用者に操作してもらい、問題点を把握する
- 出力：ペルソナ＋ユーザビリティテストレポート

### ステップ6：CRAP原則（デザイン4大原則）で詳細を点検する

- 入力：5-2の改善案
- 処理：ロビン・ウィリアムズ提唱のCRAP原則で点検
  - **C：Contrast（対比）**：強調したい要素を他と差別化し、視認性を高める。例：ボタンの色を目立たせる、見出しのフォントを太くする
  - **R：Repetition（反復）**：デザイン要素を統一し、一貫性を持たせる。例：同じフォントやカラーを繰り返し使う、見出しのデザインを統一する
  - **A：Alignment（整列）**：すべての要素を意識的に配置し、統一感を持たせる。例：左揃え・中央揃えを統一、見出しと本文の位置を整える
  - **P：Proximity（近接）**：関連する要素はグループ化し、近くに配置する。例：フォームのラベルと入力欄を近くに、ナビゲーションメニューを一か所にまとめる
- 出力：CRAP原則別チェックリスト

### ステップ7：F/Zパターンとナビゲーションを設計する

- 入力：6の点検結果
- 処理：
  - **視線パターン**：
    - **Fパターン**：左から右に視線を移しながら下に進む動き。主にテキストコンテンツに適している
    - **Zパターン**：左上→右上→左下→右下へと視線が移動。視覚的に重要な要素を配置するのに効果的
  - **グルーピングと余白の活用**：余白で情報の区切りを明確にし、読みやすさを向上
  - **ナビゲーション4種**：
    - **トップメニュー**：上から下へ、左から右へ、一般的なウェブサイトの読み方に沿う
    - **サイドバー**：サブカテゴリーの右または左に配置。フィルタリングオプションを含む
    - **フッター**：ウェブサイトの下部、主要ページへのリンクを含む
    - **パンくず**：ページ上部、ECサイトや大規模コンテンツハブで活用
- 動線設計のポイント：CTAボタンの配置など、ユーザーの行動を予測した動線設計が重要。視線が自然に向かう場所にボタンを配置
- 出力：F/Zパターン採用判断＋ナビゲーション設計

### ステップ8：フォント・コントラストを最適化する

- 入力：7の設計
- 処理：
  - **コントラストを意識する**：背景と文字の色に十分な差をつけることで視認性向上、アクセシビリティ改善
  - **フォントの選び方**：視認性の高いフォント＋サイズ・スタイルの工夫。重要な情報には大きめのフォント
- 出力：タイポグラフィ仕様

### ステップ9：改善ツールを選定する

- 入力：既存ツール状況
- 処理：以下3カテゴリから必要なツールを選定
  - **分析ツール**：Google Analytics（訪問者行動）／Microsoft Clarity（ヒートマップ・セッション録画）
  - **デザインツール**：Adobe XD・Figma（UIプロトタイプ・デザイン共有）／Canva（シンプルなデザイン作成）
  - **パフォーマンス測定ツール**：PageSpeed Insights（ページ読み込み速度測定・改善点提示）
- 出力：導入推奨ツールリスト

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| toC事業でモバイル比率が60%以上 | モバイルファースト必須 | Googleもモバイル優先評価 |
| 直帰率80%超え | ステップ5-1（現状分析）から開始 | データなしの推測改善は無駄 |
| ブランドサイトで信頼感を出したい | CRAPの「Repetition（反復）」優先 | 一貫性が信頼を生む |
| ECで購入完了率を上げたい | CRAPの「Contrast（対比）」+ F/ZパターンでCTA配置 | 視線誘導と差別化が決定打 |
| 情報量の多い記事サイト | Fパターン採用 | テキストコンテンツに最適 |
| ビジュアル中心のキャンペーンサイト | Zパターン採用 | 重要要素を視覚的に配置 |
| 大規模サイト（ECや情報ハブ） | パンくずリスト必須 | 階層理解とSEO双方に効く |
| ユーザビリティテスト未実施 | ステップ5-3で必ず実施 | データだけでは見えない問題が多い |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 「使いにくい」だけで改善着手 | ステップ5-1（GA・ヒートマップ）で数値根拠取得 | 推測改善は当て推量で失敗する |
| デザイン4原則を全部同時に適用 | サイトタイプ別に優先1〜2原則を選ぶ | 同時適用は方針がブレる |
| モバイル対応を後回し | toC事業はステップ4でモバイルファースト宣言 | Google評価・ユーザー多数派 |
| ペルソナなしでデザイン作成 | ステップ5-3でペルソナ作成してから設計 | 誰のためのデザインか不明確 |
| ユーザビリティテストを「内部のみ」で実施 | 実際の利用者にテストしてもらう | 内部の人は仕様を知っているので問題が見えない |
| ナビゲーション乱立（4種全部入り） | サイトタイプに応じて2〜3種に絞る | 多すぎると迷う |
| アクセシビリティ無視（コントラスト弱） | ステップ8でコントラスト比を必ず点検 | 色覚多様性ユーザーが離脱 |

## 連携スキル

- 前段：`hp-lp-distinction-design`（HP/LP使い分け設計）／`stp-analysis`（ターゲット明確化）／`persona-design`（ペルソナ設計）
- 後段：`lpo-improvement-design`（LP特化改善）／`web-design-guidelines`（監査）／`seo-content-strategy`（コンテンツSEO）
- 関連：`ga-gsc-diagnosis`（流入減・CV減1次診断）／`lp-design-system`（LPセクションデザイン）
- 改善ツール：Microsoft Clarity（ヒートマップ）／PageSpeed Insights（速度）／Figma・Adobe XD（プロトタイプ）
