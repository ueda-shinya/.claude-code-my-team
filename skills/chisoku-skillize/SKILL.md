# chisoku-skillize スキル

`~/.claude/reports/chisoku/` 配下の事業戦略PDFを未処理分のみ検出し、`knowledge-to-skill` を起動して順次スキル化する。履歴ファイルで処理済みを管理し、同じPDFを二重処理しない。

## 適用条件

- このスキルを使う場面：
  - `~/.claude/reports/chisoku/` に新規PDFを追加した後にまとめてスキル化したい
  - 未処理PDFがあるか確認したい（差分チェックのみのモード）
  - 過去に処理したPDFの履歴を確認したい
- このスキルを使わない場面：
  - PDF以外のドキュメント（テキスト、URL等）→ 直接 `knowledge-to-skill` を使う
  - `~/.claude/reports/chisoku/` 以外のPDF → 直接 `knowledge-to-skill` を使う
  - 単発で1つのPDFを指定してスキル化したい → 直接 `knowledge-to-skill` を使う

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 起動フレーズ | 必須 | 上記トリガーフレーズのいずれか | 「/chisoku-skillize」「chisoku 未処理ある？」 |
| モード | 任意 | 差分確認のみか、スキル化まで実行するか | 「未処理ある？」→ 差分確認のみ |
| 処理対象選択 | 任意 | 未処理PDFが複数ある場合の選択 | 「全部」「1と3」「1だけ」 |

## 実行ステップ

### ステップ1：未処理PDFを検出する

- 入力：なし（環境固定パス）
- 処理：
  1. `Glob` で `~/.claude/reports/chisoku/*.pdf` を取得し、ファイル名一覧（basename）を作る
  2. `Read` で `~/.claude/reports/chisoku/_skill-history.md` を読み込む
     - ファイルが存在しない場合 → 「処理済みなし」として全PDFを未処理扱いにする（ステップ1.4 へ）
     - ファイルが存在する場合 → 次の 1.3 に進む
  3. **履歴ファイルから処理済みキーセットを構築する（厳密一致判定）**：
     - `Grep` で履歴ファイル内の「`- key: ` で始まる行」を全行抽出する（パターンは `^- key: ` の固定前置詞）
     - 抽出した各行から `- key: ` 以降の文字列を取り出し、**そのままのバイト列**（PDFファイル名・拡張子込み・パスなし・スペースや特殊文字も原文のまま）として保持する
     - 同時に、各 `key:` 行の直近に並ぶ `- status:` 行の値（`completed` / `failed-retryable` / `skipped-permanent`）をペアで取得する
     - 「処理済みキーセット」 = `status: completed` または `status: skipped-permanent` の `key` 集合
     - 「再処理対象キーセット」 = `status: failed-retryable` の `key` 集合
  4. **未処理判定（固定文字列での厳密一致）**：
     - ステップ1.1 で取得した各PDFファイル名（basename）と、処理済みキーセットを**完全一致比較**する
     - 比較は固定文字列一致のみ。`Grep` を使う場合は必ず `-F`（固定文字列モード）を指定し、正規表現メタ文字（`.` `(` `[` `*` `?` `+` `^` `$` `|` `\` 等）や全角スペース・特殊文字を含むPDFファイル名でも誤マッチしないようにする
     - 推奨：`Grep -F` で1件ずつ照合するより、ステップ1.3 で抽出済みのキーセットと文字列イコールで比較する方が確実
     - 判定ルール：
       - キーセットに完全一致するファイル名が含まれる（`completed` または `skipped-permanent`） → 処理済み（未処理リストに入れない）
       - キーセットに含まれず、再処理対象キーセットに含まれる（`failed-retryable`） → 未処理（再処理対象として未処理リストに入れる）
       - どちらにも含まれない（履歴に key 自体がない） → 未処理（新規として未処理リストに入れる）
  5. PDFが0件の場合 → `[OK:0件]` で「reports/chisoku/ にPDFがありません」と報告して終了する
- 出力：未処理PDFのフルパス一覧（新規＋再処理対象を含む）

### ステップ2：モードを判定して分岐する

- 入力：未処理PDFリスト、ユーザーの起動フレーズ
- 処理：
  - **未処理が0件の場合** → 「未処理PDFはありません」と `[OK:0件]` 付きで報告して終了する（履歴更新なし）
  - **起動フレーズに「未処理ある？」「差分」が含まれる場合（差分確認モード）** → 未処理リストを表示するだけで終了する（履歴更新なし、ステップ3〜4をスキップ）
  - **未処理が1件以上ある通常モード** → 一覧をユーザーに提示し、以下3択で確認する：
    ```
    未処理PDFが N 件あります：
    | # | ファイル名 | 追加日（mtime） |
    |---|---|---|
    | 1 | xxx.pdf | YYYY-MM-DD |
    | 2 | yyy.pdf | YYYY-MM-DD |

    どうしますか？
    1. 全部スキル化する
    2. 番号を選んでスキル化（例：「1と3」）
    3. 差分確認のみで終了（履歴更新なし）
    ```
- 出力：処理対象PDFの確定リスト（モードによっては空リスト＝終了）

### ステップ3：各PDFをスキル化する（処理対象1件ごとにループ）

- 入力：処理対象PDFのフルパス1つ
- 処理：
  1. **総ページ数を確認する**：`Read` で `pages: "1-1"` を指定して1ページ目だけ読み込み、出力からPDFの総ページ数を把握する（読み取れない場合は10ページ以下と仮定して通常 Read を試す）
  2. **PDF全文を取得する**：
     - 10ページ以下 → `Read` で全文取得
     - 11ページ以上 → `pages: "1-10"`、`pages: "11-20"` のように10ページ刻みで分割読み込みし、結合する
     - **読み込み失敗時** → そのPDFの処理を中断し、履歴に `status: failed-retryable` で記録（次回実行時に再処理対象として検出される）。`[FAIL]` でエラー要約を残してステップ4に渡し、次のPDFへ進む
  3. **`knowledge-to-skill` スキルを起動する**：
     - ソース情報として「PDFファイル名 + 全文テキスト」を渡す
     - `knowledge-to-skill` の「ステップ2.5：スキル候補の洗い出し（複数スキル抽出）」フローに従って候補一覧を提示し、ユーザー選択を受けて生成する
     - **`knowledge-to-skill` が「手順化不能」と判定した場合** → そのPDFは履歴に `status: skipped-permanent` で記録し（再処理しない）、次のPDFへ進む。備考欄に「agent-study 推奨」等の代替候補を記録する
     - **`knowledge-to-skill` 実行中のエラー（途中失敗等）** → 履歴に `status: failed-retryable` で記録し、次回実行時に再処理対象とする。`[FAIL]` でエラー要約を残す
  4. **生成結果を記録する**（成功時）：
     - 履歴に `status: completed` で記録
     - 生成したスキル名一覧
     - 各スキルの対象エージェント
     - 各スキルのファイルパス
     - 1行説明（`knowledge-to-skill` の出力から取得）
- 出力：1PDF分のメタ情報（PDFファイル名 / 処理日 / status / 生成スキル名一覧 / 対象エージェント）

**ループ制御**：処理対象が複数ある場合、1件ずつ順番に処理する。1件失敗しても次のPDFは続行する（部分成功は最終報告で `[PARTIAL:N成功/M失敗]` マーカーを付ける）。

### ステップ4：履歴ファイルを更新する

- 入力：ステップ3の生成結果（複数PDF分）
- 処理：
  1. `~/.claude/reports/chisoku/_skill-history.md` の存在を確認する
  2. **存在しない場合** → 以下のヘッダーで新規作成する（`Write` ツールを使用）：
     ```markdown
     # chisoku PDF スキル化履歴

     `reports/chisoku/` 配下のPDFをスキル化した履歴。`/chisoku-skillize` 実行時に自動更新される。

     ## エントリフォーマット

     各エントリには以下の機械可読フィールドが含まれる:
     - `key`: PDFファイル名（拡張子込み・パスなし）。重複検出のための一意キー
     - `status`: 処理状態
       - `completed`: スキル化成功
       - `failed-retryable`: 失敗（次回実行時に再処理対象）
       - `skipped-permanent`: 手順化不能（再処理しない・agent-study 推奨等）

     ---
     ```
  3. **存在する場合** → 以下の方針で更新する：
     - 該当PDFファイル名（key）が既存エントリにない場合 → `Edit` ツールで末尾に新規エントリを追記する
     - 該当 key が既に `failed-retryable` で記録済みで、今回成功した場合 → 既存エントリを `completed` に**上書き更新**する（追記ではなく該当エントリを置換）。**同じ key の重複エントリは作らない**
     - 既存エントリが `completed` または `skipped-permanent` の場合は通常そもそもステップ3に到達しないが、何らかの理由で再処理した場合は新規結果で上書きする
  4. 1PDFごとに以下のフォーマットで追加する（`key:` 行と `status:` 行が機械可読アンカー）：
     ```markdown

     ## YYYY-MM-DD: <PDFファイル名>

     - key: <PDFファイル名（拡張子込み・パスなし・スペースや特殊文字も原文のまま）>
     - status: completed | failed-retryable | skipped-permanent
     - **ソースPDF**: `reports/chisoku/<filename>.pdf`
     - **処理日**: YYYY-MM-DD
     - **生成スキル**:
       - `<skill-name-1>` — 概要1行（対象: <エージェント名>）
       - `<skill-name-2>` — 概要1行（対象: <エージェント名>）
     - **備考**: （必要に応じて記入。`skipped-permanent` の場合は「スキル化対象外（理由：〇〇、agent-study 推奨等）」、`failed-retryable` の場合は「読み込み失敗（再処理対象）」等）
     ```
  5. 書き込みに失敗した場合 → `[FAIL]` で「履歴ファイル書き込み失敗」をログ出力し、シンヤさんに手動追記を依頼する
- 出力：更新後の履歴ファイル

### ステップ4.5：chisoku-skill-index.md への自動追記

> **設計範囲注記（リナ指摘3反映）**: 本ステップは Phase 2-B 時点の設計。「カテゴリ未分類（手動配置調整待ち）」セクションが `~/.claude/memory/chisoku-skill-index.md` に存在する前提で固定設計。同ファイルの構造変更（カテゴリ追加・表フォーマット変更・セクション削除）が発生した場合は、本ステップの手動改修が必要（自動追従不可）。

- 入力：ステップ3で `status: completed` となった生成スキルのみ（`failed-retryable` / `skipped-permanent` は対象外）
- 処理：以下の手順を completed スキル各件についてループ実行する

#### 4.5.1 インデックスファイルの存在確認

1. `Read` で `~/.claude/memory/chisoku-skill-index.md` を試行する
2. 存在しない場合 → `[FAIL]` でエラー記録し、シンヤさんに「Phase 1 を先に実行して chisoku-skill-index.md を作成してください」と報告してステップ4.5を終了する（ステップ5には進む）
3. 存在する場合 → 4.5.2 へ

#### 4.5.2 各スキルの追記ループ（completed スキル 1件ずつ）

**【重複チェック】**

- `Grep -F` で `chisoku-skill-index.md` 全体に対し、`` `<skill-name>` `` 形式（バッククォート込み・固定文字列モード必須）を検索する
  - 固定文字列モード（`-F`）を使う理由：スキル名に `-` 等の正規表現メタ文字を含む場合でも誤マッチしない（本スキルのステップ1.4 の判定方針と統一）
- 既存あり → `[SKIP]`「既存登録済（重複追記しない）」を記録し、このスキルの追記処理をスキップして次のスキルへ
- 既存なし → 追記処理へ進む

**【追記内容生成】**

| フィールド | 取得元 | 備考 |
|---|---|---|
| スキル名 | ステップ3の生成結果（skill-name） | バッククォート囲み: `` `<skill-name>` `` |
| 1行説明 | knowledge-to-skill 出力の「概要1行」 | chisoku-skill-index.md の「1行説明は凝縮版を正本」ルール準拠 |
| 主管エージェント | knowledge-to-skill 出力の対象エージェント情報 | 取得できない場合は「**未定**」と記録し、カテゴリ未分類フラグと同様にアスカへ調整依頼フラグを立てる |
| 副次 | 手動メンテのため空欄 | `—` と記入 |

**【追記先カテゴリ判定】**

- knowledge-to-skill 出力にカテゴリ指定（カテゴリ番号または名称）がある場合 → 該当カテゴリの表末尾に追記する
- カテゴリ指定がない場合 → `## カテゴリ未分類（手動配置調整待ち）` セクションの表末尾に追記し、アスカへ「本配置調整必要」フラグを立てる

**【Edit ツールでの追記実行】**（リナ指摘1・2・4反映）

以下の手順で `Edit` ツールを用いて追記する：

1. **手順1 — 対象カテゴリ範囲の Read 取得**:
   - `Read` で `chisoku-skill-index.md` を取得し、対象カテゴリのヘッダー行（`## カテゴリN：XXX` または `## カテゴリ未分類（手動配置調整待ち）`）を `Grep -F` で固定文字列照合して特定する（複数カテゴリ表が存在するため、カテゴリヘッダーから取得することで誤マッチを防止する）
   - 「表の境界」定義：対象カテゴリヘッダー行の直後から、**次の `## ` ヘッダー行の直前まで**（または `---` セパレーターの直前まで）を当該カテゴリの表範囲とする
2. **手順2 — 最終 `|` 行の特定**:
   - 手順1で取得した範囲内の最終 `|` 行を特定する（old_string として使用）
   - **空表の初回追記時の扱い**（リナ指摘2反映）：カテゴリ未分類セクション等でヘッダー行と区切り行のみの空表（`|---|---|---|---|` が最終行）になっている場合、区切り行を「最終 `|` 行」として扱い、直後に新規行を挿入する
3. **手順3 — new_string 生成と Edit 実行**:
   - `new_string` = 最終 `|` 行 + 改行 + 新規行
   - 新規行フォーマット: `` | `<skill-name>` | <1行説明> | <主管 or **未定**> | — | ``
   - `Edit` ツールで `old_string`（最終 `|` 行）を `new_string` に置換する
   - Edit 失敗時 → `[FAIL]` でエラー要約を記録し、当該スキルは追記スキップして次のスキルへ（履歴ファイルは更新済みのため、後追いで手動追記可能）

#### 4.5.3 集計と状態マーカー記録

各スキルの処理結果を以下のマーカーで集計する：

| 結果 | マーカー | 説明 |
|---|---|---|
| 追記成功 | `[OK:N件]` | 新規行を追記した件数 |
| 既存スキップ | `[SKIP]` | 重複検知でスキップした件数（M件） |
| 追記失敗 | `[FAIL]` | Edit 失敗 / ファイル不在 / 想定外エラーの件数（K件） |
| 複数スキルで混在 | `[PARTIAL:N成功/M失敗]` | N件成功・M件失敗の場合 |

集計結果はステップ5の完了報告に含める。

- 出力：インデックス追記結果サマリー（追記成功 N件 / 既存スキップ M件 / 失敗 K件）

### ステップ5：完了報告

- 入力：全PDFの処理結果
- 処理：以下のフォーマットで一括サマリーを出力する
- 出力：シンヤさんへの完了報告

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 履歴ファイルが存在しない | 全PDFを未処理扱い | 初回実行時の正常動作 |
| `reports/chisoku/` にPDFが0件 | `[OK:0件]` で報告して終了 | 処理対象なし |
| 未処理が0件 | 「未処理PDFはありません」と報告して終了 | 全件処理済み |
| 起動フレーズに「未処理ある？」「差分」を含む | 差分確認モード（リスト表示のみ・履歴更新なし） | ユーザーの意図が確認のみのため |
| 履歴に key 記載あり・`status: completed` | 処理済みとして未処理リストから除外 | 成功済みは再処理しない |
| 履歴に key 記載あり・`status: skipped-permanent` | 処理済み（手順化不能）として除外 | 同じPDFを毎回試行するのを避ける（agent-study 推奨等） |
| 履歴に key 記載あり・`status: failed-retryable` | 未処理として再処理対象に含める | 一時的失敗は次回再試行する |
| 履歴に key 記載なし | 未処理として処理対象に含める | 新規PDF |
| PDF名に正規表現メタ文字や特殊文字を含む | `Grep -F`（固定文字列モード）で厳密一致判定 | 正規表現メタ文字による誤マッチを防ぐ |
| `knowledge-to-skill` が「手順化不能」と判定 | 履歴に `status: skipped-permanent` として記録 | 再処理を防ぐ |
| PDFが10ページ超 | `pages` パラメータで10ページ刻みに分割読み込み | Read ツールの仕様（10ページ超は pages 必須） |
| PDF読み込み失敗 | 履歴に `status: failed-retryable` で記録、`[FAIL]` でエラー記録、次のPDFへ続行 | 1件の失敗で全体を止めない／次回再処理可能にする |
| 履歴ファイル書き込み失敗 | `[FAIL]` で報告、手動追記を依頼 | データロスを防ぐ |
| 同じ key を再処理して成功 | 既存エントリを `completed` に上書き更新（重複作らない） | 履歴の一意性を保つ |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 履歴ファイルがない時にエラーで止まる | 「処理済みなし」として全PDFを未処理扱いにする | 初回実行で必ず止まる設計は不適切 |
| 10ページ超のPDFを `pages` 指定なしで Read する | `pages: "1-10"` のように分割して読む | Read ツールの仕様で 10ページ超は失敗する |
| 1件失敗で全体処理を中断する | 失敗を `[FAIL]` で記録して次のPDFへ続行する | バッチ処理の基本（部分成功を許容） |
| 「スキル化対象外」を履歴に記録せず再処理対象にする | 履歴に `status: skipped-permanent` と明記して再処理を防ぐ | 同じPDFを毎回試行するのを避ける |
| 差分確認モードでも履歴を更新する | 差分確認モードでは履歴を一切更新しない | ユーザーの意図（確認のみ）に反する |
| `knowledge-to-skill` の仕様を勝手に変更する | オーケストレーションのみ行い、`knowledge-to-skill` は既存仕様のまま呼ぶ | 責任分離の原則 |
| PDF名を `Grep` の正規表現で照合し、メタ文字を含む名前で誤マッチする | `Grep -F`（固定文字列モード）または抽出済みキーセットとの完全一致比較で照合する | 正規表現メタ文字や全角スペースで誤判定する事故を防ぐ |
| 失敗したPDFを `failed-retryable` で記録せず未処理扱いだけにする（履歴に key を残さない） | 失敗時も必ず `key:` と `status: failed-retryable` で履歴に記録し、次回実行時に決定論的に再処理対象として検出されるようにする | 「未処理扱いを維持しても良い」という曖昧運用が再現性を破壊する |
| 同じ key の重複エントリを履歴に追記する | 既存エントリを上書き更新する（key は一意） | 履歴の一意性を保ち、検索結果が複数件返る事故を防ぐ |
| インデックスに同名スキルを重複追記する | `Grep -F` で `` `<skill-name>` `` を検索し、既存あれば `[SKIP]` でスキップする | 同じスキルが複数行登録されるとインデックスの信頼性が失われる |
| インデックスのカテゴリヘッダーを `Grep` の正規表現で照合し誤マッチする | カテゴリヘッダー検索は `Grep -F` 固定文字列モードを使う | `## カテゴリ1：環境分析` 等の `：` や数字が誤マッチ源になる |
| 空の「カテゴリ未分類」表（区切り行のみ）に追記できず `[FAIL]` になる | 空表では区切り行 `\|---|---|---|---\|` を最終 `\|` 行として扱い直後に挿入する | 初回追記は必ず空表から始まる（ヘッダー＋区切り行のみ） |
| `failed-retryable` / `skipped-permanent` スキルもインデックスに追記する | ステップ4.5 は `status: completed` スキルのみ対象。失敗・スキップ分は追記しない | 未完成スキルをインデックスに登録すると存在しないスキルへの参照が発生する |

## 連携スキル

- `knowledge-to-skill`（`~/.claude/skills/knowledge-to-skill/SKILL.md`）：このスキルが各PDFごとに起動する子スキル。仕様変更はしないこと。
- `agent-study`：`knowledge-to-skill` が「手順化不能」と判定した場合の代替候補。本スキルからは直接起動しないが、対象外PDFの備考欄に「agent-study 推奨」と記録する。

## 5状態出力契約

このスキルは外部API・サブプロセス実行・外部設定依存のいずれも持たないため、CLAUDE.md「スキル出力の5状態契約」の対象判定基準には該当しない。ただし以下のケースで明示マーカーを使うこと（運用上のベストプラクティス）：

- `[OK:0件]`：PDFが0件、または未処理が0件のとき
- `[PARTIAL:N成功/M失敗]`：複数PDFのうち一部だけ成功したとき
- `[FAIL]`：PDF読み込み失敗、履歴ファイル書き込み失敗時。エラー要約を必ず添える
- `[SKIP]`：差分確認モードで処理をスキップしたPDFがある場合（通常は使わない）

セクションを丸ごと非表示にせず、必ず最終報告のテーブルまたは備考欄にマーカーを残すこと。
