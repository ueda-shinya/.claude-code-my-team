# business-model-canvas

ビジネスモデルキャンバス（9要素・リーンキャンバス系）で、事業の構造を1枚で可視化し、要素間の整合性を検証する。「9要素を埋めて終わり」ではなく、必ず**要素間の連動性チェック**まで進める。

## 適用条件

- 使う場面：
  - 新規事業の立ち上げで全体像を1枚に整理したい
  - 既存事業の構造を見直し、ボトルネックを発見したい
  - チーム内で事業の共通認識を作りたい
  - 投資家・パートナーに事業を説明する1枚資料が欲しい
- 使わない場面：
  - 顧客の価値提案だけ深掘りしたい → バリュープロポジション（`value-proposition`）
  - 顧客セグメントだけ深掘りしたい → STP分析（`stp-analysis`）
  - 競合の競争構造分析 → 5フォース分析（`five-forces-analysis`）

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 分析対象 | 必須 | 事業・サービス | 「自社の新規ECサービス」 |
| 分析の目的 | 必須 | 新規立ち上げ／既存改善／投資家向け | 「新規事業立ち上げ前の整理」 |
| 既知の情報 | 任意 | 顧客・競合・コスト等 | 「20-30代女性向け／月額3,000円想定」 |

## 実行ステップ

### ステップ1：1.顧客セグメントを定義する

- 入力：分析対象、ターゲット仮説
- 処理：以下を3〜5項目で具体化する
  - **デモグラフィック**: 年齢／性別／職業／収入帯／家族構成等
  - **行動特性**: 購買行動／利用シーン／意思決定者
  - **価値観**: 何を重視するか（コスパ／環境／ブランド／信頼性等）
- 「全員」「一般消費者」は禁止。1セグメントに絞る
- 出力：顧客セグメント（3〜5項目）

### ステップ2：2.課題を特定する

- 入力：ステップ1の顧客セグメント
- 処理：その顧客が抱えている課題を3〜5項目で書く
  - 表面的な課題（「○○が見つからない」）と根本課題（「時間がない」）の両方
  - 顧客自身が認識している課題と、顧客が気づいていない潜在課題
- 各課題に「実際の顧客の声」を引用（ヒアリング・レビュー・受注理由）
- 出力：課題リスト

### ステップ3：3.独自の価値提案（UVP）を1〜2文で言語化する

- 入力：ステップ1〜2の結果
- 処理：競合と差別化されたユニークな価値を1〜2文で表現する
  - 構文：「[顧客]に対して、[課題]を、[競合と異なる方法]で解決する[サービス]」
  - 自社視点（「すごい技術」）でなく顧客価値視点で書く
- 例：「20-30代のオンライン買い物利用者に、流行・お得・環境意識を簡単に両立させるECショッピング体験」
- 出力：独自の価値提案（1〜2文）

### ステップ4：4.ソリューションを設計する

- 入力：ステップ2の課題、ステップ3のUVP
- 処理：各課題に対する具体的な解決策を3〜5項目で書く
- 課題と解決策が1対1で対応するように整理する
- 「機能」ではなく「課題をどう解決するか」で書く
- 出力：ソリューションリスト（課題とペアで）

### ステップ5：5.チャネルを設計する

- 入力：ステップ1の顧客セグメント、ステップ3のUVP
- 処理：顧客にどう価値を届けるかを以下の観点で整理する
  - **認知チャネル**: 顧客がサービスを知る経路（SNS、SEO、広告、紹介、PR等）
  - **販売チャネル**: 顧客が購入する経路（自社EC、店舗、代理店、マーケットプレイス等）
  - **デリバリチャネル**: 顧客に届ける経路（配送、デジタル配信、店頭受け渡し等）
  - **アフターチャネル**: 顧客サポートの経路（メール、チャット、電話、CRM等）
- 顧客セグメントが普段いる場所に合わせて選ぶ
- 出力：チャネル設計

### ステップ6：6.収益の流れを設計する

- 入力：ステップ1〜5の結果
- 処理：以下の観点で収益を整理する
  - **収益モデル**: 単発販売／サブスク／フリーミアム／成果報酬／広告／手数料／会員制
  - **価格帯**: 1顧客あたりの単価（最低／標準／上位プラン）
  - **追加収益**: クロスセル／アップセル／コラボ商品等
  - **想定総収益**: 単価 × 顧客数 = 月次／年次収益
- 出力：収益モデル設計（金額付き）

### ステップ7：7.主要指標（KPI）を設定する

- 入力：ステップ1〜6の結果
- 処理：成果を可視化する数値指標を3〜5個に絞る
  - **獲得指標**: CAC、流入数、登録率
  - **エンゲージ指標**: 継続率、利用頻度、コンバージョン率
  - **収益指標**: 平均購入額、LTV、解約率
- 各指標に「目標値」と「測定頻度」を付ける
- 出力：KPI一覧（3〜5個、目標値付き）

### ステップ8：8.コスト構造を整理する

- 入力：ステップ1〜7の結果
- 処理：事業運営に必要なコストを3〜5項目で整理する
  - 固定費（開発・運営・人件費等）
  - 変動費（広告・配送・原材料等）
  - 顧客獲得コスト（CAC：マーケ費 ÷ 獲得顧客数）
- 月次／年次の概算金額を添える
- 出力：コスト構造（金額付き）

### ステップ9：9.圧倒的な優位性を明確化する

- 入力：ステップ1〜8の結果
- 処理：競合に簡単に真似されない強みを1〜3項目で書く
  - **構造的優位**: 独占的なリソース、特許、認証
  - **ネットワーク効果**: ユーザー数が増えるほど価値が上がる仕組み
  - **データ優位**: 蓄積データの多さ・質
  - **コミュニティ優位**: 顧客コミュニティ・ブランドファン
  - **オペレーション優位**: 圧倒的なスピード・コスト構造
- 「いずれ競合に真似される要素」は除外する（それは優位性ではない）
- 出力：圧倒的な優位性（1〜3項目）

### ステップ10：9要素の連動性をチェックする

- 入力：ステップ1〜9の全成果物
- 処理：9要素間の整合性を以下の組み合わせでチェックする
  - **顧客 × 課題**: 課題は本当にその顧客が抱えるものか？
  - **課題 × ソリューション**: ソリューションは課題に対応しているか？
  - **UVP × チャネル**: 価値提案がチャネルで伝わるか？
  - **顧客 × チャネル**: チャネルに顧客が実際にいるか？
  - **収益 × コスト**: 利益が出る構造か？（収益 > コスト）
  - **コスト × チャネル**: チャネルコストが収益に見合うか？
  - **KPI × 全体**: KPIが全体の成功を測れているか？
  - **優位性 × 全体**: 優位性が全体に効いているか？
- 矛盾・抜けがある箇所を指摘し、再調整する
- 出力：連動性チェック結果＋修正点

### ステップ11：BMCシートを完成させる

- 入力：ステップ1〜10の全成果物
- 処理：以下のテンプレートに統合する
- 出力：ビジネスモデルキャンバス完成シート

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 顧客セグメントが「全員」「一般」 | 1セグメントに絞る | 全員ターゲットは誰にも届かない |
| UVPが3文以上 | 1〜2文に圧縮 | 長いUVPは伝わらない |
| 課題とソリューションが1対1で対応していない | 対応関係を見直す | ソリューションが空回りする |
| KPIが10個以上 | 3〜5個に絞る | 多すぎると追えない |
| 収益 < コスト | ビジネスモデル破綻。価格・コスト見直し | 赤字事業は持続不可能 |
| 圧倒的な優位性が「いずれ真似される要素」 | 構造的優位（特許・データ・ネットワーク効果）に書き直す | 簡単に真似される強みは優位性ではない |
| 9要素の連動性チェックで矛盾あり | 該当ステップに戻って修正 | 矛盾あるBMCは戦略として機能しない |
| 1度作って放置 | 半年〜1年で再作成 | 事業環境は動く |
| 検証なしで実行に移る | 必ず顧客インタビュー・データ分析で検証 | 紙の上のBMCは仮説に過ぎない |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 顧客セグメントが「全員」「一般」 | 1セグメントに絞る | 全員ターゲットは誰にも届かない |
| UVPが自社視点（「すごい技術」） | 顧客価値視点で書く | 自社視点は伝わらない |
| 課題とソリューションが対応していない | 1対1で対応させる | 空回りソリューション |
| KPIが10個以上 | 3〜5個に絞る | 多すぎると追えない |
| 収益＜コストの構造 | 価格・コストを見直す | 赤字事業は持続不可能 |
| 「圧倒的な優位性」が真似されやすい要素 | 構造的優位（特許・データ・ネットワーク効果）で書く | すぐ真似される強みは優位性ではない |
| 連動性チェックを省略 | 9要素間の整合性を必ず確認 | 矛盾BMCは戦略として機能しない |
| 検証せずに実行に移す | 顧客ヒアリングで仮説を検証 | 紙の上のBMCは仮説に過ぎない |
| 1度作って放置 | 半年〜1年で再作成 | 事業環境は変わる |
