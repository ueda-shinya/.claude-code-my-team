# /lp-create スキル

LP（ランディングページ）の新規制作を、マーケ視点を先行させたフローで進行管理するスキル。
コーディング前に訴求軸・セクション構成を固め、承認ゲートで手戻りを防止する。

## トリガー条件

以下のいずれかに該当する場合にこのスキルを実行してください：

- ユーザーが「LP作って」「LPを新規制作したい」「ランディングページを作りたい」と発言した
- ユーザーが `/lp-create` と入力した

## 制約事項（全工程共通）

- **アスカはコーディングしない。** コーディングは必ず Shu に委託すること（「Asuka Never Codes Directly」ルール）
- **承認ゲートはスキップ不可。** シンヤさんが「急いで」と言っても必ず確認を取る
- **承認なしに次工程に進んだ場合は、シンヤさんに報告の上、前工程に戻る**
- **セキュリティレビュー（Sakura）は毎回自動実行。** 省略不可
- 各工程の成果物は次の工程へ引き継ぐ。前工程の出力が不十分な場合はアスカが補足を依頼する

## 実行手順

以下の 8 ステップを順番に実行してください。各ステップの完了をシンヤさんに報告してから次へ進んでください。

---

### ステップ①：要件ヒアリング（アスカがファシリテーター）

#### 役割分担

| 役割 | 担当 | 内容 |
|---|---|---|
| ファシリテーター | アスカ | 進行管理・フェーズ切替・要約確認・次工程への引き継ぎ |
| マーケ深掘り | レン（`subagent_type: marketing-planner`） | Phase 2〜5 の深掘り（CVゴール・流入元・ターゲット・商品情報・数字・戦略系） |
| コピー深掘り | コト（`subagent_type: copywriter`） | Phase 4・6 の深掘り（悩みの言葉・断りの言葉・感情・トーン・コピー系） |

#### 基本ルール

- **ユーザーの最初の回答は「たたき台」と見なす。** 最初の回答がちゃんとしていないことを前提に深掘りする
- **深掘りは一度に一人だけ行う。** アスカがレンまたはコトのどちらに渡すかを項目ごとに判断する
- **深掘りのループ回数：最低3回・最大10回**（下記ルール参照）
  - 「最低3回」は深掘りを開始した後の最低継続回数。最初の回答が本質的であれば深掘り不要
- **アスカはフェーズ終了時に要約確認を行う**（下記手順参照）

#### 深掘りルール（レン・コト共通）

```
1. ユーザーの回答を受け取る
2. 回答が「本質的かどうか」を判断する
   - 本質的：具体的な事実・数字・生の言葉・エピソードが含まれている
   - 非本質的：「いい感じに」「なんとなく」「普通に」「特にない」等の抽象的な回答
   - 非本質的（追加）：選択肢を選んだだけで理由・背景が説明されていない回答
3. 非本質的な場合 → 以下の4パターンのいずれかで深掘りする
   a. 言い換え提示：「つまり〇〇ということですか？」
   b. 仮説提示：「もしかして〇〇が本当の課題ではないですか？」
   c. 対比提示：「AとBどちらに近いですか？」
   d. 完成文提示（3回以上繰り返して行き詰まった時）：
      レン/コトが仮の回答草案を提示し「こういう回答でいかがですか？」と確認。
      シンヤさんが修正・承認する形で回答を完成させる
4. 本質的な回答が得られた → 深掘り完了
5. 最大10回繰り返しても本質的な回答が得られない → それまでの回答で確定し次へ
6. シンヤさんが「もういい」「進めて」等で明示的に深掘り終了を指示した場合は、
   最低回数未満でも終了する。
   ただし対象項目が「必須」かつ回答がまだ抽象的な場合のみ、
   アスカが「このまま進めてよいですか？」と1回だけ確認する。
```

#### 項目ランク別の対応

| ランク | 方針 |
|---|---|
| **必須** | 必ず何らかの回答を得る。どうしても得られない場合はレン/コトが提案して選ばせる。「わからない」は回答として認めない |
| **推奨** | 深掘りを行うが、本質的な回答が得られなくてもOK。ある程度の回答で前進する |
| **任意** | レン/コトが提案して何らかの回答を引き出すよう試みる。回答が得られない場合は空欄のまま次へ |

#### Phase担当の割り振り

| Phase | 深掘り担当 | 理由 |
|---|---|---|
| Phase 1（基本情報） | アスカのみ | 事実確認なので深掘り不要。未回答は確認するだけ |
| Phase 2（CVゴール） | レン | CVの定義・KPI・導線はマーケ判断が必要 |
| Phase 3（流入元・マーケ戦略） | レン | 広告整合性・競合分析はマーケ視点で深掘り |
| Phase 4（ターゲット・ペルソナ） | レン（属性・検討段階） + コト（悩みの言葉・断りの言葉・口グセ） | 属性はレン、言葉の質はコト |
| Phase 5（商品・サービス情報） | レン（USP・実績・数字） + コト（お客様の声・エピソード） | 数字はレン、感情・言葉はコト |
| Phase 6（コピー・訴求） | コト（CTA・トーン・法的制約） | コピー判断はコトが主導 |
| Phase 7（デザイン） | アスカのみ | 事実確認ベース。深掘りは不要 |
| Phase 8（セクション構成） | レン（構成の妥当性） | マーケ視点で構成を検証 |
| Phase 9（技術・運用） | アスカのみ | 技術事実確認なので深掘り不要 |

#### 進行手順

```
1. アスカがヒアリングシート（~/.claude/clients/officeueda/hearing-sheets/lp.md）を参照し、
   Phase 1から順番にシンヤさんに質問する

2. 各Phaseの質問はまとめて1回で提示する（質問提示は1回だが、深掘りを含む往復数はPhaseによって異なる）

3. ユーザーの回答を受け取ったら、アスカが担当エージェントを判断し、深掘りを依頼する
   - 深掘りが必要な項目のみエージェントに渡す
   - 事実確認系の項目はアスカが直接処理する

4. 深掘りが完了したら、アスカがPhaseの回答をまとめてシンヤさんに確認する：
   「Phase Xの回答をまとめるとこうなります。合っていますか？」
   → シンヤさんが「OK」または修正を返す
   → 承認されたらヒアリングシートの該当Phaseに記入して次のPhaseへ

5. Phase 1〜6・Phase 8・Phase 9 の必須項目がすべて埋まったら「ヒアリング完了」とし、
   アスカがシンヤさんに完了を報告してステップ②へ進む
   （Phase 7 は Kai がワイヤーフレームを開始するために必須ではないため含めない。
   　Phase 7 の素材情報はステップ③で Kai に渡すタイミングで確認する）
```

**ステップ完了の条件：** Phase 1〜6・Phase 8・Phase 9 の必須項目がすべて埋まっていること
（Phase 7 は任意・推奨項目が中心のため含めない。内容は Kai に渡す段階で確認する）

### ステップ②：訴求軸・シナリオ設計（レンに依頼）

レン（`subagent_type: marketing-planner`）に以下を渡して依頼してください：

**渡す情報：**
- ヒアリングシート Phase 1〜6 の回答内容（全体）
- Phase 4（ターゲット・ペルソナ）の生の声・エピソード
- Phase 5（USP・実績・お客様の声）
- GA4 データ（利用可能な場合）
- 既存 LP の問題点（リニューアルの場合）

**依頼内容：**
- ターゲットに刺さる訴求軸の策定
- メッセージの優先順位
- セクション構成案（マーケ視点でのセクション順序と各セクションの役割）

**レンからの出力：**
- 訴求軸（メイン / サブ）
- セクション構成案（順序・各セクションの目的・メッセージ概要）
- CTA の配置方針

**レンの責務範囲：** セクション構成の順序・訴求軸・メッセージの決定（マーケティング責任）

---

### ステップ③：ワイヤーフレーム作成（Kai に依頼）

Kai（`subagent_type: lp-designer`）に以下を渡して依頼してください：

**渡す情報：**
- レンの訴求軸・シナリオ（ステップ②の出力全体）※ Kai はレンの出力を受け取ってから作業開始
- ヒアリングシート Phase 1〜8 の回答内容
- Phase 7（デザイン方向性）の参考URL・NGデザイン・素材情報
- Phase 8（セクション構成）のシンヤさんによる仮案（あれば）

**依頼内容：**
- レンのセクション構成案をベースにワイヤーフレームを作成
- 各セクションのコピー骨格（見出し・本文の方向性）
- 必要な画像リスト（既存利用 / 新規生成が必要なもの）を Kai が判断
- デザイン方針（カラー・フォント・レイアウトの方向性）

**Kai からの出力：**
- セクション構成（確定版）
- コピー骨格（見出し・本文概要）
- 画像リスト（既存 / 新規生成）
- デザイン方針

**Kai の責務範囲：** 各セクションの視覚設計・コピー骨格・画像リストの決定（デザイン責任）

**セクション変更時のルール：** Kai がセクションの追加・削除・順序変更が必要と判断した場合は、**アスカ経由でレンに確認**してから変更する

**コピーブラッシュアップ（コト工程）：**
Kai のワイヤーフレーム完成後、アスカが Kai のコピー骨格を確認し、以下のいずれかに該当する場合はコト（`subagent_type: copywriter`）に依頼する：
- FV キャッチコピーが訴求軸と合っていない
- 中間 CTA や各セクション見出しの表現が弱い
- シンヤさんが「コピーは任せる」と宣言している場合

上記のいずれにも該当しない場合は、コト工程をスキップする。
シンヤさんが自身でコピーを書く場合は、その旨を宣言すればコト工程はスキップする。

**判断後の報告（必須）：** コト工程を依頼した場合もスキップした場合も、アスカはシンヤさんに判断結果を報告すること。
- 依頼した場合：「コトにコピーブラッシュアップを依頼します（理由：〇〇）」
- スキップした場合：「Kai のコピー骨格で十分と判断し、コト工程をスキップします」

### 【承認ゲート①】構成承認（スキップ不可）

アスカがシンヤさんに以下を提示して承認を得てください：

```
【承認ゲート①：セクション構成】
以下の構成で LP を制作します。ご確認ください。

■ セクション構成
1. [セクション名] — [目的]
2. [セクション名] — [目的]
...

■ 主要コピー骨格
- [見出し案]
- [見出し案]
...

■ 必要画像リスト
- [画像の説明]（既存 / 新規生成）
...

この構成で進めてよいですか？
```

- **承認された場合** → セクション構成を凍結する。コーディング中にセクションの追加・削除・順序変更が必要になった場合は、コーディングを中断してシンヤさんに影響範囲を報告し、再承認を得てから再開する。CSS や表現の調整は凍結の対象外
- **修正指示があった場合** → 修正内容に応じて差し戻し先を分岐する：
  - 修正内容が「セクション順序・訴求軸・メッセージ」→ **レンに差し戻し**
  - 修正内容が「デザイン・ビジュアル・コピー表現」→ **Kai に差し戻し**
  - 修正後、再度承認ゲートを実施する

### ステップ④：デザイン設計（Kai に追加依頼）

承認済みワイヤーフレームをもとに、Kai にデザイン詳細を依頼してください：

**依頼内容：**
- カラースキーム（メイン / アクセント / 背景）
- フォント指定（見出し / 本文）
- レイアウト詳細（セクションごとの配置・余白・装飾）
- レスポンシブ対応方針（モバイルでの表示）
- CTA ボタンのデザイン

---

### 【承認ゲート②】デザイン承認（簡易確認）

デザイン方針をシンヤさんに確認してください。
ワイヤーフレームで大枠は承認済みのため、簡易確認で OK です：

```
【承認ゲート②：デザイン方針】
■ カラー：[メイン] / [アクセント]
■ フォント：[見出し] / [本文]
■ 特記：[レスポンシブ対応等]

この方針で進めますか？（簡易確認）
```

### ステップ⑤：コーディング（Shu に依頼）

Shu（`subagent_type: backend-engineer`）に以下を渡して依頼してください。Shu は本スキルにおいて**実装担当（LP PHP/HTML/CSS/JS）**として稼働する。

**渡す情報：**
- 承認済みセクション構成（ステップ③の確定版）
- デザイン仕様（ステップ④の出力）
- ヒアリングシート Phase 2（CVの定義・CV後の導線）
- ヒアリングシート Phase 7（素材リスト）
- ヒアリングシート Phase 9（CMS・設置URL・フォームツール・解析設定）
- 既存 LP のコードベース（参照先パスを明示）
- 画像プレースホルダー仕様（alt テキスト・サイズ指定）
- PC_PLATFORM（`~/.claude/.env` の `PC_PLATFORM` を確認して渡す）
- WordPress テーマ構成で PHP テンプレートとして実装すること

**Shu への注意事項：**
- セクション構成は凍結済み。構造変更は不可
- 画像は全てプレースホルダーで実装（後から差し替え）
- JSON-LD を含めること（Web Development Coding Rules に準拠）

---

### ステップ⑥：セキュリティ・コードレビュー（Sakura に依頼・自動実行）

Sakura（`subagent_type: code-reviewer`）に以下を依頼してください：

**レビュー観点：**
- XSS・エスケープ漏れ
- WordPress 作法（`wp_enqueue_style` / `wp_enqueue_script` / `esc_html` 等）
- パフォーマンス（`loading` 属性の使い分け / 画像最適化）
- モバイル対応（レスポンシブ CSS の妥当性）

**指摘の優先度と対応：**
- **重要度 高**：必ず修正
- **重要度 中**：必ず修正
- **重要度 低**：任意（シンヤさんに報告し判断を仰ぐ）

---

### ステップ⑦：修正対応（Shu に依頼）

Sakura の指摘（重要度 高・中）を Shu に修正依頼してください。

- 修正完了後、Sakura に再レビューを依頼する
- **全項目 OK になるまで⑥→⑦を繰り返す**
- 繰り返し回数に上限はないが、3 回を超えた場合はアスカがシンヤさんに状況を報告する

---

### ステップ⑧：完成報告

以下のフォーマットでシンヤさんに報告してください：

```
【LP 制作完了】

■ 作成ファイル一覧
- [ファイルパス]：[説明]
- [ファイルパス]：[説明]
...

■ 画像プレースホルダーリスト
以下の画像は後から差し替えが必要です：
- [プレースホルダー名]：[想定画像の説明]（[サイズ]）
...

■ 次のステップ
1. WordPress への配置手順：[手順概要]
2. 画像生成：Luna に依頼する場合は「この LP の画像を生成して」と指示してください
3. 公開前チェック：実機での表示確認を推奨
```

画像生成（Luna）は「後でやる」が選択可能です。シンヤさんに確認してください：
- 「今すぐ画像生成に進む」 → Luna に画像生成リストを渡して依頼
- 「後でやる」 → 画像プレースホルダーリストを保存して完了

## 中断・再開

- このスキルは工程が長いため、途中でセッションが切れる可能性がある
- 中断する場合は、現在の工程と進捗を `session-handoff.md` に記録すること
- 再開時は `session-handoff.md` を確認し、中断した工程から再開する
