# /lp-create スキル

LP（ランディングページ）の新規制作を、マーケ視点を先行させたフローで進行管理するスキル。
コーディング前に訴求軸・セクション構成を固め、承認ゲートで手戻りを防止する。

## トリガー条件

以下のいずれかに該当する場合にこのスキルを実行してください：

- ユーザーが「LP作って」「LPを新規制作したい」「ランディングページを作りたい」と発言した
- ユーザーが `/lp-create` と入力した

## 制約事項（全工程共通）

- **アスカはコーディングしない。** コーディングは必ず Shu に委託すること（「Asuka Never Codes Directly」ルール）
- **承認ゲートはスキップ不可。** シンヤさんが「急いで」と言っても必ず確認を取る
- **承認なしに次工程に進んだ場合は、シンヤさんに報告の上、前工程に戻る**
- **セキュリティレビュー（Sakura）は毎回自動実行。** 省略不可
- 各工程の成果物は次の工程へ引き継ぐ。前工程の出力が不十分な場合はアスカが補足を依頼する

## 実行手順

以下の 9 ステップを順番に実行してください。各ステップの完了をシンヤさんに報告してから次へ進んでください。

---

### ステップ①：要件ヒアリング（アスカがファシリテーター）

#### 役割分担

| 役割 | 担当 | 内容 |
|---|---|---|
| ファシリテーター | アスカ | 進行管理・フェーズ切替・要約確認・次工程への引き継ぎ |
| マーケ深掘り | レン（`subagent_type: marketing-planner`） | Phase 2〜5 の深掘り（CVゴール・流入元・ターゲット・商品情報・数字・戦略系） |
| コピー深掘り | コト（`subagent_type: copywriter`） | Phase 4・6 の深掘り（悩みの言葉・断りの言葉・感情・トーン・コピー系） |

#### 基本ルール

- **ユーザーの最初の回答は「たたき台」と見なす。** 最初の回答がちゃんとしていないことを前提に深掘りする
- **深掘りは一度に一人だけ行う。** アスカがレンまたはコトのどちらに渡すかを項目ごとに判断する
- **深掘りのループ回数：最低3回・最大10回**（下記ルール参照）
  - 「最低3回」は深掘りを開始した後の最低継続回数。最初の回答が本質的であれば深掘り不要
- **アスカはフェーズ終了時に要約確認を行う**（下記手順参照）

#### 深掘りルール（レン・コト共通）

```
1. ユーザーの回答を受け取る
2. 回答が「本質的かどうか」を判断する
   - 本質的：具体的な事実・数字・生の言葉・エピソードが含まれている
   - 非本質的：「いい感じに」「なんとなく」「普通に」「特にない」等の抽象的な回答
   - 非本質的（追加）：選択肢を選んだだけで理由・背景が説明されていない回答
3. 非本質的な場合 → 以下の4パターンのいずれかで深掘りする
   a. 言い換え提示：「つまり〇〇ということですか？」
   b. 仮説提示：「もしかして〇〇が本当の課題ではないですか？」
   c. 対比提示：「AとBどちらに近いですか？」
   d. 完成文提示（3回以上繰り返して行き詰まった時）：
      レン/コトが仮の回答草案を提示し「こういう回答でいかがですか？」と確認。
      シンヤさんが修正・承認する形で回答を完成させる
4. 本質的な回答が得られた → 深掘り完了
5. 最大10回繰り返しても本質的な回答が得られない → それまでの回答で確定し次へ
6. シンヤさんが「もういい」「進めて」等で明示的に深掘り終了を指示した場合は、
   最低回数未満でも終了する。
   ただし対象項目が「必須」かつ回答がまだ抽象的な場合のみ、
   アスカが「このまま進めてよいですか？」と1回だけ確認する。
```

#### 項目ランク別の対応

| ランク | 方針 |
|---|---|
| **必須** | 必ず何らかの回答を得る。どうしても得られない場合はレン/コトが提案して選ばせる。「わからない」は回答として認めない |
| **推奨** | 深掘りを行うが、本質的な回答が得られなくてもOK。ある程度の回答で前進する |
| **任意** | レン/コトが提案して何らかの回答を引き出すよう試みる。回答が得られない場合は空欄のまま次へ |

#### Phase担当の割り振り

| Phase | 深掘り担当 | 理由 |
|---|---|---|
| Phase 1（基本情報） | アスカのみ | 事実確認なので深掘り不要。未回答は確認するだけ |
| Phase 2（CVゴール） | レン | CVの定義・KPI・導線はマーケ判断が必要 |
| Phase 3（流入元・マーケ戦略） | レン | 広告整合性・競合分析はマーケ視点で深掘り |
| Phase 4（ターゲット・ペルソナ） | レン（属性・検討段階） + コト（悩みの言葉・断りの言葉・口グセ） | 属性はレン、言葉の質はコト |
| Phase 5（商品・サービス情報） | レン（USP・実績・数字） + コト（お客様の声・エピソード） | 数字はレン、感情・言葉はコト |
| Phase 6（コピー・訴求） | コト（CTA・トーン・法的制約） | コピー判断はコトが主導 |
| Phase 7（デザイン） | アスカのみ | 事実確認ベース。深掘りは不要 |
| Phase 8（セクション構成） | レン（構成の妥当性） | マーケ視点で構成を検証 |
| Phase 9（技術・運用） | アスカのみ | 技術事実確認なので深掘り不要 |

#### 進行手順

```
1. アスカがヒアリングシート（~/.claude/skills/lp-create/hearing-sheet.md）を参照し、
   Phase 1から順番にシンヤさんに質問する。

2. 各Phaseの質問はまとめて1回で提示する（質問提示は1回だが、深掘りを含む往復数はPhaseによって異なる）

3. ユーザーの回答を受け取ったら、アスカが担当エージェントを判断し、深掘りを依頼する
   - 深掘りが必要な項目のみエージェントに渡す
   - 事実確認系の項目はアスカが直接処理する

4. 深掘りが完了したら、アスカがPhaseの回答をまとめてシンヤさんに確認する：
   「Phase Xの回答をまとめるとこうなります。合っていますか？」
   → シンヤさんが「OK」または修正を返す
   → 承認されたらヒアリングシートの該当Phaseに記入して次のPhaseへ

5. Phase 1〜6・Phase 8・Phase 9 の必須項目がすべて埋まったら「ヒアリング完了」とし、
   アスカがシンヤさんに完了を報告してステップ②へ進む
   （Phase 7 は Kai がワイヤーフレームを開始するために必須ではないため含めない。
   　Phase 7 の素材情報はステップ④で Kai に渡すタイミングで確認する）
```

**ステップ完了の条件：** Phase 1〜6・Phase 8・Phase 9 の必須項目がすべて埋まっていること
（Phase 7 は任意・推奨項目が中心のため含めない。内容は Kai に渡す段階で確認する）

### ステップ②：ヒアリング内容レビュー（確定前チェック）

ヒアリング完了後、全回答を一覧化してシンヤさんに確認・修正の機会を提供する。
「確定」が出るまでステップ③には進まない。

#### 進行手順

**1. レン・コトに評価を依頼する**

アスカが全ヒアリング回答を以下の担当に渡し、評価とアドバイスを並列で依頼する：
- レン（`subagent_type: marketing-planner`）：Phase 1〜6・8・9 のマーケ視点での評価
- コト（`subagent_type: copywriter`）：Phase 4〜6 のコピー・言語視点での評価

依頼内容：
- 担当Phaseの各項目について回答を★1〜5で評価する
- ★3以下の項目にはアドバイスを付ける
- ★4以上は「-」でOK（特記不要）
- **レンの担当：** Phase 1〜6・8・9 のうち、マーケ系項目（CVゴール・流入元・ターゲット属性・USP・実績・数字・構成・技術）
- **コトの担当：** Phase 4〜6 のうち、言語・感情系項目（悩みの言葉・断りの言葉・口グセ・お客様の声・CTA文言・トーン）
- 1項目に対して評価するのはレン・コトどちらか一方のみ（重複評価なし）
- **項目間の整合性チェック（レンのみ）：** 以下のような矛盾がないか確認し、矛盾があれば指摘する
  - CVの定義と流入元の整合性（例：CV=購入なのに流入が潜在層向けSNS広告）
  - ターゲットの検討段階とCTA文言の整合性（例：潜在層なのに「今すぐ購入」）
  - 流入元とLPの位置づけの整合性

**★評価基準：**
```
★★★★★（5）とても参考になる（具体的・数字・生の声・エピソードあり）
★★★★☆（4）参考になる（概ね具体的）
★★★☆☆（3）普通（もう少し詳しいと良い）
★★☆☆☆（2）やや不足（深掘りが必要）
★☆☆☆☆（1）ほとんど参考にならない（抽象的すぎる）
```

**2. 一覧を出力する**

レン・コトの評価を受け取ったら、アスカが以下のフォーマットで出力する：

```
■ ヒアリング内容レビュー

| Phase | # | 項目 | 回答内容 | LP参考度 | 評価担当 | アドバイス |
|---|---|---|---|---|---|---|
| Phase 1 | 1 | 案件名 | 〇〇LP | ★★★★★ | アスカ | - |
| Phase 2 | 5 | CVの定義 | 無料相談フォーム送信 | ★★★★★ | レン | - |
| Phase 4 | 18 | ターゲットの悩み | ホームページを作ったのに問い合わせが来ない | ★★★☆☆ | コト | もう1〜2件の具体的なエピソードがあると訴求力が上がります |
...

修正したい項目があれば番号を指定してください。
修正なければ「確定」でステップ③へ進みます。
```

**3. 修正ループ**

- シンヤさんが番号を指定して修正内容を伝える → アスカがヒアリングシートの該当項目を更新
- 更新後、アスカが修正後の回答を自身で判断して★を更新する（レン・コトへの再依頼は行わない）
- 更新後、一覧を再出力（修正箇所に「（更新）」と付け、変更後の★も反映する）
- シンヤさんが「確定」と言うまで繰り返す

**4. 確定**

「確定」が出たらヒアリングシートを最終版として保存し、ステップ③へ進む旨をシンヤさんに報告する。

**ステップ完了の条件：** シンヤさんが「確定」と発言したとき

### ステップ③：訴求軸・シナリオ設計（レンに依頼）

レン（`subagent_type: marketing-planner`）に以下を渡して依頼してください：

**渡す情報：**
- ヒアリングシート Phase 1〜6 の回答内容（**ステップ②で確定した最終版**）
- Phase 4（ターゲット・ペルソナ）の生の声・エピソード
- Phase 5（USP・実績・お客様の声）
- GA4 データ（利用可能な場合）
- 既存 LP の問題点（リニューアルの場合）

**依頼内容：**
- Phase 8でシンヤさんが出した仮案を参考にしつつ、マーケ視点で最適な構成を提案する。仮案と異なる場合は理由を明示すること
- ターゲットに刺さる訴求軸の策定
- メッセージの優先順位
- セクション構成案（マーケ視点でのセクション順序と各セクションの役割）

**レンからの出力（以下のフォーマットで提出）：**

```
■ 訴求軸
- メイン訴求：___（根拠：___）
- サブ訴求：___（根拠：___）

■ セクション構成案
| 順序 | セクション名 | 目的 | メッセージ方向性 | 配置理由 |
|---|---|---|---|---|
| 1 | ___ | ___ | ___ | ___ |
| 2 | ___ | ___ | ___ | ___ |
...

■ CTA配置方針
- 配置数：___箇所
- 配置位置と理由：___

■ Phase 8 仮案との差分（ある場合のみ）
- 変更点：___
- 理由：___
```

**レンの責務範囲：** セクション構成の順序・訴求軸・メッセージの決定（マーケティング責任）

### ステップ④：ワイヤーフレーム作成（Kai に依頼）

Kai（`subagent_type: lp-designer`）に以下を渡して依頼してください：

**渡す情報：**
- レンの訴求軸・シナリオ（ステップ③の出力全体）※ Kai はレンの出力を受け取ってから作業開始
- ヒアリングシート Phase 1〜8 の回答内容
- Phase 7（デザイン方向性）の参考URL・NGデザイン・素材情報
- Phase 8（セクション構成）のシンヤさんによる仮案（あれば）
- Phase 7 #39 のモバイルファースト / PC優先の指定（FVパターン選択の前提情報として明示的に伝えること）

**依頼内容：**
- レンのセクション構成案をベースにワイヤーフレームを作成
- 各セクションのコピー骨格（見出し・本文の方向性）
- 必要な画像リスト（既存利用 / 新規生成が必要なもの）を Kai が判断 → このリストはステップ⑤のデザイン仕様書セクション10で詳細化される
- デザイン方針（トーン・雰囲気・方向性レベルのみ。HEXコード・pxサイズ等の詳細仕様はステップ⑤で決定）

**Kai からの出力：**
- セクション構成（確定版）
- コピー骨格（見出し・本文概要）
- 画像リスト（既存 / 新規生成）
- デザイン方針（方向性レベル）

**Kai の責務範囲：** 各セクションの視覚設計・コピー骨格・画像リストの決定（デザイン責任）

**セクション変更時のルール：** Kai がセクションの追加・削除・順序変更が必要と判断した場合は、**アスカ経由でレンに確認**してから変更する

**コピーブラッシュアップ（コト工程・原則実施）：**
Kai のワイヤーフレーム完成後、コト（`subagent_type: copywriter`）にコピーブラッシュアップを依頼する。

**依頼内容：**
- FV キャッチコピーの訴求軸との整合性確認・改善
- 各セクション見出しの表現強化
- 中間 CTA 周辺のコピー改善

**スキップ条件（以下の場合のみスキップ可）：**
- シンヤさんが「コピーは自分で書く」と明示的に宣言した場合

**判断後の報告（必須）：** コト工程を実施した場合もスキップした場合も、アスカはシンヤさんに判断結果を報告すること。
- 実施した場合：「コトにコピーブラッシュアップを依頼しました」
- スキップした場合：「シンヤさんのコピー自作宣言により、コト工程をスキップします」

### 【承認ゲート①】構成承認（スキップ不可）

アスカがシンヤさんに以下を提示して承認を得てください：

```
【承認ゲート①：セクション構成】
以下の構成で LP を制作します。ご確認ください。

■ セクション構成
1. [セクション名] — [目的]
2. [セクション名] — [目的]
...

■ 主要コピー骨格
- [見出し案]
- [見出し案]
...

■ 必要画像リスト
- [画像の説明]（既存 / 新規生成）
...

この構成で進めてよいですか？
```

- **承認された場合** → セクション構成を凍結する。コーディング中にセクションの追加・削除・順序変更が必要になった場合は、コーディングを中断してシンヤさんに影響範囲を報告し、再承認を得てから再開する。CSS や表現の調整は凍結の対象外
- **修正指示があった場合** → 修正内容に応じて差し戻し先を分岐する：
  - 修正内容が「セクション順序・訴求軸・メッセージ」→ **レンに差し戻し**
  - 修正内容が「デザイン・ビジュアル・コピー表現」→ **Kai に差し戻し**
  - 修正後、再度承認ゲートを実施する

### ステップ⑤：デザイン設計（Kai に追加依頼）

承認済みワイヤーフレームをもとに、**ステップ④で決めたデザイン方向性を詳細化する工程**として、Kai にデザイン詳細を依頼してください。

**Kai の成果物：** `skills/lp-create/design-spec-template.md` を使用してデザイン仕様書を作成すること。テンプレートの全セクションを埋め、セルフチェックをクリアしてから提出する。

**渡す情報：**
- ステップ④の Kai 出力（セクション構成確定版・コピー骨格・デザイン方針）
- ステップ③のレン出力（訴求軸・メッセージ）→ ビジュアルで表現すべき内容の根拠として渡す
- ステップ①のターゲット感情状態（Phase 4）→「広告クリック直後にどんな気持ちでページを開くか」
- Phase 7 の参考 LP・NG デザイン（デザイン観点の参考 URL）

**依頼内容（Kai の担当）：**

#### 1. カラースキーム・フォント

- メイン / アクセント / 背景色を HEX コードで指定
- 見出し・本文のフォントファミリー・サイズ・ウェイト・行間を指定
- CTA ボタン色は「背景に対してコントラストが最も高い色」を選ぶ。特定の色を決め打ちしない

#### 2. FV（ファーストビュー）設計仕様

キャッチ・ヒーロービジュアル・CTA の配置パターンを以下から選び、理由を添えること：

| パターン | 構成 | 向いている条件 |
|---|---|---|
| **A：左テキスト / 右ビジュアル** | キャッチ・サブコピー・CTAを左、画像を右 | BtoB・SaaS・ツール・情報量が多い商材 |
| **B：センター配置** | 全幅背景画像 or 動画の上にキャッチ・CTAを中央 | BtoC・感情訴求・ブランド訴求・旅行・ライフスタイル |
| **C：右テキスト / 左ビジュアル** | 商品・人物を左前面に出し、テキストを右 | 美容・化粧品・食品・商品を主役にしたい場合 |
| **モバイル共通** | 1カラム縦積み（キャッチ → ビジュアル → CTA） | モバイルは全パターンでこの構造に変換 |

**FV 設計の禁止事項：**
- キャッチ・機能説明・受賞歴・CTAをすべて FV に詰め込む（3秒離脱を招く）
- ビジュアルが大きすぎてキャッチが読めない配置
- FV内に CTA が収まらない構成（特にモバイル）

#### 3. 視線誘導パターン

以下の判断基準でパターンを選び、CTA の配置位置を決定すること：

| 条件 | 採用パターン | CTA の位置 |
|---|---|---|
| 情報量が少ない・2カラム・PC メイン・感情訴求 | **Z パターン** | Z の終点（右下・対角線終点）に配置 |
| 情報量が多い・1カラム・テキスト主体・比較検討ターゲット | **F パターン** | 各セクション見出し直後 + ページ下部 |
| モバイル専用・縦スクロール | **F パターン**（縦スキャン） | スクロール途中のベネフィット直後 + 末尾 |

#### 4. 信頼要素の配置設計

3 地点で内容を変えて配置すること。同じ要素を3箇所に並べない：

**FV 周辺（スクロール前）** — 「このページは信頼できる」という第一印象
- 導入実績数・登録者数などの数字（「利用者 XX 万人」）
- メディア掲載・受賞ロゴ
- BtoB の場合はクライアントロゴ
- 資格・認定バッジ

**中盤（ストーリー・比較の後）** — 「本当に効果があるのか」という疑念を解消
- 具体的な数字・Before/After を含むお客様の声・口コミ
- 事例紹介（企業名・業種・成果の3点セット）
- 第三者レビュー・評価スコア

**CTA 直前（ボタンの直上）** — 「押していいか」という最後の躊躇を除去
- 「〇〇万人が利用中」などの数値（社会的証明）
- 返金保証・無料トライアルの明示
- プライバシーマーク・SSL 表示
- マイクロコピー（→ 文言はコト担当）

#### 5. CTA 設計

**配置数・位置：**
- 標準：FV 内 / 中盤の信頼要素直後 / ページ末尾 の 3 点
- 短い LP（スクロール2〜3回）：FV と末尾の 2 点でも可
- 「ベネフィットの熱を冷まさないうちに CTA を置く」が鉄則

**ボタンデザイン：**
- 最小サイズ：44px × 44px（WCAG 基準）
- モバイル推奨：60〜72px（親指タップを想定）
- 色：背景に対して最もコントラストが高い色を選ぶ
- ホバーエフェクト：PC のみ有効（モバイルには不要）
- ボタン周辺に十分なホワイトスペースを確保する
- ラベル：「送信する」ではなくベネフィット or アクションを明示（例：「無料で相談する」「今すぐ試す」）→ **文言はコト担当**。Kai はラベルの方向性と文字数上限を定義する

**マイクロコピー（Kai の担当は配置位置・文字数制約のみ）：**
- 配置位置：ボタンの**直下**（直上はCTAより先に読まれる場合があるため直下が安全）
- 文字数上限：20〜40字
- 内容の方向性：心理的ハードルを下げる言葉（無料・解約自由・○分で完了など）→ 文言はコト担当

**スティッキー CTA（長い LP の場合）：**
- 画面下部に固定 CTA バーを設置する選択肢を提示する
- 本文の表示領域を圧迫しない高さ設計が必要

#### 6. レイアウト詳細・離脱防止

- セクションごとの配置・余白・装飾を定義
- グローバルナビゲーションは設置しない
- 外部リンクは排除（LP 外への動線を作らない）
- ボタン同士の間隔：最低 8px 以上（誤タップ防止）

#### 7. レスポンシブ・モバイル対応方針

- モバイルファーストで設計し、PC を拡張として扱う
- 2カラムは PC のみ。モバイルは全セクションで 1カラム縦積みに変換
- モバイルのフォントサイズ：見出し 20〜28px（小さすぎる失敗が多い）
- PC より余白・行間を広めに設定する
- 電話番号は `tel:` リンクにする
- フォームの入力フィールドは十分なタップ領域を確保する

#### 8. パフォーマンス設計方針

**画像フォーマット：**

| フォーマット | 使用ケース |
|---|---|
| **WebP** | 写真・イラスト・LP 上のすべての画像の基本 |
| **SVG** | アイコン・ロゴ・単純な図形 |
| **PNG** | 透過が必要でWebP非対応環境向けのフォールバックのみ |
| **JPEG** | 原則不使用（WebP で代替） |

実装は `<picture>` タグで WebP → JPEG のフォールバック構造を推奨。

**lazy loading の適用基準：**
- FV 内のヒーロービジュアル：`loading="eager"`（LCP に直結するため lazy にしない）
- FV より下の全画像：`loading="lazy"`

**禁止エフェクト：**
- パーティクルアニメーション・視差スクロール（Parallax）
- `transform` / `opacity` 以外のプロパティへの CSS アニメーション
- 背景への自動再生動画（モバイル通信環境で致命的）

**パフォーマンス目標：**
- 表示速度：3秒以内
- LCP：2.5秒以内
- FV 画像のファイルサイズ：300KB 以内

**Kai からの出力：**
- カラースキーム（HEXコード）・フォント仕様（サイズ・ウェイト・行間）
- FV 設計仕様（採用パターン + 理由）
- 視線誘導パターン（採用パターン + 理由）
- 信頼要素配置設計（3地点それぞれの要素リスト）

- レスポンシブ対応方針
- パフォーマンス設計方針（画像フォーマット・lazy loading 適用箇所・禁止エフェクト）

**コトへの追加依頼（承認ゲート②の前に完了させること）：** Kai のデザイン設計完了後、承認ゲート②に進む前に、アスカは以下をコト（`subagent_type: copywriter`）に依頼すること：
- CTA ボタンラベルの文言（Kai が定義した方向性・文字数上限をもとに）

コトの文言が確定してから承認ゲート②に進む。これにより、承認時にCTA文言の長さとボタンデザインの整合性を確認できる。

### ステップ⑤.5：画像プロンプト設計（Luna に依頼）

Kai のデザイン仕様書（セクション10：画像定義リスト）をもとに、Luna（`subagent_type: nano-banana`）に画像プロンプトの設計を依頼してください。

**このステップの目的：** 画像生成は⑨で行う。ここでは「どう生成するか」のプロンプトだけを確定させ、ファイルに保存する。

**渡す情報：**
- デザイン仕様書のセクション10（画像定義リスト）全体
- デザイン仕様書のセクション1（カラースキーム・トーン）→ 画像の色調整合に使う
- ヒアリングシート Phase 7（参考LP・NGデザイン）→ 雰囲気の参考に使う

**Luna への依頼内容：**
- セクション10の各画像（新規生成のもののみ）に対して、詳細な生成プロンプトを作成する
- 既存素材の画像はスキップ（プロンプト不要）
- 各プロンプトには以下を含めること：
  - 被写体・構図の詳細（Kai の定義をベースに具体化）
  - スタイル・トーン・照明・色調（デザイン仕様書のカラースキームと整合させる）
  - アスペクト比（Kai の指定を引き継ぐ）
  - ネガティブプロンプト（避けるべき要素があれば）

**Luna からの出力：**
- 各画像の生成プロンプト（番号・ファイル名案・プロンプト本文・アスペクト比）
- アスカが以下のパスに `image-prompts.md` として保存する（クライアントディレクトリ構造ルールに準拠）：
  - 複数事業クライアント：`~/.claude/clients/<クライアント名>/biz-<事業名>/lp-<日付>/image-prompts.md`
  - 単一事業クライアント：`~/.claude/clients/<クライアント名>/lp-<日付>/image-prompts.md`

**ステップ完了の条件：** 新規生成が必要な全画像のプロンプトが `image-prompts.md` に保存されていること

## 中断・再開

- 中断する場合は、現在の工程と進捗を `session-handoff.md` に記録すること

- 再開時は `session-handoff.md` を確認し、中断した工程から再開する
