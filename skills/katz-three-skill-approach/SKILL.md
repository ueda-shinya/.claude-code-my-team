# katz-three-skill-approach スキル

カッツモデル = アメリカの経営学者ロバート・L・カッツが提唱した理論で、組織の管理職や従業員に必要な能力を3つに分け、役職ごとにどの能力をどの程度身に付けておきたいかを階層化したもの。事業戦略スクール「知足」のフレームワーク準拠。

## カッツモデルの3スキル

| スキル | 英名 | 内容 | 重要となる役職 |
|---|---|---|---|
| **テクニカルスキル** | Technical Skill | 業務に必要な専門知識や技術。例：エンジニアのプログラミング能力、営業職の商談スキル | 現場スタッフ・スペシャリスト |
| **ヒューマンスキル** | Human Skill | コミュニケーション能力やリーダーシップ。例：部下の育成、チームのモチベーション管理 | 中間管理職・リーダー |
| **コンセプチュアルスキル** | Conceptual Skill | 抽象的な課題の本質を見抜き、戦略を立案する能力。例：経営・事業戦略の立案、新規事業開発の構想 | 経営層・上級マネジメント |

## カッツモデルと役職レベルの関係

3階層モデル（下から）：

| 階層 | 最重要スキル | 内容 |
|---|---|---|
| 【現場スタッフ】ロワーマネジメント | テクニカルスキル | 業務遂行能力が評価される |
| 【中間管理職】ミドルマネジメント | ヒューマンスキル | 部下やチームをまとめる能力が鍵 |
| 【経営層】トップマネジメント | コンセプチュアルスキル | 戦略立案や組織全体の方向性を示す能力が求められる |

**スキル比重の変化**：階層が上がるほど、テクニカルの比重が下がり、コンセプチュアルの比重が上がる。ヒューマンスキルは全階層で必要。

## 適用条件

- このスキルを使う場面：
  - 自社の役職別スキル要件を整理したい
  - 個人のキャリア成長戦略を立てたい（特に昇進前後）
  - 組織の人材育成プランを設計したい
  - 「役職に対してスキルが追いついていない」「役職と必要スキルがずれている」課題の診断
- このスキルを使わない場面：
  - 組織体制図・役割分担表の設計 → `project-team-structure`
  - 採用戦略・ガイドライン → 別スキル（採用戦略）
  - キャリアロードマップ・育成計画の詳細 → 別スキル（キャリアロードマップ）
  - 評価制度の策定 → 別スキル（評価制度）
  - 目標の階層分解 → `goal-hierarchy-design`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 診断対象 | 必須 | 個人／チーム／組織全体 | 「営業部マネージャー田中さん」 |
| 現在の役職 | 必須 | ロワー／ミドル／トップ | 「ミドル（課長）」 |
| 目指す役職 | 任意 | 次のキャリア目標 | 「2年後に部長（トップへの橋渡し）」 |
| 現状のスキル評価 | 任意 | 自己評価（主観で可） | 「テクニカル：高／ヒューマン：中／コンセプチュアル：低」 |
| 業界・職種特性 | 任意 | テクニカル要件の重要度等 | 「IT業界・エンジニア出身」 |

## 実行ステップ

### ステップ1：現在の役職と必要スキル比重を特定
- 入力：診断対象、現在の役職
- 処理：
  1. 現在の役職を3階層（ロワー／ミドル／トップ）にマッピング
  2. その階層で最重要となるスキルを特定（テクニカル／ヒューマン／コンセプチュアル）
  3. 業界特性を考慮（例：IT・コンサルはテクニカル要件が高い）
- 出力：現在の役職×必要スキル比重マッピング

### ステップ2：3スキルの現状評価
- 入力：診断対象、現状のスキル評価（あれば）
- 処理：以下4ステップで自己／他者評価を実施
  - **STEP 1 テクニカルスキル**：業務遂行能力が十分か？必要な資格や知識を習得しているか？
  - **STEP 2 ヒューマンスキル**：チームをまとめ、部下を動機づける能力を発揮できているか？
  - **STEP 3 コンセプチュアルスキル**：組織全体の目標を理解し、戦略的な意思決定ができているか？
  - **STEP 4 スキルバランス**：3スキルの中で不足している部分はどこか？改善のための行動は何か？
- 出力：3スキルの現状評価（A/B/Cまたは1〜5段階）＋不足スキル特定

### ステップ3：役職別の活用例とギャップ分析
- 入力：ステップ1・2の出力
- 処理：以下3つの活用例から該当する役職を選定し、ギャップを特定

  **活用例**：
  - **現場スタッフ**：テクニカルスキルを磨き、専門分野での成果を出す
  - **チームマネージャー**：ヒューマンスキルを活かし、部下の育成やチーム連携を強化する
  - **経営層**：コンセプチュアルスキルを発揮し、会社全体の戦略を立案する

  **ギャップ分析**：
  - 現状の役職での「最重要スキル」と「自身のスコア」の差を計算
  - 目指す役職での「最重要スキル」と「自身のスコア」の差を計算
- 出力：役職と現状のギャップ表

### ステップ4：実践ポイント（成長戦略3要素）の設計
- 入力：ギャップ分析結果
- 処理：以下3要素で成長戦略を設計
  1. **スキルバランスの意識**：必要なスキルを偏りなく発展させる
  2. **フィードバックの活用**：上司や同僚からのフィードバックを受け、スキル改善に役立てる
  3. **スキル開発プランの立案**：キャリアステージに応じたトレーニング計画を立てる
- 出力：3要素の成長戦略アクションプラン

### ステップ5：成功の鍵（2つ）と次アクションの確認・実行プラン
- 入力：ステップ4の戦略
- 処理：以下の **成功の鍵2つ** を確認した上で、**次のステップ** を実行プランに落とし込む
  - 成功の鍵① **スキルの習得順序**：現場スタッフはテクニカル／中間管理職はヒューマン／経営層はコンセプチュアル を優先的に発展させる
  - 成功の鍵② **柔軟な成長戦略**：キャリア段階に応じてスキル開発を調整
  - 次のステップ：「現在の役職で最も必要なスキルを見直し、成長プランを具体化する」を実行に移す
- 出力：実行プラン＋次の3ヶ月／6ヶ月／1年でやることリスト

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 現場スタッフ（ロワー）でテクニカルが弱い | テクニカルスキル習得を最優先 | 階層別の最重要スキル |
| 中間管理職（ミドル）でヒューマンが弱い | ヒューマンスキル（部下育成・モチベ管理）を最優先 | 階層別の最重要スキル |
| 経営層（トップ）でコンセプチュアルが弱い | コンセプチュアルスキル（戦略立案・抽象化）を最優先 | 階層別の最重要スキル |
| エンジニアからマネージャーに昇進 | テクニカル偏重から脱却し、ヒューマンスキル習得を急ぐ | 役職移行で必要スキルが変化 |
| 中間管理職で経営層を目指す | コンセプチュアルスキルを意識的に磨く（経営視点・抽象化） | 次階層の最重要スキルを先取り |
| 全スキルが同レベルで凸が無い | 現役職の最重要スキルを伸ばす | 全方位より重要スキル特化が効率的 |
| 業界がIT・コンサル | テクニカル要件が高い特殊業界として補正 | 業界特性で比重が変わる |
| FBを受けていない | 上司・同僚からのFB取得を最優先アクションに | 自己評価のバイアスを防ぐ |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 役職が変わってもテクニカルスキルだけ磨く | 役職に応じてヒューマン・コンセプチュアルへシフト | 上の役職ほどテクニカル比重が下がる |
| 全スキルを均等に伸ばそうとする | 現役職の最重要スキルを優先 | 効率が悪く成果が出にくい |
| エンジニア出身マネージャーが現場業務を抱え込む | 部下育成・チーム連携（ヒューマン）に重心移行 | プレイヤー継続だとマネジメントが回らない |
| FBを受けない自己評価のみ | 上司・同僚から定期FBを取得 | 自己評価のバイアスを補正 |
| キャリア計画なしで現状維持 | 3〜6〜12ヶ月の成長プランを具体化 | 偶然任せでは成長が遅い |
| 業界特性を無視する | IT・コンサル等はテクニカル要件高め等の補正 | 業界によって比重が変わる |
| 経営層になっても現場業務にこだわる | コンセプチュアル（戦略・抽象化）に重心移行 | 経営判断ができなくなる |

## 連携スキル

- `project-team-structure`：組織体制・役割分担の設計（カッツモデルで人材配置を設計）
- `goal-hierarchy-design`：目標の階層分解（コンセプチュアルスキルの実践）
- `teaching-coaching-leading`：階層別の指導アプローチ（ロワー＝ティーチング、ミドル＝コーチング、トップ＝リーディング）
- `mvv-design`：ビジョン提示（コンセプチュアルスキルの活用）
- `goal-execution-system`：モニタリング＋フィードバックループ（FB取得の仕組み化）
- `decision-making-framework`：戦略的意思決定（コンセプチュアルスキルの実践）
- `ms-matrix-talent-grid`：階層×3スキル×4象限のクロス分析で②理想型のリーダー育成深掘り
- `success-cycle-relationship-quality`：ヒューマンスキル開発時の関係の質構築の前提として連携
- `goal-cascade-kpt-1on1`：階層別の目標設定とKPT 1on1運用（ロワー＝Try具体化／ミドル＝Problem構造化／トップ＝Keep体系化）

## 5状態出力契約

外部API・サブプロセス・外部設定依存を持たないため、CLAUDE.md「5状態出力契約」の対象判定基準には該当しない。ただし以下のマーカーを使うこと：

- `[OK:0件]`：診断対象の役職が不明確で診断できない場合
- `[PARTIAL:N成功/M失敗]`：3スキルのうち一部しか評価できなかった場合
- `[SKIP]`：スキル評価データが不足し、推測で進める場合（前提不足）

※外部API依存なしのため `[FAIL]` は通常発生しない。前提不足は `[SKIP]` で扱う。
