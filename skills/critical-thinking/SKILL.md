# critical-thinking スキル

クリティカルシンキング（Critical Thinking）= 感情・偏見に流されず、データや事実を基に情報や主張の妥当性を検証する思考スキル。事業戦略スクール「知足」のフレームワーク準拠。

## ロジカルシンキングとの違い

| 観点 | クリティカルシンキング | ロジカルシンキング |
|---|---|---|
| **目的** | 情報や考えを批判的に検討し、妥当性を確認する | 情報を整理し、論理的に結論を導く |
| **プロセス** | 前提や根拠を疑い、多角的に分析する | 結論を中心に道筋を立てて構築する |
| **主な活用場面** | 複雑な問題や意見の妥当性を確認する場合 | 解決策を明確に示したい場合 |

## 適用条件

- このスキルを使う場面：
  - 既存の主張・仮説・結論が「本当に正しいか」を検証したい
  - 自社が選ばれる理由・売れる理由を批判的に整理したい（顧客視点で再評価）
  - 提案書や戦略仮説のレビューで前提抜けをチェックしたい
  - 「価格が高いから売れない」等の暗黙の前提を疑いたい
- このスキルを使わない場面：
  - ゼロから論理的に結論を導きたい → `logical-thinking`
  - 既存枠を超えた新発想を出したい → `lateral-thinking`
  - 競合が少ない/多い等の楽観主張の検証 → `competitive-absence-audit`（より専門的）
  - 提案資料の論理整理 → `pyramid-structure`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 検証対象 | 必須 | 妥当性を疑う主張・仮説・結論 | 「自社サービスが選ばれる理由は価格の安さ」 |
| 関連データ | 任意 | 既存の根拠データ・顧客フィードバック | 顧客満足度調査・売上データ・競合価格表 |
| 検証粒度 | 任意 | 簡易（基本ステップのみ）／詳細（3STEP実践＋3視点ずらし） | 「詳細でお願い」 |

## 4つの基本ステップ

| ステップ | 内容 | 例 |
|---|---|---|
| 1. 情報の収集と分析 | 必要なデータや事実を集め、MECE原則で整理する | 売上減少の原因を顧客データや市場調査から分析 |
| 2. 前提や仮説を疑う | 情報に隠れた前提や仮説が正しいか検討する | 「価格が高いから売れない」という仮説の妥当性を検討 |
| 3. 多角的な視点で考える | 異なる立場や視点から情報を再評価する | 顧客視点（価格の高さ）／社内視点（サービスの質） |
| 4. 結論を導き、妥当性を検証する | 根拠に基づいた判断を行い、結論に矛盾がないか確認 | 顧客満足度調査と売上データを照合し、価格ではなくサービス改善が必要と結論 |

## 3STEP 思考プロセス（実践版）

### STEP 1：問いと答えのセットをつくる
1. 問いをたてる（例：「自社サービスが選ばれる理由は何か？」）
2. 問いに対する答えの仮説を3つ程度考える（第1階層・MECEに）
3. 各答えの仮説に対する根拠を3つ程度調べる（第2階層・MECEに）

→ 結果：3階層ツリー（中心問い→仮説3本→各仮説に根拠3点）

### STEP 2：問いと答えのセットの確からしさを問う

各層に対して、両方向から検証する：

| 矢印 | 問い | 検証内容 |
|---|---|---|
| **「本当に？」** | 上から下へ（第1層→第2層） | 答えの仮説が、根拠で本当に裏付けられているか？ |
| **「だから何？」** | 下から上へ（第2層→第1層） | 集めた根拠が、本当に問いの答えになっているか？ |

→ 矛盾・抜け漏れを検出し、ツリーを修正する

### STEP 3：答えを横に広げる（ラテラル展開）

3つの視点ずらしで仮説を再評価する：

| 視点ずらし | やり方 | 例 |
|---|---|---|
| **立場をずらす** | 顧客／メーカー／60代と20代／取引先／投資家 等の視点で再評価 | 「ハッピーターンが選ばれる理由」を子供視点とシニア視点で見ると違う |
| **前提を疑う** | そもそもの前提を否定して再考 | 「ハッピーターンは選ばれていない」という前提で考えるとどう見えるか？ |
| **時間軸をずらす** | 100年前なら？／100年後なら？／販売開始時なら？ | 時代背景・市場成熟度を変えると要因が変わる |

→ 新しい仮説・盲点が浮き上がる場合は STEP 1 に戻り再構築

## 実行ステップ

### ステップ1：検証対象を明確化する
- 入力：ユーザーから受領した検証対象（主張・仮説・結論）
- 処理：
  1. 検証する主張を1文に整理する（例：「自社サービスは価格の安さが選ばれる理由」）
  2. 暗黙の前提を3つ程度書き出す（「価格が比較されている／顧客は価格を最重視している／競合より安い」等）
- 出力：検証対象1文＋暗黙前提リスト

### ステップ2：情報収集と分析（基本ステップ1）
- 入力：検証対象
- 処理：
  1. 主張を支える根拠データを集める（既存資料・顧客アンケート・売上データ等）
  2. MECE原則で分類（参考：`logical-thinking` の MECE分解軸）
  3. データが不足している箇所を「未取得」と明記
- 出力：根拠データ表

### ステップ3：問いと答えのセットを作る（STEP 1）
- 入力：検証対象＋根拠データ
- 処理：
  1. 検証対象を「中心の問い」に変換する
  2. 第1階層：答えの仮説を3つ立てる（MECEに）
  3. 第2階層：各仮説の根拠を3つ調べる（MECEに）
- 出力：3階層ツリー（中心問い→仮説3本→根拠9個）

### ステップ4：「本当に？／だから何？」で確からしさを問う（STEP 2）
- 入力：3階層ツリー
- 処理：
  1. 上から下「本当に？」：各仮説が根拠で裏付けられているか1つずつ検証
  2. 下から上「だから何？」：各根拠群が本当に問いの答えになっているか検証
  3. 矛盾・抜け漏れを修正（必要に応じてツリーを再構築）
- 出力：検証済みツリー＋検証ログ

### ステップ5：3視点ずらしで横展開（STEP 3）
- 入力：検証済みツリー
- 処理：
  1. **立場をずらす**：少なくとも2つの異なる立場（例：顧客視点／競合視点／投資家視点）で再評価
  2. **前提を疑う**：暗黙前提（ステップ1で書き出したもの）を1つずつ否定して再考
  3. **時間軸をずらす**：過去・未来の時点で再評価（市場成熟度・トレンド変化）
- 出力：新規仮説・盲点リスト

### ステップ6：結論を導き妥当性を検証（基本ステップ4）
- 入力：ステップ4・5の結果
- 処理：
  1. 当初の検証対象（主張）を「採択／却下／修正」のいずれかで判定
  2. 修正の場合は新しい結論を1文で言語化
  3. 結論の根拠を再度MECEで整理し、矛盾がないか最終確認
  4. 多角的視点（ステップ5）の結果が結論に反映されているか確認
- 出力：最終判定＋根拠＋盲点（残課題）リスト

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 「本当に？」で根拠が崩れる | 仮説を却下し、別仮説で再構築 | 根拠なき仮説は持ち越さない |
| 「だから何？」で問いに答えていない | 根拠を入れ替えるか、問い自体を修正 | ロジックの飛びを温存しない |
| 1つの視点でしか見ていない | STEP 3（3視点ずらし）を必ず実施 | 多角的検証を省くと盲点が残る |
| 暗黙前提が3つ未満 | ステップ1に戻り深掘り | 前提の洗い出しが甘い |
| 顧客視点と社内視点で結論が逆 | 顧客視点を優先し、社内視点とのギャップを論点化 | 内向き判断による戦略誤りを防ぐ |
| 統計的有意性がないデータで結論 | 「データ不十分」とマーキングし結論を保留 | 印象論で結論を確定させない |
| 競合不在を根拠にした楽観主張 | `competitive-absence-audit` を併用 | CLAUDE.md「Competitive Absence Audit Rule」遵守 |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 「本当に？」だけ問い「だから何？」を省く | 両方向で検証する（往復セット） | 片方向では論理飛躍を見逃す |
| 暗黙前提を書き出さずに分析開始 | ステップ1で必ず3つ以上書き出す | 前提を可視化しないと疑えない |
| 立場ずらしを「顧客視点」だけで済ます | 最低2つ以上の異なる立場を試す | 1視点では「多角的」と言えない |
| 否定的データを無視する（確証バイアス） | 否定的データほど厚く扱う | 確証バイアスは批判的思考の最大の敵 |
| データ不十分なまま「採択」と結論 | データ不足は「保留」または「追加検証」と明示 | 印象論で結論しない |
| 感情的な反論を「批判的」だと誤解 | 感情を排除し事実とロジックで反論する | 批判 ≠ 否定。妥当性検証である |
| 競合不在を根拠にした楽観主張を素通り | `competitive-absence-audit` 発動を提案 | CLAUDE.md ルール遵守 |

## 連携スキル

- `logical-thinking`：MECE分解で根拠を整理する基礎部分はこちらを参照
- `pyramid-structure`：検証後の結論をピラミッド構造で説明資料化する場合
- `lateral-thinking`：批判的検証で「既存枠の中で答えがない」と判明した場合に切替
- `competitive-absence-audit`：競合不在を根拠にした楽観主張のチェック専用スキル
- `as-is-to-be-gap-solution`：検証結果からギャップ分析→解決策に進める場合

## 5状態出力契約

外部API・サブプロセス・外部設定依存を持たないため、CLAUDE.md「5状態出力契約」の対象判定基準には該当しない。ただし以下のマーカーを使うこと：

- `[OK:0件]`：すべての仮説が却下になった場合（採択結論なし、要再構築）
- `[PARTIAL:N成功/M失敗]`：複数仮説のうち一部のみ採択された場合（部分採択）
- `[SKIP]`：データ不十分で検証を保留した仮説がある場合／検証対象が不明確で批判的検証を始められない場合（ステップ1に戻る、試みる前の前提不足）

※本スキルは外部API・サブプロセスを持たないため、`[FAIL]`（試行後の失敗）は通常発生しない。前提不足は全て `[SKIP]` で扱う。
