# inside-sales-sdr-bdr スキル

インサイドセールス（オンライン・電話・メール中心のリモート営業）のうちSDR（インバウンド型）とBDR（アウトバウンド型）の組織設計と運用を体系化する。

## 適用条件

- このスキルを使う場面：
  - BtoB営業組織でインサイドセールス（IS）を立ち上げたい／改善したい
  - SDRとBDRの役割分担を整理し、KPIを設計したい
  - フィールドセールスとの連携フロー（ナーチャリング→商談引き渡し）を構築したい
  - SaaS／IT／製造業BtoB等で新規開拓・商談創出のプロセスを再設計したい
- このスキルを使わない場面：
  - リード定義（MQL/SQL）と中間KPI設計 → `lead-definition-mql-sql`（前段で実施）
  - リード育成施策（メール／コンテンツ／ウェビナー詳細） → `lead-nurturing`
  - マーケと営業全体のフロー（4要素・5ステップ） → `marketing-sales-workflow`
  - 営業資料テンプレート設計 → `sales-deck-template`
  - 個別商談の提案書作成 → `proposal-builder`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 事業形態 | 必須 | BtoB SaaS／BtoB製造業／BtoBコンサル 等 | 「BtoB SaaS（HRTech）」 |
| 現在の営業体制 | 必須 | フィールドセールスのみ／IS既存／ゼロから立ち上げ | 「FSのみ・IS未導入」 |
| 月間リード件数（インバウンド） | 任意 | 問い合わせ／資料請求／ウェビナー登録 等 | 「月150件のインバウンド」 |
| ターゲット企業規模 | 任意 | SMB／中堅／エンタープライズ | 「エンタープライズ中心」 |
| 解決したい課題 | 必須 | 商談化率低い／新規開拓できない／リード捌けない 等 | 「インバウンドはあるが商談化率10%以下」 |

## 実行ステップ

### ステップ1：インサイドセールスとフィールドセールスの違いを整理する

|  | インサイドセールス | フィールドセールス |
|---|---|---|
| 営業手法 | オンライン・電話・メール | 対面営業（訪問・対面商談） |
| 目的 | 見込み顧客の獲得・育成、商談の創出 | 顧客との信頼関係構築、クロージング（契約締結） |

**インサイドセールスの目的**：
- 営業の効率化：移動時間を削減し、より多くのリード（見込み顧客）にアプローチできる
- 商談の質を向上：顧客の興味・関心を高めた状態でフィールドセールス（訪問営業）へつなぐ
- コスト削減：出張費や交通費がかからないため、営業コストを抑えられる
- リードの育成（ナーチャリング）：購入を検討中の顧客と継続的にコミュニケーションを取り、最適なタイミングで提案を行う

### ステップ2：SDRとBDRの違いを5観点で整理する

|  | SDR（Sales Development Representative） | BDR（Business Development Representative） |
|---|---|---|
| アプローチ手法 | インバウンド（問い合わせや資料請求などのリード） | アウトバウンド（未接触のターゲット企業） |
| ターゲット | すでに関心を持っているリード | まだ接点のない企業や担当者 |
| 手法 | メール、電話、SNS、ウェビナーのフォローアップ | コールドコール、コールドメール、イベント参加 |
| 目的 | 見込み顧客の関心を高め、商談化 | 新規開拓による商談機会の創出 |
| 活用される企業 | 主にSMB／BtoB SaaS、IT、マーケティング施策が強い企業 | 主にエンタープライズ／BtoB企業全般、新規開拓を重視する企業 |

**判定基準**：
- インバウンドリード（問い合わせ／資料請求／ウェビナー参加）が月一定数以上ある → SDR組織が有効
- ターゲット企業が明確で、こちらから能動的に接触したい → BDR組織が有効
- 両方該当する大企業は SDR・BDR 併設が一般的

### ステップ3：SDRの役割・業務・KPIを設計する

#### 役割
- 主な役割：インバウンドの対応を主とした **反響型** のインサイドセールス組織

#### 業務内容
- インバウンドリードの対応：ウェブサイトや広告を通じて自社に興味を持ったリードへ初期対応
- リードクオリフィケーション：BANT基準（Budget予算／Authority決裁権／Needs必要性／Timeframe導入時期）を用いてリードの質を判断し、商談に繋げる

#### SDRのプロセス
```
インバウンドリード → 初期接触 → BANT基準でクオリフィケーション → 商談化
                       ↑                  ↑
                    リード数             商談化
```

#### KPI設定と評価指標
- **コール数／メール送信数**：行動量の把握
- **商談化率**：商談の獲得に至ったリードの割合
- **リード転換率**：リードが商談に転換される割合

### ステップ4：BDRの役割・業務・KPIを設計する

#### 役割
- 主な役割：主にコールドコールによる新規市場の開拓とターゲット企業への接触を目指す **アウトバウンド型** のインサイドセールス組織

#### 業務内容
- ターゲット企業のリスト作成：特定の業界やニーズに基づく企業リストを選定
- パーソナライズドアウトリーチ：LinkedInやメールでターゲット企業への接触

#### BDRのプロセス
```
ターゲット企業リスト → パーソナライズドアウトリーチ → 商談化
        ↑                       ↑
     接触回数                  商談数
```

#### KPI設定と評価指標
- **ターゲット企業への接触回数**：アウトリーチ量の把握
- **商談化率**：リードが実際の商談に転換される割合
- **受注率**：BDRが関与した商談の受注に繋がった割合

### ステップ5：成功するインサイドセールスの3要素を設計する

| 要素 | 内容 |
|---|---|
| 要素01：シナリオ設計 | **各フェーズのコミュニケーション設計**：初回接触から商談化までのフローを設定し、適切なタイミングで適切な手法（電話、メール、SNS）を使用 |
| 要素02：リードナーチャリング | **スコアリングシステム**：リードの興味関心度に応じてホットリードを特定し、最適なアプローチを選択／**コンテンツ提供**：メールマーケティングやウェビナーを活用して関係性を深める |
| 要素03：顧客インサイトの分析 | **データ活用**：過去のデータを分析して最適なアプローチを見つける／**AI活用**：効果的なトークスクリプトを生成 |

3要素は順序関係（シナリオ設計 → ナーチャリング → インサイト分析）で運用し、データを次のシナリオ改善に還元するループを回す。

### ステップ6：組織設計とマーケティング連携を整理する

#### 役割分担と連携
- SDRとBDRはリード獲得と商談獲得の初期段階を担当
- インサイドセールスはその後、商談を進め、ナーチャリングし、フィールドセールスに引き渡す

#### マーケティングとの連携
- リードジェネレーションはマーケティング活動と密接に連携
- 例：ウェビナーやホワイトペーパーを活用してリードを集め、フィールドセールスに連携する

#### 成功事例（参考）
| 事例 | 業種 | 施策 | 成果 |
|---|---|---|---|
| ケース01 | SaaS企業 | LinkedInターゲティング、パーソナライズドメール | 商談数前年比150%増、成約率30%向上 |
| ケース02 | BtoB製造業 | ウェビナー活用、ホワイトペーパー配布後のフォローコール強化 | リード獲得単価50%削減、商談転換率20%向上 |

※成果数値はあくまで事例値であり、自社の数値目標として設定する場合は社内ベンチマーク・業界平均と照らして妥当性を検証すること。

### ステップ7：ツール選定と運用ポイントを決める

#### ツール・テクノロジー活用
| # | カテゴリ | 例 | 用途 |
|---|---|---|---|
| 01 | CRM | Salesforce, HubSpot | 営業プロセスの可視化とデータ管理 |
| 02 | オートメーションツール | Marketo, Pardot | マーケティングオートメーション、メールマーケティング |
| 03 | データ分析 | ABテストツール／AIスクリプト最適化ツール | ABテストやAIによるトークスクリプトの最適化 |

#### 成果を最大化するための運用ポイント
- **ターゲティング**：ICP（Ideal Customer Profile＝理想的な顧客像）の明確化、データを基にしたターゲティング
- **トレーニング**：ロールプレイング研修や定期的なフィードバックにより、スキル向上

### ステップ8：組織運用3ポイントでチェックする

最後に下記3ポイントで運用設計が完成しているか自己点検する。

1. **役割と業務の違い**
   - SDRはインバウンドリードの初期対応とクオリフィケーションを担当
   - BDRはアウトバウンドによる新規市場開拓とターゲット企業への接触を担当

2. **KPI設定と評価**
   - SDR：コール数や商談化率、リード転換率
   - BDR：ターゲット企業への接触回数、商談化率、受注率

3. **成功の要素（ステップ5の3要素と一致）**
   - シナリオ設計：各営業フェーズにおける適切なアプローチ（電話、メール、SNSなど）の設計
   - リードナーチャリング：スコアリングシステムでホットリード特定／メールマーケやウェビナーで関係性を深める
   - 顧客インサイトの分析：データ活用とAIを活用したトークスクリプト最適化でリードをホットに保つ

   ※ツール選定（CRM／オートメーション／データ分析）は要素ではなく「実装手段」としてステップ7で扱う。

## 判断基準

| 状況 | 判断 |
|---|---|
| インバウンドリードが月100件以上ある | SDR組織を最優先で立ち上げる |
| ターゲット企業リストが明確で能動的接触が必要 | BDR組織を立ち上げる |
| エンタープライズ＋SMB両方を狙う | SDR・BDR併設が一般的 |
| インサイドセールスとフィールドセールスの引き渡しがうまくいかない | ステップ5の「シナリオ設計」と「成功要素3」を再点検 |
| SDR/BDRのKPIが「コール数」だけで質が下がる | 商談化率・受注率を必ずKPIに含める（行動指標＋成果指標のセット） |
| マーケと営業の連携が弱い | `marketing-sales-workflow` で全体フローを再設計してから本スキルに戻る |
| 個別リード育成施策の詳細が必要 | `lead-nurturing` に引き継ぐ |
| MQL/SQL定義が曖昧 | `lead-definition-mql-sql` で前段の定義を確定させてから本スキルに戻る |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| SDRとBDRの役割を曖昧にしたまま立ち上げる | 5観点（アプローチ／ターゲット／手法／目的／活用企業）で明確に分離 | 役割が混在すると評価指標が機能しない |
| KPIを「コール数」だけに設定する | 行動指標（コール数・接触回数）＋成果指標（商談化率・受注率）をセットで設定 | 行動だけ追うと質が下がる／成果だけ追うと因果が見えない |
| ツール（CRM）を導入してから運用フローを設計する | 運用フロー（シナリオ設計）を先に設計してからツール選定 | ツール先行は形骸化する |
| BANT基準を使わずに「肌感」でクオリフィケーション | BANT（Budget／Authority／Needs／Timeframe）を必ず適用 | 商談の質が安定しない |
| マーケと連携せずIS単独で運用する | マーケのリードジェネレーション（ウェビナー／WP）と接続して運用する | リード枯渇の原因になる |
| トレーニングなしでスクリプトを配って運用 | ロールプレイング研修＋定期FBを必須化 | 個人差で成果がブレる |

## 連携スキル

- **前段（リード定義段階）**：`lead-definition-mql-sql`（MQL/SQL定義と中間KPI）
- **前段（マーケと営業の全体フロー）**：`marketing-sales-workflow`
- **並走（リード育成施策）**：`lead-nurturing`
- **並走（コンテンツ供給）**：`whitepaper-content-design` ／ `webinar-design`
- **後段（商談以降）**：`sales-playbook`（営業相談オーケストレータ）／ `proposal-builder`（提案書）／ `sales-script-builder`（商談トーク）
- **ターゲティング深化**：`persona-design`（BtoBペルソナ）／ `stp-analysis`（セグメンテーション）
