# hp-lp-distinction-design

HP（ホームページ）とLP（ランディングページ）の**使い分け判断と連携設計のみ**を担当するスキル。6項目比較表＋連携2パターン＋GA4クロスドメイン計測の3点に責任範囲を限定し、制作工程は他スキルへ引き継ぐ。LP制作判断になった時点で `lp-create` へ、HP単体制作になった時点でユイ（web-designer）へバトンを渡すのが本スキルの境界線。

## 適用条件

- 使う場面：
  - クライアントがHPを持っているがLPを作るべきか相談
  - LPを作ったがHPとの導線が整理されていない
  - HP・LPが別ドメインで運用されており計測が分断されている
  - HPリニューアルとLP新設を同時に検討中で役割分担を整理したい
- 使わない場面：
  - LP単体の制作フロー全般 → `lp-create`
  - LPのセクション/スライドごとのデザイン仕様 → `lp-design-system`
  - LPのコピー設計 → `ad-copy-7-angles` ／ Koto（copywriter）
  - LPO（既存LPの改善） → `lpo-improvement-design`
  - HP単体のサイト構造設計 → ユイ（web-designer）

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 現状資産 | 必須 | HPのみ／LPのみ／両方／なし | 「HPあり、LP未制作」 |
| マーケ目的 | 必須 | ブランディング／集客／CV獲得／キャンペーン | 「特定商品のCV獲得」 |
| 想定流入経路 | 推奨 | 検索／広告／SNS／メール／直接 | 「Meta広告→LP」「検索→HP」 |
| ターゲット数 | 任意 | 単一商品向けかブランド全体か | 「単一サービス向け」 |
| ドメイン構成 | 任意 | 同一ドメイン／サブドメイン／別ドメイン | 「HP=koujitsu.co.jp、LP=lp.koujitsu.co.jp」 |

## 実行ステップ

### ステップ1：HPの役割を整理する（HPとは何か）

- 入力：マーケ目的
- 処理：HPの3つの目的と2つの特徴を確認する
  - **HPの3目的**：
    1. **情報提供**：商品・サービス・会社情報など幅広い情報を掲載
    2. **ブランディング**：企業イメージをユーザーに伝える
    3. **ナビゲーション**：訪問者を目的のページへ誘導する
  - **HPの2特徴**：
    1. 複数のリンクやページへのアクセスを可能にする多機能設計
    2. 継続的な更新が求められる（ニュース、ブログなど）
- 出力：HPが必要かどうかの暫定判断（包括的な情報提供＋長期ブランディングが必要なら必要）

### ステップ2：LPの役割を整理する（LPとは何か）

- 入力：マーケ目的、想定流入経路
- 処理：LPの2つの目的と3つの特徴を確認する
  - **LPの2目的**：
    1. **コンバージョン促進**：購入、資料請求、会員登録など特定アクションを促す
    2. **広告の着地先として利用**：リスティング広告／SNSキャンペーンのリンク先として設計
  - **LPの3特徴**：
    1. 訴求力が高く、特定のアクションに誘導する設計
    2. 一時的なキャンペーンやプロモーションで利用されることが多い
    3. ページ内で完結するシンプルな構造
- 出力：LPが必要かどうかの暫定判断（特定CV獲得＋広告連動が必要なら必要）

### ステップ3：6項目比較表でHP/LPの違いを明示する

- 入力：ステップ1・2の判断
- 処理：以下6項目で比較表を作成し、クライアントに提示
  - 目的：HP=包括的な情報提供／ブランディング　LP=購入や登録など特定行動を促進
  - 構成：HP=複数ページ・多機能　LP=単一ページ・特定テーマに集中
  - ターゲット：HP=幅広い訪問者層　LP=特定の目的やニーズを持つ訪問者
  - 更新頻度：HP=定期的に更新　LP=短期的な運用が多い
  - 訴求力：HP=概要や幅広い情報提供に優れる　LP=メリット強調・行動促進
  - アクセス経路：HP=検索・直接訪問・SNS　LP=広告・メールキャンペーン・特定リンク経由
- 出力：6項目比較表（クライアント説明用）

### ステップ4：連携2パターンから選択する

- 入力：ステップ3の比較結果、現状資産
- 処理：HPとLPの連携方法を以下2パターンから選択
  - **パターンA：HP内にLPを配置**
    - 概要：HPの一部としてLPを特設ページにし、キャンペーンや特定商品の情報を訴求
    - 適用：単発キャンペーン・既存HPに集約したい場合
    - URL例：`koujitsu.co.jp/campaign/xxx`
  - **パターンB：HPからLPへ誘導**
    - 概要：HPのトップページやバナー広告からLPへのリンクを設置
    - 適用：LPを別ドメインまたはサブドメインで運用する場合
    - URL例：HP=`koujitsu.co.jp`、LP=`lp.koujitsu.co.jp` または `landingpage.com`
- 判断基準：
  - SEO評価をHPに集約したい → パターンA推奨
  - LPの自由度を最大化（広告クリエイティブと一体化）したい → パターンB推奨
  - 単一CV特化で計測しやすくしたい → パターンB推奨
- 出力：選定した連携パターン

### ステップ5：GA4クロスドメイン計測の必要性を判断する

- 入力：ステップ4の連携パターン、ドメイン構成
- 処理：ドメイン構成に応じてGA4設定を判定
  - **同一ドメイン内（`koujitsu.co.jp/lp/xxx`）**：標準GA4タグ設置のみで計測可能
  - **サブドメイン（HP=`koujitsu.co.jp`、LP=`lp.koujitsu.co.jp`）**：同じGA4アカウント・同じタグで計測可能
  - **完全別ドメイン（HP=`koujitsu.co.jp`、LP=`landingpage.com`）**：GA4の**クロスドメイン設定**を必須とする
- クロスドメイン設定が必要な根拠：HP・LP横断でユーザーの動きを計測するため。LPのCVRがHPコンテンツに左右される場合、計測分断は致命的
- 出力：必要なGA4設定タスク（標準／クロスドメイン）

### ステップ6：HP・LP活用事例を提示する

- 入力：マーケ目的
- 処理：以下4類型から該当する事例パターンを提示
  - **HP成功事例**：
    - 企業のブランドサイト：トップページのデザイン刷新で訪問者数を30%増加
    - ショッピングサイト：サービスメニューを整理し、直帰率を20%改善
  - **LP成功事例**：
    - 新商品プロモーションLP：簡潔なデザインと目立つCTAボタンでCVR25%向上
    - 資料請求LP：フォーム項目を減らし、完了率15%向上
- 出力：参考事例（クライアント説明・社内検討資料用）

### ステップ7：引継ぎゲート（LP制作判断 / HP単体制作判断）

- 入力：ステップ1〜6で確定した使い分け判断・連携パターン・GA4設定方針
- 処理：本スキルの責任範囲はここまで。**制作工程には踏み込まず、判断結果に応じて以下のいずれかに引き継ぐ**
  - **LP制作判断になった場合**：`lp-create` へ引継ぎ（必ず Route 判定から開始する）
    - Route F（フル制作）／Route S（戦略のみ）／Route C（コピーのみ）／Route D（デザインのみ）／Route I（実装のみ）の判定は `lp-create` の責任範囲
    - CLAUDE.md「LP案件の起動順序ルール」遵守のため、本スキルから直接 LP 実装フェーズには入らない
  - **HP単体制作判断になった場合**：ユイ（web-designer）へ委任
  - **HP・LP両方制作判断になった場合**：それぞれ上記2経路で別個に発注。連携設計（ステップ4・5の決定事項）を引継ぎ材料として渡す
- **本スキル内で制作工程の手順説明はしない**（CLAUDE.md「LP案件の起動順序ルール」遵守のため）
- 出力：引継ぎ先と引継ぎ材料（連携パターン・GA4設定・6項目比較表）

> 補足：従来の「7ステップ制作フロー（目的明確化→構成設計→デザイン→コーディング→テスト→公開→運用）」は `lp-create` Route F が同等の役割を内包しているため、本スキルでは記述しない。HP/LP どちらを作るか決まった段階で、各々の専門スキルに引き継ぐ。

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 包括的な情報提供＋長期ブランディングが必要 | HP優先 | HPの3目的の主軸 |
| 特定CV獲得＋広告連動が必要 | LP優先 | LPの2目的の主軸 |
| 単発キャンペーンを既存HPに集約したい | パターンA（HP内にLP配置） | SEO評価をHPに集約できる |
| LP自由度を最大化したい・広告と一体化 | パターンB（HPからLP誘導・別ドメイン） | LP単体最適化が可能 |
| HP・LPが別ドメイン運用 | GA4クロスドメイン必須 | 計測分断を防ぐ |
| HP・LPがサブドメイン運用 | 標準GA4タグのみで可 | 同じアカウントで計測可能 |
| 既存HPがある×LP制作依頼 | ステップ4で連携パターン確認必須 | 連携設計なしのLP制作は計測・SEO面で不利 |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 「LPを作りたい」→ いきなりLP制作着手 | まずHP/LPの役割整理→連携パターン決定→制作 | 既存HPとの連携・GA4計測を後から追加すると工数増 |
| 別ドメインLPを作ってGA4設定を後回し | ステップ5でクロスドメイン設定を最初に判断 | 計測分断はリリース後に気付くと再計測不可 |
| HPとLPで重複コンテンツ | 役割を6項目比較表で明示し責任範囲を分離 | SEO的にも重複コンテンツは評価減 |
| 単発キャンペーンを別ドメインで作る | パターンA（HP内配置）を検討 | SEO評価集約・運用工数減 |
| クライアントが「HPもLPも欲しい」と言ったら全部作る | ステップ1・2で「本当に両方必要か」を判断 | 不要なHPは運用コストが膨らむ |
| LP単体最適化したいのに同一ドメイン強制 | パターンB＋別ドメイン or サブドメインで自由度確保 | 広告と一体化したクリエイティブが組めない |

## 連携スキル

- 後段：`lp-create`（LP制作フロー全般）／`lp-design-system`（LPセクション設計）
- 前段：`marketing-mix-4p4c`（4P/4Cで全体マーケミックス確認）／`stp-analysis`（STPでターゲット明確化）
- 関連：`ga-gsc-diagnosis`（既存サイトのGA4/GSC診断）／`web-design-guidelines`（UI監査）
- 改善：`lpo-improvement-design`（公開後LP改善）／`ui-ux-improvement-fundamentals`（HP/LPのUI/UX改善）
