# roles-responsibilities スキル

役割と責任範囲 = 組織内で各メンバーや部門が担うべき業務や責任を具体的に示したもの。明確な役割と責任範囲が、業務効率やチームワークを向上させ、組織全体のパフォーマンスを底上げする。事業戦略スクール「知足」のフレームワーク準拠。

## なぜ重要か（3つ）

1. **業務の重複や抜け漏れを防ぐ**
2. **各メンバーが自分の役割を理解し、目標達成に集中できる**
3. **問題発生時に迅速な対応が可能になる**

## 役割と責任範囲を明確にする目的（3つ）

| # | 目的 | 内容 |
|---|---|---|
| 01 | **効率的な業務遂行** | 誰が何をするかを明確化し、無駄を削減 |
| 02 | **責任の所在を明確化** | 問題解決や意思決定が迅速になる |
| 03 | **チームワークの強化** | 各メンバーが役割を尊重し、連携が円滑に進む |

## 主な4要素

| # | 要素 | 内容 | 例 |
|---|---|---|---|
| 1 | **役割** | 職務内容や担当業務を定義 | 営業担当、プロジェクトリーダー |
| 2 | **責任範囲** | 役割に伴う具体的なタスクや成果物を明示 | 新規契約獲得、進行管理 |
| 3 | **権限** | 意思決定や業務遂行に必要な裁量の範囲を明確化 | 予算の配分や調整に関する決定権 |
| 4 | **報告ライン** | 誰に対して報告・相談するかを明確に設定 | プロジェクトの進捗状況を部門マネージャーに週次で報告 |

## 適用条件

- このスキルを使う場面：
  - ボードメンバー・部長職等の役職持ちメンバーの責任範囲を明文化したい
  - 業務の重複や抜け漏れが発生している問題を解決したい
  - 権限と報告ラインを整理したい
  - 役職別の評価指標を紐付けたい
- このスキルを使わない場面：
  - 組織図全体の設計（4Q組織ロードマップ） → `organization-planning`
  - PJ単位の体制管理 → `project-team-structure`
  - 評価制度の策定 → `evaluation-system-design`（別スキル）
  - 給与レンジ決定 → `salary-range-design`（別スキル）
  - キャリアロードマップ → `career-roadmap-development`（別スキル）

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 対象組織 | 必須 | 会社名・部門名 | 「ABC株式会社・営業部」 |
| 組織目標 | 必須 | 組織全体の目標 | 「売上20%増」「新製品開発の成功」 |
| 対象役職リスト | 必須 | 責任範囲を定義したい役職 | 「CEO／COO／執行役員／部長／マネージャー」 |
| 役職への配属者 | 任意 | バイネーム情報 | 「CEO：山田／COO：鈴木」 |
| 既存の役割定義 | 任意 | 現状あれば | 「営業部の責任範囲は『新規開拓』のみ定義済」 |

## 6ステップ（実行フロー）

| ステップ | 内容 | 例 |
|---|---|---|
| 1. **組織目標を設定** | 組織の目標を明確にする | 「売上20%増」「新製品開発の成功」 |
| 2. **必要な役割を洗い出す** | 部門や役職ごとに、目標達成に必要な業務やポジションをリストアップ | CEO／COO／事業部長／PM／品質管理 等 |
| 3. **責任範囲を定義** | 役割ごとに担当する業務や目標を具体化 | 営業部：「新規顧客50件獲得」／マーケ部：「SNSフォロワー数20%増加」 |
| 4. **権限と報告ラインを設定** | 各役割の意思決定権と業務の報告先を明示 | 「予算配分決定権あり／部門長に週次報告」 |
| 5. **文書化と共有** | 作成した役割と責任範囲をドキュメントにまとめ、全員に共有 | 共有ドキュメント化＋定例MTGで周知 |
| 6. **定期的な見直し** | 組織の成長や市場環境の変化に応じて柔軟に更新 | 四半期ごとにレビュー |

## 実行ステップ

### ステップ1：組織目標の確認（プロセス1）
- 入力：組織情報、目標
- 処理：
  1. 組織全体の目標をSMART原則で言語化（→ `smart-goal-setting` 連携）
  2. 達成基準を3〜5項目で具体化
- 出力：SMART化された組織目標

### ステップ2：必要な役割の洗い出し（プロセス2）
- 入力：組織目標
- 処理：
  1. 目標達成に必要な機能を全部書き出す（営業／マーケ／開発／管理／CS／人事 等）
  2. 各機能を担当する役職をリストアップ（重複・抜け漏れチェック）
  3. 必要な役職数を仮置き（規模・売上に応じた人数）
- 出力：必要役職リスト

### ステップ3：責任範囲の定義（プロセス3）
- 入力：必要役職リスト
- 処理：各役職に対し以下4要素を定義
  1. **役割**：職務内容・担当業務
  2. **責任範囲**：具体的タスク・成果物（数値目標含む）
  3. **権限**：裁量の範囲（金額・スコープ）
  4. **報告ライン**：誰に・いつ・何を報告するか
- 出力：役職別の4要素定義表

### ステップ4：アウトプット表の作成（推奨6列）
- 入力：4要素定義表
- 処理：以下6列のアウトプット表を作成（特にボードメンバー・部長職に適用）
  | 列名 | 内容 |
  |---|---|
  | 氏名 | バイネーム |
  | 役職 | 役職名（CEO／COO／部長等） |
  | 理想の姿 | 本人のキャリアビジョン・目指す姿 |
  | 期待 | 組織から本人への期待事項（複数項目） |
  | 責任範囲（組織） | 組織レベルでの責任（売上・品質・人事等） |
  | 評価指標 | 数値で測れる評価指標（売上・KPI等） |
- 出力：6列アウトプット表（ボードメンバー・部長職向け）

### ステップ5：文書化と共有（プロセス5）
- 入力：4要素定義表＋6列アウトプット表
- 処理：
  1. 共有ドキュメントにまとめる（Notion／Google Docs等）
  2. 全員に共有する場（全社会議・定例MTG）を設定
  3. 各メンバーに自分の役割を音読してもらう（理解度確認）
- 出力：共有完了状態＋共有ドキュメント

### ステップ6：定期的な見直し（プロセス6）
- 入力：運用記録
- 処理：
  1. 四半期ごとにレビュー会を設定
  2. 組織変更・市場変化があれば随時更新
  3. 「業務重複」「権限不足」「責任所在不明」の3課題が発生していないかチェック
- 出力：定期見直しカレンダー＋更新履歴管理

### ステップ7：自己点検（4観点）
- 入力：これまでの設計結果
- 処理：以下4観点で点検
  1. **業務の重複や抜け漏れの有無**：各役割が目標達成に必要な範囲を網羅しているか
  2. **権限と裁量のバランス**：権限が不十分で業務遂行が妨げられていないか
  3. **組織変更や環境変化への対応**：最近の組織変更や市場環境に対応する内容になっているか
  4. **明文化と共有状況**：設定した内容が全メンバーに適切に共有され、運用されているか
- 出力：4観点チェック結果＋改善計画

## 設定例（参考）

### 例1：営業担当者 vs プロジェクトマネージャー

| 要素 | 営業担当者 | プロジェクトマネージャー |
|---|---|---|
| 役割 | 新規顧客開拓と既存顧客フォロー | プロジェクト全体の進行管理とチーム調整 |
| 責任範囲 | ・営業目標達成<br>・契約交渉と締結 | ・プロジェクト計画の策定<br>・進捗管理と問題解決 |
| 権限 | 一定の割引決定権、営業プロモーションの提案権 | リソース割り当ての決定権 |
| 報告ライン | 営業マネージャーに報告 | 部門長に報告 |

### 例2：ボードメンバー6列表（事業戦略スクール「知足」原典より）

| 役職 | 期待 | 責任範囲（組織） | 評価指標 |
|---|---|---|---|
| 代表取締役/CEO | 全体責任（戦略、管理、広報、ブランディング、発信活動） | 全社の売上・利益 | 売上及び経常利益 |
| COO | 執行役員引き上げ・フォロー、マーケ戦略、財務戦略 | 全社の売上・労働生産性 | 売上及び経常利益 |
| 執行役員（事業部長） | 事業部の売上、LTV最大化、評価制度構築 | 事業部の売上・利益 | 売上及び経常利益・運用件数 |
| 営業部長 | 新規受注売上、マーケ戦略、顧客管理システム構築 | 営業部の労働生産性、メンバー目標達成 | 売上及び経常利益・案件普及数 |
| 採用マネージャー | 採用人数、養成講座立ち上げ、社内文化醸成 | 人事部、採用目標達成 | 採用人数、新規事業売上 |
| 品質管理部PMO | プロジェクト品質管理 | 全社の品質管理 | 案件LTV・チャーンレート |

## 課題と解決策（3つ）

| 課題 | 解決策 |
|---|---|
| 課題01：責任範囲が曖昧で業務が重複 | 各メンバーの責任範囲を明文化し、役割ごとの優先順位を明確化 |
| 課題02：権限が不明確で意思決定が遅れる | 各役割に必要な裁量と権限を明示し、迅速な対応を促す |
| 課題03：定期的に見直されない | 四半期ごとに役割と責任範囲をレビューし、変更点を反映 |

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 役職定義が「営業部」程度の粒度 | ボード/部長職は4要素＋6列表で詳細化 | 上位役職こそ責任範囲の明確化が必要 |
| 業務重複が発生している | 課題01の解決策（明文化＋優先順位付け） | 責任範囲の曖昧さが原因 |
| 意思決定が遅い | 課題02の解決策（権限と裁量の明示） | 権限不足が原因 |
| 1年以上更新されていない | 四半期レビュー再開 | 環境変化への適応漏れ |
| 組織図はあるが役割定義なし | `organization-planning` の出力を本スキルへ接続 | 組織図単独では運用不能 |
| 評価指標が抽象的（「がんばる」等） | 数値で測れる指標に変換 | 評価不能 |
| 「理想の姿」が未記入 | キャリア面談で本人に書いてもらう | エンゲージメント低下を予防 |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 役職名のみ定義し責任範囲が無い | 4要素（役割/責任範囲/権限/報告ライン）を必ず明文化 | 抽象論で運用不能 |
| 上位役職の理想の姿を聞かない | 6列表で本人のキャリアビジョンも記録 | エンゲージメント低下 |
| 責任範囲が「がんばる」等の抽象表現 | 数値目標で具体化 | 評価不能 |
| 権限の金額・スコープが曖昧 | 「予算100万円まで」等の上限を明示 | 意思決定の停滞 |
| 報告ラインの頻度が「随時」のみ | 「週次／月次」等の頻度を明示 | 報告漏れの温床 |
| 文書化せず口頭共有のみ | 共有ドキュメント化＋全員に共有 | 認識ずれが起きる |
| 1年以上更新しない | 四半期レビューサイクル設定 | 実態と乖離 |
| 業務重複・権限不明確を放置 | 3課題への解決策を即適用 | 業務効率と意思決定が低下 |

## 連携スキル

- `organization-planning`：組織図全体の設計＋4Qロードマップ
- `project-team-structure`：PJ単位の体制管理（人員・役割・コミュ・リソース・進捗）
- `evaluation-system-design`：評価制度の策定（評価指標を本スキルから引き継ぐ）
- `salary-range-design`：給与レンジ決定（役職別評価指標を活用）
- `career-roadmap-development`：キャリアロードマップ＋育成計画
- `recruitment-strategy`：採用戦略（必要な役割の採用基準）
- `smart-goal-setting`：組織目標のSMART化（ステップ1）
- `kgi-kpi-kai-design`：評価指標としてのKPI設計
- `katz-three-skill-approach`：役職別必要スキルとの紐付け
- `mvv-design`：組織の方向性（理想の姿の前提）

## 5状態出力契約

外部API・サブプロセス・外部設定依存を持たないため、CLAUDE.md「5状態出力契約」の対象判定基準には該当しない。ただし以下のマーカーを使うこと：

- `[OK:0件]`：定義対象の役職が0件で作業できない場合（要再ヒアリング）
- `[PARTIAL:N成功/M失敗]`：複数役職のうち一部のみ定義できた場合
- `[SKIP]`：組織目標が未設定で役割定義を保留した場合（前提不足）

※外部API依存なしのため `[FAIL]` は通常発生しない。前提不足は `[SKIP]` で扱う。
