# meeting-cadence-design スキル

会議体（特定の目的を持って複数の人が集まり、具体的な意思決定を行うための会議）の定義・種類・設計・運営を体系的に行い、組織全体の生産性を向上させる。事業戦略スクール「知足」フレームワーク準拠。出所：『急成長を導くマネージャーの型 ~地位・権力が通用しない時代の"イーブン"なマネジメント』長村禎庸著を参考に編集。

## 適用条件

- このスキルを使う場面：
  - 組織の会議体を体系的に設計したい
  - 既存の会議が多すぎて生産性が落ちている
  - マネジャー・役員参加会議体を半期に1度見直したい
  - 会議体ごとの目的・参加者・頻度・時間を明確化したい
  - 「情報共有型」会議をドキュメントベースに置き換えたい
- このスキルを使わない場面：
  - 個別会議のファシリテーション技法 → 一般的なファシリ知識
  - 営業の商談スクリプト → `sales-script-builder`
  - スケジュール管理全般 → `schedule-management`
  - プロジェクト体制 → `project-team-structure`
  - 役員・部長等の恒久的な役割定義 → `roles-responsibilities`
  - 意思決定フレームワーク（合意プロセス／優先順位）→ `decision-making-framework`

## 入力

| 項目 | 必須/任意 | 説明 |
|---|---|---|
| 起動フレーズ | 必須 | 上記トリガーフレーズのいずれか |
| 自社の組織情報 | 必須 | 部門数・社員数・経営層体制 |
| 既存の会議体一覧 | 任意 | 現状の会議名・参加者・頻度（あれば） |
| 主要な課題 | 任意 | 「会議が多い」「決定事項が実行されない」等 |

## 実行ステップ

### ステップ1：会議体の定義と4目的を確認する

**会議体** = 特定の目的を持って複数の人が集まり、具体的な意思決定を行うための会議。同じメンバーが定期的に集まり、重要な決定事項を協議する場として機能。**情報共有や報告を目的とする単なる会議とは異なり、参加者全員の合意によって意思決定が行われることが特徴**。

**4目的**：
| # | 目的 | 内容 |
|---|---|---|
| 01 | 意思決定の促進 | 個々の意見が反映された質の高い意思決定 |
| 02 | 透明性の確保 | 決定過程が明確で、意思決定の透明性が高まる |
| 03 | 業務効率化 | 不要な会議の削減や議論の重複防止 |
| 04 | 成果の最大化 | 各テーマに最適な形式で議論を行い、実行可能な結果を導く |

### ステップ2：5種類の会議体から目的に合うものを選定する

| # | 種類 | 目的 | 例 | 特徴 |
|---|---|---|---|---|
| 01 | 意思決定型会議 | 組織の重要事項について意思決定 | 経営会議・取締役会 | 少人数で開催／決定事項が明確に記録される |
| 02 | 情報共有型会議 | 各部門やプロジェクトの進捗状況を共有 | 部門会議・週次ミーティング | 多部門の参加が多い／双方向の情報交換 |
| 03 | 課題解決型会議 | 特定の課題に対する解決策を議論 | プロジェクト会議・問題解決ワークショップ | テーマに焦点を絞った議論／ブレストやディスカッション多 |
| 04 | 戦略策定型会議 | 中長期的な戦略や方針を立案 | 経営戦略会議・新商品開発会議 | 継続的な運営／将来を見据えた議論が中心 |
| 05 | アドホック型会議 | 特定の緊急課題やトピックに対応する一時的な会議 | 緊急対応会議・特別プロジェクト会議 | 必要に応じて設置／柔軟性が高い |

### ステップ3：会議体を設計する5要素を実行する

| # | 要素 | 内容 |
|---|---|---|
| 01 | 目的の明確化 | 会議体ごとに目的を定義（戦略立案／進捗共有／緊急課題対応） |
| 02 | 参加者の選定 | 必要なスキルや情報を持つメンバーを参加させる（意思決定型では意思決定権者を含める） |
| 03 | 会議の頻度 | 会議の目的に応じて適切な頻度を設定 |
| 04 | 議題の事前共有 | 参加者に議題を事前に共有し、議論の効率性を高める |
| 05 | フォローアップの仕組み | 決定事項やアクションプランを議事録に記録し、進捗確認を行う |

### ステップ4：会議体運営の4ポイントを徹底する

| ポイント | 内容 |
|---|---|
| 時間管理 | アジェンダに基づき、時間内で会議を完了 |
| 進行役の明確化 | 議長やファシリテーターを設定し、議論を整理 |
| 議事録の作成 | 決定事項や担当者を明確化し、共有 |
| ツールの活用 | Zoom／Teams／タスク管理ツールを活用して効率化 |

**4運営メリット**：
| # | メリット | 内容 |
|---|---|---|
| 01 | 意思決定の迅速化 | 必要なメンバーが集まり、短時間で結論を出せる |
| 02 | 情報共有の強化 | 部門間の連携がスムーズになる |
| 03 | 業務効率化 | 生産性の低い会議を排除し、価値ある議論を行う |
| 04 | チーム連携の向上 | 会議を通じて、目標の一貫性が確保される |

### ステップ5：9列の会議体一覧テンプレで現状棚卸し

**マネジャーや役員が参加している会議体は、本当に必要か？を半期に1度程度、定期見直し。**

| 列 | 内容 |
|---|---|
| 会議名 | 例：月曜CS／金曜CS／マーケ事業部会議／案件共有会／勉強会／1on1 等 |
| 目的 | 例：週頭のチェックイン／案件の深掘り／事業部タスク進捗共有 |
| 目標 | 例：今週やることが明確／成果が出てない案件の課題が明確 |
| アジェンダ | 例：チェックイン／各案件の毎週やること／予実進捗共有 |
| 参加者 | 例：マーケ全員／インターン／協力会社／早坂とメンバー |
| 頻度 | 例：週1／隔週／月1 |
| 時間 | 例：30分／45分／1時間／2時間 |
| 評価 | 開催時間変更／継続／頻度変更／統合・廃止 |
| コメント | 改善メモ：例「ざっとリスクを拾うための会議にして30分ですませる」「テーマを絞る＋1.5時間にする」 |

### ステップ6：3つの見直しアプローチで会議を圧縮する

1. **目的がかぶっているものを集約**する
2. **会議でなくドキュメントベースの報告に変更**する
3. **「情報共有型」の会議を極力減らす**

### ステップ7：成功の3ポイントと4課題分析STEPで完了

**成功の3ポイント**：
1. 目的に合致した会議体の選定と設計
2. 進行役やツールの活用で効率性の向上
3. 定期的な振り返りと改善を行い、持続可能な運営を実現

**4課題分析STEP**：
| STEP | 確認項目 |
|---|---|
| STEP01 目的の明確化 | 現在の会議体が組織の目標に合致しているか？ |
| STEP02 参加者の適切性 | 必要なスキルや権限を持つメンバーが参加しているか？ |
| STEP03 時間管理と効率性 | 会議時間が適切で無駄がないか？ |
| STEP04 フォローアップ体制 | 決定事項や進捗確認が効果的に行われているか？ |

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 同じ参加者で目的が似た会議が複数 | 集約・統合 | 重複会議は時間の無駄 |
| 「情報共有型」会議で双方向議論なし | ドキュメント報告に置換 | 会議でなくとも目的達成可能 |
| マネジャー・役員参加会議が週5以上 | 半期に1度の見直しで削減対象を特定 | 経営層の時間が会議に侵食されすぎ |
| 議事録なし・フォローアップなし | 議事録作成と進捗確認を即時導入 | 決定事項が実行されない最大原因 |
| 会議時間が毎回延長 | アジェンダと時間配分を事前に明確化 | 時間管理の基本欠如 |
| 進行役不在で発言の偏り | 議長／ファシリテーター明確化 | 議論の整理ができない |
| 緊急課題が頻発 | アドホック型会議として正規化、恒常的な会議体には混入させない | 通常会議の生産性低下を防ぐ |
| 会議体の数が増え続けている | 半期に1度の棚卸しで「廃止候補」を特定 | 自然増加した会議は半数以上が不要 |
| 「会議体は触れない／聖域」と感じる | 半期に1度の定期見直しを制度化 | 聖域化は組織硬直化の主因 |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 会議体を一度設計して放置 | 半期に1度の定期見直し | 会議体は自然増加し、半数以上が不要化する |
| 「情報共有型」会議を増やしすぎる | ドキュメントベース報告に極力置換 | 会議は意思決定の場、共有は非同期で十分 |
| 議事録なしで会議実施 | 議事録（決定事項＋担当者＋期限）を必ず作成 | 決定事項が実行されない最大原因 |
| アジェンダ事前共有なし | 議題を事前共有、参加者に準備を求める | 議論の効率性激減、時間延長の主因 |
| 進行役不在で発言の偏り | 議長／ファシリテーターを明確化 | 議論の整理不可、結論が出ない |
| 経営層の会議が週5以上で時間侵食 | 半期見直しで30%以上削減を目標 | 経営層の戦略思考時間が会議に消える |
| 緊急課題を通常会議に混入 | アドホック型として正規化 | 通常会議の生産性低下を防ぐ |
| 9列棚卸しテンプレを使わずなんとなく見直し | 必ず9列テンプレで全会議体を可視化 | 可視化なしの見直しは形骸化 |

## 連携スキル

- `decision-making-framework`：会議における意思決定プロセス（5プロセス：共有→拡散→混合→収束→纏め）
- `schedule-management`：会議スケジュール全体の最適化
- `roles-responsibilities`：会議参加者の役割・権限の定義
- `project-team-structure`：プロジェクト単位の体制と会議体
- `organization-planning`：組織体制と会議体の整合
- `growth-phase-strategy`：成長フェーズに応じた会議体の進化（初期フェーズはフラット少数→拡大フェーズはC-suite会議追加）
- `goal-execution-system`：会議体を含むモニタリング＋FBループの仕組み化
- `pdca-cycle`：会議体運営のPDCA改善
- `goal-cascade-kpt-1on1`：1on1／月次レビュー／全社目標発表会など目標カスケード関連会議の運営設計（cadence枠組設計はこちら、目標進捗管理＋KPTでのFBは `goal-cascade-kpt-1on1` 側）
- `success-cycle-relationship-quality`：朝会・1on1の運営設計（POINT 02 心理的安全性確保／POINT 04 適切なフィードバックの場として）
- `credo-language-design`：朝会クレド唱和／クレド見直し会の運営設計
- `ms-matrix-talent-grid`：人材ポートフォリオ振り返り会議体の設計

## 5状態出力契約

このスキルは外部API・サブプロセス実行・外部設定依存のいずれも持たないため、CLAUDE.md「スキル出力の5状態契約」の対象判定基準には該当しない。ただし以下のケースで明示マーカーを使うこと：

- `[OK:0件]`：自社の会議体情報が不足し、フレームワーク説明のみで終了した場合

外部API・サブプロセスを持たないため、`[FAIL]`（試行後の失敗）は通常発生しない。
