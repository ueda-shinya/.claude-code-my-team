# As-is To-be / Gap Solution スキル

現状（As-Is）と理想（To-Be）の差分を可視化し、ギャップ（課題）→ ソリューション（解決策）→ 優先順位の順に施策を導出するギャップ分析手法。1〜3項目の問題を起点に、対応する理想・課題・解決策を1対1で紐付ける構造化分析。

## 適用条件

- このスキルを使う場面：
  - 業務プロセス改善・組織課題の整理
  - 事業戦略立案前の現状診断
  - DX推進・システム刷新の要件定義
  - 中長期計画策定時の現状把握
- このスキルを使わない場面：
  - 競合との比較分析 → `3c-analysis` `swot-analysis`
  - 市場機会の発見 → `pest-analysis`
  - 価値提案の設計 → `value-proposition`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 起動フレーズ | 必須 | 上記トリガー語 | 「ギャップ分析して」 |
| 分析対象 | 必須 | 何の現状と理想を比べるか | 「営業組織」「マーケ部門」「事業全体」 |
| 現状の課題感（仮） | 任意 | ヒアリングで把握済みの問題 | 「リード獲得できていない」 |
| 理想像（仮） | 任意 | 目指したい姿（あれば短縮できる） | 「PDCAが回っている状態」 |
| 制約条件 | 任意 | 予算／期間／人員等 | 「3か月以内／追加採用なし」 |

## 実行ステップ

### ステップ1：現状の問題（As-Is）を確認する

- 入力：分析対象、現状の課題感
- 処理：
  1. 業務プロセス・組織体制・成果指標を3観点で現状を棚卸しする
  2. 問題点・非効率な部分・できていないことを洗い出す
  3. **問題点を1〜3項目に絞る**（多すぎると後段で対応しきれない）
  4. 各問題は「〜できていない」「〜が機能していない」など、ネガティブな現在形で記述する
- 出力：As-Is（現状の問題）リスト 1〜3項目

### ステップ2：理想の状態（To-Be）を定義する

- 入力：ステップ1のAs-Is、理想像（仮）
- 処理：
  1. ステップ1で挙げた各問題に対し、**1対1対応で**「理想の状態」を言語化する
     - As-Is項目1 → To-Be項目1
     - As-Is項目2 → To-Be項目2
     - As-Is項目3 → To-Be項目3
  2. 各理想は「〜できている」「〜が機能している」など、ポジティブな完了形で記述する
  3. 可能であれば定量目標（数値・期日）を含める
- 出力：To-Be（理想の状態）リスト As-Isと同じ項目数

### ステップ3：課題（Gap）を明確にする

- 入力：ステップ1のAs-Is、ステップ2のTo-Be
- 処理：
  1. As-IsとTo-Beの差分を埋めるために**何が足りないか**を1対1で言語化する
     - As-Is1 ↔ To-Be1 の差分 → Gap1
     - As-Is2 ↔ To-Be2 の差分 → Gap2
     - As-Is3 ↔ To-Be3 の差分 → Gap3
  2. 各Gapは「〇〇が△△できていないこと」のような形式で記述する
  3. 単なる現状の繰り返しではなく、**埋めるべきブランクの本質**を捉える
- 出力：Gap（課題）リスト As-Isと同じ項目数

### ステップ4：解決策のアプローチ（Solution）を立案する

- 入力：ステップ3のGap
- 処理：
  1. 各Gapに対し、**1対1で**具体的な解決策・改善策を立案する
     - Gap1 → Solution1（実施事項＋具体アクション）
     - Gap2 → Solution2
     - Gap3 → Solution3
  2. 各Solutionは「〇〇の見直し」「△△の確立」など見出しレベルの施策＋下層に具体アクション3〜5項目で記述する
  3. 必要に応じて関連スキルを案内する：
     - PDCA確立 → `pdca-cycle`
     - KPI設計 → `KGI/KPI/KAI` 関連スキル
     - ターゲット最適化 → `stp-analysis` `value-proposition`
     - 組織体制設計 → 別途専門領域
- 出力：Solution（解決策のアプローチ）リスト Gapと同じ項目数

### ステップ5：解決策に優先順位をつける

- 入力：ステップ4のSolutionリスト、制約条件
- 処理：
  1. 各Solutionを2軸で評価する：
     - **影響度**（成果へのインパクト）：大／中／小
     - **実現可能性**（コスト・期間・人員）：高／中／低
  2. 「影響度・大」×「実現可能性・高」を最優先に振る
  3. 「影響度・大」×「実現可能性・低」は2番手（中長期で取り組む）
  4. 「影響度・小」×「実現可能性・低」は対象外候補
  5. 優先順位（最優先／次／後回し）でラベル付けする
- 出力：優先順位付きSolutionリスト

### ステップ6：完了報告

ステップ1〜5を以下の出力フォーマットでまとめる。

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| As-Isを4項目以上挙げたい | 強制的に3項目以下に絞る | 多すぎると対応しきれずアウトプットが薄まる |
| As-IsとTo-Beの数が一致しない | 必ず1対1対応を強制する | 1対1構造でないとGap・Solutionが導出不能 |
| To-Beが「業界トップになる」のような曖昧表現 | 定量目標または「〇〇できている状態」に書き直す | 進捗測定不能 |
| Solutionが「頑張る」「努力する」など精神論 | 具体的アクション3〜5項目に分解する | 実行不能 |
| 全SolutionがP1（最優先）にラベル付けされる | 強制的に3段階に振り直す | 全部最優先＝優先順位なし |
| As-Isに「競合がいない」が出る | `competitive-absence-audit` を必ず経由する | CLAUDE.md「Competitive Absence Audit Rule」遵守 |
| Gap3の言語化が「単なるAs-Isの繰り返し」になる | 「埋めるべきブランクの本質」を再考する | Gap = As-Is - To-Be の意味的差分 |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| As-Isを5項目以上挙げる | 必ず1〜3項目に絞る | 対応負荷が現実的でなくなる |
| As-IsとTo-Beを1対1にしない | 必ず1対1対応にする | Gap・Solutionが導出不能 |
| To-Beを「業界トップ」のままにする | 「〇〇が達成されている状態」に具体化する | 進捗測定不能 |
| Gapを「現状を繰り返すだけ」で済ませる | As-Isとは別の意味的差分を捉える | Gap=差分の本質を欠落 |
| Solutionを「頑張る」で済ませる | 具体アクション3〜5項目に分解する | 実行不能 |
| 全Solutionに優先度を付けない | 影響度×実現可能性で3段階に振る | 着手順序が不明 |
| 1回作って終わりにする | 定期的に見直す（四半期推奨） | 継続改善サイクルが回らない |

## 連携スキル

- **前段**：`pest-analysis` `swot-analysis` `3c-analysis`（現状分析の入力情報を充実させる）
- **同時併用**：`smart-goal-setting`（To-Beを定量化）、`competitive-absence-audit`（As-Is/To-Be分析中の楽観バイアス排除）
- **後段**：`pdca-cycle`（Solution実行後の継続改善）、`goal-execution-system`（Solution運用の仕組み化）、`medium-term-business-plan`（中長期計画への組み込み）

## 5状態出力契約

外部API・サブプロセス・外部設定依存なし。運用上のマーカー：
- `[FAIL]`：必須入力（分析対象）が欠落 → 処理中断・再入力依頼
- `[OK:0件]`：As-Isが洗い出せない場合（要再ヒアリング）
