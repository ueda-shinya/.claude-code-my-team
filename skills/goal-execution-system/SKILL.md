# goal-execution-system

目標達成を継続させる「実行の仕組み」を、モニタリングシステム3要素＋フィードバックループ3要素で設計する。属人的な根性運用から、構造で回る運用に変える。

## 適用条件

- 使う場面：
  - 目標を立てたが運用が続かない／形骸化する
  - レビューや振り返りが場当たり的
  - 達成しても再現性がない／失敗の学びが共有されない
  - チーム横断でPDCAが回せていない
- 使わない場面：
  - 目標自体の設計 → `smart-goal-setting`
  - 階層分解 → `goal-hierarchy-design`
  - PDCAサイクルの設計 → `pdca-cycle`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 運用対象の目標 | 必須 | 仕組み化する対象の目標 | 「月間新規受注500万円」 |
| 関与メンバー | 必須 | 運用に関わる人数・役割 | 「営業3名・マネージャー1名」 |
| 既存の運用 | 任意 | 現状の会議体・ツール等 | 「週次MTGあり/Notion使用」 |
| サイクル期間 | 任意 | 運用サイクルの長さ | 「月次」「四半期」 |

## 実行ステップ

### ステップ1：モニタリングシステムの3要素を設計する

- 入力：運用対象の目標、関与メンバー
- 処理：以下3つを定義する

#### 1-1. 定期的なレビュー
- 頻度を決める：日次／週次／月次／四半期
- 参加者を決める：誰が出席するか（実行者／責任者／関係部署）
- 議題テンプレートを決める：「KPI実績→未達理由→次アクション」の3点固定
- 所要時間を決める：30分／60分

#### 1-2. データダッシュボード
- 表示するKPIを1〜5個に絞る（多すぎると見られない）
- 更新頻度を決める：リアルタイム／日次／週次
- 媒体を決める：スプレッドシート／Notion／Looker Studio／専用BIツール
- 閲覧権限を決める：誰が見られるか

#### 1-3. アラート設定
- 発火条件を決める：「目標達成率がX%を下回ったら」「○日連続で実績ゼロなら」等
- 通知先を決める：誰に何で通知するか（Slack／メール／チャットワーク）
- エスカレーション順を決める：1次（実行者）→2次（マネージャー）→3次（責任者）
- アラートの粒度：あまり多すぎると麻痺する。1日1〜3件以内に絞る

- 出力：モニタリングシステム設計書

### ステップ2：フィードバックループの3要素を設計する

- 入力：ステップ1の結果、関与メンバー
- 処理：以下3つを定義する

#### 2-1. 定期的な振り返り
- レビューと別物：レビューは「数値の確認」、振り返りは「学びの抽出」
- 頻度を決める：月次／四半期／プロジェクト終了時
- 形式を決める：KPT（Keep/Problem/Try）／YWT（やったこと/わかったこと/つぎやること）／Fearless Retrospective 等
- 参加者を決める：実行者全員＋ファシリテーター

#### 2-2. 改善点の特定
- 振り返りで出た「Problem」「課題」を3カテゴリに分類する
  - 仕組み起因（プロセス・ツール・体制）
  - 行動起因（実行スピード・品質・優先順位）
  - 知識起因（情報不足・スキル不足）
- 各カテゴリで最重要の改善点を1つに絞る
- 改善案には「いつまでに誰が何をやるか」を必ず付ける

#### 2-3. 成功事例の共有
- 共有対象：達成した事例、効いた施策、再現可能なノウハウ
- 共有先：チーム全体／部門全体／全社
- 共有形式：定例の場での口頭共有＋書面化（Notion/Confluence/Slack）
- 書面化テンプレート：「状況→何をやった→結果→再現するための条件」の4点固定

- 出力：フィードバックループ設計書

### ステップ3：統合運用ルールを定める

- 入力：ステップ1・2の結果
- 処理：以下を統合する
  - **運用カレンダー**：1ヶ月／四半期で「いつ何が起きるか」をスケジュール化
    - 例：毎週月曜9時=KPIレビュー、毎月最終金曜=振り返り、四半期末=戦略レビュー
  - **責任者マトリクス**：各イベントの責任者・参加者・成果物を1表にまとめる
  - **ツールマップ**：どのツールに何が記録されるかを明記（重複を避ける）
- 出力：統合運用ルール

### ステップ4：仕組みが機能しているかの自己診断項目を作る

- 入力：統合運用ルール
- 処理：以下の自己診断10項目を作る（運用開始後に毎月チェックする用）
  - □ レビューが予定通り開催されているか
  - □ ダッシュボードが更新され続けているか
  - □ アラートが発火したときに対応されているか
  - □ 振り返りが「形だけ」になっていないか
  - □ 改善点が次サイクルで実際に試されているか
  - □ 成功事例が共有されているか
  - □ 責任者マトリクスの担当が形骸化していないか
  - □ ツールマップ通りに記録が残っているか
  - □ 参加者から「やる意味あるのか」の声が出ていないか
  - □ KPI改善が実際に起きているか
- ×が3つ以上ついたら仕組みの再設計を実施する基準とする
- 出力：自己診断シート

### ステップ5：仕組み全体を1枚にまとめて出力する

- 入力：ステップ1〜4の全成果物
- 処理：以下のテンプレートで統合し、運用に乗せられる形に仕上げる
- 出力：目標達成の仕組み化設計書

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| メンバーが3名以下 | レビューは週次・振り返りは月次の最小構成 | 過剰な運用は小規模チームを潰す |
| メンバーが10名以上 | レビューを階層化（チーム単位＋全体）／ダッシュボード必須 | 全員一律の会議は効率が悪い |
| ダッシュボード未整備 | スプレッドシートで最小スタート → 必要に応じて専用ツール | 最初から完璧なBI構築は遠回り |
| アラートが多すぎて麻痺 | 発火条件を厳しくし、1日1〜3件以内に絞る | アラート慣れは重大事象を見逃す原因 |
| 振り返りで毎回同じProblemが出る | 仕組み起因の課題。改善点として最優先で取り組む | 個人努力では解決しない構造的問題 |
| 成功事例が共有されない | 振り返りに「Keep（続けるべきこと）」を必ず含める | Problem偏重の振り返りは学びが偏る |
| 自己診断で×が3つ以上 | 仕組みを再設計（過剰運用なら削減／不足なら追加） | 機能していない仕組みは害になる |
| 既存の会議体が多い | 統合・廃止を検討。本スキルの仕組みに吸収する | 並行運用は形骸化の元 |

### 記入例（要約）

営業3名チーム / 月次新規受注500万円目標 / 月次サイクル
- レビュー：毎週月曜9時30分・営業3名+マネージャー
- ダッシュボード：Notion DBで案件・受注額・確度を週次更新
- アラート：月中時点で達成率50%未満→Slackでマネージャー通知
- 振り返り：月末金曜にKPT（30分）
- 成功事例：受注時に「再現条件」を1ページ書面化、営業全員に共有
- 自己診断：毎月最終火曜にマネージャーが10項目チェック

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| レビューと振り返りを同じ場で済ます | 別の場として設計する（数値確認 ≠ 学びの抽出） | 1つの場に詰め込むとどちらも中途半端になる |
| ダッシュボードに10個以上のKPIを並べる | 1〜5個に絞る | 多いと誰も見なくなる |
| アラートを毎日10件以上発火させる | 1日1〜3件以内に絞る | 麻痺して重大事象を見逃す |
| 振り返りでProblem偏重 | KeepとProblemを必ず両方扱う | 学びが偏り、再現性が育たない |
| 成功事例を口頭共有のみで終わらせる | 必ず書面化（再現条件付き） | 口頭では再現できない |
| 仕組みを作って運用しない | 自己診断シートで毎月チェック | 仕組みは放置すると形骸化する |
| 既存の会議体に追加する形で導入する | 既存の統廃合を含めて再設計 | 並行運用は形骸化の元凶 |
| 責任者を「全員」にする | 1イベント1責任者を必ず明記 | 全員責任は無責任と同じ |
