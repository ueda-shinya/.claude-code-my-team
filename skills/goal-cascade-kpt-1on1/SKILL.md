# goal-cascade-kpt-1on1 スキル

目標設定とフィードバック = 社員一人一人に与えられた目標を全員が達成したら、会社の目標が達成される、というトップダウン構造を運用するスキル。事業戦略スクール「知足」のフレームワーク準拠。株式会社koujitsuの運用例ベース。

## 目標設定の鉄則

**目標設定の考え方**：社員一人一人に与えられた目標を全員が達成したら、会社の目標が達成されることが鉄則。そのため、目標はトップダウンで設定される。

```
会社の目標（売上、利益など）責任者：社長、役員
        │
   ┌────┼────┐
部署A   部署B   部署C  （責任者：部長）
   │
 ┌─┼─┐
チーム チーム チーム  （責任者：チームリーダー）
   │
 ┌─┼─┐
社員 社員 社員  （個人目標）
```

## 「目標」とは何か（6つの定義）

1. 「今期、あなたは何をするのか」を明確に言語化したもの
2. その人の役割・責任・仕事の「道筋」を明らかにするもの
3. その人の仕事ぶり・成果を（結果的に）客観化するもの
4. 人を育て、やる気を引き出すもの
5. 上司と部下のコミュニケーションを育むもの
6. 「理念・哲学」を浸透させていくもの

**重要原則**：目標管理（達成評価）は「賞与の計算ツール」ではなく、能力開発・部下育成・コミュニケーションが目的。

## 適用条件

- このスキルを使う場面：
  - 全社目標を部門・チーム・個人にカスケードして落とし込みたい
  - 上期/下期の目標設定運用ルールを整備したい
  - 1on1とKPTを定着させたい
  - 中間レビューと目標変更ルールを明確化したい
  - 「目標が形骸化」「個人目標の集合体が全社目標と乖離」を改善したい
- このスキルを使わない場面：
  - 個人レベルの目標階層化（年→四半期→月→個人） → `goal-hierarchy-design`
  - 個別目標のSMART点検 → `smart-goal-setting`
  - KGI/KPI/KAIの3階層指標設計 → `kgi-kpi-kai-design`
  - 評価制度（賞与査定／360度評価／らしさ評価）の本格設計 → `evaluation-system-design`（評価面談としての1on1運営はこちら）
  - モニタリングシステム＋ダッシュボード構築 → `goal-execution-system`
  - 1on1**会議体としての設計**（頻度・参加者・所要時間・運営枠組） → `meeting-cadence-design`
  - PDCAサイクル運用 → `pdca-cycle`
  - OODA意思決定 → `ooda-loop`

**本スキルの1on1スコープ限定**：本スキルは「**目標カスケード運用と紐づく**KPT 1on1（目標進捗管理＋FB）」を扱う。会議体の枠組設計（cadence設計）は `meeting-cadence-design`、人事評価面談としての1on1は `evaluation-system-design` を使う。役割が重複した場合の優先順位は以下：
- 「目標進捗管理＋KPTでのFB」が主目的 → 本スキル
- 「会議体カレンダー全体の設計／cadence管理」が主目的 → `meeting-cadence-design`
- 「賞与査定／評価面談」が主目的 → `evaluation-system-design`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 会社情報 | 必須 | 業種・規模・決算期 | 「Web制作・20名・8月決算（新期9月開始）」 |
| 全社目標 | 必須 | 売上／利益／その他主要KGI | 「売上3億・営業利益10%」 |
| 組織階層 | 必須 | 部署→チーム→個人の構造 | 「営業部7名／制作部10名／管理3名」 |
| 既存の運用 | 任意 | 評価頻度／1on1有無 | 「年1回評価／1on1なし」 |
| 課題感 | 任意 | 「目標形骸化」「乖離」「FB不足」等 | 「個人目標の達成意識が低い」 |
| 評価カレンダー | 必須 | 上期/下期の開始月（決算期から逆算） | 「8月決算→上期9月／下期3月」 |

## 実行ステップ

### ステップ1：目標カスケード構造を可視化する

- 入力：会社情報、全社目標、組織階層
- 処理：
  1. 全社目標→部門目標→チーム目標→個人目標のツリー構造を描く
  2. 各階層の責任者を明示（社長／部長／チームリーダー／個人）
  3. **整合性チェック**（バッファ最大10%許容、超過分は明示）：
     - 部門目標の集合体 ≥ 全社目標になっているか？（バッファ10%まで許容）
     - チーム目標の集合体 ≥ 部門目標になっているか？（バッファ10%まで許容）
     - 個人目標の集合体 ≥ チーム目標になっているか？（バッファ10%まで許容）
     - `goal-hierarchy-design` の整合性チェック方針（合計 ≥ 上位）と一致させる
- 出力：目標カスケードツリー（責任者明示）

### ステップ2：目標設定5ルールに沿った個人目標を設計する

- 入力：個人別の役割・責任・期待値
- 処理：個人目標が以下5ルールを満たしているかチェック

| ルール | 内容 | チェック観点 |
|---|---|---|
| **1. 3つ以上にボリューム(%)付与** | 目標は今期取り組むことにボリューム（%）をつけて設定する（3つ以上） | 各目標の重要度を%で表現／合計100% |
| **2. 背伸びレベル** | 目標は、ちょっと背伸びして到達するレベルのもの | 「達成困難だが、努力によってなんとか手が届く」ライン |
| **3. 5W1Hで明確表現** | 目標は、明確に表現されなければならない | 誰が・いつ・どこで・誰に・何を・どうやって・いつまでに |
| **4. 本人の自発的意識** | 目標は、本人からの自発的意識（やる気）が「元」にならなければならない | トップダウンの押し付けでは機能しない |
| **5. 上司・会社承認** | 目標は、上司・会社の意思が反映されて承認されたものでなければならない | 上長承認なしの個人目標は無効 |

3. **重要原則**：目標は単なる「ノルマ」ではなく、その人の今期の「役割・責任・仕事の道筋」を明確するものであり、かつ成長を実現するもの

- 出力：5ルール準拠の個人目標シート（ボリューム%付き）

### ステップ3：上期/下期の目標設定フロー（4週間）を組む

- 入力：評価カレンダー
- 処理：会社の決算期に応じて上期/下期月を確定し、以下の4ステップフローを上期・下期それぞれに組む

**月の設定方針**：
- 会社の決算期（=期末月）の翌月を「新期1月目」とする
- 例：8月決算なら新期9月開始、上期9月〜2月／下期3月〜8月
- 例：3月決算なら新期4月開始、上期4月〜9月／下期10月〜3月
- 「上期目標設定」は新期**1月目**に実施、「下期目標設定」は新期**7月目**に実施
- `evaluation-system-design` と併用する場合、両スキルで「上期/下期」の月定義を一致させること

| ステップ | 上期スケジュール | 下期スケジュール | 内容 |
|---|---|---|---|
| **1. 全社目標の設定と共有** | 新期1月目第1週 | 新期7月目第1週 | 社長・役員が全社目標を確定し全社に共有 |
| **2. 部門目標の設定と共有** | 新期1月目第2週 | 新期7月目第2週 | 全社目標を受けて部長が部門目標を策定・共有 |
| **3. 課/チーム目標の設定と共有** | 新期1月目第3週 | 新期7月目第3週 | 部門目標を受けてチームリーダーが課目標を策定・共有 |
| **4. 個人目標の設定（面談実施）** | 新期1月目第4週〜 | 新期7月目第4週〜 | 上司・部下の面談を経て個人目標を最終決定 |

**重要原則**：
- 個人目標の最終決定者は「上司」（部下が起案、上司が承認）
- 上位目標がないまま下位目標を作らない（順序厳守）

- 出力：上期/下期それぞれの4週間スケジュール（実月日に置換）

### ステップ4：四半期中間レビュー（11月・5月）を設計する

- 入力：上期/下期スケジュール
- 処理：
  1. 11月（上期中間）と5月（下期中間）に「中間レビュー（四半期）」を実施
  2. 上司の承認のもとに、必要に応じて目標の修正を行う
  3. **注意ルール**：中間レビューでの安易な目標の「引き下げ」は避ける
- 出力：中間レビュー実施計画（日程・参加者・修正承認フロー）

### ステップ5：フィードバック頻度とリカバリープランを定める

- 入力：個人目標シート
- 処理：

#### 5-1. 月次レビュー
- 目標は、少なくとも月次で達成状況を振り返る
- 進捗が悪い場合：リカバリープランを上長と社員で考え、実行する

#### 5-2. 目標変更ルール
- 会社や社員の状況の変化に応じて、期中で目標の見直しが行われる場合もある
- **POINT**：簡単に目標が変更できては、目標の意味がなくなってしまう
- 変更する場合も、上長・社長の承認を得るなど、納得できる理由と承認フローをもって変更できることが望ましい

- 出力：月次レビュー運用規定＋目標変更承認フロー

### ステップ6：1on1とKPTフレームを導入する

- 入力：個人目標シート、フィードバック頻度
- 処理：

#### 6-1. 1on1の設計
- 上長と社員で、定期的な状況共有のため、1on1を行う
- **頻度バリエーション**：
  - 週1回15分
  - 毎日5分
  - 隔週30分
  - 月1回1時間
- **目的**：目標に対するフィードバックだけではない、普段の関係性構築のためにも行う
- **共有頻度の効果**：頻度が多ければ多いほど、状況の良しあしの解像度もあがり、悪かった時のリカバリーもしやすくなる

#### 6-2. KPTフレームの導入
1on1でよく活用されるフレームワーク「KPT」：

| 要素 | 意味 | 内容 |
|---|---|---|
| **Keep** | 続けること | うまくいっていること |
| **Problem** | 問題点 | 困っていること |
| **Try** | 次の行動 | つぎにやってみること |

3つを整理する手法で、シンプルながら振り返りのフレームワークとして非常に有用。

#### 6-3. KPTシートのフォーマット

| 日付 | Keep（うまくいっていること） | Problem（困っていること） | Try（次にやってみること） | メモ |
|---|---|---|---|---|
| YYYY-MM-DD | ・SEO記事構成知識の習得<br>・自分で記事構成作成<br>・上長からFB獲得<br>→成長実感あり | ・マーケ戦略部分は知識/経験不足<br>・課題分解ができていない<br>・ソリューション/施策が考えられていない | ・案件で課題の洗い出し/分解<br>・5月のXXさん1on1までに用意<br>・必ず実践してFBまで1セット | XXX |

#### 6-4. 1on1の実施ルール
- **POINT**：1on1の際に、1on1をやってもらう側（部下）がKPTに記載し、KPTを上長と部下で共有しながら会話をすると、短時間で有意義なフィードバックができるようになる
- 部下が事前にKPT記入 → 上長と共有しながら15分で会話

- 出力：1on1運用ルール＋KPTシートテンプレート

### ステップ7：年間運用カレンダーを作成

- 入力：ステップ3〜6の運用ルール、決算期
- 処理：以下のテンプレに会社の決算期を当てはめてカレンダーに整理

```
新期1月目  第1週：全社目標発表
            第2週：部門目標発表
            第3週：課目標発表
            第4週〜：個人目標面談
新期2月目  月次レビュー（1回）
新期3月目  月次レビュー（1回）
新期4月目  月次レビュー＋四半期中間レビュー
新期5月目  月次レビュー（1回）
新期6月目  月次レビュー（1回）
新期7月目  第1週：下期全社目標発表
            第2週：下期部門目標発表
            第3週：下期課目標発表
            第4週〜：下期個人目標面談
新期8月目  月次レビュー（1回）
新期9月目  月次レビュー（1回）
新期10月目 月次レビュー＋四半期中間レビュー
新期11月目 月次レビュー（1回）
新期12月目 月次レビュー（1回）
※1on1は週次/毎日など別途実施
```

**例（8月決算＝新期9月開始の場合）**：新期1月目=9月、新期7月目=3月、四半期中間レビュー=12月と6月

- 出力：年間運用カレンダー（実月日に置換済み）

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 全社目標が未確定 | 目標カスケード設計の前に経営会議で全社目標を確定する | 上位がないと下位を作れない |
| 個人目標の集合体 < チーム目標 | チーム目標を見直すか、未割当領域を補強する目標を追加 | 「全員達成→会社達成」の鉄則を満たさない（バッファ10%以下は許容） |
| 個人目標の集合体 > チーム目標 × 1.1 | 過剰負荷の検証／個人目標の合計を再調整 | 計画破綻リスク（バッファ10%超過は禁止） |
| 個人目標がトップダウン押し付け | ステップ2のルール4「自発的意識」で再検討／面談で本人意見を組み込む | やる気が起点でない目標は機能しない |
| 個人目標が達成余裕あり（簡単すぎる） | ルール2「背伸びレベル」に違反／上方修正 | 成長を実現しない |
| 月次レビューを飛ばしがち | 1on1とセットで運用カレンダーに固定 | 月次レビューなしでは進捗管理が崩壊 |
| 期中で目標達成困難が判明 | リカバリープラン or 目標変更（上長・社長承認） | 安易な引き下げは禁止 |
| 1on1が雑談で終わる | KPTシートを部下が事前記入する仕組みに変える | 構造化なしでは有意義にならない |
| 上司が忙しくて1on1をスキップ | 頻度を週1→隔週に下げてでも継続する／毎日5分でも実施 | スキップは関係の質悪化のシグナル |
| 中間レビューで目標を引き下げる議論 | 引き下げは原則NG／環境変化が明確な場合のみ承認 | 中間レビューを言い訳の場にしない |
| 目標変更が頻発 | 変更回数の上限を設定（年1回まで等） | 簡単に変更できると目標が意味を失う |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 全社目標→部門→チーム→個人を同時並行で設計 | 上位から順に確定する（4ステップ厳守） | 上位がないと下位の整合性が取れない |
| 個人目標の集合体が全社目標を満たしていない（合計 < 上位） | カスケードの整合性チェック（合計 ≥ 上位）を必ず実施 | 「全員達成→会社達成」の鉄則を破る |
| トップダウンで個人目標を押し付ける | 本人起案→上司承認のフロー | 自発的意識がないと機能しない |
| 簡単に達成できる目標を設定する | 背伸びレベル（達成困難だが努力で手が届く）に設定 | 成長を実現しない |
| 月次レビューをスキップ | 1on1とセットで運用カレンダーに固定／月末にリマインダー設定 | リカバリープラン発動が遅れて手遅れ |
| 中間レビューで目標を引き下げる | 引き下げは原則禁止／リカバリープラン優先 | 言い訳の場になり目標が形骸化 |
| 1on1を雑談で終わらせる | KPTフレームで構造化／部下が事前記入 | 短時間で有意義にならない |
| 目標を「賞与計算ツール」として運用 | 能力開発・部下育成・コミュニケーション目的として運用 | 本来の目的を失う |
| KPTのKeep（うまくいっていること）を毎回スキップ | Keepから始める／成功体験を必ず認める | Problemばかりだと部下のモチベが下がる |
| KPTのTry（次の行動）が抽象的 | 5W1Hで具体化（「次回1on1までに〇〇を完了」） | 行動変容しない |
| 頻度バリエーションを画一的に決める | 部下のレベル/状況に応じて頻度調整（新人は毎日5分、ベテランは月1回1時間等） | 一律ルールは効率低下 |
| 目標変更が無制限 | 変更回数上限・承認フロー設定 | 簡単に変更できると目標が意味を失う |

## 連携スキル

- `goal-hierarchy-design`：個人レベルでさらに細分化（年→四半期→月→個人）
- `smart-goal-setting`：個人目標のSMART準拠点検
- `kgi-kpi-kai-design`：KGI/KPI/KAIの3階層指標で個人目標を構造化
- `evaluation-system-design`：賞与査定／360度評価／らしさ評価の本格制度設計
- `goal-execution-system`：モニタリングダッシュボード／アラート／継続化の仕組み
- `pdca-cycle`：月次レビューのPlan→Do→Check→Action構造化
- `ooda-loop`：環境変化が激しい場合の意思決定フレーム
- `meeting-cadence-design`：1on1／月次レビュー／全社朝会の**会議体枠組設計（cadence管理）**を担当。本スキルは**目標進捗管理＋KPT FB**が主担当（責任分界：cadence設計は meeting-cadence-design / 目標進捗＋FB は本スキル）
- `success-cycle-relationship-quality`：1on1で関係の質を高めるPOINT 04として連携
- `teaching-coaching-leading`：1on1での指導アプローチ（ティーチング/コーチング/リーディング）使い分け
- `roles-responsibilities`：個人目標の前提となる役割・責任範囲の明確化

## 5状態出力契約

このスキルは外部API・サブプロセス実行・外部設定依存のいずれも持たないため、CLAUDE.md「スキル出力の5状態契約」の対象判定基準には該当しない。本契約は適用しないが、運用上の**任意**ベストプラクティスとして以下のマーカー使用を**推奨**する：

- `[OK:0件]`：個人目標を設定した社員が0名のとき／月次レビュー対象0件のとき
- `[PARTIAL:N成功/M失敗]`：複数チームで並行運用して一部のチームのみ目標未確定のとき
- `[FAIL]`：上位目標未確定で下位目標が作れないとき。エラー要約を必ず添える
- `[SKIP]`：上位目標未確定で個人目標設定が走れない場合（試みなかったケースのみ）
