# delivery-build スキル

デリバリー構築（顧客への商品・サービス提供の仕組みを設計・最適化＋社内体制整備）を、5プロセス×CSモニタリング×成功5ポイントで体系設計する。

## 適用条件

- このスキルを使う場面：
  - クライアントワーク（コンサル／マーケ支援／受託開発等）の提供プロセスを標準化したい
  - 入社1年目の社員でも70点のクオリティで提供できる仕組みを作りたい
  - 属人化したサービス提供プロセスを言語化し、誰が担当しても一定品質にしたい
  - スコープクリープ（業務範囲の逸脱）を防ぐ仕組みを作りたい
  - CSモニタリングの会議体・案件管理シートを設計したい
- このスキルを使わない場面：
  - 新入社員の受け入れプログラム（社員向け） → `onboarding-design`
  - PJ全体QCDリスク管理 → `pm-risk-management`
  - PJ体制（人員・役割）設計 → `project-team-structure`
  - PJ収支管理と契約 → `project-finance-contract`
  - クライアント期待値調整の心理的ハンドリング詳細 → `client-expectation-management`
  - 営業資料テンプレート設計 → `sales-deck-template`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| サービス種別 | 必須 | コンサル／マーケ支援／受託開発／クリエイティブ制作 等 | 「マーケティング戦略コンサル」 |
| 提供形態 | 必須 | スポット／月額顧問／プロジェクト型 | 「月額顧問＋スポット」 |
| 現状の属人化度合い | 必須 | 高（特定人物に依存）／中／低 | 「特定2名に集中」 |
| 既存ドキュメント | 任意 | 提案書／報告書／契約書テンプレートの有無 | 「提案書テンプレありフォーマットバラバラ」 |
| 解決したい課題 | 必須 | 属人化／品質バラつき／スコープクリープ／引き継ぎ難 | 「担当者によって成果物の品質に大きな差」 |

## 実行ステップ

### ステップ1：デリバリー構築の定義と3目的を確認する

#### 定義
デリバリー構築とは、顧客に商品やサービスを届けるための仕組みを **設計・最適化** することに加え、社内の人材配置や業務プロセスを整備し、**組織全体で円滑に提供できる体制を構築すること** です。

#### 3つの目的
| # | 目的 | 内容 |
|---|---|---|
| 01 | 顧客満足度を高める | 一定のクオリティで安定したサービス提供 |
| 02 | ブランド価値を向上させる | 「あの会社に頼めば外れない」という信頼の蓄積 |
| 03 | プロセスを効率化し、コストを削減する | 属人性排除による業務負担軽減＋スケール対応 |

### ステップ2：デリバリー構築の役割2軸を整理する

| 軸 | 内容 |
|---|---|
| 顧客体験の向上 | 誰が対応しても会社として一定のクオリティを保ってサービス提供ができる |
| 効率化による利益向上 | 社内的にはサービス提供の属人性を減らすことで、生産性向上も見込める |

入社1年目の社員でも一定のクオリティ（**70点**）を確保できる仕組みを構築し、会社全体で安定したサービスを提供することが目的。

### ステップ3：仕組み化の5プロセスを設計する

| プロセス | 内容 |
|---|---|
| サービス提供のフレームワーク化 | クライアントの課題分析 → ソリューション設計 → 実行支援の流れを明確に定義／役務提供プロセスを言語化し、属人性を減らす |
| スコープと成果物の明確化 | クライアントとの期待値調整（スコープの認識ずれを防ぐ）／成果物（ドキュメント、レポート、提案書など）を標準化 |
| プロジェクト管理手法の確立 | タスク管理・スケジュール管理のルール設定（例：週次MTG、マイルストーン設定）／Asana, Notion, Trello などのツール活用 |
| ナレッジ共有とチーム連携の仕組み化 | 過去のプロジェクトの知見を蓄積し、次回以降の品質向上につなげる／社内での情報共有のフローを確立（社内Wiki、チェックリスト、テンプレート活用） |
| PDCAによる継続的な改善 | クライアントフィードバックを定期的に収集し、改善を回す／終了後の振り返りを実施し、次の案件に活かす |

### ステップ4：役務提供プロセスの言語化5ステップを実施する

役務提供型ビジネスにおいて、デリバリーの再現性を高めるには、サービス提供プロセスを言語化し、誰が担当しても同じ品質提供できるようにすることが重要。

| ステップ | 内容 |
|---|---|
| 現状の業務フローを可視化 | 現場の担当者とヒアリングし、案件の流れを整理／プロセスのフェーズ分け（例：ヒアリング → 課題整理 → 提案 → 実行 → 振り返り）／各フェーズの具体的なタスクを洗い出す |
| 各フェーズでの基準を明確化 | 各フェーズで「何を達成すべきか？」のゴールを定義（例：「クライアントヒアリング」では最低3つの課題を特定することを目標にする） |
| 成果物のフォーマットを統一 | 各フェーズごとに必要なドキュメントを標準化しフォーマット化（例：「提案書」には必ず「課題の背景」「提案の根拠」「期待される成果」を含める） |
| 属人性を減らすためのマニュアル化 | 各業務の進め方を「チェックリスト」や「マニュアル」に落とし込む（例：「課題整理」の手順として「クライアントのKPIを把握する」「顧客インタビューを実施する」などをリスト化） |
| 運用 & 改善 | 実際に標準化したプロセスを試行し、運用しながらブラッシュアップ／定期的にチームでフィードバックを行い、より精度の高いプロセスへ進化させる |

#### クライアントワークにおける各プロセスのフェーズ分け（参考例）

| フェーズ | 主なタスク | 使用する成果物 |
|---|---|---|
| クライアントヒアリング | 現状課題の把握、ニーズの確認 | ヒアリングシート |
| 課題整理 & 仮説立案 | 課題を整理し、仮説を立てる | 仮説立案メモ |
| 提案 & 合意形成 | 提案内容をクライアントに提示し、調整 | 提案書 |
| プロジェクト実行 | コンサルティング・ワークショップなどの実施 | コンサルレポート |
| 振り返り & 改善提案 | 成果の測定・改善提案 | 振り返りレポート |

### ステップ5：成果物の標準化5プロセスを実施する

| プロセス | 内容 |
|---|---|
| 現状の成果物の整理 | 過去の案件で作成した成果物を収集し、どのドキュメントが頻繁に使用されているかを洗い出す／形式がバラバラになっているもの、改善すべきポイントを特定 |
| コアドキュメントの選定 | すべての成果物を標準化する必要はないため、頻出する成果物（例：提案書、報告書、契約書など）に絞る／各プロジェクトで共通して必要な情報を明確化 |
| テンプレートの作成 | 必要な項目を定義し、統一フォーマットを作成（例：Google Docs, Notion, Word, Excel, PowerPoint）／企業のブランドやトーンに合わせたデザインや言葉遣いを統一 |
| 運用ルールの決定 | どのタイミングでどのドキュメントを作成するかのガイドラインを設定／ドキュメントの管理方法（クラウドストレージ、ナレッジベースなど）を統一 |
| フィードバックと改善 | 標準化したドキュメントを実際に使いながら、必要に応じて改訂を行う／クライアントのフィードバックや社内レビューを取り入れ、継続的にアップデートする |

#### 標準化が推奨されるドキュメントの例

| ドキュメントの種類 | 目的・概要 | 標準化のポイント |
|---|---|---|
| プロジェクト提案書 | クライアントに対して、課題・解決策・進め方を提示する | テンプレート化し、課題分析・提案内容・予算・スケジュールなどの項目を統一 |
| 契約書 & SOW（Statement of Work） | サービスの範囲や納品物、料金、スケジュールを明確化 | 契約範囲を明確にし、Scope Creep（業務範囲の逸脱）を防ぐためのチェックリストを作成 |
| プロジェクト計画書 | スケジュール、マイルストーン、KPIを定める | 進捗管理の項目（担当者・期限・達成基準）をフォーマット化 |
| 週次・月次報告書 | 進捗状況をクライアントと共有 | 報告すべき指標（進捗率・課題・リスク・次のアクション）を統一 |
| ワークショップ・コンサルティングレポート | クライアント向けの知見提供 | フォーマットを統一し、課題→提案→アクションプランの流れを固定 |
| ナレッジ共有用ドキュメント | 社内の知見を体系化し、再利用可能にする | ベストプラクティスを記録し、次回以降の案件で活用できる形に整理 |

### ステップ6：仕組み化の3STEP（実装版）でナレッジを集約する

実装段階では下記3STEPで集約・運用する。

| STEP | 内容 |
|---|---|
| STEP01 プロセスの整理 | 各メニューごとに時系列で流れを整理／何をどの順番で進めるべきか明確化 |
| STEP02 アウトプットの統一 | 共通で必要な成果物（アウトプット）をフォーマット化／誰が作成しても同じ品質になるように標準化 |
| STEP03 ナレッジの集約 | 情報を一か所に集める（NotionやGoogle Drive）／「誰がどのプロジェクトについて知っているか」を可視化（トランザクティブメモリー） |

抽象的なプロセス（「戦略」「課題発見」など）も、ワークシート等の具体的フォーマットに落とし込むことで誰でも実行しやすい形に整理できる。

### ステップ7：CSモニタリングを設計する

CSモニタリング（Customer Success Monitoring）とは、顧客の成功（Customer Success）を継続的に追跡・評価し、最適なサポートや施策を講じるプロセス。主にSaaS企業やサービス業で活用され、顧客満足度を高め、解約率（チャーン）を防ぐ目的で実施。

| モニタリング軸 | 方法 |
|---|---|
| 顧客満足度のモニタリング | 定期的なCSアンケート／契約時・解約時のアンケート |
| 組織全体での顧客満足度のモニタリング | 定期的な会議体を設定／各案件の状況・目標達成進捗とその課題を議論する |

#### CSモニタリング時のデリバリー構築（運用フロー）

| STEP | 内容 |
|---|---|
| STEP01 顧客満足度をモニタリング | 定期的なCSアンケート／契約時、解約時のアンケート |
| STEP02 顧客満足度の継続的モニタリング | 定期的な会議体を設定し、組織全体での顧客満足度の状況モニタリングをする |

例：既存案件管理シートで各案件の状況を担当者が記載し、週1回目標達成進捗とその課題を議論する会議を設ける。議論するための質問を出すことがジュニアメンバーにとっては難しいため、質問の視点を共有し、会議の中で思考力を磨く。

### ステップ8：デリバリー構築を成功させる5ポイントで点検する

クライアントワークの価値を最大化するポイントは5つ。

| Point | 内容 |
|---|---|
| Point01 クライアントの期待値コントロール | 「何をどこまでやるのか」を契約前に明確化／スコープずれを防ぐために進捗ごとの確認ポイントを設定 |
| Point02 ナレッジのテンプレート化・体系化 | 役務提供のベストプラクティスを社内で共有し、業務の属人性を減らす／成果物（提案書、レポート、ワークショップ資料など）を標準化 |
| Point03 プロジェクト進行の透明性確保 | クライアントと進捗を可視化し、期待値ギャップを防ぐ（例：ガントチャート、週次レポート）／定期的なコミュニケーションの仕組みを作る |
| Point04 クライアントコミュニケーションの最適化 | チャットツール・ミーティングの使い分けを明確にし、業務負担を減らす／誤解を防ぐために重要な決定事項は必ず文書化 |
| Point05 デリバリー後のフォローアップを仕組み化 | 提供した価値を最大化するためのアフターフォロー（例：1ヶ月後のフォローMTG、定期改善提案）／リピートや追加契約につなげる仕組みを作る |

### ステップ9：デリバリー構築の重要性と成功ポイントでまとめる

#### デリバリー構築の重要性
- **再現性のあるサービス提供を実現**：属人性を排除し、誰が担当しても一定の品質でクライアントに価値を提供できる仕組みを作る
- **クライアント満足度の向上と信頼構築**：期待値コントロールやプロジェクト管理を徹底することで、安定した成果を提供し、長期的な関係構築につなげる
- **業務効率化とスケーラビリティの向上**：プロセスの標準化により業務負担を軽減し、同時に複数案件を円滑に進行できる組織体制を整える

#### 成功のポイント
- **プロセスの明確化と標準化**：役務提供の流れを言語化し、チェックリストやテンプレートを活用することで品質を維持する
- **クライアントとの期待値調整**：プロジェクト開始時にScopeを明確化し、進行中も定期的な確認を行うことで、期待値ギャップを防ぐ
- **継続的な改善とフィードバックの活用**：クライアントやチームからのフィードバックをもとに、デリバリーの質を向上させるPDCAサイクルを回す

## 判断基準

| 状況 | 判断 |
|---|---|
| 属人化が高く、特定担当者でないと回らない | ステップ4「役務提供プロセスの言語化」を最優先 |
| 担当者ごとに成果物の品質がバラつく | ステップ5「成果物の標準化」を最優先 |
| クライアントとの認識ズレ・スコープクリープが頻発 | Point01「期待値コントロール」を最優先（契約書&SOWのチェックリスト整備） |
| プロジェクト進捗がブラックボックス化 | Point03「進行透明性」を強化（ガントチャート＋週次レポート） |
| クライアントとのコミュ手段が混乱（チャット/メール/MTG） | Point04「コミュ最適化」を実施（使い分けルール設定） |
| 案件終了後に次の案件につながらない | Point05「フォローアップ仕組み化」を実装（1ヶ月後MTG／定期改善提案） |
| 社員教育（オンボーディング）の文脈で混同 | `onboarding-design` に切り替え（社員向けはこのスキルではない） |
| QCDリスク管理が必要 | `pm-risk-management` を並走 |
| PJ体制設計（役割分担）が必要 | `project-team-structure` を並走 |
| PJ収支管理・契約締結が必要 | `project-finance-contract` を並走 |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 「優秀な人がやればいい」と属人運用を続ける | 入社1年目でも70点を取れる仕組みを構築する | スケールできず／離職時に大きな損失 |
| 全成果物を一斉に標準化しようとする | 頻出する成果物（提案書/週次報告書/契約書）から優先順位をつけて段階的に標準化 | 一斉着手は完成しない／優先順位なしで形骸化 |
| テンプレートを作って配布したら終わり | 運用ルール決定→FB→継続的アップデートまでセットで実施 | テンプレートは使われずに陳腐化する |
| クライアント期待値調整を口頭ベースで済ます | 契約書&SOWで明文化＋進捗ごとの確認ポイントで定期再調整 | スコープクリープ／認識ズレの主因 |
| ナレッジを個人のローカルフォルダに保管 | Notion/GDrive等の共有ストレージ＋トランザクティブメモリー（誰が何を知っているかの可視化）で運用 | 引き継ぎ不可／重複作業の温床 |
| デリバリー終了で関係終了 | Point05「フォローアップ」を仕組み化（1ヶ月後MTG／定期改善提案） | リピート・追加契約の機会損失 |
| 社員オンボーディング（社内向け）と混同 | 顧客向けはこのスキル、社員向けは `onboarding-design` で切り分け | スコープが拡散する |
