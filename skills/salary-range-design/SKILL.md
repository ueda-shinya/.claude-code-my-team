# salary-range-design スキル

給与レンジ（特定の職種や役職に応じた最低額から最高額までの給与範囲）を体系的に決定する。事業戦略スクール「知足」が定義する3目的／3重要理由／3層構成要素／6プロセス／メリット課題／4課題分析STEPに沿って、市場競争力・公平性・キャリアパスの3観点を満たす設計を行う。

## 適用条件

- このスキルを使う場面：
  - 自社の給与レンジ（年収レンジ）を決定したい
  - 役職や職種ごとの最低・中央値・最高額を設計したい
  - 既存の給与体系を市場水準と照らして見直したい
  - 採用活動での給与提示の根拠を明確化したい
  - 内部の不公平感や納得感不足を解消したい
- このスキルを使わない場面：
  - 採用戦略・採用ガイドライン → `recruitment-strategy` スキルを使う
  - キャリアロードマップ・育成計画 → `career-roadmap-development` スキルを使う
  - 評価制度の策定（評価軸・評価期間）→ `evaluation-system-design` スキルを使う
  - 役割と責任範囲の明確化 → `roles-responsibilities` スキルを使う
  - 個別の昇給判断・査定 → `evaluation-system-design` の運用部分

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 起動フレーズ | 必須 | 上記トリガーフレーズのいずれか | 「給与レンジを決めて」 |
| 会社・事業の前提 | 必須 | 業種・地域・規模・組織フェーズ | 「広島の地域密着Web制作会社、社員8名」 |
| 対象職種・役職 | 必須 | レンジを設計したい職種・役職 | 「初級エンジニア／シニアエンジニア／PM」 |
| 既存の給与水準 | 任意 | 現状の給与額（あれば） | 「初級400万、シニア650万」 |
| 業界・地域の市場水準データ | 任意 | 給与調査レポート・求人サイト情報 | 「広島IT平均500万」「Wantedly中央値550万」 |
| 人件費予算 | 任意 | 組織全体の人件費総額・上限 | 「年間5,000万、来期+15%」 |

## 実行ステップ

### ステップ1：給与レンジの3つの目的を確認する

**自社にとって給与レンジを設計する目的を以下の3観点で言語化する：**

| 目的 | 内容 |
|---|---|
| 01 公平性と透明性の確保 | 一貫した基準で給与を決定し、社内の不公平感を解消する |
| 02 組織全体の人件費管理を効率化 | 長期的な人件費計画を立てやすくし、予算逸脱を防ぐ |
| 03 採用活動での競争力を強化 | 業界・地域水準を反映させ優秀な人材を確保する |

ヒアリング項目：
- 自社で最も優先する目的はどれか？（複数可）
- 過去に「給与の不公平感」「人件費の逸脱」「採用で給与負けした」事例はあるか？

### ステップ2：給与レンジが重要な3つの理由を整理する

**給与レンジを設計しないと何が失われるかを3観点で確認する：**

| 重要な理由 | 内容 |
|---|---|
| 市場競争力を維持 | 業界・地域水準を反映させ、優秀な人材を確保する |
| 公平性の確保 | 一貫した基準で不公平感を解消する |
| キャリアパスの明確化 | キャリア開発の指針となる（どの役職でどのくらいの年収か可視化） |

### ステップ3：給与レンジの3層構成要素を設計する

**1つの職種・役職について、以下3層を金額で設定する：**

| 構成要素 | 内容 |
|---|---|
| 最高給与額 | 高度なスキルや成果に対する最大支払い額 |
| 中央値（ミッドポイント） | 一般的なスキル・経験を持つ従業員に支払われる給与額 |
| 最低給与額 | 職務の基本的なスキル・経験に対する最低支払い額 |

**具体例**：
- 初級エンジニア：年収300万円〜500万円（中央値400万円）
- シニアエンジニア：年収600万円〜1,000万円（中央値800万円）

### ステップ4：給与レンジを決定する6プロセスを実行する

#### 4-1. 市場調査（プロセス01）
- 業界や地域の給与水準を調査し、基準を把握する
- **方法**：給与調査レポート、求人サイトの情報（Wantedly／リクナビ／マイナビ／doda 等）
- **出力**：対象職種・役職の市場相場（最低／中央／最高）

#### 4-2. 職務評価（プロセス02）
- 職種や役職ごとに責任範囲やスキルを評価する
- **例**：
  - 営業職は「売上貢献度」を主軸に評価
  - エンジニアは「技術スキル」を主軸に評価
- **出力**：職種ごとの評価軸と相対的な重みづけ

#### 4-3. 内部バランスの確認（プロセス03）
- 同一企業内での役職間での公平性を確認する
- **例**：同等の責任を持つ役職間での給与格差を排除（営業マネージャーと開発マネージャーが同階層なら大きな差をつけない）
- **出力**：役職階層マップと給与の相対関係

#### 4-4. レンジ幅の設定（プロセス04）
- レンジ幅は役職ごとに異なる
- **例**：
  - 初級職：約20%〜30%の幅（300〜400万円なら33%）
  - 上級職：50%以上の幅（600〜1,000万円なら67%）
- **理由**：上級職ほど成果の差が大きく出るため、幅を広くしてパフォーマンスに連動させる

#### 4-5. 会社の予算と調整（プロセス05）
- 設定したレンジが組織の人件費予算に収まるよう調整する
- **計算式**：「対象人数 × 中央値 × 1.15（諸手当・社保事業主負担含む係数の例）」が予算内か確認
- **出力**：人件費予算とレンジの整合性チェック結果

#### 4-6. 定期的な見直し（プロセス06）
- 経済環境や市場の変化に応じて、1年に1回以上更新する
- **トリガー**：物価上昇／業界給与水準の変化／人材流出／競合の給与改定

### ステップ5：給与レンジの3メリット・3課題を確認し対策を立てる

**3メリット**：

| メリット | 内容 |
|---|---|
| モチベーション向上 | 公平な給与基準が従業員の信頼感を高める |
| 人材採用の強化 | 市場水準に合ったレンジが候補者に魅力的に映る |
| コスト管理の向上 | 長期的な人件費計画が容易になる |

**3課題と解決策**：

| 課題 | 解決策 |
|---|---|
| 市場データの不足 | 外部調査データやコンサルタントを活用 |
| レンジの不公平感 | 職務評価と市場データで公平性を確保 |
| 従業員の納得感不足 | 説明会やフィードバック面談を実施 |

**成功事例（参考）**：
- Google：柔軟な給与レンジで多様な人材を採用
- スタートアップ企業：限られた予算内で成果に基づく給与体系を構築

### ステップ6：4課題分析STEPで自社の給与レンジを振り返る

| STEP | 確認項目 |
|---|---|
| STEP01 市場データの最新性 | 業界のトレンドや地域差に基づいた最新のレンジになっているか？ |
| STEP02 内部の公平性 | 職種や役職ごとの責任範囲に応じた適切なレンジが設定されているか？ |
| STEP03 従業員への説明 | レンジの仕組みや昇給基準が十分に共有されているか？ |
| STEP04 柔軟な適用 | 希少スキルを持つ人材や高パフォーマーへの対応ができる余地があるか？ |

### ステップ7：成功の秘訣と設計のポイントをチェックする

**成功の秘訣**：定期的な見直しと従業員への透明性ある説明が、信頼性向上につながる

**給与レンジ設計の3ポイント**：
1. 市場データを活用して競争力を確保
2. 職務評価を基に公平性を担保
3. 柔軟性を持たせ、優秀な人材を惹きつける

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 初級職のレンジ幅 | 約20%〜30% | 業務範囲が限定的でパフォーマンス差が出にくい |
| 上級職のレンジ幅 | 50%以上 | 成果の差が大きく、高パフォーマーを引き留めるため |
| 市場データが不足している地方企業 | 求人サイト3社以上＋業界レポート＋競合の公開求人を組み合わせ | 単一データソースは偏りリスクがある |
| 職務評価の主軸が職種により違う | 営業＝売上貢献度、エンジニア＝技術スキル、CS＝顧客満足度／継続率等で個別設計 | 役割が違うのに同じ評価軸を使うと公平性が崩れる |
| 同階層の役職間で給与差を付けたい | 責任範囲や難易度に明確な差がない場合は付けない | 内部不公平感の最大要因 |
| 希少スキル人材を採用したい | レンジ上限を超える「特例枠」を制度化する | レンジ硬直化で採用負けを防ぐ |
| 従業員から「なぜこの給与か」と質問された | レンジ仕組み・昇給基準を説明会／面談で開示 | 透明性確保がモチベーション向上の前提 |
| 「市場データなしでもなんとなく決めたい」 | 必ず外部データかコンサルタント活用 | 推測ベースのレンジは課題の温床 |
| 採用市場で「うちの地域は給与競合がいない」「ブルーオーシャン」と楽観判断 | `competitive-absence-audit` スキルで6仮説検証 | 給与競合不在は「労働市場が成立していない＝候補者も少ない」可能性が高く、楽観判断は禁止 |
| 1年以上見直していない | 即時レビュー実施 | 物価・市場変動を反映しないレンジは形骸化する |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 「相場感」だけで給与レンジを決める | 市場調査レポート＋求人サイト3社以上＋競合公開求人を統合してレンジ設定 | 推測ベースは公平性も納得感も担保できない |
| 全職種・全役職で同じレンジ幅（例：すべて30%） | 初級は20-30%、上級は50%以上で幅を変える | 役職ごとの成果のばらつきに合っていないと不公平感の元 |
| レンジを決めて1年以上放置 | 1年に1回以上、市場変動に応じて見直し | 形骸化して市場競争力を失う |
| 従業員に給与レンジを開示しない／説明しない | レンジの仕組みと昇給基準を説明会／面談で共有 | 納得感不足は離職率上昇の主要因 |
| 営業もエンジニアも同じ評価軸（例：勤務年数）で給与を決める | 職種ごとに評価主軸を変える（営業＝売上、エンジニア＝技術スキル等） | 役割の違いを反映しないレンジは公平性が崩壊する |
| 希少スキル人材に通常レンジ上限を提示してしまう | レンジ上限を超える「特例枠」を制度化（市場価値が圧倒的に高いケースのみ） | レンジ硬直化で採用負けする典型パターン |
| 同階層の役職間で意味なく給与差を付ける | 責任範囲・難易度に明確な差がない場合は揃える | 内部不公平感の温床 |
| 採用市場で「給与競合がいない」と楽観判断する | `competitive-absence-audit` で需要不在の可能性を検証 | 給与競合不在は労働市場が成立していない可能性が高い |

## 連携スキル

- `recruitment-strategy`：採用戦略のターゲットと給与レンジを連動させる
- `career-roadmap-development`：キャリアロードマップと給与レンジの段階を整合させる
- `evaluation-system-design`：評価制度（評価軸／頻度）の結果を給与レンジ内の昇給に反映する
- `roles-responsibilities`：役職ごとの責任範囲を給与レンジの根拠にする
- `organization-planning`：組織計画の人員配置と人件費予算を整合させる
- `kgi-kpi-kai-design`：人件費KGI（年間人件費総額）の中で給与レンジを設計する
- `competitive-absence-audit`：採用市場・給与競合の楽観判断を防ぐ仮説検証
- `client-expectation-management`：従業員の納得感不足対策（説明会・面談）における期待値調整スキルを応用

## 5状態出力契約

このスキルは外部API・サブプロセス実行・外部設定依存のいずれも持たないため、CLAUDE.md「スキル出力の5状態契約」の対象判定基準には該当しない。ただし以下のケースで明示マーカーを使うこと：

- `[OK:0件]`：対象職種・役職の情報が不足し、給与レンジを設計するための前提が揃わない場合
- `[SKIP]`：競合不在の楽観判断が出た場合は `competitive-absence-audit` スキルへの誘導でスキップ／市場データが完全に欠落していて推測ベースを避けるべき場合

外部API・サブプロセスを持たないため、`[FAIL]`（試行後の失敗）は通常発生しない。
