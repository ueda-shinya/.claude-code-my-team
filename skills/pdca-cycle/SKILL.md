# pdca-cycle

PDCAサイクルの4フェーズを「いつ・何を・どう判定するか」のレベルまで具体化し、形骸化させずに回すための運用設計を出力する。

## 適用条件

- 使う場面：
  - プロジェクト・業務を継続改善したい
  - 立ち上げた施策が機能しているか定期検証したい
  - チームでPDCAを回す共通フォーマットが欲しい
  - 振り返りが「やりっぱなし」になっている
- 使わない場面：
  - 目標自体の設計（SMART化） → `smart-goal-setting`
  - 階層分解 → `goal-hierarchy-design`
  - モニタリングの仕組み化 → `goal-execution-system`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 対象プロジェクト/業務 | 必須 | PDCAを回す対象 | 「新規LP運用」「営業フロー」 |
| 目標 | 必須 | 達成すべき数値・状態 | 「月間CV30件」「失注率20%以下」 |
| サイクル期間 | 必須 | 1サイクルの長さ | 「1ヶ月」「2週間」「四半期」 |
| 既存の運用状況 | 任意 | 現在の運営体制 | 「週次MTGあり」「データ未取得」 |

## 実行ステップ

### ステップ1：Plan（計画）フェーズを設計する

- 入力：対象、目標、サイクル期間
- 処理：以下3つを定義する
  - **目標の明確化**：このサイクルで達成すべき数値・状態を1文で書く（SMART化推奨）
  - **実行計画の策定**：目標達成のための具体的アクションを3〜5個に絞り、優先順位を付ける
  - **リソース配分**：人員・予算・時間の配分を明記（誰が・どれだけ・何に使うか）
- フェーズ完了条件：
  - 目標がSMART化されている
  - アクション数が3〜5個に絞られている
  - リソース配分が数値で書かれている
- 出力：Planシート

### ステップ2：Do（実行）フェーズを設計する

- 入力：Planシート
- 処理：以下3つを定義する
  - **行動計画の実施**：Planで決めたアクションを、いつ誰が実行するかのスケジュールに落とす
  - **データ収集**：実行中に何のデータを取るかを定義する（例：CV数、CVR、CPA、商談数等）
  - **進捗記録**：記録の媒体（スプレッドシート/Notion/etc）と更新頻度（日次/週次）を決める
- データ収集ルール：
  - **必ず取るデータ**：目標KPIに直結する1〜3個（例：CV数、受注額）
  - **任意で取るデータ**：仮説検証用（例：流入元別CVR）
  - **取らないデータ**：「あったら便利そう」レベルは却下する（収集コストが効果を上回らない）
- 出力：Doシート（実施スケジュール＋データ収集計画）

### ステップ3：Check（評価）フェーズを設計する

- 入力：Doシートで集めたデータ
- 処理：以下3つを実施する
  - **進捗状況の確認**：目標KPIに対する達成率を計算する
  - **課題の特定**：未達の場合、原因仮説を3つ挙げ、データで裏付ける
  - **成果の測定**：達成・部分達成・未達の3区分で判定し、各KPIの達成率を表で示す
- 判定基準：
  - **達成（100%以上）**：そのままActionで横展開・スケール
  - **部分達成（60〜99%）**：Actionで部分修正
  - **未達（60%未満）**：Actionで根本見直し（場合によっては目標自体を再設定）
- 「課題の特定」では、原因を「環境要因／施策要因／実行要因」の3カテゴリに分類する
  - 環境要因：市場・競合・季節性等、自社でコントロール不可
  - 施策要因：施策の設計ミス・優先順位ミス
  - 実行要因：実行スピード・品質の問題
- 出力：Checkシート（進捗表＋課題分類）

### ステップ4：Action（改善）フェーズを設計する

- 入力：Checkシート
- 処理：以下3つを定義する
  - **計画の修正**：Checkの結果に基づき、次サイクルのPlanで何を変えるかを決める
  - **リソースの再配分**：効いた施策にリソースを寄せ、効かなかった施策から引き上げる
  - **新たな施策の実施**：既存施策で対応できない課題には新規施策を立てる
- Actionの判断ルール：
  - 達成 → 横展開する／規模を2倍にする
  - 部分達成 → 効いた要素を残し、効かなかった要素を差し替える
  - 未達 → 仮説の前提から見直す（顧客／競合／自社のどれかが間違っていた）
- 次サイクルのPlanへの引き継ぎ事項を明記する
- 出力：Actionシート（次サイクルへの引き継ぎ）

### ステップ5：PDCAサイクル全体を1枚にまとめる

- 入力：Plan/Do/Check/Action 4シート
- 処理：以下のテンプレートに統合する
  - 1サイクル分を1枚で見渡せる形にする
  - 次サイクル開始時に前サイクルのActionが反映されているかチェック欄を設ける
- 出力：PDCAサイクル運用シート

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| サイクル期間が1ヶ月以上 | 中間でCheckを1回入れる（半月時点） | 1ヶ月放置は軌道修正が遅すぎる |
| 達成率100%以上が3サイクル続く | 目標値を上げる | 簡単すぎる目標は成長を生まない |
| 未達が2サイクル続く | 目標自体を再設定 + 仮説の前提から見直す | 同じ目標で繰り返し失敗するのは前提が間違っている |
| 課題が「環境要因」と分類された | コントロール不可なので、内部要因の改善に集中 | 外部に原因を求めても改善できない |
| データが取れていない | Doフェーズの設計ミス。次サイクルのPlanで「何のデータを誰がいつ取るか」を最優先で決める | データなきPDCAはPDCAではない |
| アクションが10個以上ある | 3〜5個に絞る。残りは次サイクル以降に回す | 多すぎると優先順位が消える |
| Checkで「がんばった」「忙しかった」が出る | 数値で評価できていない。KPIを再定義する | 主観評価のPDCAは形骸化する |

### 記入例（要約）

新規LP運用 / 1ヶ月サイクル / 目標：CV月30件
- P: アクション「LP改善・広告配信・メルマガ」3つ、予算50万
- D: 週次でCV/CVR/CPAを記録
- C: 実績CV20件、達成率67%（部分達成）、施策要因「広告クリエイティブが弱い」
- A: 次サイクルで広告クリエイティブをABテスト、メルマガは廃止

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| Plan→Do→Check→Action のCがCheck（確認）止まり | Check で必ず「達成率＋課題分類」まで出す | 確認だけでは改善に繋がらない |
| Action が「次もがんばる」で終わる | 「継続/修正/廃止/新規」の4区分で具体施策を決める | 抽象的なActionは次サイクルに伝わらない |
| データ収集が場当たり的 | Doフェーズで取るデータを事前に定義する | 後から「データなくて評価できない」が頻発 |
| KPIを5個以上設定 | 必須KPIは1〜3個に絞る | 多すぎると焦点が分散する |
| 未達が続いても同じ目標で続行 | 2サイクル連続未達なら目標と前提を再設定 | 前提が間違っている可能性が高い |
| 環境要因を内部要因と混同する | 課題を3カテゴリ（環境/施策/実行）に分類する | コントロール不可な要因に時間を使うのは無駄 |
| 主観評価で終わる（「がんばった」） | 必ず数値で達成率を計算する | 主観PDCAは形骸化の温床 |
