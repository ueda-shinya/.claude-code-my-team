# pm-risk-management スキル

プロジェクトマネジメントにおけるリスク管理（QCDズレを防ぐプロセス）を、3プロセス×8手法×4アプローチで体系設計する。

## 適用条件

- このスキルを使う場面：
  - PJ計画段階でリスクを洗い出し、対応策を事前準備したい
  - 進行中のPJで発生確率の高いリスクを特定し、優先順位をつけたい
  - 過去のPJで発生したリスク（スコープ変更／納期遅延／品質問題）の再発を防ぎたい
  - クライアントとリスク認識を共有し、対応プランを合意したい
- このスキルを使わない場面：
  - ソフトウェア実装直前のラウンドテーブル形式の危険予知（KYT/pre-mortem） → `kyt`
  - PJ体制（人員・役割・組織図）の設計 → `project-team-structure`
  - PJ収支管理・契約締結 → `project-finance-contract`
  - スケジュール表（ガントチャート）作成 → `gantt-chart-design`
  - クライアント期待値調整の心理的ハンドリング → `client-expectation-management`
  - 全社・事業部レベルの経営リスク管理 → 本スキルは「プロジェクト単位」スコープ

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| プロジェクト概要 | 必須 | PJ名／目的／スコープ | 「マーケ戦略コンサル6ヶ月」 |
| QCD合意内容 | 必須 | 品質基準／予算／納期 | 「品質：戦略文書納品・予算300万円・納期2026-09末」 |
| ステークホルダー | 必須 | クライアント／社内／外部協力者 | 「クライアント1社・社内3名・外注1名」 |
| 既存のリスク認識 | 任意 | 既に把握している懸念事項 | 「キーマン1名の体調不安／法改正の可能性」 |
| 解決したい課題 | 必須 | リスク特定／対応策準備／優先順位付け 等 | 「リスク管理の仕組みがなく場当たり対応している」 |

## 実行ステップ

### ステップ1：リスク管理の定義と3重要性を確認する

#### 定義
リスク管理とは、企業や個人が直面するリスクを **特定・評価し、それに適切に対応するプロセス**。主な目的は損失を未然に防ぎ、影響を最小限に抑えること。リスクを管理することで、計画やプロジェクトがスムーズに進行し、戦略的目標を達成できる環境を整える。

#### 3重要性
| 観点 | 内容 |
|---|---|
| 損失の最小化 | リスクが発生した場合の影響を軽減する |
| 予測可能性の向上 | 不確実性を減らし、事業計画を安定させる |
| 企業価値の向上 | 信頼性を維持し、競争力を高める |

### ステップ2：プロジェクトにおけるリスクの定義（QCDズレ）を確認する

プロジェクトマネジメントにおける **「リスク」** とは、最初に想定・合意した **QCD**（**Quality**：品質、**Cost**：コスト、**Delivery**：納期/期限）が大きくズレる危険性 を指す。

### ステップ3：リスクの種類を内的9要因×外的3要因で洗い出す

| 分類 | 要因 | 例 |
|---|---|---|
| 内的要因 | 企業方針／運営 | 経営方針変更／組織再編 |
| 内的要因 | スコープ | 要件追加／仕様変更 |
| 内的要因 | スケジュール／時間 | 想定工数オーバー／前後工程の遅延 |
| 内的要因 | コスト | 原価高騰／追加工数発生 |
| 内的要因 | 品質 | 品質基準未達／成果物の不備 |
| 内的要因 | コミュニケーション | 認識ずれ／情報共有不足 |
| 内的要因 | 人的資源 | キーマン離脱／病欠／引き継ぎ不良 |
| 内的要因 | 技術／スキル | 技術的難度／スキルミスマッチ |
| 内的要因 | セキュリティ | 情報漏洩／不正アクセス |
| 外的要因 | 市場／社会の変化 | 競合参入／顧客行動変化 |
| 外的要因 | 法令／規制の変化 | 個人情報保護法改正／業界規制強化 |
| 外的要因 | 災害関連 | 地震／パンデミック／インフラ障害 |

#### 例
- 内的：「スコープの変更要望によりスケジュールが間に合わない可能性」
- 外的：「円安により調達費が大幅に増加」

### ステップ4：リスクの見つけ方8手法でリスクを洗い出す

| 手法 | 内容 |
|---|---|
| ステークホルダー確認 | ステークホルダー全員に影響しそうな要因をヒアリングし、潜在的な懸念事項を収集する |
| 過去プロジェクト分析 | 類似した過去のプロジェクトの問題点や教訓をレビューし、共通リスクを特定する |
| チェックリスト作成 | 業界標準や過去の経験を元にしたリスクチェックリストを使用し、抜け漏れを防ぐ |
| スケジュール精査 | プロジェクトのスケジュールを詳細に確認し、遅延やリソース不足の可能性を特定する |
| 依存関係確認 | 外部要因や他チームとの依存関係を精査し、影響を受ける可能性を洗い出す |
| 専門家ヒアリング | 専門家から直接インタビューを行い、潜在的なリスク情報を得る |
| シミュレーション実施 | プロジェクト計画を仮想的に進行させ、発生し得る問題を事前に発見する |
| 文書レビュー | 契約書や仕様書、設計書などを精査し、抜けや矛盾から発生するリスクを特定する |

8手法は併用が原則。1〜2手法だけだと網羅性が下がるので、最低でも3手法以上を組み合わせる。

### ステップ5：リスク管理3プロセスを実行する

| プロセス | 内容 |
|---|---|
| 01 リスクの特定 | 発生しうるリスクを洗い出す／手法：ブレーンストーミング、過去データ分析（→ステップ4の8手法を活用） |
| 02 リスクの評価 | 発生確率と影響度を分析／ツール：リスクマトリックス（発生確率×損害額の2軸4象限） |
| 03 対応策の決定 | 回避：リスクの発生源を排除／軽減：リスクの影響を減少／移転：保険などでリスクを第三者に転嫁／受容：リスクを許容する |

### ステップ6：対応4アプローチをマトリクスで分類する

リスク対応策として一般的に用いられる「回避」「軽減」「移転」「受容」の4つのアプローチがある。**発生確率×損害額の2軸マトリクス** で分類する。

```
       損害額
        ↑高
        |
  移転  |  回避
        |
  ───────────── 発生確率
  受容  |  軽減
        |
        ↓低
       低 ──→ 高
```

| 象限 | 対応 | 例 |
|---|---|---|
| 発生確率高×損害額高 | 回避 | 高リスク市場での事業展開を中止 |
| 発生確率低×損害額高 | 移転 | 災害保険への加入 |
| 発生確率高×損害額低 | 軽減 | データのバックアップをクラウドに保存 |
| 発生確率低×損害額低 | 受容 | 対策せずモニタリングのみで様子を見る（例：軽微なシステムエラー等） |

### ステップ7：成功3ポイントで自己点検する

最後に下記3ポイントで設計が完成しているか自己点検する。

1. **リスクの特定と評価を怠らない**：見落としがないよう、網羅的にリスクを洗い出す
2. **柔軟かつ迅速な対応策を準備**：状況に応じた適切な対応を行う
3. **定期的な見直しと改善**：リスク管理プロセスを継続的に改善する

### ステップ8：継続的モニタリングと次のアクションを設計する

リスク管理は一度実施して終わりではなく、PJ進行中に継続的に見直す。

| 観点 | 実施内容 |
|---|---|
| リスクの洗い出し | 職場やプロジェクトで発生しうるリスクをリスト化し、定期的に追加・削除する |
| リスクの評価 | 発生確率と影響度を基に、優先的に対応すべきリスクを定期的に再評価 |
| 対応策の計画 | 回避、軽減、移転、受容の中から適切な対策を選択し、責任者と期限を明示 |
| モニタリングと改善 | 定期的にリスク状況を確認し、対応策を更新する習慣をつける（週次／月次） |

## 判断基準

| 状況 | 判断 |
|---|---|
| PJ立ち上げ初期（キックオフ後1〜2週間） | ステップ1〜6を集中実施しリスク登録簿を作成 |
| PJ進行中で問題が発生 | ステップ4「リスクの見つけ方8手法」のうち「ステークホルダー確認」「依存関係確認」を最優先 |
| 過去PJで再発リスクが多い | ステップ4「過去プロジェクト分析」「チェックリスト作成」を強化 |
| 発生確率高×損害額高のリスクを発見 | 即「回避」を検討（PJスコープ変更／中止判断） |
| 発生確率低×損害額高のリスクを発見 | 「移転」を検討（保険／契約条項／ベンダー分散） |
| 発生確率高×損害額低のリスクを発見 | 「軽減」策を実装（バックアップ／冗長化／チェックリスト） |
| 発生確率低×損害額低のリスクを発見 | 「受容」（モニタリングのみで十分） |
| 実装直前のソフトウェア開発リスク | `kyt` に切り替え（ラウンドテーブル形式のpre-mortem） |
| PJ体制設計が必要 | `project-team-structure` を並走 |
| 収支リスク中心の管理が必要 | `project-finance-contract` を並走 |
| 全社・事業部レベルの経営リスク | 本スキルはPJ単位スコープ。経営リスク管理は別途検討 |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 「リスク管理＝何かあったら対応」と事後対応で運用 | 事前に「特定→評価→対応決定」の3プロセスでリスク登録簿を整備 | 場当たり対応はQCD大きなズレに直結 |
| 内的要因（スコープ／スケジュール）だけ見て、外的要因（市場／法令／災害）を無視 | 内的9要因×外的3要因を網羅的に確認 | 外的要因の見落としで致命的損失 |
| 1〜2手法だけでリスク特定を済ませる | 8手法から最低3手法以上を併用 | 網羅性が下がり重要リスクを見逃す |
| リスクマトリクスで分類せず、すべて「全力対応」 | 発生確率×損害額の2軸4象限で「回避/軽減/移転/受容」を分類 | リソース集中ができず重要リスク対策が薄まる |
| リスク登録簿を1度作って放置 | 週次／月次で見直し、対応状況を更新する | リスクは時間とともに変化する／登録簿は生きたドキュメント |
| 発生確率低×損害額低のリスクまで対策しようとする | 「受容」と判断してモニタリングのみに留める | リソース過剰投入で本当の重要リスク対策が遅れる |
| ソフトウェア実装直前のリスクを本スキルで対応 | `kyt`（ラウンドテーブル形式pre-mortem）に切り替え | スコープが違う／本スキルはPJ全体QCD観点 |
