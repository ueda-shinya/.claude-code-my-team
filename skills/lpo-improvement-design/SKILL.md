# lpo-improvement-design

既存LPのCVRを高めるLPO（ランディングページ最適化）の **現状診断・改善仮説立案** スキル。6ステップ運用プロセス＋A/B検証2観点＋3大改善ポイント（FV/CTA/フォーム）＋4要素構成（結果・実証・信頼・安心）で改善優先順位を提示する。

**重要：本スキルの責任範囲は「現状診断＋改善仮説立案＋優先順位設計」までであり、コピー変更／デザイン変更／実装変更には踏み込まない**。CLAUDE.md「LP案件の起動順序ルール」遵守のため、改善実装フェーズは以下の経路で `lp-create` Route 判定を必ず経由する：
- コピー変更（CTA文言／FV見出し／本文等） → `lp-create` Route C へ
- デザイン変更（CTA色／レイアウト／FV画像等） → `lp-create` Route D へ
- 実装変更（フォーム項目削減／追従CTA実装等） → `lp-create` Route I へ
- 戦略から作り直し → `lp-create` Route F または Route S へ

## 適用条件

- 使う場面：
  - 既存LPのCVRが目標未達で改善が必要
  - 広告予算を増やしてもCVが増えずCPAが上昇している
  - LPの直帰率が高く（FV直下で75%離脱が業界平均）対策が必要
  - フォーム入力完了率が低く改善したい
  - A/Bテストの仮説と改善シナリオを設計したい
- 使わない場面：
  - LPを新規制作する → `lp-create`
  - LPセクションのデザイン仕様 → `lp-design-system`
  - HP/LPどちらを作るかの判断 → `hp-lp-distinction-design`
  - LPコピーの執筆 → `ad-copy-7-angles` ／ Koto（copywriter）
  - サイト全体のUI/UX改善 → `ui-ux-improvement-fundamentals`
  - 流入減・CV減の1次診断 → `ga-gsc-diagnosis`
  - 広告のCPA分析 → `ad-performance-diagnosis`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 対象LP URL | 必須 | 改善対象LPのURL | 「https://example.com/lp」 |
| 現状KPI | 必須 | 現在のCVR・CV数・直帰率等 | 「CVR 1.2%、月間CV 15件」 |
| 目標KPI | 必須 | 改善後の目標値 | 「CVR 2.0%、月間CV 25件」 |
| 流入経路 | 推奨 | 検索／広告（Meta／リスティング）／メール | 「Meta広告メイン」 |
| 既存データ | 推奨 | GA・ヒートマップ・広告データの保有状況 | 「GA4＋Microsoft Clarity」 |
| LP目的 | 必須 | リード獲得／資料DL／無料トライアル／問合せ／購入 | 「無料トライアル獲得」 |

## 実行ステップ

### ステップ1：LPOの目的と重要性を整理する

- 入力：LP目的、現状KPI、目標KPI
- 処理：LPOの3つの重要性を整理
  1. **コンバージョン率向上**：訪問者が求める情報を的確に提供し、直感的に操作しやすいデザインで CVR を高める
  2. **広告費用対効果の改善**：CV増加で広告費用対効果向上、無駄な広告費削減
  3. **ユーザーエクスペリエンスの改善**：UXの質を向上させ、ブランド信頼感・満足度・リピート訪問・口コミによる新規獲得につなげる
- LPOで最適化する5要素：見出し／画像／文章／CTAボタン／ページ構成
- 対象LP例：新商品プロモーションページ／資料請求フォーム／キャンペーン専用ページ／サインアップページ
- 出力：LPO目的＋現状KPI/目標KPIギャップ

### ステップ2：6ステップ運用プロセスを開始する

- 入力：ステップ1の目的、対象LP URL
- 処理：以下6ステップで運用設計

#### STEP1：目標・KPI・評価基準を決める

- LP運用の目的・KPIを決定
  - 例：目的：リード獲得／サブスク商品の初回購入
  - KPI：リード数／LINE登録者数／購入完了数
- LPO各施策の評価対象・評価方法・評価基準を決定
  - 例：評価対象＝CTAのマイクロコピー追加
  - 評価方法＝各CTAボタンのクリック数を追加前後で比較
  - 評価基準＝クリック数が110%以上改善したら有意差あり

#### STEP2：検証仮説と改善シナリオをたてる

- 検証仮説①→結果〇なら検証終了・継続／結果×なら検証仮説②へ
- 検証仮説②→結果〇なら検証終了・継続／結果×なら検証仮説②の改善案①へ
- ※シナリオ式に「次の打ち手」を事前設計しておく（行き当たりばったりの仮説不可）

#### STEP3：運用結果を分析する

- 使用データ：ヒートマップ／GoogleAnalyticsアクセスデータ／広告データ（広告利用時）
- 改善点を洗い出す

#### STEP4：問題と改善策をリストアップする

- 観点：デザイン／ワイヤー／コーディング／状態（対応済 or 未対応）／対応者／対応期日／実施日／効果ポイント／検証ポイント／結果

#### STEP5：改善の優先順位を決めて、実行する

- 優先順位はROI（効果×実現容易性）で判断

#### STEP6：改善策の結果を評価し、PDCAを繰り返す

- STEP3と4を繰り返し実施

- 出力：6ステップ運用計画

### ステップ3：A/B検証2観点を選ぶ

- 入力：ステップ2の仮説
- 処理：以下2観点から検証アプローチを選択
  - **観点A：ターゲットと訴求**
    - ターゲットの仮説と、そのターゲットのニーズに対する訴求を出しわけて検証
    - テスト例：メッセージング・見せ方の検証／エリア別だし分けの検証
    - 訴求パターン例：「中央区にお住まいのあなたへ」「今28歳のあなた」「速さ訴求」「シーン訴求」
  - **観点B：UIUX**
    - ボタンの色や導線、LPの表示速度、モバイル対応などUIUXに関わる改善と検証
    - テスト例：LPの長さの検証／LPの表示速度改善の検証／CTAボタンのUI検証
- 出力：選定した検証観点（A/B両方並行可）

### ステップ4：3大改善ポイント（FV/CTA/フォーム）から優先着手する

- 入力：ステップ3の観点
- 処理：以下3つの順で優先着手（最初の3つが効果最大）

#### 改善ポイント①：ファーストビュー（FV）の改善

- 着眼点：①CTAの位置 ②キャッチコピー
- 重要事実：**ファーストビュー直下で75％が離脱する**
- ①CTAの位置：
  - 分かりやすい位置にある
  - 開いたときにスクロールしないで見える位置に収まっている
  - 例：Chatworkのファーストビジュアル
- ②キャッチコピー：
  - 一言で、誰のための何のサービスなのかが分かる
  - 例：UPSIDER「上場のための法人カード」

#### 改善ポイント②：CTAの改善

- **改善案①：CTAの内容の変更**（CVハードル階段で調整）
  - 代表例：お問い合わせ→資料ダウンロード（ハードル下げ）
  - 注意：CVハードルを下げると見込みの薄いリードが集まりやすくなる。ほしいリードの温度感とのバランスで決定
  - CVハードル階段（低→高）：資料ダウンロード ＜ 無料トライアル ＜ アポ設定/問合せ
- **改善案②：CTA周りのUIの変更**
  - ボタンの色の変更
  - ボタンの中の文字を動詞にする（例：「資料を請求する」「無料トライアルに申込む」）
  - マイクロコピーをつける（例：「プラン・機能の詳細を知りたい方はこちら」「まずは30日間お試し」）
  - 追従型のCTAを設置する（バナー型／フッター型）

#### 改善ポイント③：フォームの改善

- **重要事実**：フォーム項目は1個増えるごとに完了率が8%下がる
- 最低限必要な項目：会社名／氏名／メールアドレス／（電話番号）
- それ以外のリード管理用項目は入力必須・任意をよく選定
- 参考例（SmartHR）：
  - フォーム項目とCTAボタンが1画面内に収まっている
  - フォームに薄文字でガイドが入っている（何を入力したらいいかがわかる）
  - 資料DLフォームには資料のチラ見せがある

- 出力：3大改善ポイントの優先施策

### ステップ5：その次の改善ポイント5観点を順次適用する

- 入力：ステップ4の進捗
- 処理：以下5つの改善ポイントから順次適用

#### ④LP構成・コンテンツの改善

- LP基本構成は **「結果」「実証」「信頼」「安心」** の4要素で構成される
  - **結果**：CVRの改善実績や導入件数など、結果が定量で分かる訴求
  - **実証**：具体的な事例など結果の裏付けになる事実
  - **信頼**：お客様の声や選ばれる理由など、サービスの信頼要素
  - **安心**：会社概要や監修体制、ISO認証など、安心要素
- 基本構成のコンテンツに漏れがないか、また網羅されている場合はアクセス分析（ヒートマップやGA分析）からコンテンツの上下入れ替えや追加削除を検討

#### ⑤LPの長さの検証

- LP長さ別で検証
- 一般則：長すぎると離脱するので、短くする方がCVRが高くなる傾向
- 例外：toCで記事で長く読ませるタイプのLPは長いほうがCVRが高くなる場合もある

#### ⑥訴求別のだし分け検証

- ターゲットやニーズの仮説が複数ある場合はファーストビジュアルで訴求をだし分けし、A/Bテストを行う
- 例：エリア別に広告クリエイティブとLPを出しわける
- 訴求例：「中央区にお住まいのあなたへ」「今28歳のあなた」「速さ訴求」「シーン訴求」

#### ⑦速度改善・モバイル対応

- LPが表示されるまでに時間がかかると離脱する
- PageSpeed Insight（https://pagespeed.web.dev/）で現時点LP成績を調べて改善
- toCのLPは特にモバイル対応を重要視

#### ⑧FV内フォーム・追従フォーム

- ファーストビジュアル内にフォームを設置／右4分の1エリアをフォーム表示にする
- メリット：フォームのインパクトが強いので入力されやすくなる
- デメリット：FVで訴求できるエリアが少なくなる
- 例：Fileforce（FV内にフォーム）／TACT SEO（右サイドに固定でフォーム）

- 出力：5観点の改善実行計画

### ステップ6：改善内容を洗い出すワーク（4ステップ）

- 入力：ステップ4・5の結果
- 処理：以下4ステップで改善内容をクライアントワーク形式で整理
  1. **現状の課題を特定**：現在のLPで直帰率が高いページや、コンバージョンが低いポイントを分析
  2. **ゴールを設定**：「購入率を10%向上」「資料請求数を1.5倍にする」など目標を明確化
  3. **改善案の検討**：「ファーストビューの強化」「CTAボタンの文言変更」「画像や構成の見直し」などを具体化
  4. **テストと実施計画の立案**：実現可能な改善案を選び、テストと実施を計画
- 出力：改善内容の運用計画書

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| LPの直帰率が80%超え | ステップ4-①（FV改善）から着手 | FV直下で75%が離脱する事実 |
| CTA クリック率が低い | ステップ4-②（CTA改善）から着手 | クリックがなければCV発生しない |
| フォーム到達率は高いが完了率が低い | ステップ4-③（フォーム改善）優先 | 項目1個=完了率8%減 |
| 広告予算増えるがCVが増えない | ステップ4の3大改善全部+ステップ5-⑦速度改善 | 流入の品質ではなくLPがボトルネック |
| ターゲット仮説が複数ある | ステップ3観点A（ターゲット訴求）でA/B | エリア別／年齢別出しわけ |
| toC事業のモバイル比率高い | ステップ5-⑦のモバイル対応最優先 | 業態的にモバイル離脱が致命的 |
| 検討期間が長いBtoB商材 | ステップ5-⑤でLP長め＋ステップ5-④で実証強化 | 長文で信頼を積み上げる |
| 即決系BtoC商材 | ステップ5-⑤でLP短め＋ステップ4-①でFV強化 | 短時間で意思決定させる |
| 広告流入のリードクオリティが低い | ステップ4-②でCVハードル上げ（資料DL→アポ問合せ） | 質を担保 |
| ハウスリストの育成が課題 | ステップ4-②でCVハードル下げ（アポ問合せ→資料DL） | 量を増やしてナーチャリング |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| FVを変えずにフォーム改善着手 | ステップ4-①（FV改善）から着手 | FV直下で75%離脱、フォームに到達する人が少なすぎる |
| フォーム項目を増やしてリード品質を担保 | 必須項目を最低限（会社名/氏名/メール/電話）に絞る | 項目1個=完了率8%減で量が激減 |
| CVハードルを最低まで下げる | リード温度感とのバランスで決める | 見込みの薄いリードが大量発生して営業効率悪化 |
| 仮説①が×でも次の打ち手が決まっていない | ステップ2で仮説②/改善案①を事前設計 | 行き当たりばったりは時間と予算の浪費 |
| 検証期間が短すぎる/サンプル不足 | 評価基準（クリック数110%改善で有意差等）を事前定義 | 偶然の差を本物の差と誤認 |
| 全部一気に変える | 1要素ずつA/Bテスト | 何が効いたか分からない |
| toCでデスクトップ最適化のみ | ステップ5-⑦でモバイル対応最優先 | toCはモバイル比率が圧倒的 |
| LP構成「結果」だけ強化 | 4要素（結果・実証・信頼・安心）を網羅 | 要素欠如はLPの信頼性低下 |
