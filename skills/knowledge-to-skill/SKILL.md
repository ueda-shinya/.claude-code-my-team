# knowledge-to-skill スキル

**知識（What/Why）を実行スキル（How）に変換する。**

このスキルの目的は「知っている」を「できる」に変えること。
情報の解説・説明は最小限にし、実行手順を最大化する。
1つのソース情報から複数のスキルを抽出・生成できる。

## skill-creator / agent-study との違い

| 観点 | skill-creator | knowledge-to-skill | agent-study |
|---|---|---|---|
| 起動条件 | 「こういうスキルを作りたい」（ゼロから設計） | 「この資料/情報からスキルを作って」（既存情報を変換） | 「この知識をエージェントに覚えさせて」 |
| 入力 | 要件・仕様のヒアリング | 既存のドキュメント・URL・テキスト（ソース情報が必須） | 参照資料・知識 |
| 出力先 | 独立したスキルファイル（SKILL.md） | 独立したスキルファイル（SKILL.md）※複数生成可 | エージェント定義に知識ベースを追記 |
| 中身 | ヒアリングから設計した実行手順 | ソース情報から変換した実行ステップ・判断基準・出力テンプレート | 定義・概念・フレームワークの説明 |
| 効果 | エージェントが「実行できる」 | エージェントが「実行できる」 | エージェントが「知っている」 |

**判断基準：**
- ソース情報がなく、要件だけある → skill-creator
- ソース情報があり、手順化・ルール化して使いたい → knowledge-to-skill
- 情報をそのまま参照すれば役立つ → agent-study

## 実行手順

### ステップ1：入力を特定する

ユーザーの指示から以下を読み取る：

- **ソース情報**：ファイルパス、URL、またはテキスト
  - ファイルパスの断片があれば Glob で検索して特定する
  - URL があれば WebFetch で取得する
  - テキストが直接貼られていればそのまま使用する
- **対象エージェント**（任意）：名前・ニックネームからマッピングで解決する
- **スキル名**（任意）：指定があればそれを使う。なければステップ3で命名する
- **スキル指定の有無**：ユーザーが「〇〇をスキル化して」と特定のスキルを1つ指定しているか確認する

ソース情報が特定できない場合のみユーザーに確認する。

### ステップ2：ソース情報を読み込み・分析する

Read / WebFetch でソース情報を全文取得し、**まず変換可否を判断する**。

#### 変換可否の判断（最初に実施）

ソース情報の性質を確認し、以下の基準で処理を分岐する：

| ソース情報の性質 | 判断 | 処理 |
|---|---|---|
| フレームワーク、話法、プロセス、手法、戦略 | 手順化可能 | knowledge-to-skill で変換を続行する |
| 用語集、定義集、歴史年表、事実の羅列 | 手順化不能 | agent-study に回す（ユーザーにその旨を報告する） |
| 原則・哲学・マインドセット | グレーゾーン | 「判断基準」と「適用条件」に変換できるか試みる。変換できるなら続行、できないなら agent-study に回す |

判断に迷った場合はユーザーに「このまま手順化するか、知識ベースとしてエージェントに追記するか」を確認する。

#### 抽出する7要素

変換可能と判断した場合、以下の7要素を抽出する。
これが変換のコアロジックであり、このスキルの核心。
※ 7要素目（4状態出力契約）は 2026-04-21 kaizen で追加された品質ゲート。

**1. 実行ステップ（必須・最重要）**
「何を、どの順番で、どうやるか」を具体的な手順に分解する。

- 抽象的な説明 → 具体的な動詞で始まるアクションに変換する
- 「〇〇を考慮する」→ NG。「〇〇を確認し、Aなら→X、Bなら→Yを実行する」→ OK
- 順序依存がある場合は番号付きで明示する
- **各ステップは以下の構造を持つこと：**
  - **入力**：このステップで受け取るもの
  - **処理**：具体的なアクション（動詞で記述）
  - **出力**：このステップで生成するもの
- 知識の説明がステップ内に紛れ込んでいないか確認する

**NG例（説明であって実行ではない）**：
```
ステップ1：SWOTの4象限を理解する
Sは強み（Strength）、Wは弱み（Weakness）、Oは機会（Opportunity）、Tは脅威（Threat）である。
内部環境と外部環境に分けて整理する手法で...
```

**OK例（実行手順になっている）**：
```
ステップ1：ヒアリング項目を提示する
- 入力：クライアント名、業種
- 処理：以下の質問リストをクライアントに提示する
  - S: 「競合にない自社だけの強みは？」
  - W: 「社内で不満が出ている業務は？」
  - O: 「業界で伸びている領域は？」
  - T: 「競合の新サービスで脅威なものは？」
- 出力：各象限ごとの回答リスト
```

**2. 判断基準（必須）**
「どういう場合に、どう判断するか」の条件分岐を明文化する。

- if/then 形式で書く：「〇〇の場合 → Aを選択」「〇〇でない場合 → Bを選択」
- 曖昧な表現（「適切に」「状況に応じて」）は具体的な条件に置き換える
- 判断に迷うケースのデフォルト動作を定義する

**3. 入力と出力の定義（必須）**
「何を受け取って、何を出すか」を明確にする。

- 入力：必須項目と任意項目を分ける。各項目の型・例を示す
- 出力：フォーマット・構成・具体例を示す

**4. 適用条件（必須）**
「いつ・どんな場面で使うか」のトリガーを定義する。

- このスキルが有効な場面を列挙する
- このスキルを使うべきでない場面も明記する

**5. 出力フォーマット（必須）**
実際のアウトプットのテンプレートを作成する。

- プレースホルダー付きのテンプレートを記載する
- 記入例も1つ添える

**6. よくある失敗パターン**
情報から読み取れる「やりがちなミス」とその回避方法を記載する。

- NG例とOK例を対比で示す
- なぜNGなのかの理由を1行で添える

**7. 5状態出力契約（必須チェック・2026-04-21 追加）**
スキルが CLAUDE.md「スキル出力の5状態契約」の対象判定基準3箇条（外部ネットワーク呼び出し／サブプロセス実行／外部設定依存）のいずれかに該当する場合、以下の品質チェックを**全て**満たすこと。

- [ ] 5状態（データあり・該当なし・部分成功・失敗・未実行）それぞれの出力仕様が定義されている
- [ ] 失敗時に「セクション丸ごと非表示」になる構造がない（セクションタイトルは残し、本体に `[FAIL]` マーカーと詳細ログパスを表示）
- [ ] 各状態に応じた明示マーカーが出力フォーマットに含まれている（半角英語タグ固定・日本語化禁止）：
  - 成功・データあり：マーカーなし（通常表示）
  - 成功・該当なし：`[OK:0件]`
  - 部分成功：`[PARTIAL:N成功/M失敗]` ＋ 成功分データ＋失敗分 `[FAIL]` 明細
  - 失敗：`[FAIL]` ＋ エラー要約＋ログパス
  - 未実行/スキップ：`[SKIP]` ＋ スキップ理由
- [ ] `try/except: pass` 相当の無言スキップが実装されていない（HTTPステータス・例外・戻り値の明示的検査を行う）
- [ ] **制御フロー伝播禁止**: 1ステップの失敗を後続ステップが「成功扱い」で受け取らないこと。前段失敗時は後段に `[FAIL]` または `[SKIP]`（依存失敗として）を必ず伝播する
- [ ] マーカーは**ユーザーが見る最終出力**に必ず載せる（別ファイルのログだけに書いて本体を沈黙させる実装は禁止）

**この7要素チェックが完了しない限り、SKILL.md の生成は完了扱いにしない。**

### ステップ2.5：スキル候補の洗い出し（複数スキル抽出）

**このステップの発動条件：**
- ユーザーがスキル名を1つ指定している場合 → **このステップをスキップし、ステップ3に直接進む**
- ソース情報が広範（本1冊、長いドキュメント、複数の手法・フレームワークを含む等）で、独立したスキルとして成立する手法が2つ以上抽出できる場合 → **このステップを実行する**

#### 2.5-1：スキル候補を一覧化する

ステップ2で分析したソース情報から、「独立したスキルとして成立する」手法・フレームワーク・プロセスを全て洗い出す。

**「独立したスキルとして成立する」の判断基準：**
- それ単体で実行ステップ（3ステップ以上）を構成できる
- 他のスキルと組み合わせなくても単独で成果物を出せる
- 適用場面が明確に定義できる

**候補ごとに以下を整理する：**

| # | スキル名候補 | 何ができるようになるか（1行） | 対象エージェント（推定） |
|---|---|---|---|
| 1 | 例：heaven-hell-rhetoric | 理想と最悪の対比で行動を促すトークを組み立てる | タク（sales-consultant） |
| 2 | 例：spin-selling | SPIN話法で商談のヒアリングを構造化する | タク（sales-consultant） |
| 3 | 例：objection-handling | 顧客の反論に対する切り返しパターンを生成する | タク（sales-consultant） |

#### 2.5-2：ユーザーに選択を求める

候補一覧を提示し、以下の3択で確認する：

```
この情報から以下のスキルが作れそうです：

| # | スキル名候補 | 概要 | 対象エージェント |
|---|---|---|---|
| 1 | ... | ... | ... |
| 2 | ... | ... | ... |
| 3 | ... | ... | ... |

どうしますか？
1. 全部作る
2. 番号を選んで作る（例：「1と3」）
3. 1つだけ指定する
```

- 「全部作る」→ 全候補をステップ3〜6で順次生成する
- 「番号を選んで作る」→ 指定された候補のみ順次生成する
- 「1つだけ指定する」→ 指定された1つのみ生成する（従来フローと同じ）

#### 2.5-3：生成順序を決定する

複数スキルを生成する場合、以下の順序で処理する：
- 依存関係がある場合 → 前提となるスキルを先に生成する
- 依存関係がない場合 → 候補一覧の番号順に生成する

**以降、選択されたスキルごとにステップ3〜4を繰り返す。全スキルのステップ4が完了した後、ステップ5（エージェント参照追記）をエージェント単位でまとめて実行し、最後にステップ6で一括報告する。**

### ステップ3：スキル名を決定する

スキル名が未指定の場合、以下の基準で命名する：

- 動作を表す英語のケバブケース（例：`swot-analysis`, `heaven-hell-rhetoric`）
- 短く、何をするスキルか推測できる名前にする
- 単一スキル生成の場合：ユーザーに提案して確認を取る
- 複数スキル一括生成の場合：ステップ2.5で提示済みの候補名をそのまま採用する（ユーザーが候補一覧を承認済みのため再確認は不要）

### ステップ4：SKILL.md を生成する

以下のテンプレートに従い、`~/.claude/skills/<スキル名>/SKILL.md` を生成する。

```markdown
---
name: <スキル名>
description: <1文で「何ができるようになるか」を書く。起動トリガーとなるフレーズも含める>
tools: <必要最小限のツール>
external_dependencies:
  - type: <notion_db | mcp_server | external_api | subprocess | local_file | env_config | none>
    id_env: <環境変数名 / 該当なしの場合は null>
    schema_source: <スキーマ定義のパスまたは公式ドキュメントURL / なければ null>
    verification_method: <実際のスキーマを確認する方法（コマンド / 参照先 / 疎通確認の手順）>
---

# <スキル名>

<1-2行で「このスキルで何が実行できるか」を書く。知識の説明は書かない>

## 適用条件

- このスキルを使う場面：（箇条書き）
- このスキルを使わない場面：（箇条書き）

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| ... | ... | ... | ... |

## 実行ステップ

### ステップ1：<動詞で始まるアクション名>
- 入力：<このステップで受け取るもの>
- 処理：<具体的なアクション>
- 出力：<このステップで生成するもの>

### ステップ2：<動詞で始まるアクション名>
- 入力：<このステップで受け取るもの>
- 処理：<具体的なアクション>
- 出力：<このステップで生成するもの>

...

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| ... | ... | ... |

## 出力フォーマット

（テンプレート）

### 記入例

（具体的な例）

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| ... | ... | ... |
```

**記入例：**

```yaml
# 例1: Notion DB依存スキル
external_dependencies:
  - type: notion_db
    id_env: NOTION_TASKS_DB_ID
    schema_source: ~/.claude/scripts/notion_schema.py::TasksDB
    verification_method: python ~/.claude/scripts/notion-tasks.py --schema-check

# 例2: 外部依存なし
external_dependencies:
  - type: none
    reason: 純粋なテンプレート展開のみ（例：コピー生成）
```

**生成ルール：**
- 「実行ステップ」セクションが、他のどのセクションよりも長いこと（最大セクションであること）
- 知識の説明・背景解説は各セクション冒頭の1-2行以内に収めること
- 各ステップは具体的な動詞で始めること（「確認する」「分類する」「生成する」等）
- 各ステップは「入力・処理・出力」の構造を持つこと
- 曖昧な表現（「適切に」「必要に応じて」「考慮する」）は具体的な条件分岐に置き換えること
- 生成されたスキルは、そのまま `/スキル名` で実行できる品質にすること

**external_dependencies 記述ルール（2026-05-02 追加・kaizen Phase 2-B）：**

`type` 値の判定基準（優先順位順・複数該当時は上位を採用）：

| 優先順位 | type 値 | 判定境界 | 例 |
|---|---|---|---|
| 1 | `notion_db` | Notion DB を直接または MCP 経由で参照 | `NOTION_TASKS_DB_ID` を読むスキル |
| 2 | `mcp_server` | MCP プロトコル経由で外部サービス呼び出し（Notion 以外） | `mcp__hourei__*`, `mcp__search-analytics__*` |
| 3 | `external_api` | HTTP API を直接呼ぶ（subprocess や cURL 経由含む） | Anthropic SDK 直接、Gemini API 直叩き |
| 4 | `subprocess` | 外部 CLI を subprocess 起動（外部API呼び出しを伴わない） | `marp` CLI、ローカル処理のみ |
| 5 | `local_file` | ローカルファイルを構造前提で読む | `clients/<name>/README.md` 構造前提 |
| 6 | `env_config` | `.env` の特定変数を必須前提（API認証用 .env は external_api に含めるため、本タイプは認証以外の環境変数 例: PC_PLATFORM を対象） | `PC_PLATFORM` 必須スキル |
| 7 | `none` | 外部依存なし（純粋なテンプレート展開のみ） | コピー生成・営業トーク生成 |

- `type: none` の場合は `reason` フィールドに1行説明を追加すること
- type ≠ none のエントリが1件以上ある場合、本文に `## Schema Verification Steps` と `## Assumed Schema` の2セクションを必須追加する

### ステップ5：対象エージェントへの参照追記（任意）

対象エージェントが指定されている場合、そのエージェントの **日本語版定義ファイル（`.ja.md`）** の末尾に以下を追記する：

```markdown

## スキル参照

- `<スキル名>`: <スキルの1行説明>（`~/.claude/skills/<スキル名>/SKILL.md`）
```

既に「スキル参照」セクションがある場合は、そのセクションの末尾に行を追加する。
既に同名のスキルが記載されている場合は上書きする。

**編集対象は `.ja.md` のみ。英語版（`.md`）への反映はカナタ（agent-builder）に委任する。**
例：レンの場合、`agents/marketing-planner.ja.md` を編集し、`agents/marketing-planner.md` への反映はカナタに依頼する。

**複数スキル一括生成時の注意：**
- 同一エージェントに対する追記は、全スキル分をまとめて1回の編集で行う（スキルごとに個別編集しない）
- 異なるエージェントへの追記がある場合は、エージェントごとに1回ずつ編集する

### ステップ6：完了報告

**単一スキル生成の場合：**

以下の形式で報告する：

```
## knowledge-to-skill 完了

**生成したスキル**: <スキル名>
**ファイル**: ~/.claude/skills/<スキル名>/SKILL.md
**実行方法**: `/<スキル名>` または「<起動フレーズ例>」

### 変換サマリー
- ソース情報：<ドキュメント名/URL>
```

## 変換の悪い例と良い例

スキル生成時、以下を基準として常に意識すること。

### 例1：話法の変換

**ソース**：「天国と地獄話法とは、理想の状態と最悪の状態を対比させることで、行動を促す話法である」

**NG（知識のまま）**：
```
天国と地獄話法とは、理想と最悪を対比させる話法である。
営業やプレゼンで効果的に使われる。
```

**OK（実行スキル）**：
```
### ステップ1：相手の課題をヒアリングする
- 入力：クライアント情報、商談の背景
- 処理：
  - 「現在、最も困っていることは何ですか？」と聞く
  - 回答から具体的な課題を1つに絞る
- 出力：特定された課題（1つ）

### ステップ2：放置した場合の最悪シナリオを描写する
- 入力：特定された課題
- 処理：
  - 課題を放置した場合の3ヶ月後・1年後・3年後を時系列で描写する
  - 数字（コスト・損失・時間）を含めて具体化する
  - 「このまま何もしなければ...」で始める
- 出力：時系列の最悪シナリオ文

### ステップ3：解決後の理想状態を描写する
- 入力：特定された課題、最悪シナリオ
- 処理：
  - 同じ時間軸（3ヶ月後・1年後・3年後）で理想を描写する
  - 最悪シナリオと同じ指標（コスト・損失・時間）の改善値を示す
- 出力：時系列の理想状態文

### ステップ4：対比を提示する
- 入力：最悪シナリオ文、理想状態文
- 処理：
  - 最悪と理想を並べて見せる（表形式推奨）
  - 差分を強調する
- 出力：対比表

### ステップ5：行動を促す
- 入力：対比表、提案するサービス/商品
- 処理：
  - 「今日できる最初の一歩」を1つだけ提示する
- 出力：具体的なネクストアクション
```

### 例2：分析フレームワークの変換

**ソース**：「SWOT分析は、強み・弱み・機会・脅威の4象限でビジネスを分析するフレームワークである」

**NG（知識のまま）**：
```
SWOTとは強み・弱み・機会・脅威の4象限である。
内部環境と外部環境に分けて整理する。
```

**OK（実行スキル）**：
```
### ステップ1：ヒアリング項目を提示する
- 入力：クライアント名、業種
- 処理：以下の質問リストをクライアントに提示する
  - S: 「競合にない自社だけの強みは？」「顧客が選ぶ理由は？」
  - W: 「社内で不満が出ている業務は？」「失注理由の上位3つは？」
  - O: 「業界で伸びている領域は？」「未開拓の顧客層は？」
  - T: 「競合の新サービスで脅威なものは？」「法規制の変化は？」
- 出力：各象限ごとの回答リスト

### ステップ2：回答をS/W/O/Tに分類する
- 入力：ヒアリング回答
- 処理：
  - 1つの回答が複数象限にまたがる場合 → 主要因で分類し、副要因は注記する
  - 判断に迷う場合 → 「自社でコントロール可能か」で内部/外部を分ける
- 出力：分類済みSWOTマトリクス

### ステップ3：クロスSWOTで戦略候補を4パターン生成する
- 入力：分類済みSWOTマトリクス
- 処理：
  - SO戦略（強み×機会）：攻めの施策を生成する
  - WO戦略（弱み×機会）：弱み克服の施策を生成する
  - ST戦略（強み×脅威）：差別化の施策を生成する
  - WT戦略（弱み×脅威）：防御・撤退の施策を生成する
- 出力：4パターンの戦略候補リスト

### ステップ4：優先順位を付けて提案書フォーマットで出力する
- 入力：4パターンの戦略候補リスト
- 処理：
  - 実行容易性（高/中/低）× 期待効果（高/中/低）のマトリクスで優先順位を決定する
  - 上位3施策を提案書に記載する
- 出力：優先順位付き提案書
```
