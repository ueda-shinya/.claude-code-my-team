# organization-planning スキル

組織計画 = 会社・組織レベルで、役職や部門ごとの関係性を可視化し、事業計画達成のための組織構造をロードマップとして設計するスキル。事業戦略スクール「知足」のフレームワーク準拠。

## なぜ組織図をつくるのか

会社が大きくなり、一人がマネジメントできる範囲を超えると、ピラミッド型の組織にして権限移譲を進めるフェーズがくる。その時に、どのようにチームを分け、誰が誰のマネジメントをするのか、組織図を描くことで明確にする。組織図を描くことで、事業計画をどのように達成するかを見えやすくできる。

### 目的（3つ）
1. **組織の理解促進**：部門間や上下関係をひと目で把握
2. **業務効率化**：誰がどの業務を担当しているか明確化
3. **意思決定の迅速化**：誰に報告・相談するべきかが明確になる

### 利用シーン（3つ）
1. 事業計画を実現する組織の計画策定
2. 事業構造に対する事業部設計と責任者の明確化
3. 人員配置やリソース管理の最適化

## 適用条件

- このスキルを使う場面：
  - 事業計画から1年〜数年の組織ロードマップを設計したい
  - 会社全体の組織図（バイネーム入り）を作成したい
  - 組織構造（階層型／マトリクス型／フラット型）を選定したい
  - 月商200万円→2,000万円のように成長段階別の組織を設計したい
- このスキルを使わない場面：
  - プロジェクト単位の体制管理（人員・役割・コミュ・リソース・進捗の5要素） → `project-team-structure`
  - 役職別の機能・役割・責任範囲の明文化 → `roles-responsibilities`（別スキル）
  - 評価制度の策定 → `evaluation-system-design`（別スキル）
  - 採用戦略・ガイドライン → `recruitment-strategy`（別スキル）
  - 給与レンジ決定 → `salary-range-design`（別スキル）
  - キャリアロードマップと育成計画 → `career-roadmap-development`（別スキル）

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 事業計画 | 必須 | 売上目標・成長フェーズ | 「1年で月商200万円→2,000万円・10倍成長」 |
| 現状の組織 | 必須 | 現在の人員・役職・部門 | 「社長＋メンバー2名のフラット型」 |
| 想定の事業構造 | 任意 | 事業部・職能別の方向性 | 「営業／マーケ／CS／開発の4部門」 |
| 制約 | 任意 | 採用予算・期間・優先業界 | 「年間採用予算500万円」 |
| 役職への候補者 | 任意 | バイネームで配置するメンバー | 「COO候補：田中さん」 |

## 主な組織構造（3種類）

| # | 組織構造 | 特徴 | 用途 |
|---|---|---|---|
| 01 | **階層型組織図（ヒエラルキー型）** | 伝統的な形式で、上下関係を明確に示す | 大企業や官公庁で広く利用 |
| 02 | **マトリクス型組織図** | 部門とプロジェクトが交差する構造を表現 | IT企業やプロジェクト型組織 |
| 03 | **フラット型組織図** | 階層を排除したシンプルな形式 | スタートアップや小規模組織 |

### 構造選定の判断基準
- 規模・伝統重視・指揮命令系統明確化 → 階層型
- IT・PJ型・複数部門横断 → マトリクス型
- スタートアップ・スピード優先・小規模 → フラット型

## 組織のロードマップ（4Q設計）

事業計画に沿って1年間の組織図をつくる。**目標売上が達成できる組織図はどんな構成か？** を四半期ごとに考える。

### 例（月商200万→2,000万）：

| 時期 | 目標 | 組織構成 |
|---|---|---|
| **1Q** | 月商200万円 | 社長＋メンバー2名（フラット型） |
| **2Q** | 月商500万円 | 社長＋メンバー3名（フラット型・人員増） |
| **3Q** | 月商1,000万円 | 社長／Aチーム（リーダー＋メンバー2名）／Bチーム（リーダー＋メンバー1名）／営業 |
| **4Q** | 月商2,000万円 | 社長／Aチーム（部長＋メンバー）／Bチーム（新規事業）／営業（階層型へ移行） |

→ 売上に応じて段階的に階層を増やし、権限移譲を進める

## 組織図作成時のポイント（2つ）

**POINT**：組織の構成をツリーマップ上に表現し、各ポジションの役割と責任範囲を明らかにする。

1. **各ポジションに、バイネームで名前をいれる**（「○○マネージャー」ではなく「田中（マネージャー）」）
2. **現状のツリーマップと、マイルストンを置いた将来地点でのツリーマップを表現する**（6か月後・1年後・3年後に目指す組織構成等）

## 組織図可視化のメリット（3つ）

| メリット | 内容 |
|---|---|
| **業務効率化** | 担当者や報告ラインが明確になり、無駄を削減 |
| **コミュニケーションの円滑化** | 誰に相談すべきかがわかりやすくなる |
| **外部への透明性向上** | 取引先や投資家に組織体制を正確に伝えられる |

## 成功事例（3つ）

| 企業 | 組織形式 | 成功ポイント |
|---|---|---|
| Google | マトリックス型 | 柔軟性を高めることで、プロジェクトの迅速な意思決定を実現 |
| スターバックス | 階層型 | グローバル規模の店舗運営で、明確な指揮系統を活用 |
| 中小企業 | フラット型 | 社員間の連携強化と意思決定のスピードアップを達成 |

## 実行ステップ

### ステップ1：事業計画から目標売上の四半期推移を確認
- 入力：事業計画
- 処理：
  1. 1Q／2Q／3Q／4Q の目標売上を整理
  2. 売上達成に必要な機能（営業／開発／CS／管理 等）を洗い出し
  3. 各機能の必要人員数を仮置き（業界平均生産性 or 既知ベンチマーク）
- 出力：4Q目標と必要機能・人員見積もり

### ステップ2：組織構造を選定
- 入力：規模・特性・成長フェーズ
- 処理：
  1. 現状規模で適切な構造を判定
     - 〜10名：フラット型推奨
     - 10〜50名：階層型移行検討
     - PJ横断が多い：マトリクス型併用
  2. 1年後の規模を見据えて将来構造も選定
  3. 構造移行のタイミング（フラット→階層）を決定
- 出力：現在＋1年後の組織構造選定理由

### ステップ3：4Q組織ロードマップを設計
- 入力：4Q目標、構造選定
- 処理：
  1. 各四半期の組織図をツリーマップで描画（時期・目標・組織図の3行マトリクス）
  2. 1Q→4Q で人員配置の段階的変化を明示
  3. 階層数の追加・チーム分割タイミングを明示
  4. 採用が必要な四半期と職種を一覧化
- 出力：4Q組織ロードマップ＋採用計画

### ステップ4：バイネーム組織図を作成
- 入力：4Qロードマップ、現職の人員リスト、候補者
- 処理：
  1. 各ポジションにバイネームで配置（「○○マネージャー」ではなく「田中（マネージャー）」）
  2. 候補者未定のポジションは「採用予定」と明示
  3. 現状図と将来図（1年後）の2バージョンを必ず作る
  4. 役職別の機能・役割概要を1行で添える（詳細は `roles-responsibilities` スキルへ）
- 出力：バイネーム組織図（現状＋1年後）

### ステップ5：可視化メリットの確認と運用ルール設計
- 入力：バイネーム組織図
- 処理：以下3観点で運用設計
  1. **業務効率化**：報告ライン・担当者の明確化を社内周知（イントラ・契約書・名刺）
  2. **コミュニケーションの円滑化**：「○○の相談はこの人」を1ページマップ化
  3. **外部への透明性向上**：取引先・投資家・採用候補者向けに公開できる版を準備
- 出力：運用ルール（更新頻度・公開範囲・責任者）

### ステップ6：定期更新サイクルの設計
- 入力：これまでの設計結果
- 処理：以下を決定
  - **更新頻度**：四半期ごとに見直し（事業計画と連動）
  - **更新責任者**：人事責任者または代表
  - **共有方法**：全社会議・イントラ・契約書テンプレ更新
- 出力：定期更新カレンダー＋責任者リスト

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 1人がマネジメントする部下が10名超 | 階層を1段追加（チームリーダー設置） | スパンオブコントロール超過 |
| スタートアップで〜10名 | フラット型維持 | 階層化は早期にメリット少 |
| PJ横断が3つ以上 | マトリクス型を併用検討 | 部門だけだと縦割りになる |
| 役職に候補者なし | 「採用予定」と明示し採用計画を作る | 空席のままだと運用できない |
| 1年後の組織図が未設計 | 必ず将来時点ツリーも作成 | 現状だけだと採用・育成の優先順位が立たない |
| 役職の機能・役割が不明 | `roles-responsibilities` スキルへ接続 | 組織図単独では運用できない |
| 評価制度・給与レンジが未整備 | `evaluation-system-design` `salary-range-design` へ接続 | 役職定義だけでは退職リスク |
| 「ブルーオーシャンだから組織は最小限でOK」等の楽観 | `competitive-absence-audit` を併用検討 | 競合不在を理由にした楽観組織計画は要検証 |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 役職名のみで誰も配置されていない | バイネームで具体的に配置 | 抽象論で終わり実行されない |
| 現状図のみ作って終わり | 必ず1年後・3年後の将来図も作る | 採用・育成の優先順位が立たない |
| 採用予定を明示しない | 空席は「採用予定」と必ず明示 | 漏れて忘れる |
| マトリクスにすべきPJ横断業務を階層型のまま | PJ横断が3つ以上ならマトリクス型併用 | 縦割りで連携が取れない |
| 階層を増やしすぎる | スパンオブコントロール（部下5〜10名）目安 | 階層多すぎは意思決定が遅くなる |
| 役職の機能・役割が定義なし | `roles-responsibilities` スキルへ接続 | 組織図だけでは運用不能 |
| 更新せずに古い図を使い続ける | 四半期に1回必ず見直し | 実態と乖離して機能不全 |

## 連携スキル

- `project-team-structure`：プロジェクト単位の体制管理（人員・役割・コミュ・リソース・進捗の5要素）
- `roles-responsibilities`：役職別の機能・役割・責任範囲の明文化（別スキル）
- `evaluation-system-design`：評価制度の策定（別スキル）
- `recruitment-strategy`：採用戦略・ガイドライン（別スキル）
- `salary-range-design`：給与レンジ決定（別スキル）
- `career-roadmap-development`：キャリアロードマップと育成計画（別スキル）
- `medium-term-business-plan`：中長期事業計画から逆算した組織設計
- `kgi-kpi-kai-design`：組織別KPIの設計
- `katz-three-skill-approach`：役職階層と必要スキルのマッピング
- `mvv-design`：組織の方向性（MVV）が組織図設計の前提

## 5状態出力契約

外部API・サブプロセス・外部設定依存を持たないため、CLAUDE.md「5状態出力契約」の対象判定基準には該当しない。ただし以下のマーカーを使うこと：

- `[OK:0件]`：候補者・人員情報が不明で組織図が組めない場合（要再ヒアリング）
- `[PARTIAL:N成功/M失敗]`：4Qのうち一部のみ設計できた場合
- `[SKIP]`：事業計画が未確定で組織計画を保留した場合（前提不足）

※外部API依存なしのため `[FAIL]` は通常発生しない。前提不足は `[SKIP]` で扱う。
