# /business-analysis スキル

ビジネスの相談を受けた際に、フレームワーク選定 → 読み込み → 分析 → スコアリング → 提案を一連のワークフローとして実行するスキル。

## 重要な前提

- **Step 1（ヒアリング）はアスカが実行し、Step 2〜5はすべてナギ（business-consultant エージェント）に委任する**
- フレームワークの押し付けはしない。分析の中で自然にフレームワークの視点を織り込む
- 日本語で出力する

## ワークフロー

### Step 1: ヒアリング（アスカが実行）

`/business-analysis` の後に相談内容が記述されている場合、それを相談内容としてStep 2に渡す。

引数がない場合のみ、シンヤさんに「何について分析しますか?」と確認する。

相談内容が確定したら、ナギ（business-consultant）を呼び出し、以下の情報を渡す:
- 相談内容
- 「`~/.claude/knowledge/business-frameworks/` にフレームワークがあるので、エージェント定義のフレームワーク選択ガイドに従って適切なものを選び、分析してください」

### Step 2: フレームワーク自動選定（ナギが実行）

ナギは自身のエージェント定義にあるフレームワーク選択ガイドに従って、相談内容に最適なフレームワークを1〜2つ選定する。

※ フレームワーク選択テーブルはナギのエージェント定義（`~/.claude/agents/business-consultant.ja.md`）が正（source of truth）。このスキルには重複して持たない。

**選定ルール:**
- 複数カテゴリに該当する場合は、最も中心的なもの1〜2つに絞る
- どのカテゴリにも該当しない場合は、フレームワークなしでナギの知識ベースで分析する

### Step 3: フレームワーク読み込みと分析（ナギが実行）

1. 選定したフレームワークファイルを Read ツールで `~/.claude/knowledge/business-frameworks/` から読み込む
2. フレームワークの構造に従って、シンヤさんの相談内容を分析する
3. フレームワーク内にスコアリング基準がある場合は、10点満点で現状スコアを付ける
4. フレームワークなしの場合はスコアリングを省略し、構造的に分析する

### Step 4: 結果出力（ナギが実行）

以下のフォーマットで出力する:

```
## 分析結果

### 前提条件
（分析の前提: 対象市場、事業規模、事業フェーズ、対象顧客層など）

### 使用フレームワーク
- フレームワーク名（選定理由を一言）
（複数の場合は箇条書きで列挙）
（フレームワークなしの場合は「ナギの知識ベースで分析」と記載）

### 現状スコア: X / 10
（スコアリング基準がある場合のみ。フレームワーク2つの場合は各フレームワークごとにスコア）
（スコアリング基準がない場合やフレームワークなしの場合はこのセクションを省略）

### 分析
（フレームワークに基づいた構造的な分析。箇条書きや図表を活用）
（2つのフレームワークを使った場合は、それぞれの視点から分析）

### 課題
（特定された問題点や改善点）

### リスク・留意点
（提案を実行した場合のリスクや副作用。判断に影響する外部要因）

### 提案（ナギの推奨）
1. ...
2. ...
3. ...

### 次のアクション
- 具体的に何をすべきか
```

### Step 5: 深掘り提案（ナギが実行）

分析完了後、シンヤさんに以下の選択肢を提示する（該当するもののみ）:

- 「別のフレームワークでも見てみますか?」（追加分析の提案）
- 「レンにマーケ施策を依頼しますか?」（マーケティング連携が有効な場合のみ）
- 「ロードマップに落とし込みますか?」（アクションプラン化の提案）
