# llmo-compliant-coding

LP・サイト・記事の**新規作成時にLLMO 14項目を最初から満たす**ための事前遵守ルール生成スキル。`llmo-audit`（事後監査）の対となる姉妹スキル。

## 実行主体と運用構造

- **アスカ**：依頼受領→本スキル発動判定→遵守ルール生成→各実行者に渡す
- **シュウ（backend-engineer）**：HTML/JSON-LD/構造化データの実装担当
- **ハル（writer）**：本文執筆・E-E-A-T要素・引用元明記担当
- **コト（copywriter）**：見出し・キャッチ・PREP文体担当
- **サクラ（code-reviewer）**：実装後のコードレビュー＋LLMO遵守の事後検証

本スキルは「コード生成」ではなく「遵守ルール集を生成」する設計のため、CLAUDE.md「Asuka Never Codes Directly」ルール下でアスカ直接呼び出し可。

## llmo-audit との役割分担

| 観点 | llmo-compliant-coding（本スキル） | llmo-audit |
|---|---|---|
| タイミング | 新規作成**前** | 公開**後** |
| アクション | 遵守ルール注入→実装 | 監査→改善案 |
| 出力 | 実装者向け遵守チェックリスト＋コード雛形 | 採点表＋優先順位付き改善案 |
| 14項目への向き合い方 | 「最初から守る」 | 「事後に検出して直す」 |
| 連携 | `/lp-create` `/blog-post` `/feature-flow` `frontend-design` から呼び出し | 既存URL監査・3ヶ月後の再点検 |

## 適用条件

### このスキルを使う場面
- **新規LP制作**を依頼された（`/lp-create` の設計フェーズ）
- **新規ブログ記事執筆**を依頼された（`/blog-post` の構成フェーズ）
- **新規ページ・新規機能**の実装フェーズ（`/feature-flow` の実装前）
- **frontend-design** で UI/HTML を生成する前
- クライアントから「AI検索（ChatGPT/Gemini/Perplexity/Copilot）で引用されたい」と要望
- LLMO観点を**最初の1行から織り込みたい**ケース

### このスキルを使わない場面
- 既存サイト・既存記事の事後監査（→ `llmo-audit`）
- LLMO観点を意識する必要がない内部用ツール・管理画面
- 純粋なバグ修正・既存機能の小規模改修
- 流入減少診断（→ `ga-gsc-diagnosis`）
- Google向け技術SEO監査（→ `seo-audit`）

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 対象種別 | 必須 | `LP` / `article`（ブログ記事） / `page`（汎用ページ） / `site`（サイト全体） | `LP` |
| 主題タイプ | 必須 | ページの主題に対応する Schema.org タイプ | `LocalBusiness` / `Article` / `Product` / `HowTo` / `Service` / `FAQPage` |
| 対象URL（予定） | 任意 | 公開予定URL | `https://officeueda.com/lp/web` |
| 重視するAI | 任意 | 特定AIに最適化したい場合 | `Gemini` / `Perplexity` / `ChatGPT` / `Copilot` / `all`（デフォルト） |
| クライアント名 | 任意 | クライアント案件の場合 | `officeueda` |
| 業種・ターゲット | 任意 | クライアント未指定時に直接入力 | 業種・ターゲットペルソナ |
| CMS/プラットフォーム | 任意 | 実装環境 | `WordPress + SWELL` / `静的HTML` / `Next.js` 等 |
| 著者・監修者情報 | 任意 | Person スキーマに必要な情報 | 名前・経歴・資格・URL等 |

## 実行ステップ

### ステップ1：入力の確認と主題タイプの確定
- 入力：依頼内容
- 処理：
  1. 対象種別（LP/article/page/site）を確認
  2. 主題タイプ（Schema.org type）を確認。未指定なら依頼内容から推定し、ユーザーに確認
  3. クライアント指定があれば `~/.claude/clients/<name>/README.md` を Read して業種・ターゲットを取得
     - ファイル未存在 → 一般前提で続行（`[SKIP]`）
     - Read エラー → `[FAIL]` 記録のうえ続行
  4. 不足情報があればユーザーに最小限ヒアリング（著者情報・CMS等）
- 出力：構造化された入力サマリー（対象種別／主題タイプ／重視AI／業種／著者情報）

### ステップ2：14項目の遵守ルールを生成
- 入力：ステップ1のサマリー
- 処理：以下14項目について、対象種別・主題タイプに応じた**実装時の必須遵守ルール**と**コード雛形**を生成する。

| # | 優先度 | 項目 | 遵守ルールの内容 | 実装担当 |
|---|---|---|---|---|
| 1 | **高** | 見出しタグ階層 | H1=1個（ページ主題と一致）、H2以降は論理階層、レベル飛ばし禁止 | シュウ |
| 2 | **高** | 主題タイプSchema.org実装 | 主題タイプに対応する JSON-LD を `<head>` 内に配置（コード雛形提供） | シュウ |
| 3 | **高** | XMLサイトマップ | `/sitemap.xml` 自動生成設定＋Search Console送信タスク化 | シュウ |
| 4 | **高** | ブランド言及戦略 | ※実装フェーズ外。レンへ「公開後の言及獲得戦略」依頼タスクとして提示 | レン委任タスク |
| 5 | 中 | Personスキーマ | 著者プロフィールに Person スキーマ実装（コード雛形提供） | シュウ |
| 6 | 中 | ファクトベース引用元明記 | 統計・データ使用時は「出典: URL（調査年月）」を必ず本文中に明記する執筆ルール | ハル |
| 7 | 中 | AI解釈しやすいライティング | 一文50-80文字目安・PREP構成（結論→理由→例→結論）を本文構造ルールに | ハル/コト |
| 8 | 中 | 箇条書き・表・リスト活用 | **定量目安（目標値）**：`<li>` 合計 **14〜15個**（平均14.50）／`<ul>` **2〜3個**（平均2.39）／`<table>` **1個程度**（平均0.86）／本文 **4,400〜4,800字**（平均4,411字）。主要セクションごとに最低1つのリスト or 表を含める。4項目中3項目以上満たすことを目標とする。<br>出典：TechSuite「ブラインドテスト v4」p13（N=881、2025年7月調査／主にSEO記事対象・LP等では調整要） | ハル+シュウ |
| 9 | 中 | FAQセクション+FAQPageスキーマ | 主要ページに FAQ セクション設置＋FAQPage JSON-LD（コード雛形提供） | シュウ+ハル |
| 10 | 中 | 画像 alt の充実 | **定量目安**：`<img>` タグ **2個以上**（平均1.98 / 0-30秒グループの約11倍）。主要画像すべてに意味のある alt（機械的なファイル名NG）。ファイル名も英語ケバブケース＋意味あり | シュウ |
| 11 | 中 | 更新日明示 | 本文中に「最終更新: YYYY-MM-DD」可視表記＋JSON-LDの dateModified に反映 | シュウ+ハル |
| 12 | 中 | 画像・動画の戦略的活用 | **静止画以外のリッチメディア**（動画／インフォグラフィック／図解／SVG等）を最低1つ配置。`<img>` の数は項目10で評価（本項目はリッチメディアの有無のみ評価・重複回避） | シュウ+ハル |
| 13 | 中 | 権威リスト掲載戦略 | ※実装フェーズ外。レンへ「掲載獲得戦略」依頼タスクとして提示 | レン委任タスク |
| 14 | 中 | E-E-A-T 徹底 | 著者プロフィール（経歴・資格・実績件数）＋一次情報（独自調査・事例・体験談）含めた構造化 | ハル+シュウ |

- 出力：14項目それぞれについて「遵守ルール本文＋コード雛形（該当項目のみ）＋実装担当」のセット

### ステップ3：AI別重点ルールの強調
- 入力：ステップ2の遵守ルール、重視AI
- 処理：**優先度「高」4項目（項目1〜4）は常に最重要強調**。これに加えて重視AI指定がある場合、以下を追加強調：
  - `Gemini` 指定 → 項目12（マルチモーダル）・項目14（E-E-A-T）
  - `Perplexity` 指定 → 項目6（引用元明記）・項目13（権威リスト）
  - `Copilot` 指定 → 項目11（更新鮮度）
  - `ChatGPT` 指定 → 項目9（FAQ）
  - `all`（デフォルト）→ 優先度「高」4項目のみ強調
- 出力：重点強調マーク付きの遵守ルールセット

### ステップ4：実装者別の遵守チェックリストに分割
- 入力：強調済み遵守ルールセット
- 処理：実装担当別にチェックリストを再編成：
  - **シュウ向けチェックリスト**：HTML構造・JSON-LD・画像 alt・サイトマップ・更新日メタ
  - **ハル向けチェックリスト**：本文構成（PREP）・引用元明記・FAQ本文・E-E-A-T要素・更新日表記
  - **コト向けチェックリスト**：見出しコピー・キャッチ・読みやすさ
  - **レン委任タスク**：項目4（ブランド言及）・項目13（権威リスト掲載）の戦略立案
- 出力：実装担当別チェックリスト4セット

### ステップ5：連携先スキル別のサマリー出力
- 入力：実装担当別チェックリスト
- 処理：本スキルが呼び出された連携元（または独立起動）に応じてサマリーを出力。**コトのコピーチェックリストは常に「主担当（カイ/ハル/シュウ）の内部利用リソース」として同梱配布**し、コピー要素が必要な箇所が出た時点でコトに別途委任する建付けとする：
  - `/lp-create` 連携 → カイ（lp-designer）に「LP設計時にLLMO観点を最初から組み込むためのチェックリスト」として渡す。**コトのコピーチェックリストはカイ宛に同梱配布**（ファーストビューコピー設計時に参照）
  - `/blog-post` 連携 → ハルに「執筆時LLMO遵守チェックリスト」として渡す。**コトのコピーチェックリストはハル宛に同梱配布**（H1/H2コピー作成時に参照）
  - `/feature-flow` 連携 → シュウに「実装時LLMO遵守チェックリスト」として渡す。コピー要素が必要な機能の場合はシュウからコト委任
  - `frontend-design` 連携 → シュウに「コンポーネント実装時LLMO遵守チェックリスト」として渡す。UIコピーが必要な場合はシュウからコト委任
  - 独立起動 → 全実装担当向けの統合チェックリストを出力（コト分も並列で含める）
- 出力：最終遵守ルール集（Markdown形式）

**コト経路の論理空白防止ルール**：いかなる連携経路でも「コト向けチェックリスト」が誰にも届かない状態にしない。主担当が同梱配布を受け取り、コピー要素が必要な箇所で必ずコトに委任する責務を負う。

### ステップ6：事後監査タスクの予約提示
- 入力：ステップ5の最終出力
- 処理：「公開後3ヶ月で `llmo-audit` による事後監査を実施することを推奨」とNotionタスク登録の提案を出力
- 出力：事後監査リマインダー

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 主題タイプが推定できない | ユーザーに確認（候補3つ提示） | 主題タイプ次第でJSON-LDコード雛形が変わる |
| クライアント情報なし | 一般前提で生成。レポート末尾に「業種・ターゲット情報があれば精度向上」と注記 | 業種依存（YMYL等）の判断は情報がある方が正確 |
| CMS情報なし | 主要CMS（WordPress/静的HTML/Next.js）の3パターンでコード雛形を提示 | 実装環境次第で実装方法が変わる |
| 著者情報なし | Person スキーマは「実装後に著者情報追加要」とプレースホルダー雛形を提示 | E-E-A-T強化に必須だが情報待ちで止めない |
| 連携元未指定（独立起動） | 全実装担当向けの統合チェックリストを出力 | 単独利用ケースもサポート |
| 重視AI=`all`以外指定 | 優先度高4項目＋AI固有項目を**両方**強調 | AI最適化と全体底上げを両立 |
| 項目4・13（実装フェーズ外） | 「レン委任タスク」として明示し、実装チェックリストには含めない | スキルの責任範囲を明確化 |

## 実装担当別チェックリスト

### 🔧 シュウ（backend-engineer）向け

#### 必須実装項目（優先度「高」）
- [ ] H1=1個（ページ主題と完全一致）、H2以降論理階層、レベル飛ばし禁止
- [ ] 主題タイプSchema.org JSON-LD を `<head>` 内に配置（下記雛形参照）

#### コード雛形

##### Schema.org（主題タイプ別）
```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "<主題タイプ>",
  ... <タイプ別必須プロパティ>
}
</script>
```

##### Person スキーマ（著者）
```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Person",
  "name": "<著者名>",
  "jobTitle": "<肩書>",
  "url": "<プロフィールURL>",
  "sameAs": ["<SNS URL等>"]
}
</script>
```

##### FAQPage スキーマ
```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "<質問>",
      "acceptedAnswer": {"@type": "Answer", "text": "<回答>"}
    }
  ]
}
</script>
```

#### 中優先度実装項目
- [ ] 画像 alt 充実（意味あり・空alt禁止・ファイル名英語ケバブケース）
- [ ] 更新日メタ表示＋ JSON-LD `dateModified` 反映
- [ ] FAQセクション設置＋FAQPage JSON-LD

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 14項目すべてフラットに並べた巨大チェックリストを実装者に丸投げ | 実装担当別（シュウ/ハル/コト/レン）に分割し、優先度高項目を強調 | 実装者が「何から手を付けるか」を判断できる粒度が必要 |

## 連携設計（既存スキルとの呼び出し関係）

```
【新規LP制作フロー】
依頼 → /lp-create → カイ設計 → 【llmo-compliant-coding 発動】 → 遵守ルール生成
   → シュウ実装（チェックリスト遵守）+ ハル執筆 + コト copy
   → サクラレビュー（LLMO遵守チェック含む）→ 公開
   → 3ヶ月後 llmo-audit 事後監査

【新規ブログ記事フロー】
依頼 → /blog-post → 【llmo-compliant-coding 発動（articleモード）】
   → ハル執筆（遵守ルール反映）→ サクラレビュー → 公開
   → 3ヶ月後 llmo-audit 事後監査

【新機能フロー】
/feature-flow → 設計 → KYT → 【llmo-compliant-coding 発動】 → シュウ実装 → サクラ
```

**注意**: 連携先スキル（`/lp-create` `/blog-post` `/feature-flow` `frontend-design`）への自動連携機能の組み込みは別タスク（連携先スキル側の修正が必要・コード扱いのためシュウ委任）。本スキル単独で先行運用し、依頼時にアスカが手動で発動する形からスタート。

本スキルは外部ネットワーク呼び出し・サブプロセス実行・外部設定依存のいずれも含まないため、CLAUDE.md「スキル出力の5状態契約」の**対象外**（純粋なテンプレート展開スキル）。
