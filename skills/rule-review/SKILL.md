# /rule-review スキル

CLAUDE.md・メモリファイル・エージェント定義の定期棚卸しを実施します。
Claude（リナ）と Gemini API（外部視点）の両方でレビューし、ルールの矛盾・重複・肥大化を防ぎます。

## 使い方

- `/rule-review` → 当月分として棚卸しを実行
- `/rule-review 2026-04` → 指定月分として棚卸しを実行

## 実行手順

### ステップ1：対象ファイルの読み込み（アスカ）

以下のファイルをすべて Read ツールで読み込んでください：

1. `~/.claude/CLAUDE.md`
2. `~/.claude/projects/-Users-uedashinya--claude/memory/` 配下の全 `.md` ファイル（Glob で `**/*.md` を検索してから Read）
3. `~/.claude/agents/` 配下の全 `.md` ファイル（Glob で `*.md` を検索してから Read）

読み込んだファイルの一覧と合計行数を記憶しておいてください。これが棚卸しの対象範囲になります。

### ステップ2：リナによるレビュー（アスカがリナに依頼）

リナ（`subagent_type: logic-verifier`）に以下のプロンプトで依頼してください：

```
CLAUDE.md・メモリファイル・エージェント定義の棚卸しレビューを依頼します。
以下の観点でチェックし、指摘を箇条書きで返してください。

【レビュー観点】
1. ルール間の矛盾・競合（AのルールとBのルールが衝突していないか）
2. 重複・冗長な記述（同じことを別の場所で書いていないか）
3. 論理的な抜け穴・例外の未定義（「〇〇の場合はどうする？」が抜けていないか）
4. 役割が曖昧になっているルール（誰が・いつ・何をするか不明確）
5. 削除・統合を推奨するルール（不要になった・まとめられる）

【出力形式】
各指摘に重要度を付けてください：
- [高] 矛盾・競合 → 放置すると誤動作する
- [中] 重複・曖昧 → 整理すべき
- [低] 改善提案 → あると良い

以下がレビュー対象のファイル内容です：

（ステップ1で読み込んだ全ファイルの内容をここに貼る）
```

リナの回答を記憶しておいてください。

### ステップ3：Gemini API による独立レビュー（アスカが実行）

APIキーは `~/.claude/.env` から読み込んでください（`source ~/.claude/.env`）。

Gemini 2.5 Pro を使用して、CLAUDE.md の全文を投げてレビューを依頼してください。

**実行コマンド：**

まず CLAUDE.md の内容を変数に読み込み、curl で Gemini API を叩いてください。以下はテンプレートです：

```bash
source ~/.claude/.env

CLAUDE_CONTENT=$(cat ~/.claude/CLAUDE.md)

PROMPT="あなたはAIエージェントシステムの設計レビュアーです。以下はClaude Codeのメインアシスタントに対するルール定義ファイル（CLAUDE.md）です。

以下の観点でレビューし、指摘を箇条書きで返してください：
1. ルール間の矛盾・競合の指摘
2. 不要・重複ルールの指摘
3. 見落とされがちな構造的問題（ルールの依存関係、実行順序の問題など）
4. 肥大化している箇所と簡素化の提案

各指摘に重要度 [高][中][低] を付けてください。

===== CLAUDE.md 全文 =====
${CLAUDE_CONTENT}
====="

curl -s "https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-pro-exp-03-25:generateContent?key=$GEMINI_API_KEY" \
  -H "Content-Type: application/json" \
  -d "$(python3 -c "
import json, sys
prompt = sys.stdin.read()
print(json.dumps({
    'contents': [{'parts': [{'text': prompt}]}]
}))
" <<< "$PROMPT")" 2>&1
```

**フォールバック：** 上記モデルが 404 / Not Found の場合は、モデル名を `gemini-2.0-flash` に変更して再実行してください。

Gemini の回答（`candidates[0].content.parts[0].text`）を抽出して記憶してください。

### ステップ4：全員参加の協議（アスカが指揮）

ステップ2（リナ）とステップ3（Gemini）の結果を統合し、以下のエージェントにそれぞれ意見を求めてください。各エージェントには統合済みの指摘一覧を共有してください：

**4-1. リナ（`subagent_type: logic-verifier`）：**
ステップ2の自身の指摘に加え、Gemini の指摘を踏まえて追加の論理的問題がないか確認してもらう。

```
ステップ2であなたが出した指摘に加え、Gemini から以下の追加指摘が出ました。
これらを踏まえて、見落としていた論理的な問題があれば追加で指摘してください。
なければ「追加指摘なし」と回答してください。

【Gemini の指摘】
（ステップ3の結果を貼る）
```

**4-2. ツムギ（`subagent_type: process-designer`）：**
業務プロセス視点で改善提案を依頼する。

```
CLAUDE.md・メモリ・エージェント定義の棚卸しを実施中です。
以下の指摘が出ています。業務プロセス・ワークフロー設計の観点から、
改善提案やナレッジ構造の最適化提案があれば教えてください。

【リナの指摘】
（ステップ2の結果を貼る）

【Gemini の指摘】
（ステップ3の結果を貼る）
```

**4-3. サクラ（`subagent_type: code-reviewer`）：**
品質基準・ルールの明確さの観点でレビューを依頼する。

```
CLAUDE.md・メモリ・エージェント定義の棚卸しを実施中です。
以下の指摘が出ています。品質基準の観点から、
ルールの明確さ・テスト可能性・保守性について意見をください。

【リナの指摘】
（ステップ2の結果を貼る）

【Gemini の指摘】
（ステップ3の結果を貼る）
```

### ステップ5：レポート作成と変更提案（アスカ）

全エージェントの意見を統合し、以下の形式でシンヤさんに報告してください：

```
【ルール棚卸しレポート】YYYY年MM月

対象ファイル数: X件 / 合計行数: X行

■ 矛盾・競合（要対応）
- [指摘元] 内容
  → 推奨アクション

■ 重複・冗長（統合推奨）
- [指摘元] 内容
  → 推奨アクション

■ 削除推奨
- [指摘元] 内容
  → 理由

■ 改善提案
- [指摘元] 内容
  → 推奨アクション

■ 現状維持（問題なし）
- 問題が指摘されなかった領域の概要

■ Gemini の追加指摘（Claude 側で見落としていたもの）
- 内容
```

レポートの末尾に以下を追記してください：

```
変更を実施しますか？
1. すべて実施（カナタ or シュウに委譲）
2. 個別に選択して実施
3. 今回は見送り
```

シンヤさんが変更を承認した場合のみ、カナタ（`subagent_type: agent-builder`）またはシュウに実装を委譲してください。CLAUDE.md を直接変更しないでください。

## 注意事項

- CLAUDE.md・メモリファイル・エージェント定義を直接変更しないこと。変更提案のみ行い、シンヤさんの承認後に実施する
- Gemini API 呼び出しは Bash ツールで curl を使うこと（アスカは Bash ツールを持っている）
- Gemini のモデル `gemini-2.5-pro-exp-03-25` が存在しない場合は `gemini-2.0-flash` にフォールバックする
- リナ・ツムギ・サクラへの依頼は Agent ツールを使うこと
- 各エージェントの回答が長すぎる場合は、アスカが要点を整理して最終レポートに反映する
- レポートは報告のみ。ファイルへの保存はシンヤさんが指示した場合のみ行う
