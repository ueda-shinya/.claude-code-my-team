# sales-slide-builder

即決営業メソッドの個別スキルをスライド構成に変換し、提案プレゼンの骨子＋話者ノート付き原稿を生成する。最終工程はソラへ接続する。

**棲み分け：**
- 商談相談・戦略設計 → `sales-playbook`
- 読み上げ台本（トークスクリプト） → `sales-script-builder`
- 提案プレゼン用スライド骨子 → **本スキル**
- スライドの最終デザイン化（Genspark/Gamma 出力） → ソラ（`slide-create` スキル / slide-designer）

## 適用条件

- 使う場面：初回提案プレゼン、決裁者向けプレゼン、デモ＋提案のハイブリッド商談、オンライン商談
- 使わない場面：飛び込み・テレアポ（スライドを使わない場面）、スクリプトが欲しい（sales-script-builder へ）、単純な営業資料（コト/カイへ直接）

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 商材/サービス名 | 必須 | プレゼン対象 | SaaS業務効率化ツール |
| ターゲット企業 | 必須 | 業種・規模・役職 | 中小製造業・経営者 |
| 主要ベネフィット | 必須 | 顧客が得る変化 | 月間40時間の業務削減 |
| 決裁型 | 必須 | 即決型 / 高関与型 | 高関与型 |
| プレゼン想定時間 | 必須 | 10分 / 20分 / 30分 / 60分 | 20分 |
| 価格情報 | 任意 | 提示予定価格（高単価判定に使用） | 月額10万円 |
| 差別化ポイント | 任意 | 競合との違い | 先端AI統合 |

## 実行ステップ

### ステップ1：スライド枚数と構成を決定する
- 入力：プレゼン想定時間
- 処理：以下の基準で枚数を決める（1スライド≒1分）
  - 10分 → 9〜11枚
  - 20分 → 18〜22枚
  - 30分 → 25〜30枚
  - 60分 → 40〜50枚
- 出力：目標枚数

### ステップ2：スライド骨子の7ブロック構成を生成する
- 入力：目標枚数、決裁型
- 処理：以下の固定7ブロックに枚数を配分する
  - ブロック1：表紙＋自己紹介（10%）
  - ブロック2：ハロー＋アイスブレイク的つかみ（5%）
  - ブロック3：課題提示・顧客の痛み（20%）
  - ブロック4：ソリューション提示（結論先出し）（25%）
  - ブロック5：理由・証拠・事例（20%）
  - ブロック6：理想の未来（ビフォーアフター）（10%）
  - ブロック7：クロージング・ネクストアクション（10%）
- 出力：ブロック別枚数配分表

### ステップ3：呼び出すスキルを決定する
- 入力：決裁型、差別化ポイント
- 処理：各ブロックに対応する即決営業スキルをマッピングする
  - ブロック1 → `two-second-self-disclosure`（話者ノート用／スライドには載せない）
  - ブロック2 → `open-heart-praise`（話者ノート用／掴みスライドの裏話用）
  - ブロック3 → `heaven-hell-rhetoric` の「地獄」パート
  - ブロック4 → `conclusion-first-talk` の結論・理由
  - ブロック5 → `conclusion-first-talk` の証拠・追加メリット
  - ブロック6 → `heaven-hell-rhetoric` の「天国」パート
  - ブロック7 → 即決型=`sokketsu-closing` / 高関与型=`self-closing-4questions`
  - 全ブロック話者ノート → `sales-psychology-techniques` のFITD/DITF/バックトラッキング配置
- 出力：ブロック別スキル呼び出しマップ

### ステップ4：各スキルを読み込み個別成果物を生成する＋心理テク重複正規化
- 入力：ブロック別スキル呼び出しマップ、商材情報
- 処理：
  - **共通入力セットを固定する**：商材名／ターゲット／主要ベネフィット／決裁型／プレゼン時間／差別化ポイント を全スキル共通で渡す
  - 各スキルの SKILL.md を Read する
  - 各スキルの入力項目を整形して呼び出し、個別成果物を生成する
  - スライド本文用（短く視覚的）と話者ノート用（詳しくトーク調）で出力を2種類に整形する
  - **心理テク重複の正規化ルール（必須）**：FITD / DITF / バックトラッキング は骨子全体で各1回に正規化する
    - DITF：スライド面（ブロック5の松竹梅ビジュアル）に配置した場合、話者ノート側からは削除する
    - FITD：話者ノート側に配置し、スライド本文には載せない
    - バックトラッキング：話者ノートのみ（ヒアリング想定シーンに1回）
- 出力：ブロック別スライド本文＋話者ノート

### ステップ5：スライド1枚ごとの原稿に分解する
- 入力：ブロック別スライド本文＋話者ノート、目標枚数
- 処理：
  - ブロック配分枚数に従って1枚ずつに分解する
  - 各スライドに以下の要素を配置：
    - タイトル（10〜15字）
    - 本文（箇条書き3〜5行 or 1フレーズ）
    - 視覚要素指示（図／表／写真／グラフ）
    - 話者ノート（100〜200字）
- 出力：スライド別原稿リスト

### ステップ6：ソラ（slide-designer）への引き渡し形式に整形する
- 入力：スライド別原稿リスト
- 処理：
  - ソラが Genspark/Gamma プロンプトに変換しやすい Markdown 形式にする
  - 各スライドは `## スライドN：タイトル` + 本文 + `### 話者ノート` + `### 視覚指示` の4ブロック構造
  - 全体の冒頭に「プレゼン概要」「ターゲット」「主要メッセージ」のメタ情報を追加する
- 出力：ソラ引き渡し形式 Markdown

### ステップ7：完成骨子を出力し、ソラへの引き渡し方を明記する
- 入力：ソラ引き渡し形式 Markdown
- 処理：
  - 完成骨子を本スキルの最終出力として提示する
  - 話者ノートは **ソラには渡さず、本スキル最終出力の Markdown にのみ残し**、営業担当の手元資料として使う旨を明記する（Genspark/Gamma は話者ノート仕様を持たないため脱落する）
  - 末尾に以下の次工程ガイドを固定文で追加する：
    ```
    【次工程：ソラへの引き渡し手順】
    1. アスカは `/slide-create` を起動する
    2. 本骨子を「Existing Reference Materials」枠に全文投入する
    3. slide-create の Step 1 ヒアリングはスキップする旨を明示する
    4. Step 2 でソラに「この骨子のみを正とし、追加情報を捏造しない」を明示して渡す
    5. ソラが Genspark / Gamma プロンプトに変換する（本文＋視覚指示のみ。話者ノートは除外）
    6. 話者ノートは本骨子 Markdown を営業担当の手元資料として別途保管する
    ```
- 出力：完成スライド骨子＋次工程ガイド

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| プレゼン時間10分以下 | ブロック2（つかみ）を省略、話者ノートに吸収 | スライド枚数を確保するため |
| 決裁者が複数同席 | ブロック6を+5%、ブロック1・ブロック2から各-2.5%配分を削る | 合意形成には共有ビジョンが必要 |
| 競合比較が重要 | ブロック5にDITF型の比較スライド（松竹梅）を追加。話者ノート側のDITFは削除 | 重複投入防止 |
| オンライン商談 | 1スライドの箇条書きを3〜5行→2〜3行に削減、図版は1枚につき1要素まで | 画面越しは情報過多に弱い |
| 初回商談でない | ブロック1の自己紹介を省略。ブロック2（つかみ）は残す。ブロック3から本題 | 既知情報のみ飛ばす |
| 初回商談でない＋10分以下 | ブロック1・ブロック2どちらも省略、ブロック3から開始 | 2条件成立時の定義 |
| 高単価（総額100万円以上 または 月額10万円以上） | ブロック7は必ず `self-closing-4questions` | 即決クロージングは構造上不可 |
| 決裁型=高関与型 | 価格情報が無くてもブロック7は `self-closing-4questions` | 決裁型入力を優先 |

### 記入例

スライド1枚分の記入例：

```
## スライド4：御社が今抱える「見えないコスト」

**本文**
- 月間40時間の手作業工数
- 担当者1名あたり年間約80万円の機会損失
- このまま3年続くと、累計240万円の損失

### 視覚指示
- 中央にコスト推移の折れ線グラフ（3年分）
- 右下に「240万円」を赤字で強調

### 話者ノート
ここで heaven-hell-rhetoric の「地獄」パートを使います。
「このまま何もしなければ、3年後には240万円の損失が積み上がります。
担当者の残業も増え続け、離職リスクも高まります」と語りかけ、
相手の顔色を確認してから次のスライドへ進みます。
バックトラッキング用に、相手が共感した一言をメモしてください。
```

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| スライド本文に話者トークをそのまま載せる | 本文は箇条書き、トークは話者ノートへ分離 | 読まれて聞かれなくなる |
| 1スライドに7スキルを詰め込む | 1スライドに1スキル（1メッセージ） | 情報過多で刺さらない |
| 話者ノート省略 | 必ず100〜200字の話者ノートを付ける | ソラがプロンプト化できない |
| クロージングを最後の1枚で済ます | ブロック7に10%の枚数を配分 | クロージングは複数スライドで段階設計 |
| Genspark/Gamma プロンプトを本スキルで直接生成 | ソラに引き渡す（役割分担） | 工程分離が崩れる |
| 即決型なのに self-closing-4questions を選ぶ | 決裁型で必ず分岐 | 棲み分け違反 |
