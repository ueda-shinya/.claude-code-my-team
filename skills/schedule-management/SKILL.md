# schedule-management スキル

スケジュール管理 = タスクの計画・実行・進捗確認を行い、目標を効率的に達成する手法。事業戦略スクール「知足」のフレームワーク準拠。

## スケジュール管理の3つの目的

| 目的 | 内容 |
|---|---|
| 目的① 時間を有効活用する | 限られた時間を最大限に活かす |
| 目的② プロジェクトを期限内に完了させる | 締切遵守で信頼を獲得 |
| 目的③ リスクを軽減し、トラブルを未然に防ぐ | 遅延・リソース不足を予防 |

## スケジュール管理が重要な3つの理由

| 理由 | 内容 |
|---|---|
| 効率性の向上 | リソースを適切に配分し、無駄を省く |
| 締め切りの遵守 | 遅延を防ぎ、クライアントやチームの信頼を得る |
| チームの協力促進 | 全員が同じ目標に向かって取り組める環境を作る |

## 適用条件

- このスキルを使う場面：
  - 新規プロジェクトのスケジュール全体設計（目標→タスク→優先順位→ガント→進捗）
  - 既存PJの進捗管理体制構築
  - 複数手法（ガント／CPM／アジャイル）の使い分け判断
  - スケジュール課題（見落とし／リソース不足／遅延）の解決
- このスキルを使わない場面：
  - ガントチャート単体の詳細設計 → `gantt-chart-design`
  - プロジェクトの体制設計（人員・役割） → `project-team-structure`
  - KPIツリー設計 → `kgi-kpi-kai-design`
  - 目標の階層分解 → `goal-hierarchy-design`
  - 収支管理・契約締結 → `project-finance-contract`
  - SMART目標の設計 → `smart-goal-setting`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| プロジェクト概要 | 必須 | 案件・期間・チーム規模 | 「Webリニューアル・3ヶ月・5名」 |
| 目標（ゴール） | 必須 | 達成基準 | 「2026-08-31までに新サイト公開＋CV2倍」 |
| プロジェクト特性 | 任意 | IT／建設／マーケ等 | 「ITリリース系」 |
| 既知の制約 | 任意 | リソース・予算・期日制限 | 「専任2名＋兼務3名・予算300万円」 |
| 既知のリスク | 任意 | 想定される遅延要因 | 「クライアントの仕様変更頻発の懸念」 |

## スケジュール管理の5プロセス

| ステップ | 内容 | 手法・ツール |
|---|---|---|
| **1. 目標設定** | プロジェクトのゴールを明確にし、達成基準を定める | SMART原則を活用（→ `smart-goal-setting` 連携） |
| **2. タスクの洗い出しと分解** | 必要なタスクを全てリストアップし、細分化して管理しやすくする | WBS（Work Breakdown Structure） |
| **3. 優先順位の設定** | 緊急度と重要度でタスクを分類 | Eisenhowerマトリックス（緊急度×重要度4象限） |
| **4. スケジュール作成** | 各タスクの開始日・終了日を明確に設定 | ガントチャートを用いてタスクの進行状況を視覚化（→ `gantt-chart-design` 連携） |
| **5. 進捗管理** | 計画と実際の進行を比較する | 定期的なチェックポイントを設け、必要に応じて計画を修正 |

## スケジュール管理の3手法

| # | 手法 | 内容 | 向く案件 |
|---|---|---|---|
| 01 | **ガントチャート** | プロジェクト全体のタイムラインを視覚化 | 期日固定の建設・大規模PJ |
| 02 | **クリティカルパス法（CPM）** | 重要タスクを特定し、効率的な進行を実現 | 依存関係が複雑なPJ |
| 03 | **アジャイル管理** | スプリント単位で進捗を管理し、柔軟に対応 | 仕様変更が頻発するソフト開発 |

## スケジュール管理ツール（4選）

| # | ツール | 特徴 |
|---|---|---|
| 01 | **Trello** | カンバン方式でタスクを視覚的に管理 |
| 02 | **Asana** | チーム全体のタスクと進捗状況を一元管理 |
| 03 | **Googleカレンダー** | 簡易的なスケジュール管理とリマインダー機能 |
| 04 | **Microsoft Project** | プロジェクト管理専用ツールで詳細な計画が可能 |

これらのツールを適切に活用することで、時間やリソースの最適化が可能になり、プロジェクトの成功確率が高まる。

## 課題と解決策（3つ）

| 課題 | 内容 | 解決策 |
|---|---|---|
| **タスクの見落とし** | 作業工程を洗い出し切れないことがある | WBSを活用し、タスクを徹底的に分解する |
| **リソース不足** | 必要な人員や設備が足りない | 初期段階でリソースを把握し、優先順位を設定する |
| **遅延の発生** | 外部要因や変更要求でスケジュールが遅れる | バッファ期間を設定し、リスク計画を用意する |

## 成功事例（参考）

| ケース | 対応 | 結果 |
|---|---|---|
| ITプロジェクト | WBSでタスクを分解し、短期間のスプリントで進行 | 予定通りの納期でリリース成功 |
| 建設プロジェクト | ガントチャートで工程管理を徹底し、CPMで重要タスクを優先 | 工期内に完成し、コスト削減も実現 |

## 実行ステップ

### ステップ1：目標設定（プロセス1）
- 入力：プロジェクト概要、ゴール、特性
- 処理：
  1. SMART原則でゴールを明文化（具体性／測定可能／達成可能／関連性／期限）
  2. 達成基準を3〜5項目で具体化
  3. KPIがある場合は紐付け（→ `kgi-kpi-kai-design` 連携）
- 出力：SMART目標＋達成基準

### ステップ2：タスクの洗い出しと分解（プロセス2・WBS）
- 入力：ステップ1の目標
- 処理：
  1. 大項目（マイルストーン）を3〜7個に分解
  2. 各大項目を中項目（成果物単位）に分解
  3. 各中項目を小項目（実行可能な作業単位）に分解
  4. 各タスクに「担当者・所要時間・依存関係」を付与
  5. タスク数の目安：1人月のPJで20〜50タスク／3人月で60〜150タスク
- 出力：WBS（タスクの階層的リスト）

### ステップ3：優先順位の設定（プロセス3・Eisenhowerマトリックス）
- 入力：WBS
- 処理：以下4象限でタスクを分類
  | | 緊急 | 緊急でない |
  |---|---|---|
  | **重要** | 第1象限：即実施（クリティカルタスク） | 第2象限：計画的に実施（戦略的タスク） |
  | **重要でない** | 第3象限：委任を検討 | 第4象限：削除を検討 |
- 出力：優先順位付きタスクリスト＋削除候補

### ステップ4：スケジュール作成（プロセス4・手法選定）
- 入力：優先順位付きタスクリスト、PJ特性
- 処理：
  1. **手法選定**：以下の判定で手法を選ぶ
     | PJ特性 | 推奨手法 |
     |---|---|
     | 期日固定・大規模・タイムライン重視 | ガントチャート |
     | 依存関係が複雑・遅延リスクが大きい | クリティカルパス法（CPM） |
     | 仕様変更頻発・ソフト開発 | アジャイル管理 |
     | ハイブリッド（建設＋IT等） | ガント＋CPM併用 |
  2. **ガントチャート設計**（基本選定時）：→ `gantt-chart-design` 連携
  3. 各タスクに開始日・終了日を設定
  4. **バッファ期間**：全体の10〜20%を予備に確保
  5. **クリティカルパス**：遅延が許されない一連のタスクを特定（CPM選定時）
- 出力：手法別スケジュール（ガントチャート／CPM図／スプリント計画）

### ステップ5：進捗管理（プロセス5）
- 入力：完成したスケジュール
- 処理：
  1. **チェックポイント設定**：週次／隔週／月次でレビュー
  2. **報告フォーマット統一**：進捗％・課題・次アクション・リスク
  3. **計画と実績の比較**：差分が出た時の対応ルール
  4. **計画修正の手順**：どの差分でどう修正するか定義
- 出力：進捗管理ダッシュボード＋運用ルール

### ステップ6：3課題への予防策設計
- 入力：これまでの設計結果
- 処理：3課題に対し予防策を組み込む
  - **タスク見落とし**：WBSのレビュープロセスを設置（チームレビュー必須）
  - **リソース不足**：初期段階でリソース棚卸し＋優先順位による配分
  - **遅延発生**：バッファ期間設定＋リスク計画（→ KYT連携可）
- 出力：予防策チェックリスト

### ステップ7：振り返り（PJ完了後）
- 入力：実施記録
- 処理：以下4観点で振り返り
  1. **プロジェクトのゴール設定**：達成基準は具体的だったか
  2. **タスクの分解と優先順位設定**：必要な作業をリスト化し重要度に応じて整理できたか
  3. **進捗確認の定期化**：毎週または毎月の進捗確認ミーティングを実施できたか
  4. **振り返り**：成功点と課題を分析し、次回の改善策を練る
- 出力：振り返りログ＋次回PJ改善計画

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 期日固定・大規模PJ | ガントチャート主体 | タイムライン全体の見通しが必要 |
| 依存関係が複雑・遅延リスク高 | CPM併用 | クリティカルパス特定が肝要 |
| 仕様変更が頻発するソフト開発 | アジャイル管理 | スプリントで柔軟に対応 |
| WBSのタスクが10個未満 | 細分化が不足 | 1〜2人月でも20タスク以上が目安 |
| バッファなしで予定が組まれている | 全体の10〜20%をバッファに | 遅延予防の必須要素 |
| 進捗報告がない／不定期 | 週次レビュー＋報告フォーマット統一 | 計画と実績の乖離を早期検出 |
| リソースが3人月分しかないのに6人月のWBS | 優先順位で削るか期間延長 | 物理的に不可能なPJは早期再調整 |
| 「タスク見落とし」が起きた | WBSのレビュープロセスを次回から導入 | 個人での洗い出しは漏れる |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 目標が曖昧なまま着手 | SMART原則で明文化してからスタート | ゴールがないと達成判定不能 |
| WBSを作らずタスクを口頭管理 | 必ずWBSで階層的に分解 | タスク見落としの最大原因 |
| 優先順位を付けずに着手 | Eisenhowerマトリックスで4象限分類 | 優先順位なしで重要タスクが後回しに |
| ガントチャート1択で全PJ管理 | PJ特性に応じてCPM／アジャイルも検討 | ソフト開発に固定ガントは合わない |
| バッファなしで予定が組まれている | 全体の10〜20%をバッファ | 遅延が即PJ崩壊につながる |
| 進捗確認が不定期／なし | 週次／月次のチェックポイント設定 | 計画と実績の乖離を放置すると詰む |
| 仕様変更を呑み続ける | 変更管理プロセスで再合意（→ `client-expectation-management`） | スコープクリープでPJ崩壊 |
| 完了後の振り返りをしない | 4観点で振り返り＋次PJに反映 | 同じ失敗を繰り返す |

## 連携スキル

- `smart-goal-setting`：プロセス1（目標設定）でSMART原則
- `gantt-chart-design`：プロセス4（スケジュール作成）でガントチャート詳細設計
- `project-team-structure`：体制管理（人員・役割分担）
- `kgi-kpi-kai-design`：KPI設計（目標達成度の測定）
- `goal-hierarchy-design`：目標の階層分解
- `project-finance-contract`：収支管理＋契約締結
- `client-expectation-management`：仕様変更時の期待値再調整
- `pdca-cycle`：継続改善サイクル
- `kyt`：実装前のリスク予知（→ バッファ・リスク計画の入力）

## 5状態出力契約

外部API・サブプロセス・外部設定依存を持たないため、CLAUDE.md「5状態出力契約」の対象判定基準には該当しない。ただし以下のマーカーを使うこと：

- `[OK:0件]`：タスクが0件で計画が組めない場合（要再ヒアリング）
- `[PARTIAL:N成功/M失敗]`：5プロセスのうち一部のみ完成した場合
- `[SKIP]`：目標・期間が不明確で計画を保留した場合（前提不足）

※外部API依存なしのため `[FAIL]` は通常発生しない。前提不足は `[SKIP]` で扱う。
