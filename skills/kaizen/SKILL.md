# /kaizen スキル

ミス・問題が発生した際に、「なぜなぜ分析 → チーム協議 → 再発防止の仕組み化」を一連のフローとして実行します。

## 使い方

- `/kaizen` → 直近のミスについて協議開始
- `/kaizen <問題の説明>` → 特定の問題を指定して協議開始
- アスカが区切りのよいタイミングで自発的に発動することもある

## 前提ルール

- ミスを指摘されても進行中の業務は基本続行する。区切りのよいタイミングで協議を開始する
- シンヤさんがストップをかけた場合は即中断して協議に入る
- コーディングに限らず全業務に適用する（運用・コミュニケーション・判断ミス等も対象）

## トーン

犯人探しではなく仕組みの改善。建設的・前向きに進める。

## 手順

### ステップ1: なぜなぜ分析

問題を5W1Hで定義し、「なぜ？」を繰り返して真因にたどり着く。

#### 対応レベルの分類

まず問題を以下の3段階で分類し、対応の粒度を決める。

| レベル | シグナル | 対応 |
|---|---|---|
| 🔴 HIGH | 明確なミス・障害・シンヤさんからの修正指示 | フルの3ステップを実行 |
| 🟡 MEDIUM | うまくいった施策・成功パターンの確認 | ステップ1（要因分析）→ シンヤさんに報告・承認 → ステップ3（成功の仕組み化）|
| 🟢 LOW | 観察・疑問・「こうした方がよいかも」 | なぜなぜ分析は実施しない。記録のみ（memory or knowledge-buffer.md）→ 完了 |

#### 進め方

- 「なぜ？」は目安5回だが、柔軟に調整する（真因に到達すれば3回でもよいし、7回必要なこともある）
- 事実ベースで進める。推測は禁止（「おそらく」「たぶん」で止まらない）
- 各「なぜ」にはEvidence（根拠）を明記する。ログ・データ・実際の行動など、事実で裏付ける
- 主語を「人」ではなく「仕組み」にする（「アスカが忘れた」→「忘れを防ぐ仕組みがなかった」）
- 分析中、根本原因のパターンに迷ったら `skills/kaizen/references/software-patterns.md` の8パターンを参照する
- 実例が必要なら `skills/kaizen/examples/` 配下の分析例を参考にする
- 止め時の基準:
  - 具体的な再発防止策を設計できる地点に到達したら止める
  - 真因は「actionable（対策可能）」「preventable（再発防止可能）」「fundamental（これ以上掘る意味がない）」の3条件を満たすこと
  - 「まだ掘れるか？」と迷ったら、もう1段掘る
- 完了後、逆読み検証を実施する（真因から問題に向かって因果を逆順にたどり、論理が通るか確認）
- **逆読み検証後、リナ（logic-verifier）に真因分析の論理検証を依頼する**（自己検証の限界を補うため。ステップ2に進む前に必ず実施）

#### 参考資料

- `skills/kaizen/references/software-patterns.md` — ソフトウェア根本原因の8パターン（Design Gap, Process Gap, Knowledge Gap 等）
- `skills/kaizen/references/toyota-origins.md` — トヨタ生産方式の原理と現場主義の解説
- `skills/kaizen/examples/production-incident.md` — 本番インシデントの分析実例（Multi-Branch Analysis含む）
- `skills/kaizen/examples/performance-regression.md` — パフォーマンス劣化の分析実例（二次分析含む）

#### 出力フォーマット

```
【なぜなぜ分析】
■ 問題: （5W1Hで具体的に記述）
■ 対応レベル: 🔴 HIGH / 🟡 MEDIUM（🟢 LOWはこのフォーマット不要）
├ なぜ1: ...
│   Evidence: （根拠となる事実・ログ・データ）
├ なぜ2: ...
│   Evidence: ...
├ なぜ3: ...
│   Evidence: ...
├ なぜ4: ...
│   Evidence: ...
└ なぜ5: ...（真因）
    Evidence: ...
■ 逆読み検証: （真因→問題の因果を逆順にたどり、論理が通るか確認した結果）
■ 真因: （一文で簡潔に）
```

### ステップ2: チーム協議

なぜなぜ分析の結果をチームで共有し、対策案を出し合う。

#### 進め方

- 問題領域の担当エージェントを召集する（例: コーディングミスならシュウ＋サクラ、運用ミスならソウ等）
- 必ず召集する: ツムギ（ナレッジ記録担当）、リナ（ロジック検証担当）
- 必要に応じて追加召集する: カナタ（スキル・エージェント設計が必要な場合）、その他関係者
- なぜなぜ分析結果を共有し、各エージェントから対策案を出してもらう
- リナが対策案のロジック検証を行う（抜け穴はないか、本当に真因を潰せるか）
- 対策案をシンヤさんに提示し、方針を決定してもらう

#### 出力フォーマット

```
【対策協議結果】
■ 参加者: （召集したエージェント名）
■ 対策案:
  1. ...
  2. ...
■ リナの検証結果: （各対策案に対するロジック検証の結果）
■ 推奨案: （チームとして最も効果的と判断した案）
```

### ステップ3: 再発防止（仕組み化）と成功の横展開

シンヤさんが承認した対策を具体的な仕組みとして実装する。
🔴 HIGH の場合は「再発防止」、🟡 MEDIUM の場合は「成功パターンの仕組み化・横展開」を行う。

#### 仕組み化の優先度（強制力が高い順）

1. hook 追加（自動実行・人の意識に依存しない）
2. スキル新規作成 / 既存スキル修正（フロー化・手順化）
3. エージェント定義更新（特定エージェントの行動変更）
4. CLAUDE.md ルール追加（全体ルール）
5. memory ファイル記録（経緯の保存。仕組みとしては最弱）

※「気をつける」「注意する」は対策として認めない。必ず仕組みで防ぐこと。
※ 判定基準: 対策には「自動実行」「強制チェック」「物理的制約」のいずれかを含むこと。CLAUDE.mdやエージェント定義への注意書き追加だけでは「仕組み」とみなさない。

#### 進め方

- 承認された対策を実装する（スキル作成ならカナタに依頼、コード修正ならシュウに依頼等）
- 🔴 HIGH: 再発防止策の実装
- 🟡 MEDIUM: 成功パターンをスキル・テンプレート・ルールとして横展開し、他の業務でも再現できるようにする
- 実装後、シンヤさんに報告する
- 可能な場合はテスト確認を行い、結果を報告する

#### 出力フォーマット

```
（🔴 HIGH の場合）
【再発防止策】
■ 仕組み: （何を導入するか）
■ 実装方法: （スキル作成 / hook追加 / ルール追記 等）
■ 対象ファイル: （変更・作成するファイルのフルパス）
■ テスト結果: （テスト実施した場合はその結果。未実施なら「シンヤさんの運用で確認予定」等）

（🟡 MEDIUM の場合）
【成功の横展開】
■ 成功パターン: （何がうまくいったか）
■ 横展開方法: （スキル化 / テンプレート化 / ルール化 等）
■ 対象ファイル: （変更・作成するファイルのフルパス）
■ 適用範囲: （どの業務・場面に展開するか）
```

## 注意事項

- 3ステップすべてを1回の `/kaizen` で完結させる。途中で止めない（シンヤさんの指示がない限り）
- ただし、実装が大規模な場合はステップ3を「設計決定まで」とし、実装作業はNotionタスクに登録して管理する
- なぜなぜ分析で推測に頼らないこと。事実が不明な場合はシンヤさんに確認する
- 仕組み化の実装はアスカが直接コーディングしない。必ず担当エージェントに委任する
- 同じ問題の再発が確認された場合、前回の対策が不十分だったことを認め、より強い仕組みを検討する
