# ooda-loop スキル

OODAループ（ウーダループ）= Observe（観察）→ Orient（状況判断・方向付け）→ Decide（意思決定）→ Act（実行）の4ステップで構成される意思決定プロセス。米空軍ジョン・ボイド大佐が提唱。事業戦略スクール「知足」のフレームワーク準拠。

## OODAとPDCAの違い

| 観点 | OODAループ | PDCAサイクル |
|---|---|---|
| **想定外への対応** | 加味する（不完全・ジャストアイデアでも実行） | 加味しない（計画通り進める前提） |
| **スピード** | 速い（観察→即決→即実行） | 遅い（計画→実行→評価→改善） |
| **向く環境** | 変化が激しい・正解がない | 安定環境・既知の改善余地がある |
| **計画の重み** | 軽い（前段で時間をかけない） | 重い（Plan工程で精査） |
| **典型業務** | スタートアップ・現場対応・新規事業 | 既存業務改善・品質管理 |

**OODAとPDCAの関係**：細かいOODAループの繰り返し結果から、中期スパンのPDCAが作られる。両者は競合せず階層的に併用する。

## 適用条件

- このスキルを使う場面：
  - 変化の激しい市場・正解がない領域で迅速な意思決定が必要
  - スタートアップ・新規事業の運営フレーム設計
  - 現場での臨機応変な対応の仕組み化（営業・CS・店舗オペレーション）
  - 「PDCAじゃ追いつかない」と感じる業務の見直し
- このスキルを使わない場面：
  - 既知の業務をじっくり継続改善したい → `pdca-cycle`
  - 単発の意思決定で複数案を比較したい → `decision-making-framework`
  - 目標を階層分解したい → `goal-hierarchy-design`
  - KPI設計したい → `kgi-kpi-kai-design`

## 入力

| 項目 | 必須/任意 | 説明 | 例 |
|---|---|---|---|
| 対象業務／組織 | 必須 | OODAループを設計したい業務領域 | 「BtoCマーケ部門の広告運用」「新規事業のPMF探索」 |
| 現状の運営方式 | 任意 | 既存はPDCA／場当たり／不在等 | 「月次PDCAだが市場変化に追いつかない」 |
| 観察対象 | 任意 | 何を観察すべきか（顧客／競合／市場） | 「広告CTR・CV率・顧客の声」 |
| 意思決定権限 | 任意 | 現場で判断できるか／上位承認が必要か | 「広告予算調整は現場100万円まで権限委譲済み」 |

## OODAループの4ステップ

### STEP 1：Observe（観察）
**環境の変化や敵・市場の動向、顧客の反応など、必要な情報を収集する**
- 観察対象例：
  - 顧客の購買行動・離脱ポイント・問合せ内容
  - 市場の競合動向・新規参入・価格変化
  - 自社内の業績数値・現場の状況
  - 政治・経済・規制の変化
- ツール例：GA4／GSC／顧客アンケート／営業日報／SNSモニタリング／競合ウォッチ

### STEP 2：Orient（状況判断・方向付け）
**収集した情報を分析し、自社の強みや市場の特性を考慮して適切な方向性を定める**
- 判断ポイント：
  - 観察した変化のうち「対応すべきもの／無視できるもの」を仕分け
  - 自社のリソース・強みと照合する
  - 「いま打てる手」「あとで打つ手」「捨てる選択肢」を分類
- 注意：このステップが OODA の核心。情報の意味付けで打つ手の質が決まる

### STEP 3：Decide（意思決定）
**どの戦略やアクションを取るかを決定する**
- 判断ルール：
  - 完璧でなくてよい（ジャストアイデア・不完全可）
  - 撤回可能か／不可逆かをまず判定（不可逆なら追加検証）
  - 単一案より「やる／やらない／後回し」の3択にして即決
  - 5分以上迷うなら一旦実行して再観察に回す

### STEP 4：Act（実行）
**決定した戦略やアクションを素早く実行し、その結果を次の観察に活かす**
- 実行のコツ：
  - 小さく試す（プロトタイプ／A/Bテスト／限定エリア）
  - 結果データの取得方法を実行前に決めておく
  - 実行後すぐに次のObserveに戻す（ループ高速化）

## OODAループが効果的な企業条件

### A. 環境条件：変化が早く正解がない市場
- スタートアップ／新規事業部門／D2Cブランド
- 顧客ニーズや市場構造が流動的
- データより仮説検証・現場観察が重要
- 適応力とスピードで勝負する必要がある

### B. 組織条件：自律型・分散型の意思決定構造
- ティール組織／Holacracy／プロジェクト型企業
- 上意下達ではなく現場で判断できる文化がある
- ミッションや原則ベースのマネジメント
- フラットな組織構造 or セルフマネジメント型

### C. OODAが向いている業種・業態例
- スタートアップ企業（PMF・GTMフェーズ）
- BtoCマーケティング部門（顧客の声を拾い改善）
- 営業・CSチーム（現場状況に応じた臨機応変な対応）
- 店舗・現場オペレーション（接客・在庫調整など）
- 戦略コンサル・広告代理店など、クライアント対応業務

### D. 機能するための組織条件（チェックリスト）
- [ ] 経営陣が「意思決定の委譲」に前向き
- [ ] ミッション／バリュー／判断軸が明文化されている
- [ ] 成果よりも思考プロセスや試行錯誤を評価できる風土
- [ ] 学習を高速で回す文化（振り返りと対話の機会が多い）
- [ ] IT/データ環境が整っており、リアルタイムに現場状況を把握可能

## 実行ステップ

### ステップ1：対象業務とOODA適合性を診断する
- 入力：対象業務・組織情報
- 処理：
  1. 「環境条件A」「組織条件B」「業種・業態C」を1つずつチェック
  2. 「機能条件D」5項目のチェックリストで適合度を評価（5/5・4/5・3/5・2/5以下）
  3. 適合度3/5以下の場合は OODA 導入前に組織条件整備が必要と判定
  4. PDCA との使い分け方針を提示（OODA中心／OODA+PDCA併用／PDCA中心の3パターン）
- 出力：適合性診断レポート＋推奨フレーム選択

### ステップ2：観察項目（Observe対象）を設計する
- 入力：対象業務
- 処理：
  1. 観察カテゴリ4種（顧客／市場・競合／自社業績／環境変化）から該当を選定
  2. 各カテゴリで取得すべき指標・データソース・更新頻度を明確化
  3. リアルタイム性のある（日次・週次）データを優先する
- 出力：Observe項目テーブル（指標名／ソース／頻度／担当）

### ステップ3：判断軸（Orient基準）を定義する
- 入力：観察項目＋ミッション・バリュー・現状リソース
- 処理：
  1. 「対応すべき変化」と「無視する変化」の閾値を数値で定義（例：「広告CV率が前週比 -20% を超えたら対応」）
  2. 自社の強み・リソースと照合する評価軸を整理
  3. 判断のフローチャートを作成する（観察→閾値判定→対応／無視）
- 出力：Orient判断基準シート＋フローチャート

### ステップ4：意思決定プロトコル（Decide）を設計する
- 入力：Orient判断基準
- 処理：
  1. 意思決定の権限委譲ラインを明確化（現場決定／部長決定／経営決定の3レベルで予算・スコープを定義）
  2. 撤回可能／不可逆の判定基準を文書化
  3. 「5分迷ったら実行に回す」等のスピードルールを設定
- 出力：Decideプロトコル文書

### ステップ5：実行と次サイクルへの接続（Act）
- 入力：Decideプロトコル
- 処理：
  1. アクション実行前に「結果の測定方法」を必ず決める
  2. 実行後の振り返りタイミングを設定（即日／週次／月次）
  3. 振り返りで得たインサイトを次のObserveに反映するルートを明示
- 出力：Actと次サイクル接続フロー

### ステップ6：OODA × PDCA の組み合わせ設計
- 入力：ステップ1の適合性判定
- 処理：
  1. **OODA中心パターン**：日次・週次でOODA高速ループ、四半期にPDCAで方針再確認
  2. **OODA+PDCA併用パターン**：日次OODA × 月次PDCA（中期戦略の修正）
  3. **PDCA中心パターン**：OODA未導入推奨。組織条件整備を先行
  4. 各サイクルの実施タイミング・参加者・アウトプットを定義
- 出力：OODA × PDCA 階層運営シート

## 判断基準

| 状況 | 判断 | 理由 |
|---|---|---|
| 適合条件Dが3/5以下 | OODA導入前に組織条件整備 | 文化・権限委譲・データ環境がないとOODAは形骸化 |
| 業務が定型・既知・安定 | PDCAを推奨／OODA不要 | OODAは変化対応のため、安定業務では過剰 |
| 観察データがリアルタイム取得不可 | データ環境整備を先行 | 観察が遅いとループ全体が遅くなる |
| 意思決定が常に上位承認待ち | 権限委譲ラインの明文化を先行 | OODAの速度は権限委譲が前提 |
| 不可逆な意思決定（M&A・大型投資等） | OODA単独で進めない／PDCA＋意思決定フレーム併用 | 不可逆案件は速度より精度優先 |
| 観察→決定→実行が1日以上かかる | サイクル時間を短縮するボトルネックを特定 | OODAの優位性は速度。1日サイクルでは PDCA と差が出ない |
| 振り返りなしで実行を繰り返す | Act後の Observe接続を必ず実装 | 振り返りなきOODAは「思いつき施策の連発」に堕する |

## よくある失敗パターン

| NG | OK | 理由 |
|---|---|---|
| 観察データを集めずに即決から始める | Observe を必ず先行する | データ無き決定はOODAではなく「思いつき」 |
| Orient（判断軸）を曖昧にしたまま運営 | 数値閾値を必ず明文化 | 判断軸なきOODAはバラバラの場当たり対応 |
| 上位承認待ちで決定が遅い | 権限委譲ラインの明文化＋現場決定権の付与 | OODAの速度メリットが消える |
| 完璧な計画を求める | 不完全・ジャストアイデアでもActに進む | 完璧主義はPDCA向き、OODAでは害 |
| Act後に振り返らない | 振り返り→次Observeへの接続を実装 | 学習なきOODAは思いつき施策の連発 |
| 不可逆な意思決定もOODAで即決 | 撤回可否を判定し、不可逆は精度優先 | M&A等は速度より精度 |
| 組織条件Dを満たさずOODA導入 | 適合度3/5以下なら組織条件整備を先行 | 形骸化して「現場が振り回されるだけ」になる |
| OODAとPDCAを排他選択 | 階層的に併用（細かいOODA→中期PDCA） | 両者は競合しない |

## 連携スキル

- `pdca-cycle`：継続改善・既存業務の安定改善ループ。OODAと併用設計
- `decision-making-framework`：単発意思決定で複数案比較したいとき
- `goal-execution-system`：モニタリング・フィードバックループ設計（OODAのObserve・Orient強化）
- `kgi-kpi-kai-design`：観察すべき指標（KPI/KAI）の設計
- `pmf-journey`：スタートアップフェーズ判定。OODA中心運営はPMF前後で特に重要
- `project-team-structure`：自律型組織構築の体制設計

## 5状態出力契約

外部API・サブプロセス・外部設定依存を持たないため、CLAUDE.md「5状態出力契約」の対象判定基準には該当しない。ただし以下のマーカーを使うこと：

- `[OK:0件]`：適合性診断の結果すべての条件が非該当の場合（OODA非推奨）
- `[PARTIAL:N成功/M失敗]`：適合度が部分的に満たされる場合（適合3/5等）
- `[SKIP]`：観察データが取得できないためObserve設計を保留した場合／対象業務が定型・安定すぎてOODA導入価値がないため `pdca-cycle` に誘導した場合（試みる前の前提条件未満による不実行）

※本スキルは外部API・サブプロセスを持たないため、`[FAIL]`（試行後の失敗）は通常発生しない。前提不足は全て `[SKIP]` で扱う。
