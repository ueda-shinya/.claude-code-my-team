## 1. 背景・目的

### 背景
Claude Code上で動作するエージェントシステム（スキル・MCP・hook・プロンプト定義）において、実装不具合が繰り返し発生している。コードではなくプロンプト・設定ファイルが実装の主体であるため、従来のユニットテストやCIが適用しにくい。

### 目的
「実装不具合は必ず起きる」という前提に立ち、以下の4点を達成する仕組みを整備する：

1. **早期発見** — 不具合が起きたことに素早く気づける
2. **再発防止** — 同種の不具合が繰り返されない
3. **影響極小化** — 不具合が起きても被害を最小限に抑える
4. **迅速復旧** — 不具合発生後の対応を標準化し、収束を速める ← v3追加

---

- MCP設定（google-calendar等）

- CLAUDE.md・SKILL.md等のプロンプトファイル

## 4. 各対策の詳細

### No.1 変更時チェックリスト整備

**目的：** 設定変更時の見落としを構造的に防ぐ
**内容：**
変更種別ごとのチェックリストを `~/.claude/checklists/` に整備する。
構成：**共通チェック（5〜8項目）＋種別固有チェック（3項目以内）** の2層構造（10項目超は形骸化リスクあり）。

共通チェック：
- [ ] パスは絶対パスか
- [ ] 環境依存の値はないか
- [ ] 依存するMCPサーバーは起動確認済みか
- [ ] エラー時のフォールバックは記述したか
- [ ] 前回同種の変更で何が起きたか確認したか
- [ ] 変更前後のdiffを記録したか
- [ ] **ロールバック手順を用意したか（前の状態に戻せるか確認）** ← v3追加
- [ ] **変更後の動作確認手順を決めたか（何をもって正常とするか）** ← v3追加

**完了基準：** チェックリストが整備され、3回以上の設定変更で使用され、チェックリスト自体のフィードバック改善が1回以上行われた
**担当：** サクラ（作成）、ソウ（過去障害視点でのレビュー）

---

### No.2 サイレント失敗の検知ログ

**目的：** 「エラーなく素通りした」不具合を見逃さない
**内容：**
各スキル実行時に「実行されたか・成功したか」をログに残す仕組みをSKILL.md内に記述（hookよりも確実なため）。

ログ保存先：`~/.claude/logs/skill-execution.log`（追記形式）
ログフォーマット：
```
YYYY-MM-DD HH:MM:SS | スキル名 | 呼び出し元 | OK/FAIL | 失敗分類 | 詳細
```
- `呼び出し元`：hook / アスカ / ユーザー直接 等
- `OK/FAIL`：固定文字列（表記揺れ防止）
- `失敗分類`：`MCP_UNAVAILABLE / PATH_ERROR / TIMEOUT / AUTH_ERROR / UNKNOWN`（OKの場合は省略）
- `詳細`：自由記述（OKの場合は省略）

通知タイミングの二段階化：
1. **セッション中に失敗した場合** → 即座にシンヤさんに報告
2. **セッション不在中の失敗** → 次回セッション開始時（morning-briefingのステップ3相当）に通知

ログローテーション：月次でアーカイブし、直近3ヶ月分を保持する

**完了基準：** morning-briefingで実証動作確認済み、かつ1週間分のログが正常に記録されている
**担当：** カナタ（実装）

---

### No.3 スキル・設定テンプレート標準化

**目的：** 新規作成時の構造的欠落を防ぐ
**内容：**
SKILL.md・エージェント定義・hookスクリプトのテンプレートを `~/.claude/templates/` に整備する。必須セクション・記述順序・命名規則を定義する。

**完了基準（分離）：**
- Phase A：テンプレートが整備され、既存スキル3件以上が準拠している
- Phase B：No.4のバリデーション完成後、テンプレート準拠が自動チェックされる（No.4に委譲）

**担当：** サクラ（テンプレート設計・品質基準）、カナタ（技術実装・バリデーション連携はNo.4で対応）

---

### No.4 エージェント定義のスキーマバリデーション

**目的：** 必須項目の欠落による構造的不具合をゼロにする
**内容：**
エージェント・スキルのmdファイルを自動チェックするスクリプト（`~/.claude/scripts/validate-definitions.sh`）を作成する。

チェック対象（最低限）：

エージェント定義：
- `name` が存在するか
- `description` が存在し空でないか
- `tools` が存在するか
- `model` が存在するか

SKILL.md：
- `# /スキル名` の見出しがあるか
- `## トリガー条件` セクションがあるか
- `## 実行手順` セクションがあるか
- フォールバックに関する記述があるか（No.5と連動）
- テンプレート準拠のチェック（No.3と連動）

実装フェーズ：
- Phase 1：手動実行スクリプト（1セッション）
- Phase 2：カナタのエージェント定義に「生成後にバリデーション実行」を組み込む（1セッション）
- Phase 3：No.3テンプレート確定後、準拠チェックを追加（1セッション）

**完了基準：** 既存の全エージェント定義がバリデーション通過済み、かつ新規作成時に自動実行される
**担当：** カナタ

### No.5 フォールバック明記（SKILL.mdへの記述規約）

**目的：** 外部依存（MCP等）の失敗時に無言省略を防ぐ
**内容：**
各SKILL.mdに「失敗時の挙動」を必須セクションとして追加する。フォールバックの記述フォーマットはNo.3のテンプレートに組み込んで統一する。

**対象：** `~/.claude/skills/` 配下の全SKILL.md（サクラが対象リストを作成・管理）
**完了基準：** 全件に明記済み（サクラが対象リストで管理）
**担当：** サクラ（規約策定・対象リスト管理・レビュー）

---

### No.6 前提条件の明示（Fail Fast設計）

**目的：** 前提破綻状態での中途実行を防ぐ
**内容：**
各スキルの冒頭に「正常動作に必要な前提条件」を記述し、前提が満たされない場合は早い段階で明示的に失敗させる設計にする。

**完了基準：** 全スキルの50%以上（morning-briefing含む）で前提条件チェックが実装済み
（スキルが増えても基準として機能し続けるよう割合で定義）

**担当：** カナタ（設計）、サクラ（レビュー）

---

### No.7 ペアレビュー体制の正式化

**目的：** 作成者のバイアスによる見落としを防ぐ
**タイミング：** 変更後・`/commit` スキル実行前

レビュー観点：
1. **構造の一貫性** — テンプレートに準拠しているか
2. **欠落チェック** — 必須項目・フォールバック・前提条件が揃っているか
3. **可読性** — 他のエージェントが読んで意図を誤解しないか
4. **過去障害パターン照合** — ソウのナレッジと矛盾していないか

リクは内容の正確性（事実確認・矛盾検出）を補完的に担当する。

**完了基準：** 体制がCLAUDE.mdに明文化され、3件以上のレビュー実績があり形骸化していない（指摘0件の連続はNG）
**達成判定：** アスカが月次で確認
**担当：** アスカ（体制設計・最終承認）、サクラ（レビュー実施）、リク（正確性補完）

### No.12 インシデント対応フロー定義 ← v3新規追加

**目的：** 不具合発生時の初動を標準化し、場当たり的対応を防ぐ
**内容：**
障害発生時の対応フローを定義し、CLAUDE.mdに組み込む。

**標準対応フロー：**
1. **検知・記録（ソウ）** — `~/.claude/troubleshooting/active/` に記録ファイルを作成
2. **影響範囲の特定（アスカ）** — 何が動いていないか、他への波及はあるか
3. **暫定対策の実施（アスカ判断）** — まず動く状態に戻す（ロールバック or 代替手段）
4. **原因の切り分け（ソウ）** — 仮説立案 → 検証 → 絞り込み。確定した「正常領域」を宣言し再調査を禁止
5. **恒久対策の実施（カナタ/該当担当）** — 根本原因への対処
6. **再発防止策の反映（サクラ）** — チェックリスト・テンプレートへのフィードバック

**ロールバック手順の明文化：**
- gitで管理されているファイルは `git revert` または特定コミットへの `git checkout` で復元
- 手順を各チェックリストの「ロールバック確認」項目と紐付ける

**完了基準：** フローがCLAUDE.mdに明文化され、直近の障害1件以上でこのフローに沿った対応が実施されている
**担当：** アスカ（フロー設計）、ソウ（記録・切り分け）

以下がすべて達成された状態を「完了」とする。**達成判定はアスカが月次で実施する。**

1. 変更時チェックリストが整備され、3回以上の変更で使用され、フィードバック改善が1回以上行われている

4. ペアレビュー体制が明文化され、3件以上の実績があり形骸化していない（指摘0件の連続はNG）
5. インシデント対応フローがCLAUDE.mdに明文化され、実際の障害で1件以上適用されている ← v3追加
