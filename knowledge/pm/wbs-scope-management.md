# マルチエージェント運用のためのWBS・スコープ管理術

出典：マルチエージェント運用のためのプロジェクトマネジメント実務ガイド（WBS・スコープ管理）
準拠：PMI（PMBOK® Guide）/ DSDM（Dynamic Systems Development Method）

---

## 1. WBSの正しい定義（PMBOK準拠）

> WBSとは「プロジェクトチームがプロジェクト目標を達成し、必要な成果物を作成するために実施される作業の階層的分解」

**WBS = 成果物（名詞）の木構造。アクティビティ（動詞）のリストではない。**

### WBS vs スケジュールの区別

| WBS（スコープの定義） | プロジェクトスケジュール（時間の定義） |
|---|---|
| 構成要素：成果物（名詞）例「調査レポート」 | 構成要素：アクティビティ（動詞）例「調査する」 |
| 含まない：順序・期間・依存関係・担当者 | 含む：開始/終了日・期間・依存関係・リソース |
| 目的：プロジェクトの全範囲を特定する | 目的：いつ・誰がやるかを計画する |

### 100%ルール（The 100% Rule）
- 子要素をすべて足すと親要素と完全にイコール
- 上位レベルに下位レベル未収録の作業を含めない
- 下位レベルに上位レベル未収録の作業を含めない

### マルチエージェント文脈でのWBS例
```
1.0 AI記事作成プロジェクト
  1.1 リサーチフェーズ成果物
    1.1.1 検索クエリリスト
    1.1.2 検索結果サマリー
  1.2 執筆フェーズ成果物
    1.2.1 構成案
    1.2.2 記事ドラフト
  1.3 編集フェーズ成果物
    1.3.1 校正済み原稿
    1.3.2 最終納品ファイル
```

## 2. スコープ管理

### プロジェクトスコープ記述書の6要素
エージェントへのシステムプロンプト・コンテキストに含めるべき項目：

1. **目的（Objective）**：なぜこのプロジェクトを行うか
2. **成果物（Deliverables）**：何を作るかの詳細
3. **合格基準（Acceptance Criteria）**：成果物が受け入れられるための品質要件
4. **除外事項（Exclusions）**：「やらないこと」の明記 ← **最重要**
5. **制約条件（Constraints）**：守るべき制限（予算・トークン数・期限）
6. **前提条件（Assumptions）**：真実とみなして計画する事柄

### スコープクリープ（Scope Creep）
> 「プロジェクト開始後に、スコープが管理されないまま（時間・コスト・リソースの調整なしに）拡大していく現象」

エージェントへの追加指示を繰り返すことで発生しやすい。プロジェクト失敗の主因。

## 3. タスク優先順位付け

### MoSCoW分析（DSDM公式定義）

| 分類 | 定義 | 備考 |
|---|---|---|
| **Must Have** | 必須要件。これがなければキャンセルレベル。回避策なし | 全工数の60%以内を推奨 |
| **Should Have** | 重要だが回避策あり。これなしでも機能は果たせる | |
| **Could Have** | あれば良い。問題発生時に最初にスコープから削る | コンティンジェンシーの主プール |
| **Won't Have** | 今回はやらないと合意済み。将来候補として記録 | |

### アイゼンハワーマトリクス（エージェント委任の判断軸）

|  | 緊急 | 緊急でない |
|---|---|---|
| **重要** | 第1象限：Do First（今すぐ自分でやる） | 第2象限：Schedule（人間が計画） |
| **重要でない** | **第3象限：Delegate（★エージェント委任に最適）** 定型報告・単純調査・会議調整 | 第4象限：Delete（削除） |

### クリティカルパス法（CPM）
- **クリティカルパス**：依存関係にあるタスクの最長シーケンス＝プロジェクト完了の最短時間を決定
- **クリティカルタスク**：スラック（フロート）がゼロのタスク
- スラック = プロジェクト終了日を遅らせずにタスクを遅らせられる余裕時間
- ⚠️ 進捗状況によりクリティカルパスは変動する

## 4. 依存関係の4タイプ

| タイプ | 定義 | 実務での使用頻度 |
|---|---|---|
| **FS（Finish to Start）** | Aが完了しないとBは開始できない | 最も一般的 |
| **FF（Finish to Finish）** | Aが完了しないとBは完了できない | 並走可能だが完了は待つ |
| **SS（Start to Start）** | Aが開始しないとBは開始できない | 並走作業に適用 |
| **SF（Start to Finish）** | Aが開始しないとBは終了できない | 稀（実務ではほぼ使わない） |

### テキストベースの依存関係整理例
```
[ID: 1.1] 競合調査（担当: ミオ）　依存関係: なし
[ID: 1.2] 顧客分析（担当: ミオ）　依存関係: なし
[ID: 2.1] 戦略立案（担当: レン）　依存関係: [1.1] FS, [1.2] FS
```

## 5. 変更管理プロセス（スコープクリープ防止）

1. **識別**：変更リクエストを記録
2. **評価**：スケジュール・コスト（トークン）・品質への影響分析
3. **承認**：実施するか否かを決定（**人間が判断**）
4. **実施**：WBS・スケジュール・プロンプトを修正し、エージェントに指示
5. **記録**：変更履歴を残す

## 6. 実務テンプレート

### WBSミニテンプレート
```
1.0 [プロジェクト名]
  1.1 [フェーズ1: 準備/入力]
    1.1.1 [成果物A: 情報収集データセット]
    1.1.2 [成果物B: 前提条件定義書]
  1.2 [フェーズ2: 処理/生成]
    1.2.1 [成果物C: コンテンツ構成案]
    1.2.2 [成果物D: ドラフト]
  1.3 [フェーズ3: レビュー/出力]
    1.3.1 [成果物E: 校正済み最終原稿]
    1.3.2 [成果物F: 納品用フォーマット済みファイル]
```

### スコープ記述書テンプレート
```
■ プロジェクト名:
■ 目的（Objective）:
■ 主要成果物（In Scope）:
■ 除外事項（Out of Scope）:
■ 制約条件（Constraints）: 期限 / 予算/トークン数
■ 合格基準（Acceptance Criteria）:
```

```
ID  | タスク名       | MoSCoW | 依存先           | 担当エージェント

1.2 | 分析レポート   | Must   | 1.1 (FS)         | レン
```

## 7. よくある失敗パターンと対策

| 失敗パターン | 対策 |
|---|---|
| **WBSのプロセス化**：「〜を調査する」等の動詞で書いて完了基準が曖昧 | 名詞で定義。「調査結果レポート」のように成果物として定義し100%ルールを適用 |
| **依存関係の無視**：前工程完了前に次工程を起動しハルシネーション発生 | FS依存を徹底。クリティカルパス上は前タスクの出力を確認してから次を開始 |
| **スコープクリープ**：「ついでに」追加指示を繰り返して目的がブレる | 追加要望はShould/Couldとして扱い、Must要件への影響を評価してから採用 |

---

## 8. まとめ（アスカとしての実践原則）

- **WBS（成果物）は自分が定義し、作成プロセス（アクティビティ）はエージェントに任せる**
- **スコープ（境界線）は自分が定義し、その内側での探索はエージェントに任せる**
- **優先順位（MoSCoW）は自分が決定し、実行速度はエージェントに任せる**
- 追加指示の前に「これはMust/Should/Could/Won'tのどれか」を自問してスコープクリープを防ぐ
