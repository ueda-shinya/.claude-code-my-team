# GA4 離脱率の読み方・分析・改善ガイド（公式準拠改訂版）

出典：GA4バウンス率・離脱数定義の公式準拠改訂資料.pdf（2026-03-25改訂）
GA4公式ヘルプ（support.google.com/analytics）準拠

---

## 1. 公式定義（最重要）

### バウンス率
> エンゲージドセッションでなかったセッションの割合

**バウンス率 = 100% − エンゲージメント率**（補数関係。逆数ではない）

### エンゲージドセッションの条件（いずれか1つ満たせば良い）
1. **10秒超**継続した（"longer than 10 seconds"。10秒ちょうどはNG）
2. キーイベントが発生した
3. 2回以上のページ/スクリーンビューがあった

⚠️ **よくある誤解**：バウンス率 ≠ 「すぐ帰った人」。1ページしか見ていなくても10秒超ならバウンスにならない。

### 指標の違い

| 指標 | 定義 | 何を見るか |
|---|---|---|
| バウンス率 | エンゲージドでなかったセッション率 | セッション全体の質 |
| 離脱数（Exits） | そのページが最後になったセッション数 | どのページで終わったか |
| エンゲージメント率 | エンゲージドセッションの割合 | 積極的に関与したセッション率 |
| 平均エンゲージメント時間 | ページがフォーカスされていた平均時間 | 滞在の深さ |

⚠️ GA4に「Exit Rate（離脱率）」という公式標準指標は**存在しない**。Exits÷セッション数は独自計算であり公式指標ではない。

## 2. 解釈レベル3段階モデル

| レベル | 発言例 | 状態 |
|---|---|---|
| Level 1（初級） | 「高めですね」 | 数値の大小しか見ていない |
| Level 2（中級） | 「業界平均より高い。ページによって差がある」 | 比較ができる |
| Level 3（上級） | 「トップページは68%だがSNS流入に限ると93%。訴求ズレが原因と仮説」 | 原因仮説まで言える |

## 3. 数値を見たら問う5つの質問

1. **どのページ/セクションのバウンス率か？**（サイト全体 vs 特定LP）
2. **流入元は何か？**（オーガニック検索 vs SNS流入では文脈が全く違う）
3. **デバイスは？**（スマホはPCよりバウンス率が高くなりやすい傾向）
4. **コンテンツのゴールは何か？**（1ページ完結型なら高くて当然の場合も）
5. **時系列で変化しているか？**（急上昇はタグ外れ等の技術的問題を疑う）

---

## 4. 分析手順（4ステップ）

### Step 1：ランディングページレポートで入口を特定
- 場所：レポート → エンゲージメント → ランディングページ
- 見る指標：セッション数（量）、平均エンゲージメント時間（質）、バウンス率（カスタム追加）
- **「バウンス率が高い」かつ「セッション数が多い」ページを最優先で調査**

### Step 2：探索「自由形式」でExits（離脱数）を確認
- 場所：探索 → 自由形式
- ディメンション：ページタイトルとスクリーンクラス
- 指標：Exits（離脱数）、Entrances（入口数）を追加
- ⚠️ 標準の「ページとスクリーン」レポートにはExitsは既定では含まれていない

### Step 3：探索「経路データ探索」で離脱の前後を可視化
- 場所：探索 → 経路データ探索
- 「開始ポイント」に対象ページを設定 → 次のノードを展開
- 「離脱（セッション終了）」の太い線を探してドロップポイントを特定

### Step 4：探索「ファネルデータ探索」でCV経路を診断
- 設定例：ランディング → スクロール50% → CTAクリック → フォーム到達
- 「読まれていないのか」「読んでいるがボタンを押さないのか」を判別

## 5. 離脱パターン別・原因仮説マトリクス

| 離脱パターン | 考えられる原因 | 確認方法 |
|---|---|---|
| ランディング直後（10秒以内） | 訴求ズレ・ページ速度問題 | ランディングページレポート × 流入元 |
| ページ途中 | コンテンツ難易度・長すぎる・読みにくい | スクロール深度イベント・ヒートマップ |
| CTA直前 | 信頼性不足・CTAの訴求が弱い・オファー不足 | ファネル探索でCTA前ステップ完了率確認 |
| 特定デバイスのみ | スマホUI/UX問題・誤タップ・レイアウト崩れ | デバイスカテゴリでセグメント分割して比較 |
| 特定流入元のみ | ターゲットと内容のミスマッチ | セッションのソース/メディアで分解 |

## 6. 改善アクション優先度フレーム

- **すぐやる**：高インパクト × 実施しやすい
- **計画的に**：高インパクト × 実施しにくい
- **余裕があれば**：低インパクト × 実施しやすい
- **保留**：低インパクト × 実施しにくい

### よくある改善アクション

| カテゴリ | 施策 |
|---|---|
| コンテンツ | 記事冒頭に要約を追加・離脱箇所のテキスト削減・スマホでスワイプ形式へ |
| UX/デザイン | CTAをFVへ・関連記事を離脱前に配置・ページ表示速度改善 |
| 流入品質 | 広告とLPの訴求一致・検索意図と内容の整合性確認 |

## 7. 報告テンプレート

### 書面報告
```
[ページ名]のバウンス率は[X%]で、サイト全体平均の[Y%]と比べると[高い/低い]状態です。
[流入元]からの訪問者に限ると[Z%]と特に高く、
[原因仮説：例「SNS広告の訴求とページ内容のズレ」]が原因として考えられます。
経路データ探索で確認したところ、[ページ名]に到達した後、[XX%]のユーザーが
[次のページ/離脱]に進んでおり、特に[セクション名]付近でのドロップが顕著です。
改善策として[具体的なアクション]を実施し、バウンス率を[目標値]まで下げることを目指します。
```

### 口頭報告
```
「今週のバウンス率は68%でした。数字だけ見ると高いのですが、内訳を見ると問題は
[特定ページ]に集中しています。そのページへの流入の多くがSNSからで、投稿内容と
ページ内容がズレていることが原因と考えられます。来週、投稿コピーとLPのヘッドラインを
揃えるテストをして、1週間後に数値を見直す予定です。」
```

## 8. 今日からできる3アクション

1. ランディングページレポートにバウンス率列を追加（レポート右上「カスタマイズ」→ 指標 → バウンス率）
2. 探索「自由形式」でExits確認（探索 → 自由形式 → ページタイトルとスクリーンクラス ＋ Exits / Entrances）
3. 次回の報告に「なぜなら〜と仮説しています」を追加する習慣
