# 画像役割辞書 — Role Taxonomy（3軸12タイプ）

## Phase 1 注意書き（重要）

**現状のたたき台です。** 2026-04-25 のラウンドテーブル（カイ・ユイ・レン・コト・ルナ・リナ・カナタ）で合意した「3軸12タイプ + Primary/Secondary 階層 + AI向き不向きフラグ + 構造化メタデータ」を Phase 1 として最低限の形で実装したもの。

- リナから MECE 性（軸間の境界・タイプ間の重複）について指摘あり。Phase 2 で精緻化予定
- C-1（世界観設定）の判定基準は **「他のブランドLPに転用可能なら装飾扱い」** を採用済（Decorational 概念は辞書から排除）
- Phase 2 では運用ログ（実案件での適用事例）を蓄積したうえで、軸再編・タイプ統廃合・新タイプ追加を検討する
- **既存スキル（lp-create / lp-design-system / frontend-design 等）経由でルナを呼ぶ場合の運用**：
  これらのスキルは役割コード概念を持たないため、呼び出し時にアスカ（または該当スキル実行者）が
  事前に辞書を参照して役割コードを補完してからルナに渡す。Phase 2 で `/image-brief` スキル化により
  自動化予定。Phase 1 では呼び出し元に追加負担が発生することを許容する。

## 0. 辞書の使い方（運用ルール）

### 0.1 タグ付与の階層化ルール（必須）

1 枚の画像には以下を付与する：

- **Primary タグ**：1 つだけ。その画像が**最も強く担う役割**を 1 つ選ぶ
- **Secondary タグ**：最大 2 つまで。Primary を補完する副次的役割
- **複数タグの並列禁止**：「C-1 / C-2 / E-2 を同時に主役」のような並列指定は禁止。必ず階層化する

**例**：LP ファーストビューのヒーロー画像で、女性が CTA を見ている構図
- Primary: `E-2`（理想未来）
- Secondary: `B-1`（視線誘導） / `C-1`（世界観設定）

### 0.2 AI 向き不向きフラグ（運用判定用）

各タイプには以下のいずれかが付く：

| フラグ | 意味 | 取り扱い |
|---|---|---|
| `AI-READY` | AI 生成で十分到達可能。ルナで設計→ Gemini で生成して問題なし | デフォルト |
| `PHOTO-PREFERRED` | 実写優位。ストック写真または現地撮影を推奨。AI は補助的に使う | 案件ごとにシンヤさんと相談 |
| `ILLUSTRATIVE-OK` | イラスト・図解で表現可能。むしろ AI イラストの方が伝わりやすい | ルナのドメインモードで Illustration / Infographic を選択 |

### 0.3 ブリーフへの最低必須情報

辞書を引いてプロンプト変換する際、各タイプの「ブリーフで追加必須となる情報」を埋めること。空欄のままルナに渡すのは禁止。

### 0.4 メタデータ項目の標準セット（全タイプ共通）

各タイプは以下の **9 項目** で記述する。タイプ固有の特殊事項がない場合でも、すべての項目を記載する（該当なしの場合は「— なし」と明記）。

| 項目 | 必須/任意 | 内容 |
|---|---|---|
| **役割の定義** | 必須 | そのタイプが担う機能を 1〜2 文で |
| **典型配置セクション** | 必須 | LP / サイトでよく置かれる位置 |
| **プロンプト変換時に規定する要素** | 必須 | 5 コンポーネントのうち重点要素 |
| **AI 生成適性** | 必須 | ◎ / ◯ / △ / × の 4 段階 |
| **AI 向き不向きフラグ** | 必須 | `AI-READY` / `PHOTO-PREFERRED` / `ILLUSTRATIVE-OK` |
| **コピーとの接続パターン** | 必須 | アンカリング / リレー / 二重符号化 等 |
| **alt 記述ガイドライン** | 必須 | アクセシビリティ・SEO 観点の記述指針 |
| **ブリーフで追加必須となる情報** | 必須 | プロンプト化前に埋めるべき項目 |
| **使用上の注意** | 任意（空欄可） | タイプ固有の運用上の注意点。なければ「— なし」と明記。重要原則・判定基準など特に強調すべき内容はこの項目内にサブ見出しで記述する |

## 1. 軸 C：認知的機能（Cognitive Function）— 何を「分からせるか」

### C-1：世界観設定（Worldview / Brand Tone）

| 項目 | 内容 |
|---|---|
| **役割の定義** | ブランド・トーン・世界観・「ここはどんな場所か」を視覚化する。コピーが語る「誰が・何を・どう提供するか」の文脈を空気として伝える |
| **典型配置セクション** | FV ヒーロー背景、セクション間のブリッジ、フッター上部 |
| **プロンプト変換時に規定する要素** | スタイル / 照明 / ムード（被写体・構図は副次的） |
| **AI 生成適性** | ◎ |
| **AI 向き不向きフラグ** | `AI-READY` |
| **コピーとの接続パターン** | アンカリング（コピーが画像の意味を絞る） |
| **alt 記述ガイドライン** | 装飾的画像の場合は `alt=""`（空文字）。ブランド固有の意味を持つ場合は「(ブランド名)の世界観を表現する〇〇のイメージ」と簡潔に |
| **ブリーフで追加必須となる情報** | ① ブランドトーン形容詞 3 つ（例：温かい / 知的 / 軽やか）② 色相帯 ③ 「他のブランド LP に転用可能か？」のチェック（YES なら装飾扱いで再設計） |
| **使用上の注意** | **【判定基準（重要）】**：このカットを別のブランド LP に貼り替えても成立する場合は「装飾」扱いとなり、本タイプではなく再設計対象とする。ブランド固有の文化的含意（Barthes の connotation）が乗っているかを必ず確認 |

---

### C-2：オファー具体化（Offer Materialization）

| 項目 | 内容 |
|---|---|
| **役割の定義** | 商品 / サービスの実体・UI・物理的形状を見せる。「で、結局何が手に入るのか？」に答える |
| **典型配置セクション** | FV、機能説明セクション、価格表近傍 |
| **プロンプト変換時に規定する要素** | 被写体（製品・UI そのもの） / 構図（製品が主役になる） / 照明（質感を出す） |
| **AI 生成適性** | ◯（物理製品）／ △（特定 UI スクリーンショットは実物推奨） |
| **AI 向き不向きフラグ** | `AI-READY`（物理製品・コンセプトモック）／ `PHOTO-PREFERRED`（既存実 UI・実物撮影品） |
| **コピーとの接続パターン** | リレー（コピーと画像が相互補完で意味を前進） |
| **alt 記述ガイドライン** | 「(製品名)の(具体的な特徴)」を明記。例：「LapDeck Pro の本体側面、USB-C ポート 2 つを搭載」 |
| **ブリーフで追加必須となる情報** | ① 製品名 ② 製品の最重要特徴 1〜2 点 ③ 実物撮影が可能か（可能なら `PHOTO-PREFERRED`） |
| **使用上の注意** | — なし |

---

### C-3：概念可視化（Concept Visualization）

| 項目 | 内容 |
|---|---|
| **役割の定義** | 抽象的なサービス・仕組み・価値を、比喩や図解で具体化する。Carney & Levin の Interpretational に対応 |
| **典型配置セクション** | 仕組み説明セクション、特徴比較、抽象概念の導入 |
| **プロンプト変換時に規定する要素** | 被写体（メタファー対象） / スタイル（イラストレーション系が多い） / ムード |
| **AI 生成適性** | ◎ |
| **AI 向き不向きフラグ** | `ILLUSTRATIVE-OK` |
| **コピーとの接続パターン** | アンカリング（画像が抽象概念を具体化、コピーが意味を確定） |
| **alt 記述ガイドライン** | 「〇〇を表す比喩イラスト：(主要な視覚要素)」 |
| **ブリーフで追加必須となる情報** | ① 可視化したい抽象概念（1 文で）② 採用したい比喩・メタファー ③ イラスト / 図解 / 写真コラージュのいずれか |
| **使用上の注意** | — なし |

---

### C-4：プロセス図示（Process Diagram）

| 項目 | 内容 |
|---|---|
| **役割の定義** | 「どう機能するか」のフロー・ステップを順序付きで可視化する。Carney & Levin の Organizational に対応 |
| **典型配置セクション** | How it works セクション、申込フロー、サービス開始までのステップ |
| **プロンプト変換時に規定する要素** | 構図（順序が読める配置） / スタイル（インフォグラフィック系） |
| **AI 生成適性** | △（テキスト要素が入るため Figma 後処理推奨） |
| **AI 向き不向きフラグ** | `ILLUSTRATIVE-OK`（背景・アイコン部分のみ AI 生成、ステップ番号や日本語テキストは Figma 後載せ） |
| **コピーとの接続パターン** | リレー |
| **alt 記述ガイドライン** | 「(サービス名)の利用フロー：ステップ 1〇〇 → ステップ 2〇〇 → ステップ 3〇〇」 |
| **ブリーフで追加必須となる情報** | ① ステップ数 ② 各ステップの動詞 1 語 ③ アイコン主体か図解主体か |
| **使用上の注意** | — なし |

---

### C-5：結果・変化の証拠（Outcome Evidence）

| 項目 | 内容 |
|---|---|
| **役割の定義** | Before-After・実績数値・物理的変化を視覚的に提示する。Carney & Levin の Transformational に対応 |
| **典型配置セクション** | 実績セクション、ケーススタディ、成果報告 |
| **プロンプト変換時に規定する要素** | 被写体（変化対象） / 構図（左右 or 上下対比） / 照明（同条件で揃える） |
| **AI 生成適性** | △（実績写真の捏造リスクあり、原則は実例の実写） |
| **AI 向き不向きフラグ** | `PHOTO-PREFERRED` |
| **コピーとの接続パターン** | アンカリング＋数値による二重符号化 |
| **alt 記述ガイドライン** | 「(対象)のビフォーアフター：(変化前)→(変化後、数値があれば併記)」 |
| **ブリーフで追加必須となる情報** | ① Before の状態（具体的）② After の状態（具体的）③ 変化を裏付ける数値・出典 ④ 実例画像が確保可能か |
| **使用上の注意** | 実績の捏造は厳禁。数値・出典が裏付けられない Before-After は使用しない |

## 2. 軸 E：感情的機能（Emotional Function）— 何を「感じさせるか」

### E-1：課題共感（Problem Empathy）

| 項目 | 内容 |
|---|---|
| **役割の定義** | 「この悩み・困りごと、わかる」という共感を視覚化する。ターゲットの現在の不便・不満を画像で代弁 |
| **典型配置セクション** | 課題提示セクション（FV 直下）、ターゲット定義セクション |
| **プロンプト変換時に規定する要素** | 被写体（人物の表情・姿勢） / 照明（沈んだトーン） / ムード |
| **AI 生成適性** | ◯（人物の自然な負の感情表現は AI 改善中だが概ね到達可能） |
| **AI 向き不向きフラグ** | `AI-READY` |
| **コピーとの接続パターン** | リレー（コピーが「この悩み、ありませんか？」と問い、画像がその状況を見せる） |
| **alt 記述ガイドライン** | 「〇〇に困っている(ターゲット属性)の様子」 |
| **ブリーフで追加必須となる情報** | ① 描く課題（具体的に）② ターゲット属性（年代・性別・職業）③ 表情の度合い（軽い困り顔 / 深刻 / 倦怠）④ 後ろ姿 or 顔出しの選択 |
| **使用上の注意** | ネガティブ感情の連続表示は離脱を上げる。E-1 は LP 全体で 1〜2 箇所までに抑制し、必ず E-2（理想未来）とセットで配置すること |

---

### E-2：理想未来（Aspirational Future）

| 項目 | 内容 |
|---|---|
| **役割の定義** | 「使った後の自分・生活」を感情的にイメージさせる。Lifestyle Image の核心 |
| **典型配置セクション** | FV ヒーロー、ベネフィットセクション、CTA 直前 |
| **プロンプト変換時に規定する要素** | 被写体（人物の自然な表情） / 構図（人物の目線がコピー / CTA 方向） / 照明（明るい・希望的） / ムード |
| **AI 生成適性** | ◯ |
| **AI 向き不向きフラグ** | `AI-READY` |
| **コピーとの接続パターン** | アンカリング＋感情の二重符号化（コピーが文字で・画像が表情で同じ感情を表現） |
| **alt 記述ガイドライン** | 「(サービス) を使って〇〇を実現した(ターゲット)の様子」 |
| **ブリーフで追加必須となる情報** | ① 理想状態を 1 文で ② ターゲットが画像に自分を重ねられるか確認（属性近接性）③ 笑顔のレベル（自然 / 控えめ / はじける）④ 目線の方向（カメラ目線 / CTA 方向 / 自然な視線外し） |
| **使用上の注意** | **【重要原則】**：ストック写真的な過剰な笑顔は避ける。後ろ姿 / 手元 / シルエット / 自然な微笑など、ターゲットが自分を投影しやすい構図を選ぶ |

---

### E-3：信頼・安心（Trust & Authority）

| 項目 | 内容 |
|---|---|
| **役割の定義** | 本物の人・実績・権威・第三者証明を通じて安心感を与える。NNGroup の研究で「ストック写真の人物は系統的に無視される」ため、本物性が決定的 |
| **典型配置セクション** | 会社情報、代表者紹介、メディア掲載実績、認証マーク、資格・受賞 |
| **プロンプト変換時に規定する要素** | 被写体（実在人物 or 実績物） / 照明（プロフェッショナルなスタジオ照明 or 現場感） |
| **AI 生成適性** | ×（実在人物の顔は AI 生成不可・実物推奨） |
| **AI 向き不向きフラグ** | `PHOTO-PREFERRED` |
| **コピーとの接続パターン** | アンカリング（コピーが権威を主張、画像が実証） |
| **alt 記述ガイドライン** | 「(肩書) (氏名)の写真」「(媒体名)に掲載された記事」など、固有名詞を必ず含める |
| **ブリーフで追加必須となる情報** | ① 実在人物 / 実績物の特定 ② 写真素材の所在 ③ 撮影が必要な場合の手配状況 ④ AI で代替する場合は必ずシンヤさん承認 |
| **使用上の注意** | 実在人物の AI 生成は原則禁止。代替する場合は必ずシンヤさん承認を経る |

---

### E-4：親近感・共感（Affinity & Self-Identification）

| 項目 | 内容 |
|---|---|
| **役割の定義** | 「自分と似たような人が使っている」を視覚化し、自分ごと化を促進する。お客様の声・体験談に紐づく |
| **典型配置セクション** | お客様の声、体験談、ケーススタディ、UGC（ユーザー生成コンテンツ） |
| **プロンプト変換時に規定する要素** | 被写体（ターゲット属性に近い人物） / 構図（カジュアル・自然） / ムード（リアル感優先） |
| **AI 生成適性** | △（実ユーザー写真には及ばない。Dreamstime 研究で実写優位確認） |
| **AI 向き不向きフラグ** | `PHOTO-PREFERRED`（実ユーザー写真があれば必ず優先）／ `AI-READY`（実写が確保できない場合のみ） |
| **コピーとの接続パターン** | リレー（声明文＋人物画像で発話者像を補完） |
| **alt 記述ガイドライン** | 「(属性) (氏名 or イニシャル) さんの写真」 |
| **ブリーフで追加必須となる情報** | ① 体験者属性（年代 / 性別 / 職業）② 実写の確保可否 ③ AI 代替時は「ターゲット属性に近いがフェイクと明示しない構図」を回避（UGC 的な自然さで） |
| **使用上の注意** | AI 代替時、実在体験者を装う構図（特定の氏名・職業を併記する等）は厳禁。属性のみの記述に留める |

## 3. 軸 B：行動的機能（Behavioral Function）— 何を「させるか」

### B-1：視線誘導（Gaze Direction）

| 項目 | 内容 |
|---|---|
| **役割の定義** | ユーザーの視線を CTA・見出し・重要要素に向ける。人物の目線方向 / 矢印的構図 / 指差しなどを使う |
| **典型配置セクション** | CTA 近傍、価格表近傍、申込フォーム周辺 |
| **プロンプト変換時に規定する要素** | 構図（視線ベクトルの方向） / 被写体（人物 or 矢印的オブジェクト） |
| **AI 生成適性** | ◯ |
| **AI 向き不向きフラグ** | `AI-READY` |
| **コピーとの接続パターン** | リレー（画像の視線方向と CTA テキストが連動） |
| **alt 記述ガイドライン** | Primary が他にある場合は装飾扱いで `alt=""`、単独で使う場合は「CTA に視線を向ける(被写体)」 |
| **ブリーフで追加必須となる情報** | ① 視線を向けたい先（CTA / 見出し / 数値）② 画像内の人物の位置と CTA の位置関係 ③ Z 字パターン or F 字パターンのどの交点に置くか |
| **使用上の注意** | — なし |

---

### B-2：スクロール促進（Scroll Cue）

| 項目 | 内容 |
|---|---|
| **役割の定義** | 「続きを見たい」と思わせる引き。セクション境界で次への期待を作る |
| **典型配置セクション** | セクション境界、FV 下部、章の終わり |
| **プロンプト変換時に規定する要素** | 構図（縦方向のリーディングライン） / ムード（情報の密度を予感させる） |
| **AI 生成適性** | ◯ |
| **AI 向き不向きフラグ** | `AI-READY` |
| **コピーとの接続パターン** | アンカリング |
| **alt 記述ガイドライン** | 装飾扱いの場合は `alt=""`、内容を持つ場合は具体描写 |
| **ブリーフで追加必須となる情報** | ① 次セクションのテーマ ② 縦方向の余白演出の度合い |
| **使用上の注意** | — なし |

---

### B-3：離脱防止（Bounce Prevention / Self-Relevance Cue）

| 項目 | 内容 |
|---|---|
| **役割の定義** | 「自分には関係ない」という瞬時判断を防ぐ。ターゲット属性を視覚的に提示し、「これは自分向けだ」と認識させる |
| **典型配置セクション** | FV、価格提示前、ターゲット定義セクション |
| **プロンプト変換時に規定する要素** | 被写体（ターゲット属性が一目でわかる人物） / 構図（ターゲット属性の認識速度を最大化） |
| **AI 生成適性** | ◯ |
| **AI 向き不向きフラグ** | `AI-READY`（属性を表現するだけなら）／ `PHOTO-PREFERRED`（具体的なペルソナ性を出したい場合） |
| **コピーとの接続パターン** | リレー（コピーで明示 + 画像で再確認の二重符号化） |
| **alt 記述ガイドライン** | 「(具体的な属性) の(性別・年代)」 |
| **ブリーフで追加必須となる情報** | ① ターゲットの最重要属性（年代 / 性別 / 職業 / 業種）② 「自分とは違う」と即座に判断されるリスク要因の確認 ③ 多様性を持たせる必要があるか |
| **使用上の注意** | — なし |

## 4. AI 向き不向きフラグの運用補足

### 4.1 フラグ別の運用フロー

| フラグ | 運用フロー |
|---|---|
| `AI-READY` | ルナがプロンプト設計 → Gemini で生成 → 必要に応じて修正生成 |
| `PHOTO-PREFERRED` | 実写素材の確保を最優先。確保不可の場合のみ AI 生成（シンヤさん承認必須） |
| `ILLUSTRATIVE-OK` | ルナで Illustration / Infographic ドメインモード指定 → Gemini 生成 → Figma で日本語テキスト後載せ |

### 4.2 フラグの優先順位

同じタイプに複数のフラグが付く場合（例：C-2 は `AI-READY` または `PHOTO-PREFERRED`）は、案件ごとにシンヤさん判断。原則として：

- **クライアント案件の本番納品物** → `PHOTO-PREFERRED` を優先
- **社内検討・たたき台 / 試作** → `AI-READY` を優先
- **コスト制約・短納期** → `AI-READY` を許容（ただし Uncanny Valley リスクの確認は必須）

## 5. 軸間の関係（Phase 1 注意点）

### 5.1 1 画像 = 1 Primary + 最大 2 Secondary の原則

軸 C / E / B は**直交していない**。1 枚の画像が C / E / B のそれぞれから 1 タイプずつ機能を担うことが多い。例：

- LP ヒーロー画像（典型） = Primary `E-2`（理想未来） + Secondary `C-1`（世界観）+ Secondary `B-1`（視線誘導）
- ケーススタディ写真 = Primary `E-3`（信頼）+ Secondary `C-5`（結果）

**ただし Primary は必ず 1 つに絞ること**。これにより、プロンプト設計で「最も強調すべき要素」が一意に決まる。

### 5.2 リナの MECE 指摘（Phase 2 で対応予定）

- 「C-1 世界観設定」と「E-2 理想未来」の境界が文脈次第で重なる
- 「B-3 離脱防止」と「E-4 親近感」が機能的に近い
- 軸 B（行動的機能）はプロンプト変換要素が「構図」に偏り、独立軸として弱い可能性あり

→ Phase 2 で運用ログを取った上で、軸再編・統合・新軸追加を検討する。

## 7. alt 記述の共通ルール

- `alt=""`（装飾扱い）を使えるのは、Primary が「視覚的な情報を持たない純粋装飾」と判定される場合のみ。**本辞書では C-1 のうち「他の LP に転用可能なケース」のみ装飾扱い**
- それ以外はすべて、画像の意味を 1 文以内で記述する
- 固有名詞（製品名・人名・媒体名）は必ず含める（SEO・アクセシビリティの両面で重要）
- 「(写真)」「(画像)」のような冗長な接尾辞は不要
