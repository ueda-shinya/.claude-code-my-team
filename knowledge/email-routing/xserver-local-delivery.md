# Xサーバー ローカル配送問題の診断と解決

Xサーバー上で運用するドメインから、**同じXサーバーのドメイン設定に登録された別ドメイン**のメールアドレス宛にメールを送信したとき、外部MXがGoogle Workspaceを向いているにもかかわらず `user unknown. Invalid user specified.` でバウンスする問題。

共用サーバーの「ローカル配送」仕様に起因する再発性の高い問題。Xサーバー以外（さくら・ConoHa・WADAX等）でも同種の挙動あり。

---

## 適用範囲（先に必ず確認）

- 本ナレッジの手順は**Xサーバーで検証済み**。
- さくらインターネット・ConoHa WING・WADAX等、**他社の共用サーバーで同症状を確認した場合は、各社公式の「Google Workspace併用マニュアル」を参照**し、本手順を安易に流用しないこと。
- 「ローカル配送」という概念自体は共用サーバー全般にあるが、ダミーアカウント・転送・中継ドメインの具体的な指定方法は各社で異なる。

---

## 作業前の前提条件（クライアント案件では特に重要）

本手順は **Xサーバーのサーバーパネル操作** と **Google Workspace 管理コンソール操作** の**両方**が必要。クライアント案件では両方の管理権限保有者が異なるケースが多いため、作業着手前に以下を必ず確認する:

- [ ] **Xサーバー サーバーパネル** の管理権限（または操作立会い）を取得済み
- [ ] **Google Workspace 管理コンソール** の管理権限（または操作立会い）を取得済み
- [ ] Google Workspace側のルーティング設定変更は**テナント全体に影響**するため、**必ずクライアント承認を取得してから実施**（特に Split Delivery 設定を新規投入する場合）
- [ ] 同一テナントで他のドメインを運用している場合、既存のルーティングルールに影響が出ないか事前確認

※ 権限・承認が片方でも未取得の状態で着手しないこと。片側だけ進めると整合が取れなくなり、送受信が断絶する期間が発生するリスクがある。

---

## 症状パターン（このナレッジが当てはまるかの判定）

以下がすべて当てはまるときに該当する:

- 送信元: Xサーバー上のドメイン（例: `info@company.bakuchis.com`）
- 宛先: Google Workspace等、外部MXで運用しているドメイン（例: `info@bakuchis.com`）
- 宛先ドメインが**同じXサーバーのドメイン設定に登録されている**
- 外部からはその宛先に正常にメールが届く（MXは機能している）
- 送信元から**他のGmail等**には正常に届く（SMTP自体は問題ない）
- 同サーバー内の同ドメイン宛だけがバウンスする

バウンスメールの特徴:

- `Reporting-MTA`: Xサーバーのホスト名（`sv*.xbiz.ne.jp`）
- `Diagnostic-Code`: `user unknown` / `Invalid user specified.`

---

## 原因（確定）

Xサーバーの**ローカル配送仕様**:

- Xサーバーのドメイン設定に登録されているドメイン宛のメールは、送信時に**外部MXレコードを参照せず、サーバー内部で配送を試みる**
- 外部MXがGoogle Workspace等を向いていても、**Xサーバー内部からの送信には適用されない**
- サーバー内部にそのメールアカウントが存在しなければ `user unknown` で弾かれる

Xサーバー固有の表現として「ローカル配送」と呼ばれるが、さくらインターネット・ConoHa WING・WADAX等の共用サーバー全般に同種の仕様がある。

---

## 診断フロー（最速で切り分けるための手順）

### Step 1. バウンスメールのヘッダを確認

`Reporting-MTA` がXサーバーのホスト名（`sv*.xbiz.ne.jp`）か確認する。

- Xサーバーのホスト名 → **Xサーバー内部で止まっている**（次へ）
- それ以外 → 本ナレッジの対象外（Google側・経路途中の問題を疑う）

### Step 2. エラーコードを確認

`Diagnostic-Code` が `user unknown` / `Invalid user specified.` 相当か。

- 該当 → **ローカル配送疑いが濃厚**（次へ）
- 該当しない → DNS・SPF・DKIM・スパム判定など別要因を疑う

### Step 3. ドメイン設定への登録有無を確認

Xサーバーのサーバーパネル → 「ドメイン設定」を開き、**宛先ドメイン**が登録されているか確認する。

- 登録されている → **ローカル配送で確定**（解決策セクションへ）
- 登録されていない → 別要因（MX・SPFレコード・受信側のスパム判定等を疑う）

> 注: メールアカウント設定画面のドメイン切替ドロップダウンに出てこないと「登録されていない」と誤認しやすい。ドロップダウンは初期表示でデフォルトドメインしか出ない場合があるため、必ず明示的に選択し直して確認する。

**前提条件（本診断フローの適用範囲）:**

- **送信元もXサーバー上の同一サーバーアカウントに属するドメイン**であること（宛先ドメインと同じサーバーパネル内に登録されている状態）
- 確認済みの送信経路は **Xサーバー上のメールクライアント／SMTP AUTH 経由の社内送信**
- **外部SMTPサーバー経由**（例: 他社メール基盤や SendGrid 等のリレー経由）の送信で同症状が出ている場合は、本ナレッジの対象外。別途検証が必要

---

## 解決策（Xサーバー公式「ステップ5: 強制回収」手順）

Xサーバー公式の「Google Workspace併用マニュアル」で提示されている手順が有効。サーバー内部に留まろうとするメールを、Google Workspaceのテスト用ドメイン経由で強制的にGoogle側へ中継する。

### 手順

1. **Xサーバーのサーバーパネル → メールアカウント設定**
   - 対象ドメイン（例: `bakuchis.com`）をドロップダウンで選択
2. **Google運用中のアドレスと同名のメールアカウントをダミーで作成**
   - 例: `info@bakuchis.com`
   - パスワードは任意（Xサーバー側での受信用途はない）
3. **サーバーパネル → メール転送**
   - 作成したアドレスに対して転送設定を追加
   - 転送先: `<local-part>@<domain>.test-google-a.com`
     - 例: `info@bakuchis.com.test-google-a.com`
   - `.test-google-a.com` は Google Workspace が自動的に用意しているテスト用ドメインで、該当ドメインのGoogle Workspace側へ中継される
   - 「メールボックスに残す」は**「残さない」推奨**（Xサーバー側に二重保管されるため）
4. **Google Workspace側の前提設定を確認**
   - Split Delivery（ルーティングルールで「認識されない / キャッチオール」にチェック）が設定されていること
   - 設定されていない場合、Google Workspace管理コンソール → アプリ → Google Workspace → Gmail → ルーティング で設定

> **Split Delivery 未設定のまま Step 3 の転送を実行した場合の予想挙動:**
> `.test-google-a.com` 宛のメールが **Google側で受信拒否**され、Xサーバーのダミーアカウントに**滞留する可能性**がある（転送ループや再送エラーの要因）。Step 3 を実施する前に、**Google Workspace 管理コンソールで Split Delivery（キャッチオール）が有効になっていることを必ず確認**してから進めること。

### 動作確認

解決策適用後、以下のテストで正常化を確認する:

1. Xサーバー上の送信元アドレス（例: `info@company.bakuchis.com`）から、Google Workspace 運用中の宛先（例: `info@bakuchis.com`）宛にテストメールを送信
2. **Google Workspace 側の Gmail 受信箱に着信すること**を確認
3. `user unknown` / `Invalid user specified.` のバウンスが返らないことを確認
4. 念のため、Xサーバー側のダミーアカウントにメールが滞留していないか（Webメール等で）確認
5. 複数の社内送信元がある場合は、代表的な送信元から1通ずつテストして経路全体の正常性を確認

### 動作イメージ

```
[Xサーバー上の送信元]
  └→ info@bakuchis.com 宛に送信
       └→ Xサーバー内部で「ローカル配送」対象と判定
            └→ Xサーバー上のダミーアカウントに配送
                 └→ 転送設定により info@bakuchis.com.test-google-a.com へ転送
                      └→ Google Workspace の Split Delivery ルールで受信
                           └→ 実際の info@bakuchis.com のメールボックスに配送
```

## 絶対にやってはいけないこと

### 1. 宛先ドメインのMXレコードをXサーバー向けに書き換えない

- 現在Google Workspaceを向いているMXを触ると、**外部からの受信が全停止**する
- ローカル配送問題は「送信側（Xサーバー内部）の配送経路」の話であり、MXレコード（受信経路）とは別問題
- 症状が似ているため混同しやすいが、MXは触らない

### 2. Xサーバーのドメイン設定から宛先ドメインを削除しない

- 他の機能（メール・SSL・サブドメイン運用等）に影響する可能性がある
- ドメイン登録を残したまま、上記「強制回収」手順で解決する

### 3. SPF・DKIMレコードの書き換えを先行させない

- ローカル配送は内部処理のため、SPF/DKIMでは解決しない
- 本質的な原因（内部配送）を直さずDNS側をいじると、外部からの受信認証に悪影響が出る

---

## 再発時の対応チェックリスト

クライアント案件で同種問題に遭遇したら、以下の順で即時対応:

- [ ] バウンスメールの `Reporting-MTA` がXサーバー系ホスト名か確認
- [ ] `Diagnostic-Code` が `user unknown` / `Invalid user specified.` か確認
- [ ] Xサーバー管理画面のドメイン設定に宛先ドメインが登録されているか確認
- [ ] 登録されていれば「強制回収」手順を適用（ダミーアカウント作成 → 転送設定 → Split Delivery確認）
- [ ] 解決後、クライアントに「このサーバーのドメイン設定に登録しているGoogle Workspaceドメインはすべて同じ対処が必要」と共有
