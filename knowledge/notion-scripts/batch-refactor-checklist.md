## 成功要素（6つ）

依頼文を書く時に必ず含めるべき要素。**要素①〜④は High 発生リスクを下げ、要素⑤⑥はレビュー差戻し時のロスを下げる。**

### ① 基準となる流儀の原文引用
抽象的な説明ではなく、基準ファイルから該当コードを行番号付きで原文引用する。シュウが「再現すべき形」を誤解しないようにする。

### ② 対象ファイル一覧（行番号付き）
`scripts/notion-*.py L<start>-<end>` で範囲を特定。漏れ・過剰改修を防ぐ。

### ③ 対象外ファイルの明示（除外理由付き）
「特殊ロジックのため」「対応する関数がないため」等の理由付きで除外を明示。シュウが過剰な「親切」で範囲を広げるのを防ぐ。

### ④ 実装チェックリスト（保存すべきロジックの列挙）
- 流儀統一の対象範囲
- 保存すべき周辺ロジック（ページ本文取得・数値フォーマット・リスト結合等）
- 例外扱いにするフィールド（内部 dict キー・None 代替表記）

### ⑤ 動作確認範囲の明示
- シュウが担保する範囲（import・test・self-check）
- シンヤさんが本番で確認する範囲（実際の Notion API 呼び出し）

この切り分けがないと、シュウが「自分で本番テストしなきゃ」と迷走したり、逆に「self-check もしない」で出してきたりする。

### ⑥ Quality Gate 順守の明示
「コードレビュー前にシンヤさんに実装完了報告しないこと」を毎回書く。シュウ側の迷いを消す。

## 禁忌（High を引き起こすアンチパターン）

### A. 「基準ファイルを参考に」だけで原文引用しない
→ シュウが解釈ズレを起こし、部分的に流儀が乱れた実装が上がる。
（Phase 1-A バッチ4 で発生）

### B. 対象ファイルを「notion-*.py 全部」等の集合名詞で指定
→ 対象外扱いすべき特殊ファイル（例: cmd_show_seq の radar）まで巻き込まれる。
必ず個別のファイル名 + 行番号で列挙する。

### C. 保存すべき周辺ロジックを列挙しない
→ 「統一」の名の下に、数値フォーマット・リスト結合・例外扱いの個別 print まで for ループに押し込まれ、機能が壊れる。
チェックリストに「XXは現状維持」を明記する。

### D. `test_notion_schema.py` の期待結果を書かない
→ シュウが「テストをどう判定すればいいか」分からず、色々な実行方法を試して時間を無駄にする。
毎回「Phase 1-A と同じ 7 ok + 1 skip」と書く（SnsDB アクティブ化後は 8 ok + 0 skip に更新）。

※ SnsDB アクティブ化時は本ファイルの「Phase 1-A と同じ 7 ok + 1 skip」の3箇所（テンプレート内2箇所 + 本禁忌節1箇所）を同時更新すること（アスカの責務）。

### E. 先行バッチのコミット SHA を書かない
→ シュウが「前回どう書いたか」を探しに git log を漁る時間が発生。
必ず直近の成功バッチ SHA を 1-2 個書く。
