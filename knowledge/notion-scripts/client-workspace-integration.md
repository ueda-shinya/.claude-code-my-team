# クライアント別 Notion ワークスペース接続パターン

## 概要

シンヤさん本体のNotionとクライアント様のNotionは、**別ワークスペース・別インテグレーション・別トークン**で管理する。

## 理由

- Notionインテグレーションはワークスペース単位で発行される
- 1つのトークンで複数ワークスペースにまたがってアクセスすることはできない
- クライアント様のワークスペースに招待されても、そのワークスペース内で発行されたインテグレーションを別途「ページに接続」する必要がある

## 命名規則（環境変数）

`.env` の命名：

```
NOTION_<CLIENT>_TOKEN           # クライアント専用インテグレーショントークン
NOTION_<CLIENT>_ROOT_PAGE_ID    # ルートページID
NOTION_<CLIENT>_<DB_NAME>_DB_ID # 子DB ID（用途別に複数可）
```

### クライアント識別子の命名ルール

- **英字のみ・4〜8文字推奨**（長い場合は意味の通る短縮形を使う）
- ハイフン・アンダースコアはクライアント識別子の中では使わない（環境変数のセクション区切りとして `_` を使うため）
- ディレクトリ名（`clients/<name>/`）と環境変数識別子は**一致させない**ことがある（ディレクトリは `inada-ryota`、環境変数は `INADA` 等）

### クライアント識別子 対応表

`.env` の冒頭（または「Notion: クライアント」セクションの冒頭）に対応表をコメントで記載：

```
# Notion: クライアント別ワークスペース
# ------------------------------------
# 対応表（CLIENT識別子 → clients/ ディレクトリ）:
#   LANDO        → clients/lando-planning/
#   INADA        → clients/inada-ryota/
#   USSAIJO      → clients/us-saijo/
#   LINNOA       → clients/linnoa/
#   YOSHINO      → clients/yoshino-mayumi/
# ------------------------------------
```

例（ランドプランニング様）：
```
NOTION_LANDO_TOKEN=ntn_...
NOTION_LANDO_ROOT_PAGE_ID=...
NOTION_LANDO_CUSTOMER_DB_ID=...
NOTION_LANDO_COMPANY_DB_ID=...
```

## 接続手順

### ① クライアント様に依頼すること

1. Notionインテグレーションを発行（https://www.notion.so/profile/integrations）
   - 関連ワークスペースにクライアント様のワークスペースを指定
2. アクセスさせたい**親ページ**を開く
3. 右上「•••」→「コネクト」→ 発行したインテグレーションを追加
4. **親ページのURLとトークンをシンヤさんに共有**

### ② アスカがやること

1. curlで読み取りテスト：`GET /v1/pages/<page_id>`
2. 子ブロック一覧取得で子DBを発見：`GET /v1/blocks/<page_id>/children`
3. 各DBのプロパティ構造を確認：`GET /v1/databases/<db_id>`
4. 書き込みテスト：`POST /v1/pages`（parent: page_id）→ 成功したら即 `PATCH` で `archived: true` にする
   - ※ Notion API には物理削除エンドポイントが存在しないため、`archived: true` による論理削除（アーカイブ領域への移動）が上限
   - 完全削除が必要な場合は、**クライアント側にアーカイブページの手動削除を依頼**する（通常はテストページであればアーカイブで十分）
5. `.env` に環境変数を追加
6. `clients/<client>/README.md` にNotion接続情報セクションを追記

## 運用ルール

### 対象ワークスペースの明示ルール（誤爆防止）

依頼文からアスカが判定する際の**離散基準**：

| 判定条件 | アスカの行動 |
|---|---|
| クライアント名が依頼文に明示されている（「ランドプランニングの〜」「ランド様の〜」等） | そのクライアントのNotionで実行 |
| 直近3ターン以内にクライアント名が出ており、文脈が明らかに継続している | そのクライアントのNotionで実行（ただし書き込み操作は1度確認） |
| シンヤさん本体と明示（「アスカ室の」「本体の」「私の」「案件DBに」「CRMに」等、本体固有の用語） | `NOTION_API_TOKEN` で実行 |
| クライアント名が無く、「顧客DB」「顧客管理」等のクライアント・本体どちらにも存在しうる単語のみ | **必ず確認**（「シンヤさん本体のNotionですか？それともクライアントのNotionですか？」） |
| 完全に曖昧 | 必ず確認 |

既存 `notion-tasks.py` 等のスクリプト実行時も同じ判定を適用する。

### その他

- 大量データ操作（100件以上の一括更新等）は事前相談
- 既存スクリプト（`notion-tasks.py` 等）はシンヤさん本体用。クライアント専用スクリプトが必要になったら `notion-<client>-*.py` として別途作成

## 冪等性原則との住み分け

`knowledge/notion-scripts/design-principles.md` の「DB作成コマンドの冪等性必須」は**シンヤさん本体のNotionのみ**に適用される。クライアント様のNotionではDBはクライアント側が手動で作成・管理するため、アスカ側でDB作成スクリプトを書かない（書く場合も、対象DBの存在確認のみで作成処理は含めない）。

## README記載テンプレート

`clients/<client>/README.md` に以下セクションを追加：

```markdown
## Notion 接続情報

- **ワークスペース**: <クライアント名>様（シンヤさん本体とは別ワークスペース）
- **ルートページ**: <ページ名>（<URL>）
- **子DB**:
  - <DB名1>（<プロパティ数>プロパティ、<用途>）
  - <DB名2>（...）
- **環境変数**（`~/.claude/.env`）:
  - `NOTION_<CLIENT>_TOKEN` — 専用インテグレーショントークン
  - `NOTION_<CLIENT>_ROOT_PAGE_ID` — ルートページID
  - `NOTION_<CLIENT>_<NAME>_DB_ID` — ...
- **権限**: 読み書き両方（YYYY-MM-DD 接続テスト済み）
- **同期**: 手動のみ / 自動同期あり（頻度）
```
