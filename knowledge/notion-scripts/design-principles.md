# Notion スクリプト設計原則

## 1. DB作成コマンドには冪等性を必須とする

### ルール
`--create-db` のようなDB作成系コマンドは、何度実行しても同じ結果になること（冪等性）を保証する設計にする。

### 必須チェック（2段階）

1. **ID存在チェック**: `.env` に DB ID が既にある場合、API で存在を確認し、存在すれば作成を中断する
2. **同名検索チェック**: ID がない or 404 の場合、親ページ配下を同名で検索し、既存DBがあれば案内する

### オプション
- `--force`: 既存DBがあっても強制的に新規作成する（意図的な再作成用）
- `--reuse`: 検索で見つかった既存DBの ID を `.env` に設定する（PC間共有が漏れた場合のリカバリ用）

### 背景
- `.env` は gitignore 対象であり、DB作成結果が複数 PC 間で自動共有されない
- 「一度だけ実行する」という暗黙の前提は、複数PC環境では破綻する
- 実際に Mac/Win 両方で `--create-db` が実行され、重複 DB が発生したインシデントあり（2026-04-11）

### 対象スクリプト
- `~/.claude/scripts/notion-kaizen.py`
- `~/.claude/scripts/notion-tasks.py`
- `~/.claude/scripts/notion-projects.py`
- 今後追加される `notion-*.py` 系スクリプト全て

## 2. 複数PC環境での .env 依存に注意する

### ルール
`.env` にしか保存されない情報（DB ID等）に依存する機能は、もう一方のPCでその情報がない場合の振る舞いを必ず設計する。

### 具体策
- 情報がない場合のフォールバック（検索して案内する等）を実装する
- 将来的には `config/notion-db-ids.json` 等で git 管理に移行することも検討（P4タスクとして積み済み）
