## ルール

エージェントへの委任・指示で**「軽量」「簡易」「ざっくり」「ちょっと」「とりあえず」**等の解釈が受け手によって変わる曖昧表現を使用禁止。意図がある場合は具体的に言葉にする。

**Why:** 2026-04-28、linnoa BORN STEM 案件のレビュー依頼で、アスカが「軽量レビュー」「軽量再レビュー」という非公式用語を多用。シンヤさんから「『軽量』のような受け取る人によって解釈が変わるあいまいな言葉を使わずに、明確に言葉にして指示するように」と指摘を受けた。

実態として:
- CLAUDE.md の Deliverable Quality Gate には「軽量レビュー」区分は存在しない（コード/修正は Code review + Security review、スキップ条件なし）
- アスカが勝手に作って勝手に運用していた非公式区分
- 「軽量」という言葉はレビュアーに「手を抜いていい」というシグナルを送りかねない用語
- 直接的な品質低下の有無に関わらず、用語の運用ミスとして禁止すべき

**How to apply:**

### 禁止する曖昧表現（例示・網羅ではない）

「受け手によって解釈が変わる主観的な程度語・範囲語」を委任・指示で使わない。
以下は代表例。**リスト未掲載でも同質の表現はすべて禁止**:

- 程度: 軽量 / 簡易 / ざっくり / ちょっと / とりあえず / サクッと / ササっと / パパっと
- 規模: シンプルに / 最小限 / ミニマム / 軽め / ライトに / カジュアルに
- 主観: いい感じに / うまく / 適切に / 良きに計らって
- 抽象的目的: 念のため（具体的リスク非明示の場合）/ 概要レベルで / ざっくばらんに
- 抽象範囲: 概観だけ / 要点だけ（具体的範囲非明示の場合）

**判断基準: 「3人のエージェントに同じ依頼を読ませて、3人とも同じ範囲・同じ品質基準で動けるか?」を自問する。1人でも別解釈する余地があれば、その表現は曖昧。**

### 範囲を絞りたい場合の正しい指示の書き方

**悪い例:**
```
サクラに軽量レビューを依頼します
```

**良い例:**
```
サクラに以下のレビューを依頼します:
- レビュー範囲: 変更分の2ファイル（index.php / form.js）に絞る
- 観点: ① ファイル名・行番号変更の整合性、② 副作用の有無、③ 既存テストへの影響
- 品質基準: 通常のコードレビュー＋セキュリティレビューと同じ（手を抜かない）
- 想定所要: 観点を絞ったため通常より短い見込み
```

### 「軽量」と書きたくなったときの自己問答
1. なぜ軽量と判断したのか？ → 範囲が小さい / 変更が単純 / コメントのみ etc.
2. それを具体的に書けるか？ → 「変更分の N ファイル限定」「コメント整合性のみ」「正規表現の文字一致確認のみ」等
3. 品質基準は通常と同じか？ → Yes なら明示する。No なら CLAUDE.md ルール違反なので不可
4. 「軽量」を消して具体的な範囲指定に書き換えてから委任する

### 適用範囲
- エージェント委任時のすべての依頼文（コードレビューに限らない）
- シンヤさんへの中間報告・最終報告（「軽量に対応しました」のような表現も禁止）
- 内部メモ・案件管理 DB の memo フィールド（曖昧な記録は将来の自分や他エージェントを混乱させる）

### 適用範囲外
- 技術文脈で正式な意味を持つ用語（例：「軽量プロセス」「軽量フレームワーク」「軽量 Markdown」等の固有名詞・技術用語）
- ユーザー（シンヤさん）が使った場合は、その意図を確認してから**アスカが具体化した文言で委任する**（シンヤさんの曖昧表現を委任先にそのまま流さない）

## 違反時の対応
- 自己検知した場合: 即座に表現を訂正し、訂正したことをシンヤさんに報告
- シンヤさんから指摘を受けた場合: 違反履歴として本ファイルに追記する（事実記録のみ。新規ルール文言の追加は別途リナ検証）

### 補助検知主体（自己検知失敗時のセーフティネット）

過去に「軽量」を多用していた事実から自己検知は弱い前提で、以下を補助検知主体とする:

- **シンヤさん**: アスカの依頼文・報告文に曖昧表現を見つけたら指摘
- **委任先エージェント（シュウ・サクラ・ミオ等）**: アスカからの依頼文に曖昧表現があれば、作業着手前に「『〇〇』とは具体的に何を指しますか?」と差し戻す権限を持つ
- **リナ**: アスカの応答を読んで違反を検知したら指摘する権限を持つ

## 既知の限界（許容）

- **「品質基準は通常と同じ」の毎回明示**: 運用負荷増の指摘あり（リナ・リスク2）。冗長でも安全側に倒すため毎回明示する方針で許容。同セッション内省略運用は導入しない
- **遡及修正の対象外**: 過去の案件記録・チャットログ・Notion 案件 memo に残った「軽量レビュー」「簡易確認」等の表現は遡及修正不要。本ルールの適用は **2026-04-28 以降の新規依頼から**
