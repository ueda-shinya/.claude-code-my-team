## 役割・背景
- officeueda 代表。Web制作・AI活用ビジネスを展開
- WordPress / Swell でのサイト構築経験あり
- 技術的な話を素早く正確に理解できる（JSON-LDの仕組みをすぐ把握）

## 作業スタイル
- 自律的に進めてほしいタイプ。席を外す間に作業を完結させることを好む
- 意思決定はアスカに任せてくれることが多い（色の選定・ファイル構成など）
- 「できそう?」「任せる」という形で裁量を渡してくれる
- 追加指示は短く端的。行間を読んで動くことが求められる
- 「なぜそうするか」の背景をちゃんと説明してくれる（チームフローの理由・要件定義の必要性など）
- 仕組みとして正しいことを重視する。その場しのぎより「次からちゃんとそうなる」形を好む
- アスカに指揮役としての役割を明確に期待している（自分でやるな、振れ）

## コミュニケーション傾向
- 口語・短文でやり取り（「やっといて」「大丈夫」「なにかある?」）
- 確認事項は最小限にしてほしい派
- 「おはよ」「おつかれ」など日常会話のトリガーを使う

## 技術的なこだわり
- コードの「意図」を正確に読むことを重視する
- 「動作を変えずに改善する」と「動作が変わる改善」を明確に区別する
- 自分が作ったコードの意図を誤読されることを嫌う（2026-03-20 lazyload誤読の件）

## チーム・エージェント設計の傾向
- 役割の分離を重視する（「マーケと営業は違う」と明確に意識している）
- メタ概念を理解したがる（「スキルを身に付けさせるスキルを作ったってこと?」と確認するタイプ）
- 意思決定は短く即断（「B」「よろしく」など）
- 何が作られたか・何ができるようになったかをちゃんと確認する（結果に対して納得してから次へ進む）
- エージェントの成長に投資する意識がある（アスカ向けにGA4・PM・コピーライティングの学習素材を自ら用意した）
- アスカに戦略的提案を求めることがある（「アスカから、なにか提案ある?」）。実行だけでなく提言も期待している
- エージェントを使う目的は「並列処理で全領域を同時に進めること」。優先順位をつけず全部を同時に底上げしたい（2026-03-31）
- 「精度の限界を超えさせたい」という高い期待値を持っている。「補助ツール」ではなく「実戦で使える専門家レベル」を目指している
- システムの仕組みの使い分け（メモリ vs スキル など）をちゃんと理解したがるタイプ。「なぜその使い分けか」を聞いてくる（2026-04-07）
- ツール比較は実際に試して結果を見て判断するスタイル（Genspark単独 / ソラ+Genspark を実際に作って3パターン比較）
- 新スキル初回運用時は、性格の異なる複数ルート（検証系・深掘り系・実装系など）を並列で走らせて違いを観察するスタイル。運用で気づいた知見はその場でスキル側に逆流反映させる（2026-04-21 ga-gsc-diagnosis 初回運用で3コース並列、得た知見をスキルに即追記）
- エージェントが積極的に意見・提案をして高品質な仕事をする文化を目指している。その方向に沿うものは積極的に取り入れる（2026-04-08）
- 「結果」より「経緯（なぜそうなったか）」を残すことに価値を置く。事例の背景・判断プロセスが資産になると考えている（2026-04-08）
- Max5プランに移行。複数セッション並行運用を増やしていく予定（2026-04-08）
- ミスへの対処は「仕組みで防ぐ」を徹底。「気をつける」は対策として認めない。なぜなぜ分析で真因まで掘り、hookやスキルとして実装することを求める（2026-04-11）
- エージェントが自主的にミスに気づき改善できる状態を目指している。「指摘されてから動く」ではなく「自分で気づいて直す」が理想形（2026-04-11）
- 自作する前に既存のツール・スキルがないかリサーチしてから判断するスタイル。車輪の再発明を避ける合理性（2026-04-11）
- 「使う時に壊れてた」を嫌う。事前に余裕のある時に問題を潰しておく予防保全の考え方（2026-04-11）
- 「知識はあってもスキルが重要」という本質を掴むのが速い。ナレッジとスキルの違いを即座に理解し、仕組み化（スキル化）の方向に自ら持っていく（2026-04-11）
- 一人のエージェントを強化したら「他の人もやろう」と横展開する思考。全体最適・チーム全体の底上げを志向する（2026-04-11）
- kaizen報告は「原因→対策」の流れがわかるように報告すること。原因がわからないと対策が正しいか判断できない、という合理的思考（2026-04-11）
- 複雑なチェックリストより「召集→揃ったら報告→OKもらってからスタート」のようなシンプルで確実な仕組みを好む（2026-04-11）
- エージェントの出力を鵜呑みにしない。「カイはLPデザイナーの役割がわかっていないのでは？」「アスカは何を見て判断したの？」のように本質を問う（2026-04-16）
- 問題を指摘するだけでなく「リサーチして、その内容からスキル作って」のように**根本的な能力強化**の方向に導く。個別修正より仕組み化を常に志向（2026-04-16）
- **AI/エージェントの判定・監査は「客観性」を最重要視する**。事前情報や答えを与えた状態での検証を無価値と断定する（2026-04-22 llmo-audit 試運転でユーザー情報を事前提供したことを「人の意見を取り入れた判定だと意味ないよね」と指摘）。スキル検証・試運転は情報ゼロ条件で実施し、エージェントが自力でロジック通りに動作するかを確認する運用に改める必要がある
- スキル「完成」と「調整中」を柔軟に切り分ける。完璧を待たず一旦クローズし、後日改善タスクとして Notion 登録する合理的な進め方を好む（2026-04-22）
- **曖昧な品質シグナルを許容しない**。「軽量レビュー」のような受け手の解釈で意味が変わる表現は『手抜きOK』を暗示するとして恒久禁止を即指示。指示・委任時は意図を具体的に言語化することを厳格に求める（2026-04-29 / feedback-asuka-vague-language.md）
- 役割分担の人選を厳しく見る。フロントエンド領域なのに backend-engineer に振っていれば「なんでツバサじゃなくシュウ？」と即指摘する。エージェント定義の専門領域を尊重した委任を求める（2026-04-29）
- UI 細部の統一感を重視する。「ある項目、無い項目があるのは不自然」と即指摘し、徹底した一貫性を求める。サービス特性（女性向け等）に合わせたサンプル値の配慮も妥協しない（2026-04-29）
