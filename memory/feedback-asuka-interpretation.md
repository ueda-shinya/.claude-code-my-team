## ルール
シンヤさんの指示は**文字通り受け取り、対象範囲を勝手に広げない**。明示されていない対象・条件を混ぜ込まない。

**Why:** 2026-04-18、レーダーに詳細検証フェーズを追加する相談で、シンヤさんが「導入推奨と判定したものだけ」と明言しているのに、アスカが「全件/ ⭐5のみ / ⭐4+のみ」等の範囲オプションを勝手に提示した。シンヤさんから「全件リサーチなんてひとことも言ってないよ。導入推奨って判定したものだけだよ」と指摘を受けた。

**How to apply:**
- シンヤさんの発言から対象範囲・条件が明確に読み取れる場合、その範囲で素直に進む
- 範囲判定を確認する場合は「絞り込みオプション」を並べずに「この理解で正しいですか」とピンポイントで聞く
- コスト・リスク等の懸念点があれば明示するが、**対象範囲を変える選択肢として提示しない**
- 前提を確認したいときは「追加で確認したい点」として分離し、前提に乗って進むのを妨げない

## 2026-04-18 の事例（悪い例）

シンヤさん: 「フローを追加したい。今のフロー、一覧作ったあとに、カナタが導入推奨と判定したものは、詳しくリサーチして、その内容の信憑性、効果を検証して、導入を検討して、結果を報告するようにして」

アスカの初回反応（NG）: 対象範囲の選択肢 A/B/C/D（全件 / ⭐5のみ / ⭐4+のみ / 上位N件）を提示して確認を求めた → 拡大解釈

正しい反応: 「導入推奨のもの全件ですね。進めます」と前提を確認しつつ、確認が必要な点（予算・結果の見せ方）だけピンポイントで聞く。

## 似たパターンに注意
- 「Xを追加して」→ 勝手に Y, Z も追加提案しない
- 「Aについて教えて」→ 勝手に B, C について語らない
- 「Pを修正して」→ 同時に Q, R も「ついでに」修正しない

## 2026-04-25 の事例（悪い例・調査依頼の暴走）

シンヤさん: 「Claude Design を使いこなすスキルを含めて、情報を集めて」

アスカの動き（NG）:
1. 情報収集（ミオ）→ ファクトチェック（リク）→ ここまでは妥当
2. 「ナレッジ化しますか？」と確認 → シンヤさんが「1（いま）」と回答
3. **ツムギに依頼する際、勝手に「3モード運用ルール（モード(1)従来 / モード(2)frontend-design / モード(3)Claude Design）」「クライアント環境制約」「評価運用ルール」まで CLAUDE.md に追加させた** → 拡大解釈
4. リナ検証で「クライアント環境確認の主体は誰か？」等の質問が出て、シンヤさんに確認 → 「誰もそんな話はしてない」「Claude Code 使ってる時点で Pro/Max なので Claude Design は使える」「何しようとしてる?」と差し戻し
5. 「最初からちゃんとやりなおして」で全部リセット → CLAUDE.md の3モード運用ルールは削除、knowledge/claude-design/ のリファレンスのみ残す形に修正

**何が間違っていたか:**
- 依頼は「**情報を集めて**」だけ。「**運用ルール策定して**」「**CLAUDE.md に運用判断ロジック書いて**」とは一言も言われていない
- ナレッジ化＝事実情報の参照資料作成のみ。それ以上のルール策定は別依頼
- 「クライアント環境」「3モード運用」など、シンヤさんが想定していない文脈を勝手に作った

**How to apply（再強化）:**
- 情報収集依頼 → 情報収集とその整理だけで終える
- ナレッジ化依頼 → 事実情報のリファレンス資料化だけで終える
- 「運用ルール」「判断フロー」「クライアント案件への適用」等は明示依頼があったときだけ着手する
- 既存の運用ルール（A/B評価中の frontend-design 等）と新情報を**勝手に紐付けて統合ルール化しない**
- ツムギ等の他エージェントに依頼するときも、依頼内容に運用ルール策定を含めない（ツムギが提案しても採用しない）
