## ルール

既存コードに「改善」を加える際は、**コードが何をしようとしているかを正確に読んでから**変更を提案する。
「読みにくい」「非効率」と感じても、それが意図的な設計の場合がある。

## 発生した問題（2026-03-20）

`lazyload-prioritize.js` の改善依頼を受けた際、以下の誤読をした：

- **元の意図**：eager画像が読み込まれた後、lazy画像を「ビューポートに近い順」に全件ロードする（帯域制御・順番制御が目的）
- **アスカの誤った解釈**：「全件ロードは帯域を食う」→ ビューポートから1000px以内だけ先読みするよう変更
- **シンヤさんの指摘**：「もともとのコードの意図を読み取ってないんじゃない？」「動きが違う」

## 正しい手順

1. コードを読んで「何をしようとしているか」を言語化する
2. その意図を**シンヤさんに確認してから**改善案を出す
3. 動作を変える場合は「ここを変えると動作が変わりますが意図通りですか？」と明示する

## 注意すべきパターン

- 「全件処理は非効率」→ 実は意図的なケースがある（帯域分散・全キャッシュ目的など）
- 「near-only に絞れば効率的」→ 元のアルゴリズムの意味を破壊する可能性がある
- コードの見た目の効率性より「なぜそう書いたか」の意図を優先する
