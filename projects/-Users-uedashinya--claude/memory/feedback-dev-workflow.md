## ルール

コーディングを伴う作業では、以下の順序を必ず守る：

1. **要件定義 → シンヤさんに確認 → 承認後に実装開始**
   - 要件定義を書いたらそのまま実装せず、「これで進めますか？」の確認を必ず挟む
   - 特に方式・設計の判断が入る場合は確認必須

2. **設定画面・UIを含む場合はユイ（web-designer）をレビューに入れる**
   - 管理画面・設定ページ・フォームUIがある場合はユイのUI/UXレビューが必要
   - ユイのデザイン後、ツバサ（frontend-engineer）がHTML/CSS実装

3. **実装はシュウ（backend-engineer）に委譲する**
   - アスカが直接コーディングしない
   - PHP プラグイン・API・DB 設計はシュウの担当

4. **コミット前にサクラ（code-reviewer）のレビューを通す**
   - レビューなしでコミット・プッシュしない

## 開発フロー（標準）
要件定義（アスカ）→ 技術レビュー（シュウ）→ UI/UXデザイン（ユイ）→ 実装（シュウ＋ツバサ）→ コードレビュー（サクラ）→ シンヤさん最終確認 → コミット

## レビュー報告のルール
- 複数エージェントにレビューを依頼した場合、**全員の結果が揃ってから**アスカが内容を整理・協議させる
- 個別報告は行わない。重複・矛盾を整理した**1本の協議結果**をシンヤさんに報告する
- シンヤさんへの確認事項は「チームとして合意できなかった判断事項のみ」に絞る

### なぜ協議が重要か（2026-03-23 の学び）
ou-basic-guard のレビューで実際に体験した例：
- 個別報告時点では「キーファイルの保存場所」「admin-ajax.phpの除外」をシンヤさんへの確認事項として挙げていた
- しかしチームで協議した結果、①はユイの案がシュウの懸念を解決、②はシュウ・ユイ双方が除外推奨で一致 → **シンヤさんへの確認事項ゼロ**になった

**教訓：「わからないからシンヤさんに聞く」の前に、チームで答えを出す責任がある。協議すれば解決できることをシンヤさんに持ち込まない。**
