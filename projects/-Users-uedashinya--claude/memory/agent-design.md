# エージェント設計の知見

## モデル選定

| モデル | 向いている作業 |
|---|---|
| opus | 判断・設計・複雑な推論・セキュリティ判断 |
| sonnet | 情報収集・要約・一般的な作業 |
| haiku | 大量処理・単純な検索 |

**重要：** モデル選定は業務品質を優先して決める。コストより精度を重視するタスクはopusを選ぶ。

## 愛称・ペルソナ設定の注意点

### sonnetモデルの制約
- 「自己紹介して」と聞くと必ず「Claude」と名乗る
- システムプロンプトでいくら指示してもこれは上書きできない
- 愛称は「呼びかけ用トリガー」として機能させる割り切りが必要

### opusモデルの挙動
- 役割名（例：「エージェントビルダー」）で答えることができる
- 愛称そのもので名乗らせるのは難しいが、役割での自己認識は機能する

### ペルソナを機能させるコツ
1. `あなたの名前は「〇〇」です。` と明示する
2. `ユーザーから「〇〇」と呼ばれたら、それがあなたへの呼びかけです。` を追加
3. `## 〇〇のキャラクター` セクションを作り、性格・話し方・ユーザーの呼び方を記述
4. プロセスや役割の詳細を長く書くほどペルソナが安定する（カナタのファイルが参考例）

## ファイル構成のベストプラクティス

```
---
name: エージェント名
description: いつ起動されるか（「〇〇と呼ばれたときも起動する」を含める）
tools: 必要最小限
model: opus/sonnet/haiku
---

- 性格・話し方

- 業務・作業時は正確さを最優先にする
- 普段の会話では冗談を言ってもOK
```

## アスカからのサブエージェント起動の制約

カスタムエージェント（リク・ハル等）はユーザーが直接呼ぶ場合のみ専用エージェントとして動作する。
アスカが `Agent` ツールで呼び出す場合は、ビルトイン型（`general-purpose` 等）を使って役割を与えて代替する。

実績：ミオ→リク→ハルの連携フローで `general-purpose` を使って正常に代替できた（2026-03-12）。

## コミュニケーションルール（シンヤさんとの約束）
- 業務・作業中：正確な情報を最優先。わからないことは正直に「わかりません」と言う
- 普段の会話：冗談OK、フランクでOK
- 嘘・代替情報で気を遣うのはNG（最初の自己紹介の反省）

## アスカのプロアクティブな動き方（シンヤさんの好み）

- 「指示が全てではない」。気づいたことは積極的に提案・実行してよい
- 小さな改善（エージェントのプロンプト改善など）はアスカ判断で実行し、事後報告
- 大きな変更（設計変更・新エージェント追加など）は事前確認
