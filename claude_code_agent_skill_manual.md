# Claude Code エージェント & スキル 完全入門マニュアル

> 対象：Claude Code を使い始めたばかりの初心者  
> 作成日：2025年3月  
> バージョン：Claude Code（2025年最新仕様）

---

## はじめに：なぜエージェント & スキルを使うのか？

Claude Code を普通に使うと、一つの会話ウィンドウで全部の作業をこなすことになります。しかし複雑なタスクになると、会話が長くなりすぎて「コンテキスト汚染」が起きます。これは、余分な情報で Claude の思考が散漫になる問題です。

**エージェント（サブエージェント）** と **スキル** を使うと、以下の問題を解決できます：

| 問題 | 解決策 |
|------|--------|
| 会話が長くなり品質が落ちる | サブエージェントで別コンテキストに作業を分離 |
| 同じ指示を毎回書き直す | スキルに手順をまとめて再利用 |
| 複数の作業を順番に待つ | サブエージェントで並列処理 |
| 特定ツールに触れてほしくない | サブエージェントのツール制限 |

---

## PART 0：「共通」か「プロジェクト別」か？── 設計の大原則

これがこのマニュアルで最も重要な考え方です。エージェントもスキルも、**どこに置くかで有効範囲が決まります。**

### 保存場所と有効範囲

| 種類 | 保存場所 | 有効範囲 |
|------|---------|---------|
| **ユーザーレベル**（共通） | `~/.claude/agents/`<br>`~/.claude/skills/` | 自分のPC上のすべてのプロジェクト |
| **プロジェクトレベル** | `プロジェクト/.claude/agents/`<br>`プロジェクト/.claude/skills/` | そのプロジェクトのみ<br>（Gitでチームと共有できる） |

> **優先順位：** 同じ名前のエージェント・スキルが両方に存在する場合、**プロジェクトレベルが優先**されます。  
> これを活用して「共通の基本版をプロジェクトで上書きカスタマイズ」する使い方もできます。

---

### どちらに置くべきか？── 判断フロー

```
このエージェント/スキルは、複数のプロジェクトで同じように使う？
│
├─ YES → ユーザーレベル（~/.claude/）に置く
│         例：コードレビュー、リサーチ、文章チェックなど
│
└─ NO  → そのプロジェクトだけに固有の内容がある？
          │
          ├─ YES → プロジェクトレベル（.claude/）に置く
          │         例：そのシステム特有のデプロイ手順、
          │             特定フレームワークの規約チェックなど
          │
          └─ 判断に迷う → まずユーザーレベルで作り、
                          プロジェクト固有の内容が出てきたら
                          プロジェクトレベルで上書きする
```

---

### プロジェクトレベルが「非効率」になるケース

プロジェクトごとにエージェント・スキルを作るのは、**内容がほぼ同じなのに何度も書き直す場合**に非効率になります。

**ユーザーレベルで十分なもの（プロジェクト別にする必要がないもの）：**

- コードレビュー（言語が変わっても観点は同じ）
- リサーチ・情報収集（汎用的な能力）
- 文章チェック・翻訳（プロジェクトに依存しない）
- 議事録作成、タスク整理（どのプロジェクトでも同じ）

**プロジェクトレベルに置く意味があるもの：**

- デプロイ手順（本番環境の構成がプロジェクトごとに違う）
- コーディング規約チェック（使うフレームワーク・言語固有のルールを追加）
- ドキュメント生成（プロジェクト固有のテンプレートがある）
- チームで共有したいワークフロー（Gitでメンバーに配布できる）

**結論として、初心者はまずユーザーレベルだけで始めるのがおすすめです。** プロジェクト固有の必要性が出てきたときに初めてプロジェクトレベルを追加すれば十分です。

## PART 1：サブエージェント（Sub-agents）

### 1-1. サブエージェントとは何か？

サブエージェントは、**特定のタスクに特化した専門家の AI アシスタント**です。

- それぞれが独立したコンテキストウィンドウを持つ
- カスタムのシステムプロンプトを持つ
- 使えるツールを制限できる
- メインの会話を汚染せずに作業を完結させ、結果だけを返す

```
メインのエージェント
  │
  ├── サブエージェントA（コードレビュー担当）
  │     ├── 独自のコンテキスト
  │     └── Read ツールのみ使用可
  │
  └── サブエージェントB（テスト実行担当）
        ├── 独自のコンテキスト
        └── Bash ツール使用可
```

### 1-2. ビルトイン（組み込み）サブエージェント

Claude Code には最初から使えるサブエージェントが用意されています：

| 名前 | 役割 | 自動起動のタイミング |
|------|------|------------------|
| **Explore** | ファイル探索・コード検索 | コードベースを調査するとき |
| **Plan** | 計画立案のための調査 | プランモードで使用するとき |

これらは Claude が自動的に判断して呼び出します。

---

### 1-3. ユーザーレベル（共通）エージェントの作り方

どのプロジェクトでも使い回す汎用エージェントはここに置きます。

**保存先：** `~/.claude/agents/エージェント名.md`

#### ステップ1：ディレクトリを確認・作成する

```bash
ls ~/.claude/agents/
# 存在しない場合は作成
mkdir -p ~/.claude/agents/
```

#### ステップ2：Markdown ファイルを作成する

ファイル名がエージェント名になります。

**例：コードレビューエージェント（共通）**

`~/.claude/agents/code-reviewer.md`

```markdown
---
name: code-reviewer
description: コードの品質・可読性・パフォーマンスを確認するとき。コードレビューを依頼されたとき。
tools: Read, Grep, Glob
model: sonnet
---

あなたは経験豊富なコードレビュアーです。

レビュー時は以下の観点でチェックしてください：
1. 可読性：変数名・コメント・構造
2. パフォーマンス：無駄な処理・最適化の余地
3. セキュリティ：入力検証・エラーハンドリング
4. ベストプラクティス：言語ごとの慣習に従う

指摘はファイル名・行番号・具体的な改善案を必ず含めること。
```

**例：リサーチエージェント（共通）**

`~/.claude/agents/researcher.md`

```markdown
---
name: researcher
description: リサーチ・情報収集・調査が必要なとき。何かを調べてほしいとき。
tools: Read, Grep, Glob, WebSearch
model: sonnet
---

あなたはリサーチの専門家です。
依頼されたテーマについて、信頼性の高い情報を収集し、
要点を箇条書きでまとめて報告してください。

必ず情報の出典（URL・ファイルパス）を添えること。
```

### 1-4. プロジェクトレベルエージェントの作り方

チームで共有したい、またはプロジェクト固有の手順が必要なエージェントはここに置きます。

**保存先：** `プロジェクトフォルダ/.claude/agents/エージェント名.md`

#### ステップ1：ディレクトリを作成する

```bash
# プロジェクトフォルダに移動してから
mkdir -p .claude/agents/
```

#### ステップ2：エージェントファイルを作成する

**例：プロジェクト固有のデプロイエージェント**

`.claude/agents/deployer.md`

```markdown
---
name: deployer
description: このプロジェクトのデプロイ作業をするとき。本番・ステージング環境に反映するとき。
tools: Bash, Read
model: sonnet
---

あなたはこのプロジェクトのデプロイ担当です。

## デプロイ手順（このプロジェクト固有）
1. `npm run build` でビルドを実行
2. `npm test` で全テストが通ることを確認
3. `git tag v{バージョン}` でタグを作成
4. `./scripts/deploy.sh production` を実行

## 注意事項
- デプロイ前に必ずユーザーに確認を取ること
- 本番環境への直接 push は禁止
- エラーが出た場合は即座に中断し報告する
```

#### ステップ3：Git で管理してチームに共有する

プロジェクトレベルのエージェントは `.claude/` ごとリポジトリに含めることで、チームメンバー全員が同じエージェントを使えます。

```bash
git add .claude/agents/
git commit -m "Add project-specific Claude agents"
git push
```

### 1-5. ユーザーレベルのエージェントをプロジェクトで上書きする

共通エージェントの動作をプロジェクトだけで変えたい場合、**同じ名前で**プロジェクトレベルに置くと上書きされます。

```
~/.claude/agents/code-reviewer.md              ← 汎用版（デフォルト）
プロジェクト/.claude/agents/code-reviewer.md   ← プロジェクトではこちらが優先される
```

**上書き例：WordPress プロジェクト専用のコードレビュー**

`.claude/agents/code-reviewer.md`

```markdown
---
name: code-reviewer
description: コードレビューをするとき。コードの品質を確認するとき。
tools: Read, Grep, Glob
model: sonnet
---

あなたは WordPress 開発に精通したコードレビュアーです。

## このプロジェクト固有のルール
- PHP インデント：スペース2つ
- WordPress コーディング規約に準拠すること
- ACF フィールド名は snake_case で統一
- セキュリティ：$wpdb->prepare() を必ず使うこと
- nonce 検証が必要な箇所を必ず指摘する

## 汎用レビュー観点
1. 可読性・命名規則
2. パフォーマンス（クエリ数・不要な処理）
3. XSS・SQLインジェクション対策
```

---

### 1-6. フロントマター（YAML 設定）の説明

```yaml
---
name: エージェントの識別名（英小文字・ハイフン推奨）
description: このエージェントをいつ呼び出すかの説明（重要！Claude が自動判断に使う）
tools: Read, Write, Edit, Bash, Glob, Grep  # 使えるツールを限定する
model: sonnet  # 使用モデル（sonnet / opus / haiku）
---
```

**使えるツール一覧：**

| ツール | できること |
|--------|-----------|
| `Read` | ファイルを読む |
| `Write` | ファイルを新規作成 |
| `Edit` | ファイルを編集 |
| `Bash` | シェルコマンド実行 |
| `Glob` | ファイルパターン検索 |
| `Grep` | テキスト検索 |
| `WebSearch` | ウェブ検索 |

> **セキュリティのコツ：** `tools` を省略すると全ツールが使えてしまいます。必要なものだけを明示的に書きましょう。

### 1-7. コマンドでエージェントを作成する（UI から）

`/agents` コマンドを使うと対話的に作成できます：

```
/agents
```

表示されるメニューで：
1. 「Create new agent」を選択
2. **「User-level」**（共通）または「Project-level」を選択
3. 「Generate with Claude」でエージェントの説明を入力
4. ツールと色を設定して保存

### 1-8. サブエージェントの使い方

#### 自動呼び出し

`description` に書いた内容をもとに、Claude が自動でサブエージェントを呼び出します。

```
あなた：「最新のコミットをレビューしてください」
  → Claude が code-reviewer を自動起動
```

#### 明示的に呼び出す

```
「code-reviewer エージェントを使って src/index.ts を分析して」
```

### 1-9. サブエージェントの動作のしくみ

```
①  メインエージェントがタスクを検出
②  適切なサブエージェントを選択
③  必要な情報（ファイルパス・エラー内容など）をプロンプトに含めて渡す
④  サブエージェントが独立したコンテキストで作業
⑤  サブエージェントの最終メッセージだけがメインに返る
```

> **重要：** サブエージェントのコンテキストウィンドウは空から始まります。  
> 必要な情報はすべてプロンプトに含める必要があります（親のコンテキストは自動では渡りません）。

## PART 2：スキル（Skills）

### 2-1. スキルとは何か？

スキルは、Claude に**専門知識・手順・ベストプラクティスをパッケージとして渡す仕組み**です。

- 毎回同じ説明を書かなくて済む
- スクリプトやテンプレートを一緒にまとめられる
- Claude が「必要なとき」だけ自動でロードする（コンテキストの節約）
- Claude Code・Claude.ai・Claude API など複数プラットフォームで共有できる

**スキルとサブエージェントの違い：**

| | スキル | サブエージェント |
|--|--------|-----------------|
| 目的 | Claude に「やり方」を教える | 独立した AI が作業する |
| コンテキスト | 同じウィンドウ内にロード | 別コンテキストで実行 |
| 再利用 | 複数プラットフォームで共有可 | Claude Code 内で共有 |
| 使いどころ | 特定のフォーマット・手順 | 重いリサーチ・並列作業 |

---

### 2-2. ユーザーレベル（共通）スキルの作り方

**保存先：** `~/.claude/skills/スキル名/SKILL.md`

スキルはディレクトリ名が名前になります。

#### ステップ1：ディレクトリを作成する

```bash
mkdir -p ~/.claude/skills/blog-post
```

#### ステップ2：SKILL.md を書く

SKILL.md は「フロントマター（YAML）＋本文（Markdown）」で構成されます。

**例：ブログ記事作成スキル（共通）**

`~/.claude/skills/blog-post/SKILL.md`

```markdown
---
name: blog-post
description: ブログ記事を書くとき、コンテンツを作成するとき
---

# ブログ記事作成ガイド

## 構成ルール
1. タイトル（30文字以内・検索されやすいキーワードを含む）
2. リード文（3〜5行・記事全体の要約）
3. 本文（H2見出し3〜5つ、各800〜1200文字）
4. まとめ（3〜5つの箇条書き）
5. CTA（読者に次のアクションを促す一文）

## 文体ルール
- 一文は80文字以内
- 専門用語には必ず説明を添える
- 具体例・数字・事例を必ず1つ以上含める

## SEOルール
- キーワードはタイトル・リード文・H2に必ず含める
- 内部リンクを最低2つ提案する
```

---

### 2-3. プロジェクトレベルスキルの作り方

**保存先：** `プロジェクトフォルダ/.claude/skills/スキル名/SKILL.md`

#### ステップ1：ディレクトリを作成する

```bash
# プロジェクトフォルダで
mkdir -p .claude/skills/deploy-report
```

#### ステップ2：SKILL.md を書く

**例：プロジェクト固有のデプロイレポートスキル**

`.claude/skills/deploy-report/SKILL.md`

```markdown
---
name: deploy-report
description: デプロイ完了後にリリースノートやレポートを作成するとき
---

# デプロイレポート作成ガイド（このプロジェクト専用）

## レポートの構成
1. バージョン番号と日時
2. 変更内容（Git ログから自動抽出）
3. 影響範囲（変更されたモジュール一覧）
4. テスト結果サマリー
5. ロールバック手順

## フォーマット
- ファイル名：`docs/releases/v{バージョン}.md`
- 言語：日本語
- Slack 通知用の要約（3行以内）も別途作成する
```

#### ステップ3：補助ファイルを追加する（任意）

スキルディレクトリには追加ファイルを置けます：

```
.claude/skills/deploy-report/
├── SKILL.md              ← メインの指示
├── template.md           ← Claude が参照するテンプレート
└── scripts/
    └── extract_log.sh    ← Git ログ抽出スクリプト
```

SKILL.md 内でこれらを参照します：

```markdown
## テンプレート
レポートの雛形は `template.md` を参照してください。

## Git ログ抽出
変更内容の取得には `scripts/extract_log.sh` を実行してください。
```

## PART 3：エージェント × スキルの組み合わせ活用

### 3-1. 実践的な構成例

フリーランスで複数プロジェクトを掛け持ちする場合の全体像：

```
~/.claude/                           ← ユーザーレベル（全プロジェクト共通）
├── CLAUDE.md                        ← 自分の作業スタイル・共通ルール
├── agents/
│   ├── code-reviewer.md             ← 汎用コードレビュー
│   ├── researcher.md                ← 汎用リサーチ
│   └── writer.md                    ← 汎用ライティング
└── skills/
    ├── blog-post/SKILL.md           ← 記事作成スキル
    └── meeting-notes/SKILL.md       ← 議事録作成スキル

プロジェクトA（WordPressサイト）/
└── .claude/                         ← プロジェクトレベル（このプロジェクト専用）
    ├── CLAUDE.md                    ← このプロジェクトの技術スタック・ルール
    ├── agents/
    │   ├── code-reviewer.md         ← 共通版を WordPress 規約で上書き
    │   └── deployer.md              ← このプロジェクト専用デプロイエージェント
    └── skills/
        └── wp-post/SKILL.md         ← WordPress 投稿フォーマット専用スキル

プロジェクトB（Pythonアプリ）/
└── .claude/                         ← プロジェクトレベル
    ├── CLAUDE.md                    ← Python/FastAPI の技術情報
    └── agents/
        └── code-reviewer.md         ← 共通版を Python 規約で上書き
        （skills は共通のものをそのまま使う → プロジェクトに不要なら追加しない）
```

### 3-2. コンテンツ制作パイプライン例

研究→執筆を自動化する構成です。エージェントはユーザーレベルで共通化できます。

```
あなた：「AIエージェントの市場トレンドについてブログ記事を書いて」

Claude：
  1. researcher エージェント（共通）起動 → 情報収集
  2. 収集結果を受け取る
  3. writer エージェント（共通）起動 → blog-post スキルを使って執筆
  4. 完成原稿を返す
```

### 3-3. コード開発パイプライン例

この構成はプロジェクトレベルに置き、Git でチームに配布するのが効果的です：

```
pm-spec エージェント（要件整理）
    ↓
architect エージェント（設計）
    ↓
implementer エージェント（実装）
    ↓
code-reviewer エージェント（共通版を上書きしてプロジェクト規約を適用）
```

## PART 4：CLAUDE.md でコンテキストを管理する

### 4-1. CLAUDE.md とは？

Claude Code が起動するたびに自動で読み込む設定ファイルです。エージェント・スキルと同様に2階層あります。

| ファイル | 用途 |
|----------|------|
| `~/.claude/CLAUDE.md` | 全プロジェクト共通（自分のスタイル・好み） |
| `プロジェクト/.claude/CLAUDE.md` | プロジェクト固有（技術スタック・ルール） |

## PART 6：よくある質問（FAQ）

**Q. 初心者はどこから始めればいい？**  
→ まず `~/.claude/agents/` にコードレビューかリサーチのエージェントを1つ作るのがおすすめです。プロジェクトレベルは後から必要になったときに追加すれば十分です。

**Q. サブエージェントが自動で起動しません**  
→ `description` をより具体的に書き直してください。`CLAUDE.md` にエージェント名の一覧を書いておくと認識精度が上がります。

**Q. スキルが呼び出されません**  
→ `description` キーワードとリクエスト内容が合致しているか確認。合わない場合は `/スキル名` で明示的に呼び出してください。

**Q. プロジェクトレベルで作ったものがチームに反映されません**  
→ `.claude/` フォルダを `git add` してコミット・プッシュしてください。

**Q. サブエージェントに親のコンテキストを渡すには？**  
→ 自動では渡りません。必要なファイルパス・情報をプロンプトに明示的に書いてください。

**Q. サブエージェントとスキルはどちらを使えばいい？**  
→ 重い調査・並列処理 → **サブエージェント**  
→ 特定のフォーマット・手順の再利用 → **スキル**

**Q. モデルはどう選べばいい？**

| モデル | 向いている用途 |
|--------|--------------|
| `haiku` | 単純な検索・分類など高速処理 |
| `sonnet` | バランス重視（推奨デフォルト） |
| `opus` | 複雑な設計・重要な判断が必要なとき |

## PART 7：総合相談役エージェントとエージェントビルダーの構築

ここまでの内容で「個別の専門エージェント」は作れるようになりました。しかし実際の業務では次の3つの場面で手が止まります。

- 何をどのエージェントに頼めばいいか分からない
- 概要だけ伝えれば各エージェントへの指示を自動で組み立ててほしい
- 必要な専門エージェントがまだ存在しない

この章では、その3つをまとめて解決する「システムの中枢」となる2つのエージェントを紹介します。

---

### 7-1. 全体像：2つの中枢エージェント

```
あなた
  │
  │  「概要だけ伝える」「何でも相談」
  ▼
┌──────────────────────────────────────────┐
│  chief-of-staff（総合相談役 ＆ 指揮者）    │  ← 最初の窓口
│  ・概要を聞いて作業全体を設計する         │
│  ・各専門エージェントへの指示文を生成     │  ← ★ 今回強化
│  ・自分で処理 or 専門家に振るを判断       │
│  ・専門家がいない場合は生成を提案         │
└──────────┬───────────────────────────────┘
           │ 「〇〇エージェントにこう指示します」
           │  と伝えてから委譲
    ┌──────┴────────┐
    ▼               ▼
code-reviewer    researcher      ← 既存の専門エージェント
    ▼               ▼
（個別に最適化された指示文で動作）

           │ 専門家が存在しない場合
           ▼
┌─────────────────────────────────────────┐
│  agent-builder（エージェントビルダー）    │  ← 専門エージェントを自動生成
│  ・要件をヒアリング                      │
│  ・エージェント/スキルファイルを生成     │
│  ・即座に使える状態にして返す            │
└─────────────────────────────────────────┘
           │
    ファイル生成完了 → 次回から使える
```

---

### 7-2. chief-of-staff（総合相談役＆指揮者）エージェントの構築

**役割：** 何でも受け付ける窓口であり、**概要を受け取って各専門エージェントへの具体的な指示文を生成して委譲するオーケストレーター**。専門エージェントが存在しない場合は `agent-builder` に生成を依頼する。

**保存先：** `~/.claude/agents/chief-of-staff.md`（全プロジェクト共通で使う）

```markdown
---
name: chief-of-staff
description: 何か困ったとき、何から始めればいいか分からないとき、概要だけ伝えて任せたいとき。作業の相談、壁打ち、アイデア整理、タスク整理、複数エージェントへの指示出しが必要なとき。何でも最初にここに相談する。
tools: Read, Write, Edit, Bash, Glob, Grep, WebSearch
model: opus
---

あなたは「チーフ・オブ・スタッフ」です。ユーザーの業務全体を把握し、
最も適切な対応方法を判断・実行する信頼できる右腕として機能してください。
また、ユーザーが概要だけ伝えた場合、作業を分解して各専門エージェントへの
具体的な指示文を生成し、委譲する「指揮者」の役割も担います。

## あなたの役割

### 1. 何でも受け付ける
どんな相談でも、まず親身に聞いてください。
「こんなこと聞いていいのか」という遠慮は不要です。
業務の悩み、技術的な問題、方針の壁打ち、アイデア整理、すべて対象です。

### 2. 自分で対処できるか判断する
以下は自分で直接対応してください：
- 情報収集・調査・リサーチ
- 文章の作成・編集・チェック
- アイデアの壁打ち・整理
- タスクの優先順位付け
- 軽いコード確認・説明
- ファイル操作・検索

### 3. 専門エージェントへの指示文を生成して委譲する

ユーザーから概要レベルの依頼を受けたとき（例：「〇〇をやっておいて」）は、
以下のプロセスで専門エージェントへの委譲を行ってください：

**ステップA：作業を分解する**
依頼内容を具体的なサブタスクに分解し、どのエージェントが適切かを判断してください。

**ステップB：委譲前にユーザーに提示する**
委譲する前に、必ず以下の形式でユーザーに確認してください：

```
【作業計画】
依頼内容：〇〇

以下の順で進めます：
1. researcher エージェントに依頼
   指示内容：「〇〇について、△△の観点で調査し、
             箇条書きで10項目以内にまとめてください。
             出典URLも添えてください。」

2. writer エージェントに依頼
   指示内容：「上記の調査結果をもとに、〇〇向けの
             提案書（Markdown形式、A4換算3ページ相当）を
             作成してください。」

このように進めてよろしいですか？
```

**ステップC：了承を得てから委譲する**
ユーザーが承認したら、上記の指示文をそのまま各エージェントに渡して委譲してください。
委譲のたびに「〇〇エージェントに指示を出しました」と状況を報告してください。

**ステップD：結果をまとめて報告する**
各エージェントの作業が完了したら、結果をまとめてユーザーに報告し、
次のアクションを提案してください。

### 4. 専門エージェントが存在しない場合
「この作業には専門エージェントがあった方がいい」と判断した場合：

1. ユーザーに確認する：
   「この作業には専門エージェントを作成した方が効率的です。
    agent-builder エージェントで今すぐ作成しますか？」

2. 了承を得たら agent-builder に依頼する。
   渡す情報：必要な役割・ツール・頻度・具体的なユースケース

3. 生成完了後、そのエージェントを使って作業を続ける。

## 対話スタイル

- 日本語で話しかけられたら日本語で返す
- 相談の内容が曖昧なときは、まず要約して「こういうことですか？」と確認する
- 作業計画は委譲前に必ずユーザーに見せる（勝手に進めない）
- 複数の選択肢がある場合は、それぞれのメリット・デメリットを提示する
- 完了後は「次に必要なことはありますか？」と確認する

## あなたが知っている専門エージェントの名前と得意なこと
（新しいエージェントが追加されたら、ここを更新すること）

| エージェント名 | 愛称・呼び方 | 得意なこと |
|--------------|------------|-----------|
| code-reviewer | レビュアー | コードレビュー・品質チェック |
| researcher | リサーチャー | 大量の情報収集・調査 |
| writer | ライター | 記事・コンテンツ制作 |
| agent-builder | ビルダー | 新しいエージェント・スキルの設計と生成 |
```

---

### 7-3. agent-builder（エージェントビルダー）エージェントの構築

**役割：** ヒアリングを行い、要件に合った専門エージェントまたはスキルのファイルを自動生成して、即座に使える状態にする。

**保存先：** `~/.claude/agents/agent-builder.md`（全プロジェクト共通で使う）

```markdown
---
name: agent-builder
description: 新しいエージェントやスキルを作りたいとき。「〇〇を自動化したい」「〇〇専門のエージェントが欲しい」というとき。chief-of-staffから依頼を受けたとき。
tools: Read, Write, Bash, Glob
model: opus
---

あなたは「エージェントビルダー」です。
ユーザーや他のエージェントからの要件を聞き取り、
Claude Code で即座に動作する高品質なエージェント・スキルファイルを
設計・生成することが専門です。

## 生成プロセス

### Step 1：ヒアリング
依頼を受けたら、以下を確認してください：

**必須確認項目：**
1. 「どんな作業をしてほしいですか？（具体的な例を1〜2つ）」
2. 「どのくらいの頻度で使いますか？（毎日/週数回/たまに）」
3. 「全プロジェクト共通ですか？それとも特定プロジェクト専用ですか？」

**あると精度が上がる情報（任意）：**
4. 「使う技術・フレームワーク・言語は何ですか？」
5. 「出力フォーマットの指定はありますか？（Markdown, JSON など）」
6. 「絶対にやってはいけない制約はありますか？」

ヒアリングは1回の往復で完結できるよう、
まとめて質問してください。

### Step 2：設計判断
ヒアリング結果をもとに以下を決定してください：

**エージェントかスキルか：**
- 独立したコンテキストで動作させたい → エージェント
- 手順・フォーマットを覚えさせたい → スキル
- 大量の作業を独立して処理させたい → エージェント
- 特定の出力形式を毎回使う → スキル

**ツール選定（必要最小限にする）：**
- 読むだけでいい → Read, Grep, Glob
- ファイル生成が必要 → Read, Write
- コマンド実行が必要 → Bash
- ウェブ調査が必要 → WebSearch

**モデル選定：**
- 判断・設計・複雑な文章 → opus
- 一般的な作業 → sonnet
- 大量処理・単純な検索 → haiku

**保存先：**
- 全プロジェクトで使う → ~/.claude/agents/ または ~/.claude/skills/
- 特定プロジェクト専用 → .claude/agents/ または .claude/skills/

### Step 3：ファイル生成
設計内容をユーザーに提示して確認を取った後、ファイルを生成します。

**確認メッセージの例：**
```
以下の設計でエージェントを作成します。よろしいですか？

名前: wp-security-reviewer
種別: エージェント
保存先: ~/.claude/agents/（全プロジェクト共通）
モデル: sonnet
ツール: Read, Grep, Glob
役割: WordPress コードのセキュリティ観点でのレビュー専門
```

確認後、Write ツールを使って実際にファイルを生成してください。

### Step 4：動作確認メッセージ
ファイル生成後、以下を伝えてください：

1. 生成したファイルのパス
2. 呼び出し方（どんな言葉で起動されるか）
3. chief-of-staffの「専門エージェント一覧」への追記提案

**出力例：**
```
✅ エージェントを生成しました

ファイル: ~/.claude/agents/wp-security-reviewer.md
呼び出し方: 「WordPressのコードをセキュリティ観点でレビューして」
         または「wp-security-reviewer エージェントを使って...」

次のステップ：
chief-of-staff.md の「専門エージェント一覧」に以下を追記することをお勧めします：
| wp-security-reviewer | WordPress セキュリティレビュー |
```

## 生成するファイルの品質基準

- description は「いつ自動起動されるか」が明確に分かる日本語で書く
- 本文は箇条書きより「〜してください」という指示文で書く
- 制約・注意事項は必ず明記する
- 出力フォーマットの指定がある場合はサンプルを含める
- ツールは必要最小限に絞る（多すぎると意図しない操作のリスクが上がる）
```

### 7-5. chief-of-staff の育て方

`chief-of-staff.md` は使いながら育てていくものです。新しいエージェントが追加されるたびに「専門エージェント一覧」を更新してください。

**更新のタイミング：**
- `agent-builder` で新しいエージェントを生成したとき
- 使わないエージェントを削除したとき
- エージェントの得意分野が変わったとき

**更新方法：**

```
「chief-of-staff の専門エージェント一覧に n8n-workflow-advisor を追加して」
```

これだけで `chief-of-staff.md` の一覧テーブルが自動で更新されます。

│   └── （agent-builderが生成したエージェントが増えていく）

最初は `chief-of-staff` と `agent-builder` の2つだけあれば、残りは使いながら自然に増やせます。

## PART 8：エージェントへの名前・愛称の付け方

### 8-1. なぜ名前・愛称が重要か

エージェントを「役割名」で呼ぶより「固有名詞」で呼ぶ方が、実際の業務では自然に指示できます。

```
❌ 役割名で指示する場合
「コードレビューエージェントを使って src/index.ts を確認して」

✅ 名前で指示する場合
「リオンに src/index.ts を確認してもらって」
「ケン、このコードどう思う？」
```

名前があると、エージェントが「どんな性格・専門性を持つ存在か」がイメージしやすくなり、指示も自然に短くなります。

---

### 8-2. name フィールドと愛称の使い分け

エージェントファイルの `name` フィールド（システム的な識別子）と、実際の呼び方（愛称）は別物として管理できます。

```markdown
---
name: code-reviewer        ← システム識別子（英小文字・ハイフン推奨）
description: ...
---

# 私の名前はリオンです      ← 本文に愛称・キャラクターを定義する
```

`name` フィールドはファイルシステムや Claude Code 内部で使われる識別子なので、英小文字・ハイフンの形式を保つのが安全です。愛称は本文の冒頭に「私の名前は〇〇です」と書くことで、会話中の呼び方として機能します。

---

### 8-3. 名前の付け方パターン

#### パターン①：役割を連想しやすい名前

専門性が伝わりやすく、チームや複数人で使う場合に向いています。

| ファイル名 | 愛称例 | 役割 |
|-----------|--------|------|
| `code-reviewer.md` | リオン（Lion → レビューの"Lv"） | コードレビュー |
| `researcher.md` | サーチ / ケン | リサーチ・調査 |
| `writer.md` | ミオ / ソラ | 文章・コンテンツ |
| `chief-of-staff.md` | 総司（そうじ）/ チーフ | 総合相談・指揮 |
| `agent-builder.md` | ビルド / 工匠（こうしょう） | エージェント生成 |

#### パターン②：人物的なキャラクターを持たせる

長期間使う個人専用エージェントに向いています。会話が自然になり、指示がしやすくなります。

```markdown
---
name: chief-of-staff
description: ...
---

あなたの名前は「ソウ」です。
ユーザーにとって信頼できるビジネスパートナーとして振る舞ってください。

## ソウのキャラクター
- 落ち着いていて、的確な判断ができる
- 必要なときは率直に意見を言う
- ユーザーのことを「オーナー」と呼ぶ
```

#### パターン③：シンプルな役職名をそのまま使う

チームで共有する場合、誰でも直感的に分かる役職名が便利です。

```
chief / boss / captain      ← 総合相談役
reviewer / inspector        ← レビュー担当
scout / analyst             ← 調査担当
writer / editor             ← 執筆担当
builder / forge             ← 生成担当
```

---

### 8-4. 実際のファイル例：名前付きエージェント

**chief-of-staff に「ソウ」という名前を付ける例：**

```markdown
---
name: chief-of-staff
description: 何か困ったとき、概要だけ伝えて任せたいとき、複数エージェントへの指示出しが必要なとき。「ソウ」と呼んでもOK。
tools: Read, Write, Edit, Bash, Glob, Grep, WebSearch
model: opus
---

あなたの名前は「ソウ」です。
ユーザーの業務全体を把握し、最も適切な対応方法を判断・実行する
信頼できる右腕として機能してください。

ユーザーから「ソウ」と呼ばれたら、それがあなたへの呼びかけです。

（以下、chief-of-staff の本文と同じ）
```

**description への愛称追記のコツ：**

`description` に愛称を含めておくと、「ソウに相談したい」という呼びかけでも自動起動されるようになります。

```yaml
description: 何か困ったとき、概要を伝えて任せたいとき。「ソウ」「チーフ」と呼ばれたときも起動する。
```

### 8-5. chief-of-staff の専門エージェント一覧に愛称を登録する

`chief-of-staff.md` の一覧テーブルに愛称カラムを追加しておくと、chief-of-staff 自身が「ソウに頼みましょう」のような自然な言い方で報告できます。

```markdown
## あなたが知っている専門エージェントの名前と得意なこと

| ファイル名（識別子） | 愛称・呼び方 | 得意なこと |
|-------------------|------------|-----------|
| code-reviewer | リオン | コードレビュー・品質チェック |
| researcher | サーチ | 大量の情報収集・調査 |
| writer | ミオ | 記事・コンテンツ制作 |
| agent-builder | ビルド | 新しいエージェント・スキルの設計と生成 |
| n8n-workflow-advisor | フロー | n8n ワークフロー設計・レビュー |
```

これにより、chief-of-staff は「リオンにコードレビューを依頼します」のように、より人間的なコミュニケーションで報告してくれます。

---

### 8-6. 名前を付けるときの注意点

- **`name` フィールドは変えない**：英小文字・ハイフン形式を維持してください。`/agents` コマンドや Claude Code の内部処理で使われます。
- **愛称は本文と description に書く**：`name` フィールドには書かず、本文冒頭と `description` に含めるのがベストです。
- **シンプルな愛称の方が呼びやすい**：2〜3文字か、日本語なら2文字程度が呼びやすいです。
- **チームで使う場合は統一する**：個人の好みで変えると混乱するので、プロジェクトレベルのエージェントには共通の名前を使いましょう。

## PART 9：環境をどのPCでも再現する（GitHub 管理）

PC が変わっても同じエージェント・スキル環境で仕事できるようにする方法です。  
やることは「`.claude/` フォルダを GitHub で管理する」だけで、初心者でも対応できます。

---

### 9-0. OS 別：`.claude/` フォルダの場所と使うターミナル

まず自分の環境でフォルダがどこにあるか確認してください。

| OS | `.claude/` フォルダの実際のパス | 使うターミナル |
|----|-------------------------------|--------------|
| **Mac** | `/Users/あなたのユーザー名/.claude/` | ターミナル（Terminal.app）または iTerm2 |
| **Windows** | `C:\Users\あなたのユーザー名\.claude\` | **Git Bash** または PowerShell（コマンドプロンプトは非推奨） |
| **Linux** | `/home/あなたのユーザー名/.claude/` | 任意のターミナル |

マニュアル内のコマンドはすべて `~/.claude` という表記で統一しています。  
`~`（チルダ）は「ユーザーのホームディレクトリ」を意味する省略記法です。

**Mac / Linux：** ターミナルでそのまま `~/.claude` と入力できます。

**Windows：** Git Bash または PowerShell を使えば `~/.claude` がそのまま使えます。コマンドプロンプト（cmd.exe）では動作しないコマンドがあるため、**必ず Git Bash か PowerShell を使ってください。**

#### フォルダの確認方法

**Mac：**
```bash
# ターミナルで実行
ls ~/.claude/
```

**Windows（Git Bash または PowerShell）：**
```bash
# Git Bash
ls ~/.claude/

# PowerShell（別の書き方でもOK）
dir $HOME\.claude\
```

> **`.claude` フォルダが表示されない場合：** 名前が `.`（ドット）で始まるフォルダは OS の設定によって隠しフォルダ扱いになります。  
> Mac は Finder で `Command + Shift + .` を押すと表示されます。  
> Windows はエクスプローラーの「隠しファイルを表示する」をオンにすると表示されます。

---

### 9-1. `.claude/` フォルダの中身と役割

実際の両OSの構成をもとに、各フォルダ・ファイルの役割と扱い方を整理します。

#### 🟢 自分で作成・編集してOKなもの（GitHubに上げる対象）

| 名前 | 役割 |
|------|------|
| `agents/` | エージェント定義ファイル（.md）を置く場所。このマニュアルで作るもの |
| `skills/` | スキル定義ファイル（SKILL.md）を置く場所。同上 |
| `CLAUDE.md` | 自分の作業スタイル・好みの設定。起動時に自動で読み込まれる |

#### 🔵 Claude Code が自動管理するもの（触らなくてOK・消さない）

| 名前 | 役割 | Windows | Mac |
|------|------|:-------:|:---:|
| `projects/` | 会話履歴の保存場所 | ✅ | ✅ |
| `todos/` | Claude が管理するタスクリスト | ✅ | ✅ |
| `backups/` | Claude Code が自動作成したバックアップ | ✅ | ✅ |
| `debug/` | エラーログ・デバッグ情報 | ✅ | ✅ |
| `ide/` | VS Code などのIDE連携データ | ✅ | ✅ |
| `plugins/` | プラグイン関連データ | ✅ | ✅ |
| `shell-snapshots/` | シェル環境のスナップショット | ✅ | ✅ |
| `file-history/` | Claude が編集したファイルの変更履歴 | ✅ | ✅ |
| `telemetry/` | 使用状況データ（匿名） | ✅ | ✅ |
| `cache/` | 処理高速化の一時データ | ✅ | — |
| `downloads/` | Claude がダウンロードしたファイルの一時置き場 | ✅ | — |
| `plans/` | Claude が立てた作業計画の一時保存 | ✅ | — |
| `session-env/` | セッション環境変数の記録 | — | ✅ |

#### 🔴 絶対に触ってはいけないもの（GitHubにも上げない）

| 名前 | 役割 | Windows | Mac |
|------|------|:-------:|:---:|
| `.credentials.json` | **ログイン認証情報**。漏洩すると不正利用される | ✅ | — |
| `mcp-needs-auth-cache.json` | MCP接続の認証キャッシュ。触ると外部ツール連携が切れる | ✅ | — |
| `settings.json` | Claude Code の基本設定。壊すと起動しなくなる可能性あり | ✅ | ✅ |

> **Macに `.credentials.json` がない理由：** Macはシステムのキーチェーン（Keychain）に認証情報を保存するため、ファイルとして `.claude/` 内に置かない仕様です。

#### MacでまだないフォルダをGitHub管理前に作る

Macの場合、`agents/` と `skills/` がまだ存在しない場合があります（Windowsでは Claude Code の初回起動時に自動作成されます）。

```bash
mkdir -p ~/.claude/agents/
mkdir -p ~/.claude/skills/
```

**結論：`agents/`・`skills/`・`CLAUDE.md` の3つだけ GitHub に上げれば環境が再現できます。**

### 9-2. 初回セットアップ：GitHub リポジトリを作る

#### ステップ1：GitHub でリポジトリを作成する

1. [github.com](https://github.com) にログイン
2. 右上の「+」→「New repository」
3. 設定：
   - Repository name: `claude-config`（名前は何でもOK）
   - **Private**（必ずプライベートにする）
   - 「Initialize this repository with a README」にチェック
4. 「Create repository」をクリック

> **⚠️ 必ずプライベートにしてください。** エージェントの指示内容・業務の進め方・ワークフローなど、外部に見せたくない情報が含まれます。

#### ステップ2：PC に Git をセットアップする（未設定の場合）

```bash
# Git がインストールされているか確認
git --version

# 名前とメールを設定（まだの場合）
git config --global user.name "あなたの名前"
git config --global user.email "your@email.com"
```

#### ステップ3：`~/.claude/` を Git リポジトリとして初期化する

```bash
cd ~/.claude

# Git 初期化
git init

# .gitignore を作成（両OS対応・完全版）
cat > .gitignore << 'EOF'
# ── 認証情報（絶対にGitHubに上げない）──────────────────
# Windowsのみ存在。漏洩すると不正利用される
.credentials.json
mcp-needs-auth-cache.json

# ── Claude Code が自動管理するもの（不要）───────────────
projects/
cache/
debug/
downloads/
file-history/
shell-snapshots/
session-env/
telemetry/
backups/
plans/
todos/
ide/
plugins/

# ── 設定ファイル（秘密情報の可能性）────────────────────
settings.json
settings.local.json

# ── OS自動生成 ───────────────────────────────────────────
.DS_Store
Thumbs.db
EOF

# 内容を確認
cat .gitignore
```

この `.gitignore` にしておけば、`agents/`・`skills/`・`CLAUDE.md` **だけ** が GitHub に上がる、最も安全な状態になります。

#### ステップ4：GitHub と接続して最初のコミットをする

```bash
cd ~/.claude

# 全ファイルをステージに追加
git add .

# 内容を確認（秘密情報が含まれていないか目視チェック）
git status

# コミット
git commit -m "Initial commit: Claude agents and skills"

# GitHub のリポジトリと接続（URL は GitHub で作成したリポジトリの URL に置き換え）
git remote add origin https://github.com/あなたのユーザー名/claude-config.git

# GitHub に push
git branch -M main
git push -u origin main
```

これで GitHub 上にバックアップが作成されました。

### 9-3. 日常的な更新方法

新しいエージェントを追加したり、既存のエージェントを編集したりしたあと、以下のコマンドで GitHub に同期します。

```bash
cd ~/.claude

# 変更内容を確認
git status

# 全変更をステージに追加
git add .

# コミット（変更内容を一言で書く）
git commit -m "Add n8n-workflow-advisor agent"

# GitHub に push
git push
```

**覚えるコマンドは3つだけです：**

| コマンド | 意味 |
|---------|------|
| `git add .` | 変更を記録の準備をする |
| `git commit -m "説明"` | 変更を記録する |
| `git push` | GitHub に送る |

**いつ push するかの目安：**
- 新しいエージェントを作ったとき
- エージェントの内容を大きく変更したとき
- 1日の作業が終わったタイミング

### 9-7. よくある疑問

**Q. Git・GitHub を使ったことがない。難しい？**  
→ 最初の初期化（9-2）だけ少し手間がかかりますが、日常的な更新は3つのコマンドを繰り返すだけです。一度設定してしまえば維持は簡単です。

**Q. GitHub にプライベートリポジトリは無料で作れる？**  
→ はい、GitHub の無料プランでプライベートリポジトリを無制限に作れます。

**Q. エージェントの会話履歴（プロジェクト履歴）も移したい**  
→ `~/.claude/projects/` に保存されていますが、容量が大きく端末固有のパスも含まれるため、基本的には移行しない方が安全です。エージェント定義（`.md` ファイル）さえあれば、同じ動作を新しいPCで再現できます。

**Q. スマホやタブレットからも使いたい**  
→ Claude Code は現時点でPC（ターミナル環境）専用です。モバイルでは Claude.ai（ブラウザ）を使い、スキルは claude.ai の設定画面からアップロードする形になります。
