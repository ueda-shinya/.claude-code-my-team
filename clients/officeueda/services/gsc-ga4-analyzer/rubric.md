## 対象成果物の振り分け

| 成果物 | 営業 rubric 対象 | 備考 |
|---|---|---|
| 機能A: 計測診断レポート（Markdown） | ◯ 対象（rubric-A） | 営業先に「現状診断」として提示する一次成果物 |
| 機能B: 計測補強JSスニペット | × 対象外 | コード品質はサクラのレビューで担保。営業で見せるのは「実装後のGA4スクショ」であってコード本体ではない |
| 機能C: GSC/GA4分析＋改善提案レポート（Markdown） | ◯ 対象（rubric-C） | 営業先に「改修すべき内容と効果」として提示する主成果物 |

機能A と機能C は性格が異なるため**別 rubric** とし、両方の合格を完了条件とする。
機能B は「機能C のデータソース」として、rubric-C 内の「データ駆動性」項目で間接的に評価する。

## rubric-A: 計測診断レポート（機能A）— 6項目

### A1: 診断観点の網羅性
- **観点**: GA4基本設置／CV設定／重複計測／必須イベント計測／同意管理／クロスドメイン等、計測診断で押さえるべき観点が漏れていないか
- **合格ライン**: 診断項目が **10項目以上** チェック済み（Pass / Warn / Fail のいずれか判定が付与されている）
- **NG例**: 「GA4 入ってます ◯ / CV設定 ◯」の2項目だけで終わっている
- **OK例**: 10項目以上について Pass/Warn/Fail のステータスと「なぜ Warn/Fail か」の理由が1〜2行で記載されている

### A2: 問題箇所の特定具体性
- **観点**: Warn/Fail 項目について、抽象論ではなく具体的な箇所が特定されているか
- **合格ライン**: 全ての Warn/Fail 項目に「具体的な対象URL／タグ名／イベント名」のいずれかが記載されている
- **NG例**: 「コンバージョン設定が不十分です」
- **OK例**: 「`/contact/thanks.html` の page_view が GA4 のキーイベント設定に登録されていない（GA4 管理画面 → イベント → 該当イベント横のスイッチがOFF）」

### A3: 影響度の明示
- **観点**: 各問題が「データ計測上どんな影響があるか」が読み手にわかるか
- **合格ライン**: Warn/Fail 項目の **80%以上** に「データへの影響（数値が出ない／重複カウント／誤計測 等）」が1行以上で記載されている
- **NG例**: 影響範囲が書かれず、なぜ修正が必要かわからない
- **OK例**: 「→ コンバージョン数が GA4 に反映されないため、広告ROAS の正しい評価ができない」

### A4: フォーマット遵守
- **観点**: 営業先にそのまま渡せる Markdown 構造になっているか
- **合格ライン**: 以下を **すべて** 満たす（Yes/No）
  - サマリー（先頭1ページ相当）に「総診断項目数 / Pass / Warn / Fail の件数」が表形式で記載
  - 章立てが **3〜5章**（深さは見出し3階層以内）
  - 表組みが2つ以上含まれる（人間の目で一覧できる）
  - 専門用語（GA4・GTM・dataLayer・キーイベント 等）は**初出時に1行で説明**
- **NG例**: 上から順にダラダラとリスト羅列、サマリーなし
- **OK例**: 「サマリー → 診断結果一覧表 → 各カテゴリ詳細 → 推奨対応 → 次アクション」の構造

### A5: クライアント視点（経営層が読めるか）
- **観点**: 技術者向けでなく、Web担当・経営層が読んで意思決定できる文章になっているか
- **合格ライン**: 専門用語の初出説明（A4で必須化）に加え、**Warn/Fail の影響説明（A3）に「ビジネス上の損失」が1件以上の項目で言及されている**
- **NG例**: 「dataLayer の push が gtag.js より遅延している」だけで終わる
- **OK例**: 「→ ページ表示直後の離脱が計測されず、実際より直帰率が低く見えている。改善の優先判断を誤るリスクがある」

### A6: 次アクションへの動線（営業ツールとしての機能）
- **観点**: 「読んで終わり」でなく、改修依頼への動線があるか
- **合格ライン**: レポート末尾に「次にできる3つのアクション」セクションがあり、**最低1つは「officeueda に改修を依頼する選択肢」が明示されている**
- **NG例**: 「以上、ご参考まで」で終わる
- **OK例**: 「① 自社で修正する（推定工数: ◯時間、必要スキル: GTM初級） / ② 計測ツール導入を検討する / ③ officeueda に改修を依頼する（無料お見積り対応中・連絡先: ◯◯）」

## rubric-C: 改善提案レポート（機能C）— 7項目

### C1: 課題特定数
- **観点**: GSC/GA4 データから引き出した課題が十分な件数あるか
- **合格ライン**: 改善対象の課題が **5件以上** 特定されている（rev.8 ドラフトの「3件以上」より厳しめ。営業ツールとしてボリューム不足を防ぐ）
- **NG例**: 「直帰率が高い」1件だけ
- **OK例**: 「①特定LPの直帰率超過 / ②キーワードAでクリックされるが滞在短い / ③CV導線でフォーム到達率が低い / ④モバイルだけCVR低下 / ⑤特定流入元の質が低い」等

### C2: 優先度の付与
- **観点**: 全課題に優先度が付き、優先順位の根拠が示されているか
- **合格ライン**: 各課題に**「優先度（高/中/低）」+「優先度根拠」**が記載されている（Yes/No）
- **NG例**: 「優先度: 高」とだけ書く
- **OK例**: 「優先度: 高（理由: 月間流入の30%を占めるLPで、直帰率が他LP平均比+25pt と顕著に悪い）」

### C3: 改善提案の具体性
- **観点**: 提案が「やれる」レベルまで落ちているか（5W1H）
- **合格ライン**: 各課題に対する改善提案が以下を**すべて**含む
  - **What**: 何を変更するか（具体的な要素・テキスト・配置）
  - **Where**: どのページ／どのセクションか（URL or 該当箇所明記）
  - **How**: 実施手段（A/Bテスト／一斉差し替え／段階導入 等）
  - **想定工数**: 時間 or 人日（レンジ可、例: 4〜8時間）
- **NG例**: 「ファーストビューを改善しましょう」
- **OK例**: 「`/biz-ai/` のFV見出しを『AI で何ができるか』訴求から『広島の中小企業に1社1名のAI担当をアサイン』訴求に変更（What） / FV 直下の H1 と CTA テキストを差し替え（Where） / 1週間A/Bテスト後に勝者採用（How） / 想定工数: 8〜16時間」

### C4: 期待効果の数値化
- **観点**: 「やったら何が良くなるか」が数値で示されているか
- **合格ライン**: 課題の **80%以上** に「期待効果（数値レンジ可）」が記載されている。可能なら算出根拠も併記
- **NG例**: 「効果が見込めます」
- **OK例**: 「直帰率を現状78% → 60%に改善（同業他社中央値65%基準・改善幅18pt） / 月間問合せ数を現状1件 → 3〜5件に増加見込み」

### C5: データ駆動性（機能Bデータの反映含む）
- **観点**: 提案が GSC/GA4 の実データに基づいているか、機能Bで計測したイベントデータが反映されているか
- **合格ライン**: 各課題に「根拠データ（数値+出典）」が記載され、**全課題のうち1件以上で機能B計測イベント（scroll/click/form_start 等）のデータが引用されている**
- **NG例**: 「一般的に〜と言われています」「業界平均では〜」のみで自社データが出てこない
- **OK例**: 「該当LPの form_start イベント発火率は12%（過去30日 / 機能B計測） / 一方で form_submit は3%。フォーム着手後の脱落率が75%と高い」

### C6: フォーマット遵守 + クライアント視点
- **観点**: 営業先にそのまま渡せる Markdown 構造で、専門用語が経営層に伝わる形か
- **合格ライン**: 以下を**すべて**満たす（Yes/No）
  - サマリー（先頭）に「課題件数 / 優先度内訳 / 期待効果合計レンジ」が記載
  - 章立てが**4〜6章**（深さは見出し3階層以内）
  - 専門用語（直帰率・CTR・CVR・セッション 等）は初出時に1行説明
  - 経営層向けの「経営インパクト要約」が1セクションある（「この提案を全部やったら売上にどう跳ねるか」を3〜5行）
- **NG例**: 技術用語のままダラダラ書かれている
- **OK例**: 経営層が30秒で全体像を把握できるサマリーがある

### C7: 改修受注への動線（営業ツールとしての機能）
- **観点**: 改修依頼への自然な誘導があるか
- **合格ライン**: 以下を**すべて**満たす
  - レポート末尾に「次にできる3つのアクション」セクションあり
  - 各課題に「自社対応 / 外注対応」の難易度ラベル（自社対応可・要外注 等）が付与されている
  - 「外注対応」とラベルされた課題が**最低1件**ある（受注リードを残す）
  - 末尾CTA に officeueda の連絡先 or 見積もりフォームURL が明記されている
- **NG例**: CTA がない、または「気軽にお問い合わせください」だけで連絡先がない
- **OK例**: 「優先度高3件のうち2件は外注推奨（C3工数見積で合計32時間規模）→ officeueda 無料相談はこちら: [URL]」

## シンヤさん最終承認の主観チェックポイント

rubric-A・rubric-C の全項目パス後、シンヤさんが以下の主観確認を行う。1つでも「No」なら不合格として差し戻し。

1. **「これ、知り合いの社長に営業で渡せるか？」**
   恥ずかしくないか、内容の薄さで信頼を失わないか
2. **「読み手が"自社でなんとかなりそう"と全部閉じてしまわないか？」**
   外注したくなる難易度の課題が混ざっているか
3. **「改修依頼の見積もりに、このレポートの数字をそのまま使えるか？」**
   工数見積が現実的か、期待効果の数値根拠が客先で説明できるレベルか
4. **「3つのレポート（コーポ／biz-ai LP／biz-web LP）で内容がコピペっぽくなっていないか？」**
   3対象でドッグフードする以上、それぞれの個性が出ているか
5. **「officeueda が改修を受注した場合、提案通りに着手して効果が出る確信があるか？」**
   レンの自信レベルで言えば、最低でも「中の上」以上であること

## 不合格時の対応フロー

| 落ちた項目 | 対応 | 担当 |
|---|---|---|
| rubric-A の A1〜A3（網羅性・具体性・影響度） | 機能A の診断ロジック改修 | シュウ（実装） |
| rubric-A の A4〜A5（フォーマット・クライアント視点） | 機能A のレポートテンプレート改修 | シュウ＋ハル（writer 視点レビュー） |
| rubric-A の A6（次アクション動線） | レポートテンプレートに CTA セクション追加 | シュウ |
| rubric-C の C1〜C5（課題特定・具体性・期待効果・データ駆動性） | Claude プロンプト改修（`prompts/analysis_proposal.md`） | シュウ（実装）+ レン（プロンプト方向性） |
| rubric-C の C6（フォーマット・クライアント視点） | プロンプト＋テンプレート改修 | シュウ＋ハル |
| rubric-C の C7（改修受注動線） | プロンプト＋テンプレートに CTA・難易度ラベル追加 | シュウ＋レン |
| シンヤさん主観チェックで不合格 | レンが原因項目を特定し、上記の該当行に振り分け | レン |

不合格→修正→再評価のサイクルは**最大3回**。3回目でも不合格の場合は、設計（rev.8）レベルまで戻ってシンヤさん＋レン＋アスカで再設計を検討する。
